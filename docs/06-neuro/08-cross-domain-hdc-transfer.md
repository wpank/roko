# Cross-Domain HDC Transfer

## Abstract

The most novel capability of Neuro's HDC encoding is **cross-domain insight resonance** — the automatic detection of structural analogies across different problem domains, in nanoseconds. When a coding agent learns "complex code needs more review," the structural pattern transfers to chain agents as "volatile markets need more caution." The surface-level Engrams are different, but the HDC vectors are similar because both encode the abstract relationship `BIND(high_uncertainty, more_verification)`.

This is not metaphorical similarity — it is geometric. In 10,240-dimensional binary space, structural relationships are preserved by the bind and bundle operations. Two entries that encode the same abstract relationship (even using different domain-specific concepts) will have measurably higher Hamming similarity than unrelated entries. The recommended threshold for cross-domain resonance detection is 0.526 (see [09-false-positive-math.md](./09-false-positive-math.md)), which guarantees <1% false positive rate against a 100K-entry knowledge base.

This capability produces insights that **no human would consider**, because no human simultaneously monitors code quality, financial signals, and operational metrics with structural pattern matching across all domains.

## How Cross-Domain Transfer Works

### The Encoding Mechanism

Consider three knowledge entries from three different domains:

**Coding domain**:
- Content: "High-complexity modules need more code review"
- Encoding: `BIND(role_risk_factor, hv_high_complexity) ⊕ BIND(role_response, hv_more_review)`

**Chain domain**:
- Content: "High-volatility assets need more caution before trading"
- Encoding: `BIND(role_risk_factor, hv_high_volatility) ⊕ BIND(role_response, hv_more_caution)`

**Research domain**:
- Content: "Contradictory sources need more verification before citing"
- Encoding: `BIND(role_risk_factor, hv_contradictory_sources) ⊕ BIND(role_response, hv_more_verification)`

All three share the **same abstract structure**:
```
BIND(role_risk_factor, hv_high_X) ⊕ BIND(role_response, hv_more_Y)
```

Where `hv_high_X` and `hv_more_Y` are domain-specific fillers, but the **roles** (`role_risk_factor`, `role_response`) are shared across domains.

### Why the Vectors Are Similar

The similarity arises from the shared role structure. Even though `hv_high_complexity` and `hv_high_volatility` are quasi-orthogonal (they are different concepts), the role bindings create a shared substructure:

1. Both entries bind `role_risk_factor` with some concept → the `role_risk_factor` component contributes similarly to both vectors
2. Both entries bind `role_response` with some concept → the `role_response` component contributes similarly to both vectors
3. The overall bundle preserves these shared structural features

The similarity is not 1.0 (the entries are not identical) but is measurably above the noise floor of 0.5. With well-chosen role vectors and 5-10 structural components per entry, cross-domain similarity typically falls in the range 0.53–0.58 — well above the 0.526 threshold for statistical significance.

### Insight Resonance Detection

Neuro continuously cross-correlates new knowledge entries against the full HDC knowledge base across all domains:

```
For each new entry E:
    For each stored entry S in domains ≠ E.domain:
        sim = E.hdc_vector.similarity(S.hdc_vector)
        if sim > CROSS_DOMAIN_THRESHOLD (0.526):
            emit Resonance alert:
                "Pattern in {E.domain} has structural similarity to pattern in {S.domain}"
                "Both encode: {abstract_pattern_description}"
```

The `abstract_pattern_description` can be generated by unbinding the shared roles from both vectors and comparing the results against a codebook of abstract concepts.

### Concrete Example

An agent monitoring a Rust codebase detects increasing churn in the authentication module:
```
Coding entry: BIND(role_pattern, hv_high_churn) ⊕ BIND(role_location, hv_auth_module)
              ⊕ BIND(role_risk, hv_hidden_accumulation)
```

Meanwhile, a financial agent detects increasing options activity before earnings:
```
Chain entry: BIND(role_pattern, hv_increased_activity) ⊕ BIND(role_location, hv_options_market)
             ⊕ BIND(role_risk, hv_hidden_accumulation)
```

Both share `BIND(role_risk, hv_hidden_accumulation)` — the abstract pattern "hidden risk accumulating in a critical subsystem." Cross-domain resonance detects this overlap and generates:

> "A pattern in your codebase (high churn in auth module) has structural similarity to a pattern a financial agent detected (increased options activity before earnings). Both correlate with the abstract pattern 'hidden risk accumulating in a critical subsystem.' Investigate."

## Design of Abstract Role Vectors

Cross-domain transfer works best when roles are defined at the right level of abstraction. Too specific (e.g., `role_rust_borrow_checker`) and no cross-domain transfer occurs. Too generic (e.g., `role_thing`) and everything matches everything.

### Recommended Role Hierarchy

**Abstract roles** (shared across all domains):
| Role | Encodes |
|---|---|
| `role_risk_factor` | What creates risk in this context |
| `role_response` | How the agent should respond |
| `role_pattern` | Observable pattern or signal |
| `role_severity` | How serious the situation is |
| `role_temporal` | Time dimension of the observation |
| `role_confidence` | How certain the agent is |

**Domain-specific roles** (used within a single domain):
| Role | Domain | Encodes |
|---|---|---|
| `role_crate` | Coding | Which Rust crate |
| `role_protocol` | Chain | Which DeFi protocol |
| `role_source` | Research | Which information source |
| `role_asset` | Chain | Which token/asset |
| `role_function` | Coding | Which function/method |

**Cross-domain transfer happens through the abstract roles.** Domain-specific roles provide precision within a domain but do not contribute to cross-domain similarity.

### Encoding Strategy

When encoding a knowledge entry, both abstract and domain-specific roles are included:

```rust
let entry_hv = HdcVector::bundle(&[
    // Abstract roles (enable cross-domain transfer)
    &role_risk_factor.bind(&hv_high_complexity),
    &role_response.bind(&hv_more_review),
    &role_severity.bind(&hv_medium),
    // Domain-specific roles (enable within-domain precision)
    &role_crate.bind(&hv_roko_gate),
    &role_function.bind(&hv_run_pipeline),
]);
```

Cross-domain queries compare against abstract role components; within-domain queries compare against both.

## Applications Beyond Knowledge Retrieval

### Strategy-Regime Bidirectional Lookup

HDC enables looking up strategies by regime and regimes by strategy:

**Forward**: "Given this regime (high volatility, low liquidity), what strategies worked?"
```
regime_hv = BIND(role_volatility, hv_high) ⊕ BIND(role_liquidity, hv_low)
// Query StrategyFragments for entries similar to regime_hv
```

**Reverse**: "Given this strategy (small position sizes, tight stops), when has it been used?"
```
strategy_hv = BIND(role_position_size, hv_small) ⊕ BIND(role_stop_loss, hv_tight)
// Query episode summaries for entries similar to strategy_hv
```

### Analogical Reasoning

HDC can answer "A is to B as C is to ?" using binding:
```
// "Rust is to borrow checker as Java is to ?"
analogy = BIND(BIND(hv_rust, hv_borrow_checker), hv_java)
// Query codebook for nearest match → hv_garbage_collector
```

This follows the classic analogy encoding: `BIND(A, B)` captures the relationship; binding with C produces a vector similar to D if the structural relationship `A:B :: C:D` holds.

## Limitations and Caveats

1. **Spurious resonance**: Not all structural similarities are meaningful. Two entries may share abstract structure by coincidence. The 0.526 threshold reduces false positives to <1%, but the remaining 1% must be validated by a higher-level mechanism (e.g., LLM evaluation or human review).

2. **Role vector design is manual**: The choice of abstract roles determines what cross-domain transfers are possible. If no abstract role captures a particular structural relationship, the transfer will not be detected. Expanding the role vocabulary requires human judgment about what abstractions are useful.

3. **Encoding quality varies**: Cross-domain transfer depends on the quality of the HDC encoding. If a knowledge entry is encoded with only content fingerprinting (no structured role-filler bindings), cross-domain similarity will be low regardless of the actual structural relationship.

4. **Confirmation requirement**: The refactoring-prd recommends requiring that cross-domain analogies be confirmed by at least 2 independent agents before being accepted as genuine. This reduces false positives quadratically (if each agent has a 1% FP rate, joint confirmation has a 0.01% FP rate).

## Implementation Details: Cross-Domain Resonance Detection Loop

### Detection algorithm

The resonance detection loop runs when a new knowledge entry is ingested. It cross-correlates the new entry's HDC vector against all stored entries in *other* domains.

```rust
use crate::{HdcVector, KnowledgeEntry, NeuroStore};

/// A detected cross-domain resonance between two knowledge entries.
pub struct Resonance {
    /// The new entry that triggered detection.
    pub source_entry_id: String,
    /// The stored entry that resonates.
    pub target_entry_id: String,
    /// Hamming similarity between the two HDC vectors.
    pub similarity: f32,
    /// Source entry's domain.
    pub source_domain: String,
    /// Target entry's domain.
    pub target_domain: String,
    /// Abstract pattern description (generated by role decomposition).
    pub abstract_pattern: Option<String>,
}

/// Configuration for resonance detection.
pub struct ResonanceConfig {
    /// Minimum similarity for cross-domain resonance. Default: 0.526.
    /// This threshold guarantees <1% FPR against 100K entries (Bonferroni-corrected).
    pub cross_domain_threshold: f32,
    /// Maximum number of resonances to emit per new entry. Default: 5.
    /// Prevents flooding when an entry is structurally generic.
    pub max_resonances_per_entry: usize,
    /// Whether to generate abstract pattern descriptions. Default: true.
    /// Requires role codebook and ResonatorNetwork. Adds ~1ms per resonance.
    pub generate_descriptions: bool,
}

impl Default for ResonanceConfig {
    fn default() -> Self {
        Self {
            cross_domain_threshold: 0.526,
            max_resonances_per_entry: 5,
            generate_descriptions: true,
        }
    }
}
```

**Algorithm pseudocode**:

```
fn detect_resonances(
    new_entry: &KnowledgeEntry,
    store: &NeuroStore,
    config: &ResonanceConfig,
) -> Vec<Resonance>:
    // Skip entries without HDC vectors
    new_hv = decode(new_entry.hdc_vector)?
    new_domain = extract_domain(new_entry)

    candidates = []
    for stored_entry in store.all_entries():
        // Only cross-domain comparisons
        stored_domain = extract_domain(stored_entry)
        if stored_domain == new_domain:
            continue

        stored_hv = decode(stored_entry.hdc_vector)?
        sim = new_hv.similarity(stored_hv)

        if sim > config.cross_domain_threshold:
            candidates.push((stored_entry, sim))

    // Sort by similarity descending, take top N
    candidates.sort_by(|a, b| b.1.partial_cmp(&a.1))
    candidates.truncate(config.max_resonances_per_entry)

    // Optionally generate abstract pattern descriptions
    resonances = []
    for (stored_entry, sim) in candidates:
        abstract_pattern = if config.generate_descriptions:
            generate_abstract_pattern(new_hv, stored_hv, role_codebook)
        else:
            None

        resonances.push(Resonance {
            source_entry_id: new_entry.id,
            target_entry_id: stored_entry.id,
            similarity: sim,
            source_domain: new_domain,
            target_domain: extract_domain(stored_entry),
            abstract_pattern,
        })

    resonances
```

**Trigger conditions**: The detection loop fires on every call to `NeuroStore::ingest()`. For bulk ingestion (e.g., importing a batch of entries), detection runs after each entry. If performance is a concern during bulk import, pass a flag to defer detection and run it once after the batch completes.

**Performance**: Brute-force scan of 100K entries takes ~1.3ms. For stores under 100K entries, this is acceptable latency to add to ingestion. For larger stores, use the three-tier index (see [04-hdc-vsa-foundations.md](./04-hdc-vsa-foundations.md)).

### Abstract role vector hierarchy

The role hierarchy determines which structural patterns transfer across domains. Roles at the top of the hierarchy are abstract enough to match across domains; roles at the bottom are domain-specific and only match within a domain.

```
Abstract roles (cross-domain transfer)
├── role:risk_factor     — What creates risk
├── role:response        — How to respond
├── role:pattern         — Observable signal
├── role:severity        — How serious
├── role:temporal        — Time dimension
├── role:confidence      — Certainty level
│
Domain-specific roles (within-domain precision)
├── Coding
│   ├── role:crate       — Which Rust crate
│   ├── role:function    — Which function
│   └── role:module      — Which module
├── Chain
│   ├── role:protocol    — Which DeFi protocol
│   ├── role:asset       — Which token
│   └── role:pool        — Which liquidity pool
└── Research
    ├── role:source      — Which information source
    ├── role:citation    — Which paper
    └── role:method      — Which methodology
```

**Design principle**: Cross-domain transfer happens through abstract roles only. When two entries share `BIND(role:risk_factor, hv_X)` structure, their vectors have elevated similarity regardless of what X is — the shared role structure creates the signal. Domain-specific roles add precision within a domain but do not contribute to cross-domain similarity because their seeds differ.

**Adding new abstract roles**: Adding a new abstract role (e.g., `role:opportunity`) requires:
1. Add the seed to `KnowledgeHdcEncoder::new()` role registry
2. Update the encoding logic to populate the role when relevant
3. Re-encode existing entries (or accept that old entries will not benefit from the new role)

### Confirmation protocol (2+ agents)

Cross-domain analogies are prone to false positives. The confirmation protocol requires at least 2 independent agents to validate a detected resonance before it is accepted as genuine.

```rust
/// Message sent to agents for resonance confirmation.
pub struct ConfirmationRequest {
    /// Unique ID for tracking this request.
    pub request_id: String,
    /// The detected resonance to confirm or reject.
    pub resonance: Resonance,
    /// Deadline for responses. Default: 5 minutes.
    pub deadline: std::time::Duration,
    /// Number of confirmations required. Default: 2.
    pub required_confirmations: usize,
}

/// Agent's response to a confirmation request.
pub struct ConfirmationResponse {
    pub request_id: String,
    pub agent_id: String,
    /// true = confirmed, false = rejected.
    pub confirmed: bool,
    /// Optional reasoning for the decision.
    pub reasoning: Option<String>,
    /// Agent's independent similarity assessment (if re-computed).
    pub independent_similarity: Option<f32>,
}

/// Tracks confirmation state for a pending resonance.
pub struct ConfirmationTracker {
    pub request: ConfirmationRequest,
    pub responses: Vec<ConfirmationResponse>,
    pub created_at: std::time::Instant,
}

impl ConfirmationTracker {
    /// Check whether the resonance has been confirmed.
    pub fn is_confirmed(&self) -> bool {
        let confirms = self.responses.iter().filter(|r| r.confirmed).count();
        confirms >= self.request.required_confirmations
    }

    /// Check whether the resonance has been rejected (enough explicit rejections
    /// that confirmation is impossible).
    pub fn is_rejected(&self) -> bool {
        let rejects = self.responses.iter().filter(|r| !r.confirmed).count();
        let remaining = self.expected_total() - self.responses.len();
        let confirms = self.responses.iter().filter(|r| r.confirmed).count();
        confirms + remaining < self.request.required_confirmations
    }

    /// Check whether the deadline has passed.
    pub fn is_expired(&self) -> bool {
        self.created_at.elapsed() > self.request.deadline
    }

    fn expected_total(&self) -> usize {
        // Total agents that will be asked (set by the coordinator)
        3
    }
}
```

**Consensus algorithm**:

```
1. Resonance detected between entry A (domain X) and entry B (domain Y)
2. Select N=3 agents with access to both domains X and Y
3. Send ConfirmationRequest to all N agents
4. Each agent independently:
   a. Re-computes HDC similarity (guards against corrupted vectors)
   b. Evaluates whether the structural analogy is meaningful
   c. Returns confirmed/rejected with reasoning
5. If confirmed >= required_confirmations (default 2):
   → Accept resonance, create a CrossDomainInsight entry
6. If rejected:
   → Discard resonance, log for diagnostic review
7. If expired (no quorum within deadline):
   → Discard resonance, log timeout
```

**False positive reduction**: If each agent has a 1% FP rate independently, requiring 2 confirmations from 3 agents reduces the joint FP rate to approximately 0.03% (3 choose 2 times 0.01^2 times 0.99).

**Error handling**: Agents that fail to respond within the deadline are counted as abstentions (not rejections). If not enough agents are available for the confirmation, the resonance is queued until agents become available, up to a maximum queue age of 1 hour.

### Analogical reasoning query API

HDC enables answering "A is to B as C is to ?" using vector arithmetic:

```rust
/// Result of an analogical reasoning query.
pub struct AnalogyResult {
    /// The recovered concept (D in "A:B :: C:D").
    pub concept: String,
    /// Similarity of the recovered vector to the codebook match.
    pub similarity: f32,
    /// Whether the analogy is statistically significant (above threshold).
    pub significant: bool,
}

impl NeuroStore {
    /// Answer an analogical reasoning query: "A is to B as C is to ?"
    ///
    /// Uses the classic HDC analogy encoding:
    ///   relationship = BIND(A, B)     — captures the A→B relationship
    ///   answer = BIND(relationship, C) — applies the relationship to C
    ///   result = nearest(answer, codebook)
    ///
    /// The answer vector is similar to D if the structural relationship
    /// A:B :: C:D holds in the knowledge base.
    pub fn analogy(
        &self,
        a: &str,
        b: &str,
        c: &str,
        codebook: &ItemMemory,
    ) -> Option<AnalogyResult> {
        let hv_a = HdcVector::from_seed(a.as_bytes());
        let hv_b = HdcVector::from_seed(b.as_bytes());
        let hv_c = HdcVector::from_seed(c.as_bytes());

        // Capture A→B relationship
        let relationship = hv_a.bind(&hv_b);
        // Apply to C
        let answer = relationship.bind(&hv_c);

        // Look up nearest concept
        let (concept, similarity) = codebook.nearest(&answer)?;

        Some(AnalogyResult {
            concept: concept.to_string(),
            similarity,
            significant: similarity > 0.526,
        })
    }

    /// Batch analogy: find all concepts related to C
    /// by the same relationship that links A to B.
    ///
    /// Returns top K matches sorted by similarity.
    pub fn analogy_top_k(
        &self,
        a: &str,
        b: &str,
        c: &str,
        codebook: &ItemMemory,
        k: usize,
    ) -> Vec<AnalogyResult> {
        let hv_a = HdcVector::from_seed(a.as_bytes());
        let hv_b = HdcVector::from_seed(b.as_bytes());
        let hv_c = HdcVector::from_seed(c.as_bytes());

        let relationship = hv_a.bind(&hv_b);
        let answer = relationship.bind(&hv_c);

        codebook.top_k(&answer, k).into_iter()
            .map(|(name, sim)| AnalogyResult {
                concept: name.to_string(),
                similarity: sim,
                significant: sim > 0.526,
            })
            .collect()
    }
}
```

**Response format**: `AnalogyResult` contains the concept name, raw similarity, and a boolean `significant` flag. Callers should only act on results where `significant == true`. Results below threshold may appear in diagnostic output but should not drive agent behavior.

**Test criteria**:
- Resonance detection: two entries sharing `BIND(role:risk_factor, X)` structure produce similarity > 0.526
- Resonance detection: two entries in the same domain are skipped
- `max_resonances_per_entry` limits output size
- Confirmation: 2 of 3 confirms leads to `is_confirmed() == true`
- Confirmation: 2 of 3 rejects leads to `is_rejected() == true`
- Expired tracker: `is_expired()` returns true after deadline
- Analogy: "rust" to "borrow_checker" as "java" to ? returns "garbage_collector" when codebook contains it
- Analogy: result similarity below threshold leads to `significant == false`
