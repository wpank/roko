## What Dreams Are in the Roko Architecture

Dreams are a **Delta-frequency cognitive process** — the slowest of three cognitive speeds that govern how a Roko agent thinks. Where Gamma (~5–15 seconds) handles reactive perception-action loops and Theta (~75 seconds) handles strategic re-planning, Delta operates on the scale of hours. Dreams are what happens when an agent has nothing urgent to do: it turns inward, replays its past experiences, imagines counterfactual alternatives, and consolidates the results into durable knowledge.

The term "dream" is not metaphorical. The system implements biological sleep consolidation mechanisms — hippocampal replay, emotional depotentiation, counterfactual simulation — mapped onto LLM-based agents. The research basis is extensive:

- **Complementary Learning Systems (CLS)**: McClelland et al. (1995) demonstrated that biological brains maintain two learning systems — a fast episodic system (hippocampus) and a slow semantic system (neocortex). Dreams are the bridge: during sleep, episodic memories are replayed and gradually integrated into semantic knowledge. Roko's dream system implements this exact architecture, with the episode log as the fast system and NeuroStore as the slow system.

- **Sleep-time compute**: Lin et al. (2025, arXiv:2504.11651) showed that dedicating computation to offline processing yields a 5× reduction in test-time compute requirements. The insight is that agents can do significant cognitive work during idle periods, making their waking performance dramatically more efficient.

- **DreamerV3**: Hafner et al. (2025, Nature) demonstrated that agents learning world models and planning within them can master diverse domains. Roko's dream system uses a similar architecture — the agent builds internal models during dreaming and plans within those models via counterfactual simulation.

- **Generative Agents**: Park et al. (2023, UIST, arXiv:2304.03442) introduced periodic reflection cycles where agents synthesize higher-order observations from raw experiences. Roko's dream consolidation directly implements this pattern.

The Roko dream system produces five categories of output:

| Output | Description | Destination |
|--------|-------------|-------------|
| **Insights** | Patterns extracted from episode replay (e.g., "tasks involving refactoring consistently fail the test gate when the model is under-specified") | NeuroStore via `Substrate` trait |
| **Heuristics** | Promoted insights with 3+ confirmations, becoming actionable rules | NeuroStore + PLAYBOOK.md |
| **Counterfactual strategies** | Novel approaches generated by imagining alternative histories | Staging buffer → NeuroStore on validation |
| **Meta-patterns** | Cross-episode structural similarities discovered via HDC clustering | NeuroStore |
| **Confidence updates** | Strengthened or weakened belief in existing knowledge entries | NeuroStore confidence scores |

## The Critical Reframe: Dreams Are NOT Death-Triggered

### What Changed and Why

The legacy Bardo architecture framed dreams as a response to approaching death. In that system, agents had "stochastic death clocks" — random countdowns to mortality — and dream frequency intensified as death approached. The metaphor was evocative but architecturally harmful:

1. **It coupled dreams to mortality**, making the dream subsystem depend on a death-clock subsystem that has no place in a general-purpose agent framework.
2. **It made dream frequency unpredictable** — driven by a PRNG countdown rather than by actual cognitive need.
3. **It excluded non-mortal agents** from dreaming, even though every agent benefits from offline consolidation regardless of whether it has a finite lifespan.
4. **It introduced morbid language** throughout the codebase that made the project harder to explain to investors, researchers, and contributors.

### What Replaces It

In the Roko architecture, dreams are triggered by two mechanisms:

**1. Idle-time triggering**: When an agent has no active tasks and has been idle for a configurable threshold (default: 15 minutes), the dream scheduler fires a consolidation cycle. This is the primary trigger. The logic is simple: if you have nothing to do, process what you've learned.

```rust
pub struct DreamLoopConfig {
    /// Whether automatic dreaming is enabled.
    pub auto_dream: bool,
    /// Idle threshold in minutes before a dream may run.
    pub idle_threshold_mins: u64,
    /// Minimum number of new episodes required before dreaming.
    pub min_episodes_for_dream: usize,
    /// Agent backend used to review the dream batch.
    pub agent: DreamAgentConfig,
}
```

**2. Scheduled consolidation**: The Delta-frequency scheduler can trigger dreams on a fixed schedule (e.g., every 4 hours, or at specific times of day). This ensures that even busy agents with continuous task queues still get periodic consolidation time.

Both triggers are configurable in `roko.toml`:

```toml
[dreams]
auto_dream = true
idle_threshold_mins = 15
min_episodes_for_dream = 5
```

### What the Legacy System Called "Death Dreams"

The legacy PRD described several dream-adjacent concepts in death-related terms. Here is the mapping to their Roko equivalents:

| Legacy Concept | Legacy Description | Roko Equivalent | Roko Description |
|----------------|-------------------|-----------------|------------------|
| **Dream-as-approaching-death** | Dream frequency increases as the agent's death clocks tick down | **Idle-triggered / scheduled consolidation** | Dream frequency is driven by idle time and schedule, not mortality |
| **Terminal dreaming** | Frantic dream cycles every 67 ticks in the Terminal behavioral phase | **Intensive consolidation mode** | When an agent has accumulated many unprocessed episodes (high backlog), consolidation cycles run more frequently — not because of approaching death but because there is more material to process |
| **Death-proximate replay** | Replaying traumatic episodes as the agent nears death | **High-salience replay** | Episodes with high emotional intensity (measured by the Daimon's PAD vectors) are prioritized for replay regardless of the agent's lifecycle state |
| **Thanatopsis dream output** | Final dream producing a "death testament" | **Knowledge export** | An agent can export its accumulated knowledge at any time via user-controlled backup/restore — this is not death-specific |
| **Stochastic death clock** | Random countdown to agent termination that influences dream scheduling | **REMOVED** | No concept of random death in Roko. Agents run until budget exhaustion (economic) or knowledge plateau (epistemic), both of which are continuous metrics, not clocks |
| **Dream urgency from mortality** | Urgency score weighted by proximity to death | **Dream urgency from backlog** | Urgency score weighted by number of unprocessed episodes and time since last consolidation |

### The Underlying Mechanism Is Preserved

The computational mechanisms that made the legacy dream system powerful are fully retained. What changes is *when and why* dreams fire, not *what happens during* a dream:

- **NREM replay** still replays episodes weighted by the Mattar & Daw (2018, Nature Neuroscience) utility formula
- **REM imagination** still generates counterfactuals using Pearl's structural causal models and Boden's three creativity modes
- **Integration** still uses a staging buffer with confidence-gated promotion to permanent knowledge
- **Emotional depotentiation** still reduces arousal markers per Walker & van der Helm (2009, Psychological Bulletin)
- **HDC counterfactual synthesis** still uses 10,240-bit BSC vector permutation operations

The academic foundations are unchanged. The trigger logic is simplified. The death framing is removed.

## Where Dreams Fit in the Five-Layer Taxonomy

Dreams are a **cognitive cross-cut** — they are not confined to a single layer but inject behavior across multiple layers:

| Layer | Dream Integration |
|-------|-------------------|
| **L0 Runtime** | Dream scheduling: the Delta-frequency scheduler lives in the runtime and decides when to fire dream cycles based on idle time and backlog |
| **L1 Framework** | Agent dispatch: during dreaming, the framework dispatches a review agent (typically a Haiku-class model for NREM, Sonnet-class for REM) to process the dream batch |
| **L2 Scaffold** | Context assembly: dream outputs (insights, heuristics) are injected into future waking prompts via the NeuroStore retrieval pipeline |
| **L3 Harness** | Gate validation: dream-generated hypotheses enter the staging buffer and must pass validation gates before promotion to permanent knowledge |
| **L4 Orchestration** | Dream scheduling is coordinated with task orchestration — the orchestrator determines idle periods and delegates to the Delta scheduler |

Dreams flow through the Synapse Architecture:

1. **Substrate**: Episode logs and NeuroStore entries are the substrate from which dreams draw raw material
2. **Scorer**: The Mattar-Daw utility formula scores episodes for replay priority
3. **Gate**: The staging buffer's confidence threshold (0.70) gates promotion of dream-generated hypotheses
4. **Router**: Model routing selects the appropriate LLM tier for each dream phase (cheap models for NREM replay, more capable models for REM imagination)
5. **Composer**: Dream outputs are composed into Engrams (currently `Signal` in the codebase — will be renamed to `Engram` in Tier 0D) and written to NeuroStore
6. **Policy**: The dream scheduling policy observes the Engram stream and emits scheduling decisions

## Domain-Agnostic Dreaming

Dreams are **not specific to any domain**. The legacy PRD heavily framed dreams in DeFi terms (replaying liquidation cascades, imagining hedging strategies). In the Roko architecture, dreams are domain-agnostic at the kernel level:

**Coding agent example**: A coding agent replays past implementation episodes. During NREM, it reviews tasks where the compile gate failed and extracts patterns ("when I modify error types without updating the match arms downstream, the compile gate always fails"). During REM, it imagines: "what if I had run clippy before the test gate?" The counterfactual generates a process improvement heuristic that enters the staging buffer.

**Research agent example**: A research agent replays past literature review episodes. During NREM, it identifies citation patterns across successful searches. During REM, it combines topics from unrelated research domains to generate novel research hypotheses.

**Chain agent example**: A chain agent (the `roko-chain` domain plugin) replays past transaction episodes. During NREM, it extracts gas price patterns. During REM, it simulates alternative execution strategies.

The dream engine provides the consolidation mechanism. The domain plugin provides the episodes and the interpretation. This separation is enforced by the `roko-dreams` crate's lack of dependency on any domain-specific crate.

## Relationship to Other Cognitive Cross-Cuts

Dreams interact deeply with the other cognitive cross-cuts:

- **Neuro (NeuroStore)**: Dreams *consume* episodes and knowledge entries from NeuroStore and *produce* new insights, heuristics, and confidence updates that are written back to NeuroStore. This bidirectional flow is the CLS architecture in action.

- **Daimon (Affect Engine)**: The Daimon's PAD (Pleasure-Arousal-Dominance) emotional state vectors influence dream processing. High-arousal episodes are prioritized for replay. Emotional depotentiation during REM reduces the Daimon's arousal markers. The Daimon provides somatic markers that weight episode significance.

- **Hypnagogia**: The hypnagogia engine is a sub-component of the dream system that operates at the transition between waking and sleep. It uses controlled noise injection (stochastic resonance) to generate novel associations that the structured dream phases then process. See [07-hypnagogia-engine.md](07-hypnagogia-engine.md).

## Dream Quality Metrics and Self-Assessment

Over time, the dream subsystem should be able to evaluate its own effectiveness. Are dreams actually improving waking performance, or are they burning compute for no return? The `DreamQualityDashboard` aggregates quality signals across dream cycles:

```rust
/// Aggregate dream quality metrics tracked across cycles.
pub struct DreamQualityDashboard {
    /// Cycles tracked.
    pub total_cycles: usize,
    /// Mean promotion rate: fraction of staged hypotheses eventually promoted.
    pub mean_promotion_rate: f64,
    /// Mean diversity: average pairwise HDC distance of generated hypotheses.
    pub mean_hypothesis_diversity: f64,
    /// Mean waking improvement: estimated reduction in task retries attributable to dreams.
    pub mean_waking_improvement: f64,
    /// Nightmare rate: nightmares detected per dream cycle.
    pub nightmare_rate: f64,
    /// Cost efficiency: waking improvement per dollar spent on dreams.
    pub cost_efficiency: f64,
    /// Trend: is dream quality improving, declining, or stable?
    pub trend: DreamQualityTrend,
}

pub enum DreamQualityTrend {
    Improving { slope: f64 },
    Stable,
    Declining { slope: f64 },
}
```

Key metrics explained:

- **Promotion rate** measures whether dreams generate *useful* hypotheses. A low promotion rate (< 0.10) suggests dreams are producing noise. A high rate (> 0.40) suggests dreams may not be exploring broadly enough. The healthy range is 0.15-0.35.
- **Hypothesis diversity** uses pairwise HDC cosine distance to measure whether dreams are exploring diverse territory or converging on the same patterns. Low diversity triggers the Hypnagogia engine's stochastic resonance to inject more noise.
- **Waking improvement** estimates the causal effect of dreams on waking performance by comparing task retry rates and gate pass rates before and after dream consolidation cycles. This is the bottom-line metric: are dreams making the agent better at its job?
- **Nightmare rate** tracks how often the nightmare detection system (see [17-advanced-dream-concepts.md](17-advanced-dream-concepts.md)) fires. A rising nightmare rate may indicate the agent is encountering genuinely novel failure modes that warrant attention.
- **Cost efficiency** normalizes waking improvement by dream compute cost, ensuring that dreams provide positive ROI. If cost efficiency drops below 1.0, the dream scheduler should reduce frequency or switch to cheaper models.
- **Trend** fits a linear regression over the last N cycles (default: 20) to determine whether dream quality is improving, stable, or declining. A declining trend triggers diagnostic logging and may adjust dream parameters (e.g., increasing REM allocation, changing model tier).
