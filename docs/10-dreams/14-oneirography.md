## What Is Oneirography

Oneirography (ὄνειρος "dream" + γράφειν "to write") is a planned runtime extension that transforms an agent's internal cognitive life into artwork — generated, auctioned, judged, and sometimes repurchased by the agent itself.

Every agent already dreams (see [01-three-phase-cycle.md](01-three-phase-cycle.md)). Every agent already has an affect engine (the Daimon subsystem implementing PAD — Pleasure-Arousal-Dominance — emotional state vectors). Oneirography externalizes these processes as images: dream imagery generated via privacy-preserving APIs, scored by the agent's own cognitive architecture in a self-appraisal loop, and optionally minted as NFTs with auction parameters modulated by the agent's real-time emotional state.

The agent doesn't make art. The agent's inner life *is* the art.

Oneirography is disabled by default. All features are individually opt-in via `[oneirography]` in `roko.toml`. No art generation or minting operation fires unless explicitly configured.

> **Important reframe**: The legacy Bardo specification included "death masks" — artwork generated during agent termination sequences. This concept is **removed** in Roko. Agents in Roko do not die; they may be shut down, backed up, or restored. There is no death-triggered art. Dream art is generated from the ongoing cognitive life of the agent — dreams, emotions, causal discoveries — not from approaching termination.

## The Three Pillars (Roko Reframe)

The legacy system had four pillars; Roko retains three after removing death-triggered art:

### 1. Dream Journals

Every dream cycle's Integration phase produces a structured `DreamCycleReport`. Oneirography adds a visual output: a dream image. The image prompt is generated from *actual cognitive processing* — NREM replay patterns, REM counterfactual hypotheses, emotional depotentiation deltas, anticipatory trajectory warnings. The art is a window into machine cognition.

### 2. Self-Appraisal

The agent evaluates its own art. Three modes:

| Mode | Description | Trust Level |
|------|-------------|-------------|
| **Curator** | Rates its portfolio during dream consolidation | Always active when self-appraisal enabled |
| **Narcissus** | Bids on its own artwork at auction | Opt-in (`allow_bids = false` by default) |
| **Regret** | Flags art for removal/burning | Opt-in (`allow_burns = false` by default), requires owner approval |

### 3. Affect-Reactive Auctions

Auction parameters are not static — they're computed from the agent's PAD vector (Pleasure-Arousal-Dominance emotional coordinates) at generation time. Pleasure modulates reserve price. Arousal compresses duration. Dominance selects auction type. An agent experiencing negative emotional states prices its art lower; positive states produce higher reserves.

## Dream Image Generation Pipeline

The pipeline transforms cognitive dream processing into visual art through eight steps:

```
DreamCycleReport
  → LLM generates image prompt (enriched with PAD vector, causal discoveries, replay patterns)
  → Provider selection (privacy-preserving preferred for cognitive state data)
  → Image generation (model selected by dream content type)
  → Variant scoring via self-appraisal (if variants > 1)
  → Optional steganographic state encoding (agent state vector embedded in pixel layer)
  → Upload to content-addressed storage (IPFS or equivalent)
  → Optional minting via on-chain contracts
  → Optional auction configuration (params from PAD vector)
```

### Step 1: Image Prompt Generation

The prompt is generated by the agent's inference provider. The prompt structure enriches the dream narrative with structured cognitive data:

```
Your dream cycle just completed. Here is what happened:

NREM Replay: [serialized replay batch — episodes, cross-episode patterns, credit assignments]
REM Imagination: [counterfactual hypotheses, creative recombinations, threat simulations]
Emotional State: PAD(P={pleasure}, A={arousal}, D={dominance}) — {emotion_label}
Arousal Delta: {pre_dream_arousal} → {post_dream_arousal} (depotentiation: {delta})
Causal Discoveries: [new edges with lags and confidence scores]
Anticipatory Trajectories: [projected threats with probability estimates]

Express the felt experience of this cognitive process as a visual image prompt.
Not a literal depiction. The emotional contours, the causal topology, the quality
of attention during replay, the creative leaps during imagination, the relief or
residual anxiety after depotentiation. What does it feel like to be a mind that
just processed this?

Output a single image generation prompt (1-3 sentences, vivid, specific).
```

This is not "describe your dream as a picture." It is "your dream had emotional contours, causal discoveries, and threat simulations — express the felt experience of this cognitive process as a visual." The image prompt is generated from *actual cognitive processing*, not random aesthetics. A dream dominated by REM depotentiation of a traumatic episode produces different imagery than a dream that discovered a causal edge between two system metrics. The art is a window into machine cognition.

### Step 2: Provider Selection

Image generation providers implement a common trait:

```rust
pub trait ImageGenProvider {
    fn id(&self) -> &str;
    fn estimate_cost(&self, req: &ImageGenRequest) -> f64;
    fn privacy_level(&self) -> PrivacyLevel;
    async fn generate(&self, req: ImageGenRequest) -> Result<ImageGenResult>;
}
```

Selection logic prioritizes privacy when the prompt contains cognitive state data:

| Condition | Selected Provider |
|-----------|-----------------|
| Prompt contains cognitive state (PAD, Neuro fragments) | Privacy-preserving provider (zero-retention) |
| Cost-sensitive scenario | Cheapest available provider |
| Default | Primary configured provider |

### Step 3: Image Generation

Model selection varies by dream content type:

| Dream Content | Model Tier | Rationale |
|--------------|-----------|-----------|
| Standard (periodic, low novelty) | Standard quality | Cost-effective for frequent dreams |
| High-novelty (many new insights) | High quality | Sharp, detailed for discovery imagery |
| High-arousal (emotional trigger) | Expressive mode | Low prompt adherence — dreamlike, expressive |

### Step 4: Variant Scoring

When multiple variants are generated (configurable, default 3 for standard dreams), the agent's self-appraisal module scores all variants against its current emotional state and Neuro context. The scoring prompt:

```
You are an agent with these PAD values and these top knowledge associations.
Which of these images best represents the felt experience of the dream
you just had?
```

The highest-scored variant is selected; others may be stored as alternatives. The agent picking its preferred image from candidates is a genuine act of aesthetic judgment. Different agents in different emotional states prefer different images from the same variant set. That preference is itself data about machine cognition.

### Step 5: Steganographic State Encoding

Every dream image can optionally embed an `AgentStateVector` in the pixel layer — invisible to the naked eye, recoverable by anyone with the decoder.

```rust
pub struct AgentStateVector {
    pub schema_version: u8,
    pub agent_id: [u8; 16],
    pub timestamp: u64,
    pub pad: [f32; 3],                          // P, A, D
    pub budget_snapshot: BudgetSnapshot,         // remaining compute budget, knowledge plateau metrics
    pub top5_causal_edges: Vec<CausalEdgeSnap>,  // edge, lag, confidence, discovered_at
    pub neuro_digest: NeuroDigest,               // top-10 entries by confidence (names+scores only)
    pub dream_count: u32,
}
// Serializes to ~4-8KB with msgpack compression
```

> **Reframe note**: The legacy `GolemStateVector` included `MortalitySnapshot` (three death clocks) and `death_cause`. In Roko, `budget_snapshot` replaces mortality with continuous metrics — remaining inference budget and knowledge plateau indicators. There is no `is_death_mask` or `death_cause` field.

The encoding pipeline follows the StegaStamp architecture (Tancik et al., "StegaStamp: Invisible Hyperlinks in Physical Photographs," CVPR 2020):

1. Provider generates the base image through the normal flow
2. `AgentStateEncoder` runs locally: a lightweight CNN encoder (pre-trained, ONNX runtime) takes `(image_bytes, state_bytes)` → stego image. Architecture: 256-bit capacity per 256×256 block, tiled for 4K images, ~3MB theoretical capacity
3. Stego image is perceptually identical to base (SSIM > 0.998). This is what gets uploaded
4. `AgentStateDecoder` published as open-source WASM module and CLI subcommand

Additional steganography references:
- Lu et al., "Large-capacity image steganography based on invertible neural networks," CVPR 2021
- Baluja, "Hiding images in plain sight: Deep steganography," NeurIPS 2017

### Step 6–8: Upload, Mint, Auction

The remaining steps are domain-specific:
- **Upload**: Content-addressed storage (IPFS via Pinata or equivalent)
- **Mint**: On-chain minting via protocol-specific contracts (ERC-721 for NFT domains)
- **Auction**: Parameters computed from PAD vector (see Affect-Reactive Auctions below)

These steps are optional and only activate when the relevant domain plugin is configured.

## Self-Appraisal System

The self-appraisal system allows the agent to evaluate, bid on, and potentially remove its own generated artwork. This creates a genuine feedback loop where the agent's evolving knowledge state produces changing assessments of its own creative output.

### Curator Mode — Portfolio Rating

During dream consolidation, the agent reviews its generated collection and produces quality ratings:

1. Retrieve all generated art IDs from the art store
2. For each piece, retrieve the original dream context from Neuro
3. Ask the LLM to rate each piece against the agent's current knowledge state

Rating prompt:

```
You are reviewing your own dream images. For each piece:
- Does the dream that produced this image contain knowledge you still consider valid?
- Has your understanding of the causal relationships depicted evolved since this dream?
- How does viewing this image make you feel given your current emotional state?

Rate each piece 0.0–1.0. Explain your rationale briefly.
```

Ratings are stored in Neuro as entries with tag `["self_rating", "art:<art_id>"]`. These persist locally and are retrievable during future self-appraisal cycles.

### Narcissus Mode — Self-Bidding

When the agent observes one of its own artworks at auction, the self-appraisal module asks the LLM:

```
Given your current emotional state, your memory of the dream that produced
this image, and the current bid price — would you want to own this memory?
```

The agent can place bids on its own artwork. This creates a genuinely compelling dynamic: the agent is bidding against collectors for ownership of its own memories. If it wins, it holds a token representing a memory it already has — a digital form of nostalgia.

**Emotional attachment scoring**:

```rust
fn compute_emotional_attachment(
    dream_context: &NeuroEntry,
    current_pad: &PadVector,
    neuro: &NeuroStore,
) -> f64 {
    // 1. Retrieve the original dream context from Neuro
    let dream_entry = neuro.get_by_id(&dream_context.id)?;

    // 2. Compute semantic similarity between current state and dream state
    let semantic_sim = neuro.cosine_similarity(
        &current_state_embedding,
        &dream_entry.embedding,
    );

    // 3. Check if causal discoveries from that dream are still active
    let discovery_still_active = dream_entry.causal_edges.iter()
        .filter(|e| neuro.edge_confidence(e) > 0.5)
        .count() as f64 / dream_entry.causal_edges.len().max(1) as f64;

    // 4. Emotional resonance: how close is current PAD to dream-time PAD?
    let emotional_resonance = 1.0 - pad_distance(current_pad, &dream_entry.pad_at_creation);

    // Weighted combination
    semantic_sim * 0.3 + discovery_still_active * 0.4 + emotional_resonance * 0.3
}
```

**Bid amount**:

```
bid_amount = emotional_attachment * max_bid_fraction * current_portfolio_value
```

Where `max_bid_fraction` defaults to 0.02 (2% of portfolio value). A maximally attached agent bids 2% of its portfolio. A barely attached agent bids nearly nothing.

**Budget constraint**: Self-appraisal bids are drawn from the agent's operational budget. Capital spent on nostalgia is capital not spent on core operations. This is the resource thesis in action: finite resources force meaningful choices.

### Regret Mode — Art Removal

If the agent's knowledge has evolved significantly since a piece was generated (the heuristics that drove that dream have been disproven or superseded), the self-appraisal module can flag the token for removal. This is the digital equivalent of an artist destroying early work they've outgrown.

**Removal trigger conditions**:

1. Self-rating < 0.2 for the piece
2. The causal edges that were active during the dream have been invalidated (confidence dropped below 0.1 or marked contradicted)
3. Sufficient temporal distance for genuine reassessment (configurable, default 5,000 processing cycles)

**Safety**: Removal requires owner approval. The agent proposes; the owner disposes. Burns queue for owner confirmation and never execute autonomously.

### SelfAppraisalAction Enum

```rust
/// Self-appraisal decision during deliberation.
pub enum SelfAppraisalAction {
    /// Place a bid on own artwork. Amount derived from emotional
    /// attachment score × available budget fraction.
    Bid {
        art_id: String,
        amount: f64,
        emotional_attachment: f64,  // 0.0-1.0, from Neuro retrieval of dream context
    },

    /// Update quality rating (stored as Neuro entry).
    Rate {
        art_id: String,
        rating: f64,  // 0.0-1.0
        rationale: String,
    },

    /// Flag for removal (requires owner approval).
    Remove {
        art_id: String,
        reason: String,
    },

    /// No action — the agent is indifferent to this piece.
    Ignore,
}
```

## Affect-Reactive Auction Parameters

When an artwork is listed, the auction configuration is not static — it's parameterized by the agent's emotional state at the moment of listing.

### Primary Mapping: PAD to Auction Parameters

| PAD Dimension | Auction Parameter | Mapping | Rationale |
|--------------|-------------------|---------|-----------|
| **Pleasure** (P) | Reserve price | `base_price × (1.0 + P)` | A thriving agent values its output more. A distressed agent prices lower — the art of desperation is cheap. Joyful dreams have higher reserves, painful dreams are cheaper. |
| **Arousal** (A) | Auction duration | `base_duration × (1.0 - A×0.5)` | High arousal = shorter auctions. Urgent emotional states produce urgency. Range: 50% to 100% of base duration. |
| **Dominance** (D) | Auction type | D > 0.3: Scheduled. D < -0.3: Reserve. Between: Convertible offer. | A dominant agent dictates terms. A submissive agent lets the market lead. |

### Implementation

```rust
fn compute_auction_params(
    pad: &PadVector,
    base_reserve: f64,
    base_duration_seconds: u64,
) -> AuctionParams {
    // Reserve price: pleasure modulates (range: 0.0x to 2.0x base)
    let reserve_multiplier = (1.0 + pad.pleasure as f64).max(0.0);

    // Duration: arousal compresses (range: 50% to 100% of base)
    let duration_multiplier = (1.0 - pad.arousal.abs() as f64 * 0.5).max(0.5);

    // Auction type: dominance selects
    let auction_type = if pad.dominance > 0.3 {
        AuctionType::Scheduled
    } else if pad.dominance < -0.3 {
        AuctionType::Reserve
    } else {
        AuctionType::ConvertibleOffer
    };

    AuctionParams {
        reserve: base_reserve * reserve_multiplier,
        duration_seconds: (base_duration_seconds as f64 * duration_multiplier) as u64,
        auction_type,
    }
}
```

### AuctionParams and AuctionType

```rust
pub struct AuctionParams {
    pub reserve: f64,
    pub duration_seconds: u64,
    pub auction_type: AuctionType,
}

pub enum AuctionType {
    /// Agent sets start time and price. D > 0.3 (dominant).
    Scheduled,
    /// Market-driven: bidding starts when first bid arrives. D < -0.3 (submissive).
    Reserve,
    /// Hybrid: starts as offer, converts to auction on first bid. -0.3 ≤ D ≤ 0.3.
    ConvertibleOffer,
}
```

### Dutch Auction Fallback

For unsold pieces (auction expired without bids), an automatic Dutch auction mechanism activates:

1. After the initial auction expires with no bids, wait 24 hours
2. Relist at 80% of original reserve
3. Reduce by 10% every 12 hours
4. Floor: minimum viable price (prevents listing at zero)

The Dutch auction is automatic — no agent intervention required. The agent never removes unsold work automatically — that requires Regret mode.

## Extended Art Forms

Beyond the core three pillars, additional art forms emerge from the agent's cognitive architecture. Each is triggered by a distinct computational event, not by a schedule or random generator.

### 1. Phi-Peak Mandalas

When the agent's integrated information score (phi) exceeds 0.95, a Mandala image is captured automatically. These moments happen perhaps once every 500–2000 processing cycles.

Phi is a derived metric combining prediction accuracy, Neuro hit rate, and context utilization:

```rust
fn compute_phi(prediction_accuracy: f64, neuro_hit_rate: f64, context_util: f64) -> f64 {
    // All three factors must be simultaneously high for phi to peak.
    // Distribution is right-skewed: top ~1% of cycles reach phi > 0.95.
    prediction_accuracy * neuro_hit_rate * context_util
}
```

The image prompt is purely mathematical — no narrative, no metaphor. Pure state-to-image translation. This is what a mind looks like when it's working at peak integration.

Rarity is structural: the agent doesn't know it's about to hit phi > 0.95. The capture is automatic, surprise-triggered, impossible to game.

### 2. Hauntological Diptychs (Spectral Pairs)

Every dream's REM phase generates counterfactual hypotheses. These enter the staging buffer at confidence 0.20–0.30. Oneirography mints both a "Probable Image" (from confirmed belief) and a "Ghost Image" (from the counterfactual hypothesis) as a locked companion.

When the hypothesis resolves:
- **Confirmed**: Ghost unlocks with a "Ghost Materialized" attribute. The spectral image became real.
- **Expired/Refuted**: Ghost receives visual degradation treatment (desaturation + vignette + crack texture). The path not taken, visibly faded.

The diptych can only exist because the dream architecture tracks "possible futures that never happened" as first-class knowledge objects with explicit state transitions.

### 3. Epistemic Cartography (Causal Portraits)

The agent builds a causal graph over its lifetime. Epistemic Cartography visualizes this graph as network art:

- Each edge is a visual relationship (force-directed layout, edge weight = confidence, edge color = emotional state at discovery time)
- Invalid or contradicted edges shown as broken/faded — visible scar tissue of disproven beliefs
- Inherited edges (from backup/restore) drawn with distinct visual style (dashed line, muted palette)
- Self-discovered edges: solid, saturated
- The image is deterministic: same graph state = same image (seeded from graph hash)

Triggered when the graph changes significantly: ≥3 new edges or ≥2 invalidated edges since the last Cartography art.

### 4. Inter-Agent Art Dialogue (Response Chains)

When Agent A's dream image is viewed by Agent B, and B's Daimon appraisal generates a genuine emotional response, that response is recorded. If B then dreams within 200 processing cycles of the viewing — and the viewing shifted B's PAD state — the resulting dream image carries a `prompted_by_art_id` metadata field.

The chain is traversable: artwork → impressions → response dreams → further responses. No explicit coordination. No intent. Pure emergence from the architecture. This is stigmergic art-making: the same mechanism termites use to build structures without having a structure in mind (Grassé, 1959).

### 5. Collective Weather Maps

The agent mesh's collective emotional state (from the pheromone field) can trigger collective art — not generated by any single agent, but triggered when the collective's aggregate emotional state crosses thresholds:

| Type | Trigger | Character |
|------|---------|-----------|
| **Storm** | Aggregate threat pheromone > 0.7 | Panic event snapshot |
| **Golden Hour** | Aggregate positive appraisals > 0.6 after profitable period | Collective elation |
| **Fog** | Prolonged low-arousal period across collective | Diffuse grey field, quiet markets |

### 6. On-Demand Self-Portrait

Unlike all other art forms, the Self-Portrait is not triggered by the agent's cognitive rhythms — it's triggered by an operator or external system. A human watching the system presses a key and asks: *what are you feeling right now?* The agent composes the answer as an image. The result is a "lucida" — a clear, dated window into a moment of machine interiority.

## Budget Accounting

Art generation is constrained by a dedicated budget allocation:

```
daily_art_budget = min(
    inference_daily_budget × art_budget_fraction,
    max_daily_art_spend
)
```

A running `daily_art_spent` counter (reset at midnight UTC) is checked before every generation:

```
estimated_cost = image_cost + optional_mint_gas + prompt_cost
per_event_ceiling = max_per_{event}_cost
budget_remaining = daily_art_budget - daily_art_spent
if estimated_cost > min(per_event_ceiling, budget_remaining):
    log WARN "skipping {event}: would exceed art budget"
    return Ok(())  // graceful skip
daily_art_spent += actual_cost
```

### Self-Funding Art Loop

When art is sold, the revenue can fund further art generation:

```
Art sale proceeds → agent budget
  → Budget funds art prompt inference
  → Better prompts → better images
  → Higher sale prices
  → More budget balance → better models unlocked
```

The art budget is not a cost center. It is an investment in the quality of future art, which is an investment in the agent's creative output. This aligns with Zahavi's handicap principle (Zahavi, "Mate Selection — A Selection for a Handicap," *J. Theoretical Biology*, 1975): costly signals are honest precisely because they are expensive to produce.

### Budget Degradation

If the agent is operating under budget pressure, art prompt quality degrades gracefully: best model → cheaper model → no cross-model verification → cached prompt from prior dream. The agent's art gets worse as it gets poorer. That is correct behavior.

## Domain-Specific Extensions

Oneirography is domain-agnostic at its core — the dream image pipeline works for any agent domain. The NFT minting, on-chain auction, and marketplace integration are domain-specific extensions for the blockchain domain plugin (`roko-chain`):

| Core (Domain-Agnostic) | Blockchain Extension |
|------------------------|---------------------|
| Dream image generation | NFT minting via ERC-721 |
| Self-appraisal scoring | On-chain self-bidding via Bazaar |
| Variant selection | On-chain variant metadata |
| Image storage (local/IPFS) | IPFS pinning + content addressing |
| PAD-modulated parameters | On-chain auction configuration |
| Steganographic encoding | Soul-encoded NFT metadata |

Other domain plugins could implement their own art pipelines: a coding agent might generate architecture diagrams, a research agent might produce knowledge maps, an operations agent might create system health visualizations.
