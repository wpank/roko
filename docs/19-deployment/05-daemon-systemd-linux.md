# Daemon Mode: systemd (Linux)

> On Linux, Roko can run as a persistent background daemon managed by systemd. The
> `roko daemon --install` command generates a systemd user unit file, installs it to
> `~/.config/systemd/user/`, and enables the service. This document covers the unit file
> generation, restart policies, journal integration, socket activation, and the differences
> from the macOS launchd path.

---

## Overview

The Linux daemon mode mirrors the macOS launchd path (see `04-daemon-launchd-macos.md`) with
identical functionality: repository watching, automatic plan execution, IPC over Unix socket,
and event streaming. The difference is the service manager integration — systemd instead of
launchd.

The `roko daemon` subcommand detects the platform at runtime and generates the appropriate
service definition:

```rust
fn daemon_install(config: &DaemonConfig) -> Result<()> {
    #[cfg(target_os = "macos")]
    return install_launchd(config);

    #[cfg(target_os = "linux")]
    return install_systemd(config);

    #[cfg(not(any(target_os = "macos", target_os = "linux")))]
    anyhow::bail!("Daemon mode is only supported on macOS (launchd) and Linux (systemd)");
}
```

## The `roko daemon` Subcommand (Linux)

The same CLI interface as macOS, backed by `systemctl --user`:

```
roko daemon [COMMAND]

Commands:
  install      Generate systemd unit and enable it
  uninstall    Disable and remove systemd unit
  start        Start the daemon (systemctl --user start roko)
  stop         Stop the daemon (systemctl --user stop roko)
  restart      Restart the daemon (systemctl --user restart roko)
  status       Show daemon status (systemctl --user status roko)
  logs         Tail daemon logs (journalctl --user-unit roko)
  send <cmd>   Send a command to the running daemon via IPC
```

## systemd Unit File

The generated unit file lives at `~/.config/systemd/user/roko.service`. It uses the user-level
systemd instance (not system-level), so no root access is required.

```ini
# ~/.config/systemd/user/roko.service
#
# Roko cognitive agent daemon.
# Generated by: roko daemon install
# To modify: edit this file, then run: systemctl --user daemon-reload

[Unit]
Description=Roko cognitive agent daemon
Documentation=https://github.com/nunchi/roko
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart=/home/USERNAME/.cargo/bin/roko daemon run \
    --socket %t/roko-daemon.sock

# Restart policy: restart on failure, with exponential backoff
Restart=on-failure
RestartSec=10
RestartMaxDelaySec=300
RestartSteps=5

# Environment
Environment=RUST_LOG=info
Environment=HOME=/home/USERNAME
EnvironmentFile=-%h/.config/roko/daemon.env

# Resource limits
LimitNOFILE=4096

# Security hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=read-only
ReadWritePaths=%h/.roko %h/.local/state/roko %h/.config/roko
PrivateTmp=true

# Working directory
WorkingDirectory=%h

# Watchdog: systemd kills the process if it doesn't ping within 60s
WatchdogSec=60

[Install]
WantedBy=default.target
```

### Key Configuration Choices

**`Type=simple`**: The daemon process itself is the main process. systemd considers the service
started as soon as the `ExecStart` process is running. This is correct because the daemon
initializes quickly (< 1 second) and starts accepting IPC commands immediately.

**`Restart=on-failure`**: systemd restarts the daemon if it exits with a non-zero exit code.
Clean shutdowns (exit 0) do not trigger restart. This matches the launchd `SuccessfulExit =
false` KeepAlive behavior.

**`RestartSec=10` with `RestartMaxDelaySec=300` and `RestartSteps=5`**: Exponential backoff
for restarts. First restart after 10 seconds, then progressively longer delays up to 5 minutes.
This prevents restart storms if the daemon has a persistent failure (e.g., missing config file,
permission error).

**`%t/roko-daemon.sock`**: The `%t` specifier expands to `$XDG_RUNTIME_DIR` (typically
`/run/user/1000` on Linux). This directory is per-user, tmpfs-backed, and cleaned up on logout
— ideal for Unix domain sockets.

**`EnvironmentFile=-%h/.config/roko/daemon.env`**: The `-` prefix means systemd ignores the
file if it does not exist. This file can contain API keys and other environment variables that
should not be in the unit file:

```bash
# ~/.config/roko/daemon.env
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
```

**Security hardening**:
- `NoNewPrivileges=true` — prevents the daemon from gaining new privileges
- `ProtectSystem=strict` — mounts the entire filesystem read-only except explicitly allowed paths
- `ProtectHome=read-only` — mounts the home directory read-only
- `ReadWritePaths=...` — allows write access only to Roko's state directories
- `PrivateTmp=true` — gives the daemon its own `/tmp` namespace

## Unit File Generation in Rust

```rust
/// Generate systemd unit file content for the daemon.
fn generate_systemd_unit(config: &DaemonConfig) -> String {
    let binary_path = std::env::current_exe()
        .unwrap_or_else(|_| PathBuf::from("roko"));
    let home = std::env::var("HOME")
        .unwrap_or_else(|_| String::from("/tmp"));

    format!(r#"[Unit]
Description=Roko cognitive agent daemon
Documentation=https://github.com/nunchi/roko
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart={binary} daemon run \
    --socket %t/roko-daemon.sock

Restart=on-failure
RestartSec=10
RestartMaxDelaySec=300
RestartSteps=5

Environment=RUST_LOG={log_level}
Environment=HOME={home}
EnvironmentFile=-{home}/.config/roko/daemon.env

LimitNOFILE=4096

NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=read-only
ReadWritePaths={home}/.roko {home}/.local/state/roko {home}/.config/roko
PrivateTmp=true

WorkingDirectory={home}

WatchdogSec=60

[Install]
WantedBy=default.target
"#,
        binary = binary_path.display(),
        home = home,
        log_level = config.log_level,
    )
}
```

## Lifecycle Commands (systemd Implementation)

### Install and Enable

```rust
fn install_systemd(config: &DaemonConfig) -> Result<()> {
    let unit_dir = dirs::config_dir()
        .unwrap_or_else(|| PathBuf::from("~/.config"))
        .join("systemd/user");
    std::fs::create_dir_all(&unit_dir)?;

    let unit_path = unit_dir.join("roko.service");

    if unit_path.exists() {
        eprintln!("Daemon already installed at {}", unit_path.display());
        eprintln!("Use 'roko daemon restart' or 'roko daemon uninstall' first.");
        return Ok(());
    }

    // Write the unit file
    let content = generate_systemd_unit(config);
    std::fs::write(&unit_path, &content)?;

    // Reload systemd to pick up the new unit
    run_systemctl(&["daemon-reload"])?;

    // Enable the service (start on login)
    run_systemctl(&["enable", "roko.service"])?;

    // Start the service now
    run_systemctl(&["start", "roko.service"])?;

    // Verify it started
    std::thread::sleep(std::time::Duration::from_secs(2));
    daemon_status_systemd()?;

    Ok(())
}

fn run_systemctl(args: &[&str]) -> Result<()> {
    let status = std::process::Command::new("systemctl")
        .arg("--user")
        .args(args)
        .status()?;

    if !status.success() {
        anyhow::bail!("systemctl --user {} failed", args.join(" "));
    }
    Ok(())
}
```

### Uninstall

```rust
fn uninstall_systemd() -> Result<()> {
    // Stop the service
    let _ = run_systemctl(&["stop", "roko.service"]);

    // Disable (remove from login startup)
    let _ = run_systemctl(&["disable", "roko.service"]);

    // Remove the unit file
    let unit_path = dirs::config_dir()
        .unwrap_or_else(|| PathBuf::from("~/.config"))
        .join("systemd/user/roko.service");

    if unit_path.exists() {
        std::fs::remove_file(&unit_path)?;
    }

    // Reload systemd
    run_systemctl(&["daemon-reload"])?;

    eprintln!("Daemon uninstalled.");
    Ok(())
}
```

## Log Management via journald

systemd's journald captures all stdout/stderr from the daemon. No separate log files needed —
the journal handles rotation, compression, and retention automatically.

```bash
# Tail daemon logs in real time
$ roko daemon logs
# Equivalent to: journalctl --user-unit roko.service -f

# Show last 100 lines
$ roko daemon logs --lines 100
# Equivalent to: journalctl --user-unit roko.service -n 100

# Show logs since last boot
$ roko daemon logs --boot
# Equivalent to: journalctl --user-unit roko.service -b

# Show logs from a specific time range
$ journalctl --user-unit roko.service --since "2026-04-10" --until "2026-04-11"

# JSON output for machine parsing
$ journalctl --user-unit roko.service -o json
```

The `roko daemon logs` command wraps `journalctl` for convenience, but users can always use
journalctl directly for advanced filtering.

### Structured Logging

The daemon uses `tracing` with the `tracing-journald` subscriber to emit structured log entries
that journald stores with metadata fields:

```rust
use tracing::{info, warn, error};

info!(
    repo = %repo_path.display(),
    plan = %plan_name,
    "Plan execution started"
);

warn!(
    subscription = %sub_id,
    error = %err,
    "Subscription check failed, will retry"
);
```

These structured fields are queryable via journalctl:

```bash
# Find all log entries for a specific repo
journalctl --user-unit roko.service REPO=/home/user/dev/project-a
```

## Watchdog Integration

The unit file includes `WatchdogSec=60`, which means the daemon must notify systemd at least
once every 60 seconds that it is still healthy. If it fails to do so, systemd considers it
hung and restarts it.

The daemon pings the watchdog from its main event loop:

```rust
use sd_notify::NotifyState;

// In the daemon's main event loop (runs every tick)
async fn daemon_tick(state: &DaemonState) {
    // Process subscriptions, IPC commands, events...

    // Ping the systemd watchdog
    let _ = sd_notify::notify(false, &[NotifyState::Watchdog]);
}

// On startup, notify systemd that the daemon is ready
fn notify_ready() {
    let _ = sd_notify::notify(false, &[NotifyState::Ready]);
}
```

This uses the `sd-notify` crate (lightweight, no C dependency) to communicate with systemd's
notification protocol. On macOS (where systemd is not available), the watchdog calls are no-ops.

## Socket Path Differences: macOS vs Linux

| Platform | Socket Path | Why |
|---|---|---|
| macOS | `/tmp/roko-daemon.sock` | macOS has no per-user runtime directory by default |
| Linux | `/run/user/$UID/roko-daemon.sock` | `$XDG_RUNTIME_DIR` is tmpfs, per-user, auto-cleaned |

The `%t` specifier in the systemd unit file expands to `$XDG_RUNTIME_DIR`. On Linux, this is
typically `/run/user/1000` for UID 1000.

The CLI automatically detects the correct socket path:

```rust
fn default_socket_path() -> PathBuf {
    #[cfg(target_os = "linux")]
    {
        if let Ok(runtime_dir) = std::env::var("XDG_RUNTIME_DIR") {
            return PathBuf::from(runtime_dir).join("roko-daemon.sock");
        }
    }

    // Fallback for macOS and other platforms
    PathBuf::from("/tmp/roko-daemon.sock")
}
```

## Lingering: Daemon Without Login Session

By default, systemd user services only run while the user has an active login session. For
headless servers or CI machines where the daemon should run persistently (even when no user
is logged in), enable lingering:

```bash
# Enable lingering for the current user (one-time, requires root or polkit)
loginctl enable-linger $USER
```

With lingering enabled, the user's systemd instance starts at boot and persists after logout.
The `roko daemon install` command checks for this and prints a note if lingering is not enabled:

```
Note: systemd user services only run while you're logged in.
For persistent daemon operation on a server, run:
  sudo loginctl enable-linger $USER
```

## Comparison: launchd vs systemd

| Feature | launchd (macOS) | systemd (Linux) |
|---|---|---|
| Unit file location | `~/Library/LaunchAgents/` | `~/.config/systemd/user/` |
| Start on login | `RunAtLoad = true` | `WantedBy=default.target` + `enable` |
| Restart on crash | `KeepAlive.SuccessfulExit = false` | `Restart=on-failure` |
| Restart backoff | `ThrottleInterval` (flat) | `RestartSec` + `RestartMaxDelaySec` (exponential) |
| Logs | Custom log files | journald (built-in rotation and querying) |
| Socket path | `/tmp/roko-daemon.sock` | `/run/user/$UID/roko-daemon.sock` |
| Security hardening | Sandbox profiles (limited) | `ProtectSystem`, `NoNewPrivileges`, namespaces |
| Watchdog | Not built-in | `WatchdogSec` with `sd_notify` |
| Persistent without login | Always (launchd runs at boot for user agents) | Requires `loginctl enable-linger` |
| Environment | `EnvironmentVariables` dict in plist | `Environment=` directives + `EnvironmentFile=` |

The daemon implementation provides identical functionality on both platforms. The platform
differences are abstracted behind the `roko daemon` CLI — users do not need to know whether
they are on macOS or Linux.
