## What this task does

Threads `Arc<dyn EventEmitter>` through the `Scenario` trait and into every scenario spine, then wires event emission into every step of `yield_routing.rs`. The result: running `yield-routing --events ndjson` prints a structured NDJSON stream that describes the full scenario lifecycle.

This task modifies the `Scenario` trait. Every existing scenario implementation must be updated to match the new signature, even if they ignore the emitter.

## Step 4 — Wire events into yield_routing.rs

Read the current yield-routing spine before editing. Then add event emission at each point listed below.

Use a helper pattern to fire-and-forget without failing the scenario on emission errors:

```rust
// inline every call — do not extract a helper function
events.emit(DemoEvent::SomeVariant { .. }).await;
```

Emission is always fire-and-forget: a failure inside `emit` (if the backend is down) must not propagate as an error. The `EventEmitter` trait's `emit` method already swallows errors internally (see `NullEmitter`, `NdjsonEmitter`). You do not need a wrapper.

### Emission points

#### Scenario start

Before the first round, emit:

```rust
events.emit(DemoEvent::ScenarioStarted {
    name: "yield-routing".to_string(),
    agents: agents
        .iter()
        .map(|a| AgentInfo {
            id: a.id.clone(),
            model: llm_label(a.index, &llm),   // see note on model labels below
            wallet: format!("{:#x}", a.wallet),
        })
        .collect(),
}).await;
```

**Model label note:** For `StubLlm`, use `"stub"`. For `ClaudeApiProvider`, use `"claude"`. For `OllamaProvider`, use `"ollama"`. For `MultiProvider`, use `"multi"`. The simplest approach: add a `fn model_label(llm: &Arc<dyn LlmProvider>) -> String` free function in `yield_routing.rs` that uses `std::any::type_name` or just returns `"stub"` for the default case. Alternatively, add a `fn label(&self) -> &'static str` method to `LlmProvider` with a default implementation returning `"unknown"`. Do whichever requires fewer changes across files. Record the actual start timestamp (`std::time::Instant::now()`) for use in `ScenarioCompleted`.

#### Each round — start

```rust
events.emit(DemoEvent::RoundStarted { round, total_rounds: 2 }).await;
```

#### Job posted

After the on-chain `postJob` call succeeds:

```rust
events.emit(DemoEvent::JobPosted {
    job_id,
    description: job_spec.clone(),
    bounty_amount: format!("{bounty}"),
}).await;
```

#### Knowledge query (per agent, before bidding)

After each agent's knowledge query (to `InsightBoard` or a local cache):

```rust
events.emit(DemoEvent::KnowledgeQueried {
    agent_id: agent.id.clone(),
    insights_found,   // count of relevant insights retrieved
}).await;
```

If the current spine does not query knowledge before T2.2, emit with `insights_found: 0` as a placeholder.

#### Agent bid decision (per agent)

After each agent's LLM call returns a `BidDecision`:

```rust
events.emit(DemoEvent::AgentBid {
    agent_id: agent.id.clone(),
    model: model_label(&llm),
    expected_output: route_decision.expected_output,
    strategy_summary: route_decision.strategy_summary.clone(),
}).await;
```

The `route_decision` comes from the `fill("route_proposal", ...)` call. Parse the returned JSON into a local struct:

```rust
#[derive(serde::Deserialize)]
struct RouteProposal {
    hops: Vec<String>,
    expected_output: f64,
    #[serde(default)]
    strategy_summary: String,
}
```

#### Job assigned

After the on-chain `assign(job_id, winner_wallet)` call:

```rust
events.emit(DemoEvent::JobAssigned {
    job_id,
    agent_id: winner.id.clone(),
}).await;
```

#### Execution started

Before the winner's on-chain `submit(job_id, result_hash)` call:

```rust
events.emit(DemoEvent::ExecutionStarted {
    agent_id: winner.id.clone(),
    route: serde_json::to_value(&winner_proposal.hops).unwrap_or(serde_json::Value::Null),
}).await;
```

Use `unwrap_or(serde_json::Value::Null)` — serialization of a `Vec<String>` will never fail in practice, but the pattern avoids `unwrap`.

#### Execution completed

After `submit` succeeds:

```rust
events.emit(DemoEvent::ExecutionCompleted {
    agent_id: winner.id.clone(),
    output_amount: winner_proposal.expected_output,
    gas_used: 0,
}).await;
```

#### Validator vote (per validator)

After each validator's LLM call returns a `VoteDecision`:

```rust
events.emit(DemoEvent::ValidationVote {
    validator_id: validator.id.clone(),
    approved: vote.approve,
    reason: vote.reason.clone(),
}).await;
```

#### Validation complete

After the on-chain `resolve(job_id, accepted)` call:

```rust
events.emit(DemoEvent::ValidationComplete {
    job_id,
    accepted,
}).await;
```

#### Insight posted (per agent, if the spine posts insights)

After each on-chain `postInsight` call:

```rust
events.emit(DemoEvent::InsightPosted {
    agent_id: agent.id.clone(),
    insight_id,
    content: insight_content.clone(),
}).await;
```

If the current spine does not post insights yet, skip this emission point and add a `// TODO T2.2` comment.

#### Insight confirmed (per confirmer, if the spine confirms insights)

```rust
events.emit(DemoEvent::InsightConfirmed {
    confirmer_id: confirmer.id.clone(),
    insight_id,
}).await;
```

Same: skip and mark `// TODO T2.2` if not yet implemented.

#### Round completed

At the end of each round, before the loop continues:

```rust
events.emit(DemoEvent::RoundCompleted { round }).await;
```

#### C-Factor measurement

After both rounds complete, store each round's winning `expected_output`:

```rust
let round1_output: f64 = /* winner_proposal.expected_output from round 1 */;
let round2_output: f64 = /* winner_proposal.expected_output from round 2 */;
let improvement_pct = if round1_output != 0.0 {
    ((round2_output - round1_output) / round1_output) * 100.0
} else {
    0.0
};

events.emit(DemoEvent::CFactorMeasured {
    run_a_output: round1_output,
    run_b_output: round2_output,
    improvement_pct,
}).await;
```

With `StubLlm` the stub counter increments between rounds, so the values will differ artificially. With a real LLM and knowledge injection (T2.2), the difference should reflect genuine improvement.

#### Scenario completed

After `CFactorMeasured`:

```rust
let duration_ms = scenario_start.elapsed().as_millis() as u64;
events.emit(DemoEvent::ScenarioCompleted {
    name: "yield-routing".to_string(),
    duration_ms,
}).await;
```

   ```bash
   cargo run -p roko-demo -- --demo-dir demo up yield-routing --events ndjson 2>/dev/null | head -20
   ```

   ```bash
   cargo run -p roko-demo -- --demo-dir demo up yield-routing --events ndjson 2>/dev/null \
   ```

7. The scenario completes in under 60 seconds with `--llm-backend stub`.
8. `ScenarioStarted` is the first event; `ScenarioCompleted` is the last.
9. `CFactorMeasured` appears exactly once, between `RoundCompleted { round: 2 }` and `ScenarioCompleted`.
10. No `unwrap()` in any new code. Propagate errors with `?` or absorb them with `tracing::warn!`.
