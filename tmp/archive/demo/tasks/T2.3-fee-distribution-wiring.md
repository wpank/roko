## Context

Today, when `BountyMarket::resolve(jobId, true)` is called, the bounty transfers directly to the
winning worker. No fees flow to validators who checked the work or to the data providers whose
insights the winning agent used.

This task routes payment through `FeeDistributor` after resolution. The split is 40% validators,
30% data providers, 20% agent, 10% treasury. It does not modify `BountyMarket.sol` — the worker
receives the full bounty first (via the existing resolve path), then immediately forwards it to
`FeeDistributor`, which pulls from the worker and redistributes.

This is Option A from the design doc: non-invasive, no contract changes required.

## Step 1 — Add FeeDistributor to bindings.rs

Open `crates/roko-demo/src/bindings.rs`. After the `InsightBoard` sol! block (line 97), add:

```rust
#[sol(rpc)]
contract FeeDistributor {
    function distribute(
        uint256 jobId,
        uint256 totalAmount,
        address agent,
        address[] calldata validators,
        address[] calldata dataProviders
    ) external;
    function cumulativeEarnings(address recipient) external view returns (uint256);
    function setTreasury(address newTreasury) external;
    event FeesDistributed(
        uint256 indexed jobId,
        uint256 totalAmount,
        uint256 validatorShare,
        uint256 dataProviderShare,
        uint256 agentShare,
        uint256 treasuryShare
    );
    event EarningsCredited(
        address indexed recipient,
        uint256 amount,
        string role
    );
}
```

The function signatures must exactly match `contracts/src/FeeDistributor.sol`. If the Solidity
contract uses different parameter names, adjust. The event names must match exactly.

## Step 3 — Wire distribute() into yield_routing.rs

### 3a. Import FeeDistributor

In `yield_routing.rs`, add `FeeDistributor` to the existing bindings import:

```rust
use crate::bindings::{BountyMarket, FeeDistributor, InsightBoard, MockERC20, WorkerRegistry};
```

### 3b. Approve FeeDistributor in prepare()

In the `prepare` function, after each worker receives their DAEJI mint, approve the
`FeeDistributor` to spend their tokens:

```rust
let fee_dist_addr = ctx.address_of("FeeDistributor")?;
// ...existing approve for WorkerRegistry...
let token_w = MockERC20::new(token_addr, worker_provider.clone());
token_w.approve(fee_dist_addr, U256::MAX).send().await?.watch().await?;
```

### 3c. Collect validator addresses

The yield-routing scenario uses a `ConsortiumValidator` for final approval (added in T2.1).
After the consortium's `Tallied` event fires, you have the three committee members. Store them
as a `Vec<Address>` called `validator_addrs`.

If the scenario does not use ConsortiumValidator (bare-resolve path), use `[deployer_addr]` as
a one-element validator slice.

### 3d. Collect data-provider addresses

Data providers are the agents whose insights the winning agent used. Read this from the insight
query result (from T2.2): collect the `poster` field of each insight whose `id` appears in the
winning proposal's context. For the stub case, use the addresses of the workers who seeded the
initial insights (workers 0, 1, 2).

Simplest implementation: pass the three seeders from `prepare` as data providers in every round.

```rust
let data_provider_addrs: Vec<Address> = (0..3)
    .map(|i| ctx.wallet_address(&format!("worker{i}")))
    .collect::<anyhow::Result<Vec<_>>>()?;
```

### 3e. Call distribute() after resolve

After `BountyMarket::resolve(job_id, true)` completes, add:

```rust
// Winner has received the bounty from BountyMarket. Now forward it to FeeDistributor.
let bounty_amount: U256 = /* same value used in postJob */;
let fee_dist = FeeDistributor::new(fee_dist_addr, winner_provider.clone());

// Winner approves FeeDistributor for this exact amount (already approved MAX in prepare).
fee_dist
    .distribute(
        job_id,
        bounty_amount,
        winner_addr,
        validator_addrs.clone(),
        data_provider_addrs.clone(),
    )
    .send()
    .await?
    .watch()
    .await?;
tracing::info!(round, %job_id, "fees distributed");
```

`winner_provider` is the alloy provider for the winning worker wallet. `bounty_amount` is the
same `U256` value passed to `postJob` earlier in the round — store it in a variable at post time
and reuse it here.

### 3f. Compute share amounts for the event

The on-chain split is 4000/3000/2000/1000 basis points. Compute the amounts in Rust for the
`FeesDistributed` DemoEvent:

```rust
let validator_share = bounty_amount * U256::from(4000u64) / U256::from(10000u64);
let data_provider_share = bounty_amount * U256::from(3000u64) / U256::from(10000u64);
let agent_share = bounty_amount * U256::from(2000u64) / U256::from(10000u64);
let treasury_share = bounty_amount - validator_share - data_provider_share - agent_share;

fn fmt_ether(wei: U256) -> String {
    format!("{} DAEJI", wei / U256::from(10u128.pow(18)))
}

events.emit(DemoEvent::FeesDistributed {
    job_id: job_id.to::<u64>(),
    validator_share: fmt_ether(validator_share),
    data_provider_share: fmt_ether(data_provider_share),
    agent_share: fmt_ether(agent_share),
    treasury_share: fmt_ether(treasury_share),
}).await;
```

## FeeDistributor constructor note

`FeeDistributor::new(token_, treasury_)` — the first argument is the ERC20 token address, the
second is the treasury address. In the manifest, `$deployer` maps to the sender wallet. In Rust,
retrieve the deployer address via `ctx.wallet_address("deployer")?`.
