## Context

C-Factor measures how much shared knowledge improves agent performance. To measure it, you run
the yield-routing scenario twice:

- **Run A (cold):** empty InsightBoard — no prior knowledge.
- **Run B (warm):** the insights posted during Run A are present — agents see prior knowledge.

The improvement in the winning agent's `expected_output` between Run A and Run B is the
C-Factor. A positive percentage means shared knowledge helped.

This task adds a `benchmark c-factor` subcommand to `roko-demo` that executes both runs and
emits a structured JSON report.

## Step 1 — Create `crates/roko-demo/src/benchmark.rs`

### Types

```rust
use serde::{Deserialize, Serialize};

/// Result of one benchmark run (cold or warm).
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct RunResult {
    /// Identifier of the winning worker (e.g. "worker2").
    pub winner_agent: String,
    /// LLM backend label used (e.g. "stub", "claude", "ollama").
    pub winner_model: String,
    /// `expected_output` field from the winning route proposal.
    pub expected_output: f64,
    /// Number of insights available to agents at bid time.
    pub insights_available: usize,
    /// Wall-clock milliseconds for this run.
    pub duration_ms: u64,
}

/// Computed improvement metrics.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CFactorMetrics {
    /// `(run_b.expected_output - run_a.expected_output) / run_a.expected_output * 100`.
    /// Negative if warm run performed worse (indicates overfitting or noise).
    pub output_improvement_pct: f64,
}

/// Full C-Factor benchmark report.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CFactorReport {
    /// Block number at which the benchmark forked (0 for Anvil default).
    pub fork_block: u64,
    /// Cold run — no prior knowledge.
    pub run_a: RunResult,
    /// Warm run — with knowledge from Run A.
    pub run_b: RunResult,
    /// Computed improvement.
    pub c_factor: CFactorMetrics,
}
```

### `run_c_factor` function signature

```rust
use std::sync::Arc;
use crate::chain_ctx::ChainCtx;
use crate::events::EventEmitter;
use crate::scenarios::llm::LlmProvider;

/// Run the C-Factor benchmark: cold run followed by warm run.
///
/// Returns a `CFactorReport` with both results and the computed improvement percentage.
pub async fn run_c_factor(
    ctx: Arc<ChainCtx>,
    llm: Arc<dyn LlmProvider>,
    events: Arc<dyn EventEmitter>,
) -> anyhow::Result<CFactorReport>
```

### `run_c_factor` implementation

The function calls into the yield-routing scenario's single-round logic. That logic must be
factored out as described in Step 4. The benchmark function:

1. Calls `crate::scenarios::yield_routing::run_cold_round(&ctx, llm.clone(), events.clone())`
   — deploys + prepares fresh contracts, then runs round 1 only (no seeded insights). Records
   `winner_agent`, `winner_model`, `expected_output`, `insights_available = 0`, `duration_ms`.

2. Calls `crate::scenarios::yield_routing::run_warm_round(&ctx, llm.clone(), events.clone())`
   — reuses the same deployed contracts (InsightBoard now has the round-1 insight). Runs round 2
   only. Records the same fields, `insights_available = board.nextInsightId()` before the round.

3. Computes:
   ```rust
   let improvement = if run_a.expected_output > 0.0 {
       (run_b.expected_output - run_a.expected_output) / run_a.expected_output * 100.0
   } else {
       0.0
   };
   ```

4. Emits `DemoEvent::CFactorMeasured` via the event emitter:
   ```rust
   events.emit(DemoEvent::CFactorMeasured {
       run_a_output: run_a.expected_output,
       run_b_output: run_b.expected_output,
       improvement_pct: improvement,
   }).await;
   ```

5. Returns the assembled `CFactorReport`.

### Human-readable summary

Print to stderr (not stdout — stdout is reserved for the JSON report):

```rust
eprintln!("\nC-Factor Benchmark Results");
eprintln!("==========================");
eprintln!(
    "Run A (cold): winner={} ({}), output={:.2}, insights={}, time={:.1}s",
    run_a.winner_agent, run_a.winner_model, run_a.expected_output,
    run_a.insights_available, run_a.duration_ms as f64 / 1000.0
);
eprintln!(
    "Run B (warm): winner={} ({}), output={:.2}, insights={}, time={:.1}s",
    run_b.winner_agent, run_b.winner_model, run_b.expected_output,
    run_b.insights_available, run_b.duration_ms as f64 / 1000.0
);
let sign = if improvement >= 0.0 { "+" } else { "" };
eprintln!("Improvement: {sign}{improvement:.1}%");
```

Print this before returning the report. The JSON is printed by the caller in `main.rs`.

### `benchmark_cmd` function

```rust
async fn benchmark_cmd(cli: &Cli, loaded: &LoadedManifest, kind: &BenchmarkKind) -> anyhow::Result<()> {
    match kind {
        BenchmarkKind::CFactor { output } => {
            let deploy_ctx = loaded.build_deploy_ctx(cli.rpc_url.clone())?;
            let deployments = load_deployments(&cli.runtime_dir, "yield-routing")?;
            let ctx = Arc::new(ChainCtx {
                rpc_url: deploy_ctx.rpc_url.clone(),
                chain_id: deploy_ctx.chain_id,
                wallets: deploy_ctx.wallets.clone(),
                addresses: deployments.contracts,
                deployed_at_block: deployments.deployed_at_block,
            });
            let llm = create_provider(&cli.llm_backend)?;
            let events = create_emitter(&cli.events, cli.ws_port).await?;
            let report = run_c_factor(ctx, llm, events).await?;
            let json = serde_json::to_string_pretty(&report)?;
            match output {
                Some(path) => std::fs::write(path, &json)?,
                None => println!("{json}"),
            }
        }
    }
    Ok(())
}
```

`benchmark_cmd` relies on a prior `roko-demo deploy yield-routing` having been run. It reuses
the existing deployment rather than redeploying. For a self-contained benchmark that deploys
fresh contracts, the caller should run:

```bash
cargo run -p roko-demo -- --demo-dir demo deploy yield-routing
cargo run -p roko-demo -- --demo-dir demo benchmark c-factor
```

## Step 4 — Factor out single-round logic in yield_routing.rs

The benchmark needs to run individual rounds in isolation. Refactor `yield_routing.rs` so the
two-round spine calls out to reusable functions:

```rust
/// Run round 1 with empty InsightBoard (cold start).
/// Returns the winner's name, their expected_output, and how many insights were available.
pub async fn run_cold_round(
    ctx: &ChainCtx,
    llm: Arc<dyn LlmProvider>,
    events: Arc<dyn EventEmitter>,
) -> anyhow::Result<(String, f64, usize)>

/// Run round 2, assuming InsightBoard already has insights from a prior run.
/// Returns the same triple.
pub async fn run_warm_round(
    ctx: &ChainCtx,
    llm: Arc<dyn LlmProvider>,
    events: Arc<dyn EventEmitter>,
) -> anyhow::Result<(String, f64, usize)>
```

The existing `Scenario::spine` implementation should become:

```rust
async fn spine(&self, ctx: Arc<ChainCtx>, _manifest: &ScenarioManifest, llm: Arc<dyn LlmProvider>) -> anyhow::Result<()> {
    prepare(&ctx).await?;
    // For the scenario run, use a null emitter when no events flag is set.
    let null_events: Arc<dyn EventEmitter> = Arc::new(crate::events::NullEmitter);
    run_cold_round(&ctx, llm.clone(), null_events.clone()).await?;
    run_warm_round(&ctx, llm, null_events).await?;
    Ok(())
}
```

If `EventEmitter` is passed into `spine` (from T2.1), use the real emitter instead of
`NullEmitter`.

`run_cold_round` and `run_warm_round` must be `pub` so `benchmark.rs` can import them as
`crate::scenarios::yield_routing::run_cold_round`.

## StubLlm behaviour with benchmark

With `StubLlm`, the `route_proposal` slot returns:

```rust
"route_proposal" => serde_json::json!({
    "hops": ["0xaaa", "0xbbb"],
    "expected_output": 1000 + n * 10,
})
```

Because the counter advances between calls, Run A will receive `expected_output: 1000` (counter
0) and Run B will receive `expected_output: 1010` (counter 1). This gives a measurable 1%
improvement, which is enough to verify the pipeline.
