## What to add

### Option A — struct return (preferred)

alloy's `sol!` macro supports struct return types when the struct is defined in the same macro
block. Add the struct definition and the function:

```rust
#[sol(rpc)]
contract InsightBoard {
    // Existing functions...
    function post(bytes32 contentHash, string calldata uri) external returns (uint256);
    function confirm(uint256 id) external;
    function claim() external returns (uint256);
    function earningsOf(address poster) external view returns (uint256);
    function nextInsightId() external view returns (uint256);
    event InsightPosted(uint256 indexed id, address indexed poster, bytes32 contentHash, string uri);
    event InsightConfirmed(uint256 indexed id, address indexed confirmer, uint64 pheromone);

    // New: struct type + getter
    struct Insight {
        address poster;
        bytes32 contentHash;
        string uri;
        uint64 postedAt;
        uint64 pheromone;
    }
    function getInsight(uint256 id) external view returns (Insight memory);
}
```

When alloy generates bindings for this, the return type of `getInsight` will be
`InsightBoard::Insight` (or similar). Fields are accessed as `.poster`, `.contentHash`,
`.uri`, `.postedAt`, `.pheromone`.

### Option B — tuple return (fallback)

If alloy does not support the `struct` keyword inside a `sol!` block, flatten to named output
parameters instead:

```rust
function getInsight(uint256 id) external view returns (
    address poster,
    bytes32 contentHash,
    string memory uri,
    uint64 postedAt,
    uint64 pheromone
);
```

alloy generates a `getInsightReturn` struct with those field names. Fields are accessed the same
way: `.poster`, `.contentHash`, `.uri`, `.postedAt`, `.pheromone`.

Try Option A first. If `cargo build -p roko-demo` errors with a parse error in the sol! macro,
fall back to Option B.
