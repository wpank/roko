## What this task does

Adds a `tui` subcommand to `roko-demo` that runs the yield-routing scenario while rendering a
live 4-panel terminal UI. The TUI consumes `DemoEvent` values from a channel and updates display
state in real time.

### 1. State model in `crates/roko-demo/src/tui.rs`

### 2. Event processing

Add a method `TuiState::apply(&mut self, event: &DemoEvent)` that updates state:

| Event | State update |
|---|---|
| `ScenarioStarted { agents, .. }` | Populate `self.agents` from `AgentInfo` list; set all statuses to `Idle` |
| `RoundStarted { round, total_rounds }` | Set `current_round`; set `total_rounds`; reset `knowledge.new_this_round = 0` |
| `JobPosted { description, .. }` | Add log entry in Cyan: `"Job posted: {description}"` |
| `KnowledgeQueried { agent_id, .. }` | Set matching agent status to `Querying`; add log entry |
| `AgentBid { agent_id, model, expected_output, .. }` | Set matching agent status to `Bidding`; add log entry with `expected_output` |
| `JobAssigned { agent_id, .. }` | Set matching agent status to `Winner`; set all others to `Idle`; add Green log entry |
| `ExecutionStarted { agent_id, .. }` | Set matching agent status to `Executing`; add log entry |
| `ExecutionCompleted { agent_id, output_amount, .. }` | Add `(current_round, output_amount)` to `economics.round_outputs`; add Green log entry |
| `ValidationVote { validator_id, approved, .. }` | Add log entry — Green if `approved`, Red if not |
| `ValidationComplete { accepted, .. }` | Add log entry — Green if accepted, Red if not |
| `FeesDistributed { agent_share, .. }` | Parse `agent_share` as u64; update `economics.total_fees`; add log entry |
| `InsightPosted { .. }` | Increment `knowledge.total_insights` and `knowledge.new_this_round`; add Cyan log entry |
| `InsightConfirmed { .. }` | Increment `knowledge.total_confirms`; add log entry |
| `ReputationUpdated { agent_id, new_rep, .. }` | Update matching agent's `reputation` |
| `CFactorMeasured { improvement_pct, .. }` | Set `economics.c_factor = Some(improvement_pct)` |
| `RoundCompleted { .. }` | Add log entry |
| `ScenarioCompleted { .. }` | Set `self.done = true`; add Green log entry |
| `Error { message }` | Add Red log entry |

All events also call a shared helper that prepends a timestamp and appends to `activity_log`. Cap
`activity_log` at 500 entries — drop the oldest when the limit is reached.

Timestamp format: `chrono` is not available. Use
`std::time::SystemTime::now().duration_since(std::time::UNIX_EPOCH)` to get seconds since epoch,
then format as `HH:MM:SS` manually (modulo arithmetic). Or just use a monotonic counter
`entry_count` as `{count:04}` if time formatting adds too much complexity.

    widgets::{Block, Borders, List, ListItem, Paragraph},

pub fn render(frame: &mut Frame, state: &TuiState) { ... }

**Top bar** — single-line `Paragraph` with title and round counter:

```
NUNCHI DEMO — Yield Routing                          Round 2/2
```

**Middle row** — horizontal split with `Constraint::Percentage(20)` (agents) and
`Constraint::Percentage(80)` (activity log).

**Bottom row** — horizontal split with `Constraint::Percentage(20)` (knowledge) and
`Constraint::Percentage(80)` (economics).

**Agents panel** — `List` widget. Each agent renders as two `ListItem` rows:
- Row 1: `"● {id}"` — bullet is Green if `Winner`, Yellow if `Querying`/`Bidding`/`Executing`,
  White if `Idle`
- Row 2: `"  {model}  rep:{:.2}  earned:{}"` in dim style

**Activity log panel** — `List` of `LogEntry` values, newest-at-bottom. Apply `scroll_offset` for
keyboard scrolling. Each entry uses the entry's `color` field as the foreground color.

**Knowledge panel** — `Paragraph` with four lines:
```
Insights: {total_insights}
New: +{new_this_round}
Confirms: {total_confirms}
Top: {top_weight:.2}
```

**Economics panel** — `Paragraph`. Show one line per completed round and the C-Factor line if set:
```
Round 1: {output:.1} ETH
Round 2: {output:.1} ETH
C-Factor: +{pct:.1}%
```
If C-Factor is not yet set, omit that line.

All panels use `Block::default().borders(Borders::ALL).title("...")`.

### 4. Event loop

```rust
pub async fn run_tui(
    demo_dir: PathBuf,
    runtime_dir: PathBuf,
    rpc_url: Option<String>,
    scenario: String,
    llm_backend: String,
) -> anyhow::Result<()>
```

Steps:

1. Create an `mpsc::channel::<DemoEvent>(512)`.

2. Create a `ChannelEmitter` that wraps the `mpsc::Sender`:

   ```rust
   struct ChannelEmitter { tx: tokio::sync::mpsc::Sender<DemoEvent> }

   #[async_trait::async_trait]
   impl EventEmitter for ChannelEmitter {
       async fn emit(&self, event: DemoEvent) {
           let _ = self.tx.send(event).await;
       }
   }
   ```

3. Load the manifest via `LoadedManifest::load(&demo_dir)?`.

4. Build `ChainCtx` from the loaded manifest.

5. Look up the scenario via `scenarios::find(&scenario)`.

6. Spawn the scenario spine in a background tokio task, passing `Arc::new(ChannelEmitter { tx })`.
   Wrap it in `tokio::spawn`.

7. Initialize crossterm and ratatui terminal:

   ```rust
   use crossterm::{
       event::{self, Event, KeyCode, KeyEventKind},
       execute,
       terminal::{EnterAlternateScreen, LeaveAlternateScreen, disable_raw_mode, enable_raw_mode},
   };
   use ratatui::{Terminal, backend::CrosstermBackend};

   enable_raw_mode()?;
   let mut stdout = std::io::stdout();
   execute!(stdout, EnterAlternateScreen)?;
   let backend = CrosstermBackend::new(stdout);
   let mut terminal = Terminal::new(backend)?;
   ```

8. Run the render loop:

   ```rust
   let mut state = TuiState::new();
   loop {
       // Drain all pending events from the channel without blocking.
       while let Ok(event) = rx.try_recv() {
           state.apply(&event);
       }
       terminal.draw(|f| render(f, &state))?;
       // Poll for keyboard input with a 50ms timeout.
       if event::poll(std::time::Duration::from_millis(50))? {
           if let Event::Key(key) = event::read()? {
               if key.kind == KeyEventKind::Press {
                   match key.code {
                       KeyCode::Char('q') | KeyCode::Esc => break,
                       KeyCode::Char('j') | KeyCode::Down => {
                           state.scroll_offset = state.scroll_offset.saturating_add(1);
                       }
                       KeyCode::Char('k') | KeyCode::Up => {
                           state.scroll_offset = state.scroll_offset.saturating_sub(1);
                       }
                       _ => {}
                   }
               }
           }
       }
       // Break naturally when the scenario finishes.
       if state.done {
           tokio::time::sleep(std::time::Duration::from_secs(3)).await;
           break;
       }
   }
   ```

9. Always restore terminal, even on error. Use a guard or explicit cleanup:

   ```rust
   disable_raw_mode()?;
   execute!(terminal.backend_mut(), LeaveAlternateScreen)?;
   terminal.show_cursor()?;
   ```

   Wrap the entire loop in a closure or use `scopeguard` (not available — use explicit cleanup in
   all return paths). Pattern: put the loop body after `enable_raw_mode`, wrap in a block that
   catches the result, then call the three cleanup lines unconditionally before returning the
   result.

## Acceptance criteria

1. `cargo build -p roko-demo` compiles without errors.
2. `cargo clippy -p roko-demo -- -D warnings` passes.
3. `cargo run -p roko-demo -- --demo-dir demo tui` launches a terminal UI without panicking.
4. All four panels render and have `Borders::ALL` visible.
5. The agents panel lists agents by name with a model label once `ScenarioStarted` arrives.
6. The activity log scrolls with `j`/`k` and arrow keys.
7. The knowledge panel updates its count when `InsightPosted` events arrive.
8. The economics panel shows C-Factor after `CFactorMeasured` arrives.
9. `q` and `Esc` quit cleanly — the terminal is restored to its original state (no leftover raw
   mode or alternate screen).
10. No panic on terminal resize.
11. No `unwrap()` in any new code. Propagate errors with `?` or log with `tracing::warn!`.

## Do NOT

- Do not import from `crate::tui` inside scenario files — the dependency is one-way (tui imports
  from scenarios, not the reverse).
- Do not use `unwrap()` anywhere in the new code.
- Do not use `panic!` to handle resize events — ratatui handles resize automatically.
- Do not add `#[allow(dead_code)]` — make everything reachable.
- Do not use `{ workspace = true }` for `ratatui` or `crossterm` — they are not workspace deps.
- Do not modify any existing scenario file.
- Do not skip the terminal restore step — a process that leaves raw mode active on exit is
  broken.
