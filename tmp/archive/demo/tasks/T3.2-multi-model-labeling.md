## What this task does

Ensures every agent's LLM model label is accurate in all emitted events and in the TUI agents
panel. Today the model field is likely `"stub"` for all agents regardless of backend. After this
task, each of the five workers has a distinct, readable model label — both when `StubLlm` runs
(using the static `AGENT_MODELS` constants) and when a real backend runs (using the actual model
name from the provider).

---

## Design decision

The simplest approach that requires the fewest changes across files: assign each worker a fixed
provider index at scenario start rather than sharing a single round-robin provider. This avoids
adding a `label` method to `LlmProvider` (a breaking interface change) and keeps model labels
deterministic and visible to the TUI.

In `yield_routing.rs`, the `AGENT_MODELS` constant already exists:

```rust
const AGENT_MODELS: [&str; 5] = [
    "claude-sonnet-4",
    "gemma-27b",
    "gemma-7b",
    "claude-haiku",
    "llama-3.2",
];
```

Use this array as the label source regardless of which backend actually handles the call.

## Step 1 — Add `label` to `LlmProvider`

In `scenarios/llm.rs`, add a default method to the trait:

```rust
pub trait LlmProvider: Send + Sync {
    async fn fill(&self, req: LlmRequest) -> anyhow::Result<serde_json::Value>;

    /// Human-readable model identifier shown in events and the TUI.
    /// Override this in concrete implementations.
    fn label(&self) -> &str {
        "unknown"
    }
}
```

Adding a default implementation is not a breaking change — all existing `impl LlmProvider` blocks
continue to compile. Override it in each struct:

```rust
impl LlmProvider for StubLlm {
    fn label(&self) -> &str { "stub" }
    // fill unchanged
}

impl LlmProvider for ClaudeApiProvider {
    fn label(&self) -> &str { &self.model }
    // fill unchanged
}

impl LlmProvider for OllamaProvider {
    fn label(&self) -> &str { &self.model }
    // fill unchanged
}

impl LlmProvider for MultiProvider {
    fn label(&self) -> &str { "multi" }
    // fill unchanged
}
```

`&self.model` is a `String` field — return `self.model.as_str()` if the borrow checker requires
it.

## Step 2 — Per-agent model labels in `yield_routing.rs`

### Current behavior

The `model_label` helper in `yield_routing.rs` (added in T2.1) probably delegates to
`llm.label()` and returns the same string for every agent. For `StubLlm` this is `"stub"`.

### Target behavior

When emitting `ScenarioStarted`, `AgentBid`, `ExecutionStarted`, or `ExecutionCompleted`, each
event should carry the model label for that specific worker, not the provider's global label.

The rule is:

```
worker_i uses AGENT_MODELS[i] as its label
```

This is always correct because:
- With `--llm-backend stub`: `StubLlm` handles all calls, but the agents notionally represent
  those models, so the label stays meaningful.
- With `--llm-backend multi`: `MultiProvider` round-robins, but agent `i` is conceptually
  associated with `AGENT_MODELS[i]` for the demo's story.
- With `--llm-backend claude`: all calls go to Claude; the label still shows the per-agent model
  from the demo script.

### Changes in `yield_routing.rs`

Replace the `model_label` helper (or update it to accept an agent index):

```rust
fn agent_model(index: usize) -> &'static str {
    AGENT_MODELS.get(index).copied().unwrap_or("unknown")
}
```

Update `ScenarioStarted` emission to use per-agent label:

```rust
events.emit(DemoEvent::ScenarioStarted {
    name: "yield-routing".to_string(),
    agents: agents
        .iter()
        .enumerate()
        .map(|(i, a)| AgentInfo {
            id: a.id.clone(),
            model: agent_model(i).to_string(),
            wallet: format!("{:#x}", a.wallet),
        })
        .collect(),
}).await;
```

Update `AgentBid` emission inside the bidding loop to use the agent's index-based label:

```rust
events.emit(DemoEvent::AgentBid {
    agent_id: agent.id.clone(),
    model: agent_model(i).to_string(),  // i is the loop index
    expected_output: proposal.expected_output,
    strategy_summary: proposal.strategy_summary.clone(),
}).await;
```

Update `ExecutionStarted` and `ExecutionCompleted` to pass `agent_model(winner_idx)`:

```rust
events.emit(DemoEvent::ExecutionStarted {
    agent_id: winner.id.clone(),
    route: ...,
}).await;
```

`ExecutionStarted` does not have a `model` field in the current `DemoEvent` definition, so no
change is needed there. If you added one locally, remove it — do not change the `DemoEvent` enum.

With `--llm-backend multi`, the `AgentBid.model` field still shows `AGENT_MODELS[i]` (the demo
story label), while the actual LLM call is handled by whichever backend `MultiProvider` picks.
This is intentional.
