## What this task does

Adds a `tournament` subcommand that runs N rounds of the yield-routing scenario with persistent
knowledge and reputation state accumulating across rounds. The goal is to show a measurable
learning curve: agents with access to more confirmed insights bid more accurately and win more
often over time.

## Tournament runner

```rust
pub async fn run_tournament(
    config: TournamentConfig,
    demo_dir: std::path::PathBuf,
    runtime_dir: std::path::PathBuf,
    rpc_url: Option<String>,
    llm_backend: String,
    events: std::sync::Arc<dyn crate::events::EventEmitter>,
) -> anyhow::Result<TournamentReport>
```

### Architecture

The tournament runner does not re-implement the yield-routing spine. Instead it calls into
`yield_routing.rs` via a public function you expose there (see step 2 below). Chain state persists
across rounds because all calls go to the same local chain.

State that accumulates across rounds:
- `WorkerRegistry` reputation scores — each win calls `updateReputation(worker, true)`; each
  round the registry state carries forward.
- `InsightBoard` pheromone weights — each posted insight accumulates confirms; weights carry
  forward.
- Agent `prior_insights` list — grows each round.

The tournament does not redeploy contracts between rounds. It does not call `reset` or wipe state.
It calls `prepare` once before round 1, then `post_job` + `execute_and_validate` + `post_insight`
in a loop.

### Required public API in `yield_routing.rs`

Before writing `tournament.rs`, add these public functions to `yield_routing.rs`:

```rust
/// One-time setup: mint tokens, register workers, promote validators, seed initial insights.
/// Safe to call multiple times — handles already-registered errors gracefully.
pub async fn prepare(
    ctx: &Arc<ChainCtx>,
    llm: &Arc<dyn LlmProvider>,
    events: &Arc<dyn EventEmitter>,
) -> anyhow::Result<()>

/// Run a single job round: post job, collect bids, assign winner, execute, validate.
/// Returns the job result with winner index and route proposals.
pub async fn run_round(
    ctx: &Arc<ChainCtx>,
    round: u32,
    prior_insights: &[String],
    llm: &Arc<dyn LlmProvider>,
    events: &Arc<dyn EventEmitter>,
) -> anyhow::Result<RoundOutcome>

pub struct RoundOutcome {
    pub job_id: u64,
    pub winner_idx: usize,
    pub winner_id: String,
    pub winning_output: f64,
    pub insights_available: usize,
}

/// Post the winning insight and have validators confirm it.
/// Returns the insight text for appending to prior_insights.
pub async fn post_winner_insight(
    ctx: &Arc<ChainCtx>,
    round: u32,
    outcome: &RoundOutcome,
    events: &Arc<dyn EventEmitter>,
) -> anyhow::Result<String>
```

These functions extract the three phases currently embedded in `YieldRouting::spine`. The `spine`
method then calls them in sequence:

```rust
async fn spine(...) -> anyhow::Result<()> {
    let llm = /* from arg */;
    prepare(&ctx, &llm, &events).await?;
    let mut prior_insights: Vec<String> = SEED_INSIGHTS.iter().map(|s| s.to_string()).collect();
    for round in 1..=2 {
        let outcome = run_round(&ctx, round, &prior_insights, &llm, &events).await?;
        let new_insight = post_winner_insight(&ctx, round, &outcome, &events).await?;
        prior_insights.push(new_insight);
    }
    Ok(())
}
```

If refactoring `spine` in this way touches the event emission order defined in T2.1, preserve it
exactly — `ScenarioStarted` still fires before the first round, `ScenarioCompleted` and
`CFactorMeasured` still fire after the last round.

### Tournament loop in `tournament.rs`

```rust
let loaded = LoadedManifest::load(&demo_dir)?;
let deploy_ctx = loaded.build_deploy_ctx(rpc_url)?;

// Deploy once.
let suite = deploy_suite(&deploy_ctx, &scenario_manifest.deploy).await?;
write_deployments(&runtime_dir, &config.scenario, &suite.addresses, ...)?;

// Build shared ChainCtx.
let ctx = Arc::new(ChainCtx { ... });

// Seed fixtures once.
run_fixtures(&ctx, &registry, &scenario_manifest.fixtures, ...).await?;

// Create LLM provider.
let llm = create_provider(&llm_backend)?;

// One-time prepare.
prepare(&ctx, &llm, &events).await?;

// Collect seed insights.
let mut prior_insights: Vec<String> = SEED_INSIGHTS.iter().map(|s| s.to_string()).collect();

// Round loop.
let mut round_results: Vec<RoundResult> = Vec::new();

for round in 1..=config.rounds {
    events.emit(DemoEvent::RoundStarted {
        round,
        total_rounds: config.rounds,
    }).await;

    let outcome = run_round(&ctx, round, &prior_insights, &llm, &events).await?;
    let new_insight = post_winner_insight(&ctx, round, &outcome, &events).await?;
    prior_insights.push(new_insight);

    round_results.push(RoundResult {
        round,
        winner_id: outcome.winner_id.clone(),
        winner_model: agent_model(outcome.winner_idx).to_string(),
        winning_output: outcome.winning_output,
        insights_available: outcome.insights_available,
    });

    events.emit(DemoEvent::RoundCompleted { round }).await;
}
```

## Expected output (with `--llm-backend stub`, 5 rounds)

The winning output should strictly increase because:
1. `StubLlm` increments its counter, so `expected_output` grows with each call.
2. Agents see more confirmed insights in later rounds and are notionally more informed.
3. `WorkerRegistry` reputation changes do not currently affect bidding in the stub path, but they
   accumulate for the rankings.

Example output:

```json
{
  "learning_curve": [
    [1, 1000],
    [2, 1040],
    [3, 1080],
    [4, 1120],
    [5, 1160]
  ],
  "final_rankings": [
    { "agent_id": "worker0", "model": "claude-sonnet-4", "wins": 5, "total_earned": 0, "final_reputation": 100 },
    ...
  ]
}
```

The exact values will differ based on `StubLlm` counter behavior. The key requirement is that
`learning_curve[N][1] >= learning_curve[N-1][1]` for at least 3 of the 4 transitions in a 5-round
run. If `StubLlm` does not satisfy this, note it in a `tracing::warn!` but do not fail.
