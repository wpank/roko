## What this task does

Adds a `KnowledgeGraphUpdate` event type that carries a full snapshot of the `InsightBoard` state
as a graph: nodes are insights, edges are confirmation relationships and agent queries. After each
round completes, `yield_routing.rs` reads the on-chain state and emits one `KnowledgeGraphUpdate`
event with the current graph.

The event is consumed by the TUI (T3.1) and by external dashboards via the NDJSON stream or
WebSocket. The format is stable — downstream consumers (Sam's dashboard) depend on the exact
field names.

```rust
/// A single insight node in the knowledge graph.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct KnowledgeNode {
    /// On-chain insight ID (u64 cast from U256).
    pub id: u64,
    /// Content hash as a hex string, or the insight text if available off-chain.
    pub content: String,
    /// Wallet address of the agent that posted this insight.
    pub poster: String,
    /// Pheromone weight as returned by InsightBoard.pheromoneWeight(id).
    /// If the contract does not expose this method, use confirmation_count * 1000 as a proxy.
    pub pheromone_weight: u64,
    /// Number of confirmations received.
    pub confirmations: u32,
}

/// A directed edge in the knowledge graph.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct KnowledgeEdge {
    /// Source node: either an agent_id ("worker0") or an insight id as a string ("insight:42").
    pub from: String,
    /// Target node: same format as `from`.
    pub to: String,
    /// Relationship type. One of: "posted", "confirmed", "queried".
    pub kind: String,
}
```

The `DemoEvent` variant to add inside the enum body:

```rust
KnowledgeGraphUpdate {
    round: u32,
    nodes: Vec<KnowledgeNode>,
    edges: Vec<KnowledgeEdge>,
},
```

## Graph construction rules

### Nodes

Read the on-chain `InsightBoard` state after each round using the pattern already established
in the scenario:

1. Call `InsightBoard.nextInsightId()` to get the count N.
2. For each id in `0..N`:
   - Read `InsightBoard.contentOf(id)` — returns a `bytes32` content hash. Format as `"0x{hex}"`.
   - Read `InsightBoard.pheromoneWeight(id)` — returns a `uint256`. If this method does not exist
     on the current `InsightBoard` ABI, approximate it as `confirmations * 1_000`. Add a
     `// TODO: read actual pheromone weight` comment.
   - Count confirmations by reading `InsightBoard.confirmationCount(id)` if available, or by
     tracking them from `InsightConfirmed` events accumulated in local state.

### Edges

Track edges in local state during the round:

- After each `InsightPosted` call: add an edge `{ from: agent_id, to: "insight:{id}", kind: "posted" }`.
- After each `InsightConfirmed` call: add an edge `{ from: confirmer_id, to: "insight:{id}", kind: "confirmed" }`.
- After each `KnowledgeQueried` event (before bidding): add an edge
  `{ from: agent_id, to: "insight:{id}", kind: "queried" }` for each insight the agent retrieved.
  If query results are not tracked individually, add one edge per agent to a synthetic node
  `"insight:all"` to at least represent the query relationship.

Maintain a `Vec<KnowledgeEdge>` that accumulates across the full run (not just the current round),
since edges from previous rounds remain meaningful for the graph.

## Emission point in `yield_routing.rs`

Emit `KnowledgeGraphUpdate` at the end of each round, after `InsightPosted` and `InsightConfirmed`
events and before `RoundCompleted`:

```rust
// After all InsightConfirmed events for this round:
let nodes = build_knowledge_nodes(&ctx).await?;
let edges = accumulated_edges.clone(); // all edges so far, including this round

events.emit(DemoEvent::KnowledgeGraphUpdate {
    round,
    nodes,
    edges,
}).await;

events.emit(DemoEvent::RoundCompleted { round }).await;
```

`build_knowledge_nodes` is a local async function that queries `InsightBoard` and returns
`Vec<KnowledgeNode>`. Extract it to keep the spine readable.

## Acceptance criteria

1. `cargo build -p roko-demo` compiles without errors.
2. `cargo clippy -p roko-demo -- -D warnings` passes.
3. `KnowledgeGraphUpdate` serializes with `"type": "knowledge_graph_update"`.
4. Exactly one `KnowledgeGraphUpdate` event is emitted per round.
5. After round 2, `nodes.len() >= 4` (3 seeded + at least 1 winner insight).
6. Every node has `confirmations >= 3` (validators confirm each insight).
7. Edges include at least one `"posted"` and one `"confirmed"` entry.
8. The JSON is parseable by a frontend framework:
   ```bash
   cargo run -p roko-demo -- --demo-dir demo up yield-routing \
     --events ndjson 2>/dev/null \
     | grep '"type":"knowledge_graph_update"' \
     | python3 -c "import sys, json; [json.loads(l) for l in sys.stdin]; print('parseable')"
   ```
9. No `unwrap()` in any new or modified code.
