## Context

Each `roko-demo up yield-routing` run starts fresh: workers register with zero reputation, get promoted through the `prepare` phase, then race for jobs. When the process exits, all state is gone. Validators and agents can never build track records between sessions.

This task adds optional persistence: after a run, serialize each worker's reputation to `demo/.runtime/reputation.json`. On the next run, restore those scores before the scenario starts competing. Workers who performed well in earlier sessions carry that advantage into the next one. A `--persist-reputation` flag controls the behaviour; without it, runs stay independent.

## Contract bindings reference

All bindings are in `crates/roko-demo/src/bindings.rs`.

```rust
// WorkerRegistry ‚Äî key methods for this task
fn reputationOf(address worker) -> uint256;
fn tier(address worker) -> uint8;
fn updateReputation(address worker, bool outcome);
fn setAuthorized(address caller, bool allowed);
```

Reputation scale: `SCALE = 1_000_000`. Tier thresholds:

| Value | Tier |
|---|---|
| 0 | 0 ‚Äî Unregistered |
| < 350_000 | 1 ‚Äî Probation |
| < 550_000 | 2 ‚Äî Standard |
| < 800_000 | 3 ‚Äî Trusted |
| >= 800_000 | 4 ‚Äî Elite |

Each `updateReputation(worker, true)` call increments reputation by a fixed step defined in the contract. Read `WorkerRegistry.sol` to confirm the step size before computing how many calls are needed to restore a target score.

## What to implement

### 1. Persistence types

Add a private module at the top of `yield_routing.rs`:

```rust
mod persistence {
    use serde::{Deserialize, Serialize};

    #[derive(Debug, Clone, Serialize, Deserialize)]
    pub struct WorkerSnapshot {
        pub address: String,   // checksum hex: "0x..."
        pub name: String,      // "worker0" ‚Ä¶ "worker4", "validator0" ‚Ä¶ "validator2"
        pub reputation: u64,
        pub tier: u8,
    }

    #[derive(Debug, Clone, Serialize, Deserialize)]
    pub struct ReputationFile {
        pub saved_at: String,  // RFC 3339, e.g. "2026-04-10T12:00:00Z"
        pub workers: Vec<WorkerSnapshot>,
    }
}
```

Place these types in `yield_routing.rs`, not in a separate file. The module keeps them out of the
public namespace.

### 2. Save reputation after the scenario completes

Add a function at the module level:

```rust
async fn save_reputation(
    ctx: &ChainCtx,
    runtime_dir: &std::path::Path,
) -> anyhow::Result<()>
```

Steps:

1. For each of `worker0`‚Äì`worker4` and `validator0`‚Äì`validator2` (eight names total):
   - Resolve the address with `ctx.wallet_address(name)?`.
   - Call `WorkerRegistry.reputationOf(address)` using `ctx.read_provider()?`.
   - Call `WorkerRegistry.tier(address)` using `ctx.read_provider()?`.
   - Build a `WorkerSnapshot`.
2. Build a `ReputationFile` with `saved_at` set to the current UTC time in RFC 3339 format.
   Use `chrono::Utc::now().to_rfc3339()` if `chrono` is available; otherwise construct the
   string with `std::time::SystemTime` and format it manually as `YYYY-MM-DDTHH:MM:SSZ`.
   Check `Cargo.toml` before adding `chrono` ‚Äî it may already be a dependency.
3. Write the file to `runtime_dir.join("reputation.json")`. Create `runtime_dir` with
   `std::fs::create_dir_all` if it does not exist.
4. Log: `tracing::info!(path = %runtime_dir.join("reputation.json").display(), "reputation saved")`.

Do not fail the scenario if saving fails. Wrap the call site with:

```rust
if let Err(e) = save_reputation(&ctx, &runtime_dir).await {
    tracing::warn!("failed to save reputation: {e}");
}
```

### 3. Restore reputation during Phase 1 (prepare)

Add a function at the module level:

```rust
async fn restore_reputation(
    ctx: &ChainCtx,
    runtime_dir: &std::path::Path,
) -> anyhow::Result<()>
```

Steps:

1. Check if `runtime_dir.join("reputation.json")` exists. If not, return `Ok(())` immediately ‚Äî
   this is a first run and there is nothing to restore.
2. Read and deserialize the file into a `ReputationFile`.
3. For each `WorkerSnapshot`:
   - Resolve the current reputation via `WorkerRegistry.reputationOf`.
   - Compute how many positive `updateReputation` calls are needed to reach `snapshot.reputation`.
     The step size comes from the contract. If the current reputation already equals or exceeds the
     saved value, skip this worker and log a warning.
   - Call `WorkerRegistry.updateReputation(address, true)` the computed number of times from the
     deployer wallet. Batch the calls in a loop ‚Äî the same pattern as `consortium.rs` lines 140‚Äì160.
4. Log: `tracing::info!(path = %runtime_dir.join("reputation.json").display(), "reputation restored from previous run")`.

If the file exists but deserialization fails, log a warning and return `Ok(())` rather than
propagating the error. The scenario should proceed with default reputation, not abort.

### 4. Thread `runtime_dir` and `persist_reputation` into the spine

The `spine` method signature does not currently receive the runtime directory or the flag value.
Pass them through `ScenarioManifest` or via a new field on `YieldRouting`. The cleanest option:

```rust
pub struct YieldRouting {
    pub runtime_dir: std::path::PathBuf,
    pub persist_reputation: bool,
}
```

Construct it in `agents_cmd` (or wherever scenarios are instantiated) using the values from `Cli`.

Inside `spine`:
- If `self.persist_reputation`, call `restore_reputation` at the end of Phase 1 (after workers
  are registered and validators are promoted, but before Phase 2 begins).
- After the scenario completes all rounds, if `self.persist_reputation`, call `save_reputation`.

### 5. CLI flag

In `crates/roko-demo/src/main.rs`, add to the `Cli` struct:

```rust
/// Persist worker reputation scores across demo runs (default: false).
/// When set, reputation is loaded from demo/.runtime/reputation.json at startup
/// and saved back after each run.
#[arg(long)]
persist_reputation: bool,
```

Pass the value down to `YieldRouting` when constructing the scenario object. Other scenarios
receive no change.
