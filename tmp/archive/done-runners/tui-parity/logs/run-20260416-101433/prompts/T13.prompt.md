## Task

Three modal problems to fix in one batch:

1. **Modals open empty** — WaveOverview, QueueOverview, TaskPicker pass `Vec::new()` instead of real data from `TuiState`.
2. **PlanDetail modal has no render path** — `show_plan_detail` flag exists, but no `ModalState::PlanDetail` variant and no rendering when Enter pressed on a plan.
3. **Key intercepts leak** — modal-open keys (help, plan_detail, wave_overview, agent_pool) don't intercept Tab, arrows, j/k, etc. — they leak to the background tab.

## Implementation

### 1. Pass real data into modal constructors

Search for modal construction sites. For each:

- `WaveOverview::new(waves: Vec::new())` → `WaveOverview::new(tui_state.execution_waves.clone())`
- `QueueOverview::new(milestones: Vec::new())` → use `tui_state.milestones` (or compute from plans)
- `TaskPicker::new(tasks: Vec::new())` → `TaskPicker::new(tui_state.available_tasks.clone())`

### 2. Add `ModalState::PlanDetail` variant

In `state.rs`:

```rust
pub enum ModalState {
    // ...existing
    PlanDetail { plan_id: String },
}
```

Create `modals/plan_detail.rs` if missing with a `render()` function that takes `&PlanEntry` and draws title, status, tasks, timing, failure summary.

### 3. Wire Enter-on-plan → open PlanDetail

In `input.rs` key handler for Enter when focus is on plan tree:

```rust
if let Some(plan) = selected_plan_id {
    state.active_modal = Some(ModalState::PlanDetail { plan_id: plan });
}
```

### 4. Fix modal key intercepts

In `input.rs:292-302` (the intercept section), extend the match to include:

- `show_wave_overview`
- `show_help`
- `show_plan_detail` (or `ModalState::PlanDetail` once unified in T14)
- `show_agent_pool_modal`

Intercept pattern: when any modal is open, only Escape/q/Ctrl-C + modal-internal keys (j/k scroll, arrows) reach the handler — everything else is swallowed.
