## Task

Today, when the user enters Inject or Filter mode, keystrokes are accepted but NOTHING is rendered on screen — no cursor, no input box, no mode label. Users type blind.

Additionally, the Filter mode has a double bug: `filter_text` is accumulated but never copied to `filter`, so the filter never actually applies.

Fix both.

## Implementation

### 1. Bottom input line renderer

Add a one-line input bar rendered at the bottom of the screen when `input_mode != InputMode::Normal`. Format:

```
[INJECT] > <buffer text>│   or   [FILTER] > <buffer text>│
```

Where `│` is the cursor. Use `ratatui::widgets::Paragraph` with a styled `Span`.

Trigger: inside the main frame render function in `app.rs`, after the primary layout, reserve the bottom 1 row when in a non-Normal input mode.

### 2. Fix filter-text → filter copy

In the filter key handler in `input.rs`, when a character is pressed in Filter mode:

```rust
state.filter_text.push(ch);
state.filter = state.filter_text.clone();   // NEW — keep them in sync
```

On Backspace:

```rust
state.filter_text.pop();
state.filter = state.filter_text.clone();
```

On Enter: also copy, then exit filter mode.

### 3. Mode label

Expose a helper on `TuiState` or `App`:

```rust
pub fn input_mode_label(&self) -> &'static str {
    match self.input_mode {
        InputMode::Normal => "",
        InputMode::Inject => "INJECT",
        InputMode::Filter => "FILTER",
    }
}
```

## Rules

1. When `input_mode == Normal`, the input bar must not render — don't steal a row unnecessarily.
2. Cursor must blink or at least be visible as a styled character.
3. Backspace, arrow keys, and Enter must all be handled sensibly.
4. Escape exits the mode AND clears the buffer.

## Done when

- `cargo check -p roko-cli` succeeds
- Pressing `i` (inject) or `/` (filter) shows a visible input bar with mode label + cursor
- Typing updates the buffer and cursor position
- In Filter mode, typing a character immediately filters the current view
- Escape returns to Normal and hides the bar
