## Task

Add a new `roko plan validate <plans-dir>` subcommand that lints every
`tasks.toml` under the given directory without executing the plan. Today the
only way to discover plan errors is to run the plan and watch the executor
explode — UX04 lifts that validation into a pre-flight CLI check.

## Current state

- `crates/roko-cli/src/main.rs` has `plan list`, `plan show`, `plan create`,
  `plan run`.
- Validation logic (task schema, gate rung refs, DAG cycle detection, missing
  role templates) is scattered across the executor. Most is reachable through
  the parser + `PlanRunner::from_plans_dir`.
- No standalone validation entry-point.

## Implementation

### 1. Clap / CLI surface

In `crates/roko-cli/src/main.rs`:

```rust
#[derive(Subcommand)]
enum PlanCmd {
    // existing: List, Show, Create, Run
    Validate {
        /// Plans root directory (default: plans/)
        #[arg(default_value = "plans/")]
        dir: PathBuf,

        /// Fail on warnings, not only errors
        #[arg(long)]
        strict: bool,

        /// Output machine-readable JSON instead of text
        #[arg(long)]
        json: bool,
    },
}
```

Dispatch to `fn cmd_plan_validate(dir: &Path, strict: bool, json: bool) -> Result<()>`.

### 2. Validation rule set

Each rule emits `Diagnostic { severity, rule_id, plan_id, task_id, message }`.
Implement in `crates/roko-cli/src/plan_validate.rs` (new):

| Rule id | Severity | Check |
|---------|----------|-------|
| `PLAN_001` | error | `tasks.toml` parses as valid TOML |
| `PLAN_002` | error | `tasks` array is non-empty |
| `PLAN_003` | error | Each task has `id` + `title` + `role` |
| `PLAN_004` | error | Task IDs unique per plan |
| `PLAN_005` | error | `depends_on` entries reference known task IDs |
| `PLAN_006` | error | DAG has no cycles (Tarjan or Kahn) |
| `PLAN_007` | error | `gate_rung` (if set) ∈ `{0..=6}` |
| `PLAN_008` | warning | `role` resolves to a template in `crates/roko-compose/src/templates/` |
| `PLAN_009` | warning | `model` (if set) is a known alias in roko.toml |
| `PLAN_010` | warning | Unreferenced task (reachable from no root) |
| `PLAN_011` | warning | `verify` steps present if `gate_rung` is 0 (compile-only) |

### 3. Output format

Text (human):
```
plans/test/tasks.toml
  error  PLAN_006  task 'foo' forms a cycle with 'bar'
  warn   PLAN_008  task 'implement-api' uses role 'implementer-v2' which has no template

2 diagnostics in 1 plan
```

JSON (`--json`):
```json
{
  "plans": [
    {
      "plan_id": "test",
      "path": "plans/test/tasks.toml",
      "diagnostics": [
        { "severity": "error", "rule_id": "PLAN_006", "task_id": "foo",
          "message": "task 'foo' forms a cycle with 'bar'" }
      ]
    }
  ],
  "totals": { "errors": 1, "warnings": 1, "plans_checked": 1 }
}
```

### 4. Exit codes

- 0 — no errors (warnings allowed unless `--strict`)
- 1 — errors present (or `--strict` with warnings)
- 2 — I/O or malformed CLI args

### 5. Reuse executor's existing checks

Where the orchestrator already performs a check (e.g. cycle detection in
`PlanRunner::from_plans_dir`), lift the check into a standalone
`validate_plan(tasks_file: &TasksFile) -> Vec<Diagnostic>` helper and have
both call-sites use it. The executor keeps its runtime error path; the
validator gets a pre-flight reporter.
