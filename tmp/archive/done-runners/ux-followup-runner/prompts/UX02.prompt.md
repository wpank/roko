## Task

The CLI half of auto-plan-on-PRD-publish is already wired at
`crates/roko-cli/src/prd.rs:628` (`maybe_generate_plan_after_promote`). What
is missing: an event emission + an orchestrator-side subscriber so that when
**any** process promotes a PRD (CLI, HTTP, another agent), the running
orchestrator picks it up and auto-plans + auto-runs it.

## Current state

- `prd draft promote` inline-runs `generate_plan_from_prd` when `[prd]
  auto_plan = true` is in `roko.toml`. One process only.
- External promote events (another CLI invocation, an HTTP POST) do not
  notify a long-running `roko serve`.
- No `RokoEvent::PrdPublished` variant exists.

## Implementation

### 1. Add `RokoEvent::PrdPublished` (depends on UX01's event_bus extension)

```rust
// crates/roko-runtime/src/event_bus.rs
pub enum RokoEvent {
    PlanRevision { /* from UX01 */ },
    PrdPublished {
        slug: String,
        path: PathBuf,
        published_at: chrono::DateTime<chrono::Utc>,
        origin: PublishOrigin,
    },
}

pub enum PublishOrigin {
    Cli,
    Http,
    Import,
}
```

If UX01 has not yet landed an event-bus extension pattern, follow the
existing `RokoEvent` cases in `event_bus.rs`.

### 2. Emit from `prd draft promote`

In `crates/roko-cli/src/prd.rs`, right after the successful promote (line
~627 `println!("✅ Promoted: {}", dst.display());`):

```rust
if let Some(bus) = event_bus() {
    let _ = bus.publish(RokoEvent::PrdPublished {
        slug: slug.to_string(),
        path: dst.clone(),
        published_at: chrono::Utc::now(),
        origin: PublishOrigin::Cli,
    });
}
let _ = maybe_generate_plan_after_promote(workdir, slug, &dst, auto_execute).await?;
```

Keep the existing `maybe_generate_plan_after_promote` call. It handles the
in-process fast path; the event is the fan-out for external subscribers.

### 3. Subscriber inside `roko serve`

In `crates/roko-serve/src/lib.rs` (or equivalent), on server boot:

```rust
let mut rx = event_bus.subscribe();
tokio::spawn(async move {
    while let Ok(evt) = rx.recv().await {
        if let RokoEvent::PrdPublished { slug, path, origin, .. } = evt {
            if !config.auto_orchestrate { continue; }
            tracing::info!(%slug, %origin, "auto-orchestrating published PRD");
            if let Err(err) = run_plan_for_slug(&slug, &path).await {
                tracing::error!(%slug, %err, "auto-orchestrate failed");
            }
        }
    }
});
```

### 4. Dedupe

Two promotes of the same slug within 60 s → run once:

```rust
static RECENT_PUBLISHES: Lazy<Mutex<HashMap<String, Instant>>> = ..;

fn should_run(slug: &str) -> bool {
    let mut map = RECENT_PUBLISHES.lock().unwrap();
    let now = Instant::now();
    map.retain(|_, t| now.duration_since(*t) < Duration::from_secs(60));
    !map.insert(slug.to_string(), now).is_some()
}
```

## Rules

1. **In-process path unchanged.** `maybe_generate_plan_after_promote` keeps
   working; the event is an additive fan-out.
2. **Idempotent subscriber.** Two publishes of the same slug within 60 s →
   one run. Plan generator is itself idempotent (`generate_plan_from_prd`
   overwrites on re-run).
3. **Non-blocking.** Subscriber must not block the event-bus dispatch thread;
   use `tokio::spawn` for the actual plan-run.
4. **Observable.** Each PrdPublished event is recorded to
   `.roko/episodes.jsonl` as `kind = "prd_published"` for audit.
5. **No HTTP coupling.** `roko-runtime::event_bus` must not depend on
   `roko-serve`. The subscriber lives in `roko-serve`.
