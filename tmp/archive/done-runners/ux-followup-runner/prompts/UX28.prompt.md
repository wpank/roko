## Task

`roko-compose::enrichment::*` already implements a full estimate → select →
pipeline flow with unit tests. It is exported from `lib.rs` but has zero
external callers.

The correct runtime seam is the CLI orchestrator's existing `Enriching`
phase in `crates/roko-cli/src/orchestrate.rs`, not
`crates/roko-agent/src/dispatcher/mod.rs`. Wire the pipeline there. Do not
turn this batch into a seam-selection exercise and do not add a new
`roko-agent -> roko-compose` dependency edge.

## Implementation

### 1. Use the existing Enriching phase

Anchor on these current seams:

- `dispatch_action(..., AgentRole::Strategist, "enrich")`
- `handle_enriching(plan_id)`
- `build_enrichment_system_prompt(plan_id)`
- `PhaseKind::Enriching` / `PlanPhase::Enriching`

`handle_enriching` is the batch's call-site. Keep phase transitions,
episode logging, and failure behavior owned by the orchestrator.

### 2. Build a real pipeline call from the plan/workdir context

Inside the Enriching phase:

- build `EnrichmentConfig` from the current workdir / repo root and plan id
- derive the `LlmBackend` from the current runtime model/provider choice as
  directly as the current code allows
- use `estimate_enrichment(...)` and `StepSelector` to decide what should run
  for the current plan complexity
- execute the selected steps via `EnrichmentPipeline`

Do not add a new `[agent.enrichment]` or dispatcher-local config surface.
If an existing config/helper in `roko-core` already describes the needed
pipeline tiering, reuse it.

### 3. Make the Enriching phase materially better, not parallel dead code

After this batch, the existing strategist round-trip in `handle_enriching`
must no longer be the only enrichment path.

Choose one of these, but be explicit in code and final notes:

- run the enrichment pipeline first, then keep the strategist dispatch and
  make it consume/reference the generated artifacts
- or replace the strategist round-trip entirely if the pipeline output now
  covers the phase's required behavior

Either way, `roko-compose::enrichment::*` must become a real runtime caller
from `roko-cli`.

### 4. Focused tests / observability

Add the narrowest useful coverage you can at the orchestrator seam:

- a helper test for backend/step selection, or
- a small Enriching-phase test that proves the pipeline is now invoked

At minimum, log which enrichment steps were selected and whether they ran or
were skipped, so the phase is inspectable in logs.

### 5. Reality check

If any of the files or symbols above moved in the current tree, verify with
`rg --files` / `rg -n` before editing. Do not move the work into
`roko-agent::dispatcher`, and do not add a reverse dependency from
`roko-agent` back into `roko-compose`.

4. **Decision captured.** Final message must state whether the strategist
   round-trip remained, changed, or was replaced.
