## Task

Top-10 hottest files have 70 `.unwrap()` calls combined. Replace with
`Result` propagation (for I/O / parse) or `.expect("invariant: …")` (for
genuine invariants). Each removed unwrap improves crash-resistance.

## Implementation

### 1. Prioritize the two 20+ files

- `system_prompt_builder.rs` (20)
- `routes/middleware.rs` (22)

These are hot paths on any request / dispatch; crashes here take down
entire workflows.

### 2. Conversion patterns

- `.lock().unwrap()` on a `Mutex<T>`: replace with `.lock().expect("lock
  poisoned")` if the mutex is process-local invariant-guarded, or propagate
  via `try_lock().map_err(|_| anyhow!("lock contention"))?`
- `.parse::<T>().unwrap()` on user input: propagate via `?` with a context
  message
- `.get(key).unwrap()` on a map: `.get(key).ok_or_else(|| anyhow!("missing {key}"))?`
- `serde_json::from_str(…).unwrap()` on runtime JSON: propagate `?`
- Testing unwraps (inside `#[cfg(test)]`): leave alone

### 4. Error-type hygiene

Prefer `anyhow::Result<T>` at public boundaries (HTTP routes, CLI commands)
and `thiserror`-based typed errors at crate-internal boundaries. Do not
introduce new `thiserror` enums just to placate this batch; reuse existing
errors where possible.

### 5. Tests

No new tests are strictly required, but existing tests must pass. Any
test that formerly relied on panic-on-unwrap may need `.unwrap()` kept at
test boundaries — OK.

## Rules

1. **No blanket `.expect()`.** Each `.expect("…")` carries a specific
   message that names the invariant.
2. **No logic drift.** Replaced unwrap → error propagation must not change
   successful-path output.
3. **No new panics.** `.unwrap()` count after the batch = 0 for these files.
4. **Tests stay green.** All existing tests compile + pass.
