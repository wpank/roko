# A2A Protocol Integration for Roko — Context, Implementation Plan & Architectural Details

> **Status**: Research & Specification
> **Date**: April 25, 2026
> **Scope**: Full A2A (Agent-to-Agent) protocol support in the Roko runtime, Nunchi dashboard, and Korai chain — covering both server (Roko agents as A2A endpoints) and client (Roko consuming external A2A agents) roles.

---

## 1. What A2A Is and Why It Matters for Roko

### 1.1 Protocol Landscape in April 2026

Three protocols now define the agentic communication stack:

- **MCP (Model Context Protocol)** — Anthropic, Nov 2024. Agent-to-tool. JSON-RPC client-server for context ingestion and tool invocation. 97M+ monthly SDK downloads. Natively supported in Claude, ChatGPT, Gemini, Cursor, VS Code, JetBrains. Roko already has MCP integration (`roko_agent::mcp::McpConfig`).

- **A2A (Agent-to-Agent Protocol)** — Google, Apr 2025, donated to Linux Foundation Jun 2025. Agent-to-agent. Enables opaque agents to discover capabilities, delegate tasks, stream progress, and exchange artifacts over HTTP/JSON-RPC 2.0/SSE. IBM's ACP merged into A2A in Sep 2025 — ACP is now A2A. 50+ enterprise partners (Atlassian, Salesforce, SAP, ServiceNow, PayPal, MongoDB). Official SDKs in Python, Go, Java, JS, C#/.NET, and Rust (`a2a-rs` by the Linux Foundation). Current spec version: v1.0.

- **ANP (Agent Network Protocol)** — P2P discovery and mesh networking for agents. Still emerging, less relevant to Roko's current architecture.

The mental model: **MCP is vertical (agent ↔ tools), A2A is horizontal (agent ↔ agent)**. Roko already has vertical (MCP). A2A adds the horizontal layer.

### 1.2 What A2A Provides

The protocol defines five core primitives:

**Agent Card** — A JSON metadata document served at `/.well-known/agent.json` describing identity, capabilities, skills, authentication requirements, and endpoint URL. Think of it as an agent's business card + résumé. This maps naturally to Roko's existing agent Config + on-chain Passport.

**Task** — The fundamental unit of work. Tasks have a lifecycle: `submitted → working → input-required → completed | failed | canceled`. Tasks carry a unique ID, belong to a context (conversation), and accumulate status updates and artifacts. This maps to Roko's `PlanState` / episode model.

**Message** — A communication turn between client and server agent, with a role (`user` or `agent`) and one or more Parts. Messages carry context across turns within a task.

**Part** — The smallest unit of content. Three types: `TextPart` (text), `FilePart` (files/images), `DataPart` (structured JSON). Multi-modal by design.

**Artifact** — An output generated by the agent as a result of a task (documents, images, code, structured data). Composed of Parts.

Transport: HTTP(S) with JSON-RPC 2.0, SSE for streaming, optional gRPC. Push notifications via webhooks for long-running tasks.

### 1.3 Why A2A Is the Missing Piece for Roko

Roko's architecture already has most of the internal primitives that map cleanly to A2A:

| Roko Concept | A2A Concept | Current Gap |
|---|---|---|
| Agent Config + Passport | Agent Card | No A2A card generation or serving |
| Plan / Task / Episode | Task lifecycle | No external task delegation protocol |
| Bus Pulses + StateHub projections | SSE streaming / status updates | No external streaming protocol |
| Knowledge Entries (Engrams) | Artifacts | No standardized artifact exchange format |
| Agent roles (Strategist, Implementer, Reviewer) | Skills | No external skill advertisement |
| MCP integration | Tool layer | Already exists — A2A sits above this |
| Conductor watchers + Gate pipeline | Task monitoring | Internal only — not exposed externally |
| Tier system (Probation → Sovereign) | Trust / authentication | No external trust federation |
| Agent Detail Config tab | Endpoint configuration | **Already has A2A/MCP/REST/WebSocket fields — all "not configured"** |

The last row is critical: **the Nunchi UI already has A2A endpoint configuration fields on every agent's Config tab** (observed in the current state audit). The backend just isn't wired.

### 1.4 What A2A Enables

**Immediate capabilities:**

1. **External agent consumption** — Roko orchestrator can delegate subtasks to any A2A-compliant agent (from any framework — LangGraph, CrewAI, Google ADK, Semantic Kernel, BeeAI). The PlanRunner treats external A2A agents as execution targets alongside local `ClaudeCliAgent` and `ExecAgent`.

2. **Roko agents as services** — Any Roko agent can be exposed as an A2A endpoint that external systems can discover and invoke. Roko becomes a participant in multi-vendor agent ecosystems.

3. **Cross-instance coordination** — Multiple Roko deployments can coordinate via A2A. A `laptop` Roko instance delegates expensive GPU tasks to a `clustered` Roko instance. Different teams run their own Roko instances and agents communicate across boundaries.

4. **IDE integration** — VS Code and JetBrains already support MCP natively. As A2A adoption grows, IDEs will support agent discovery and delegation. A Roko coding agent exposed via A2A becomes usable from any IDE that implements A2A client support, without requiring the Roko CLI or Nunchi dashboard.

5. **Agent marketplace** — Nunchi's planned community/marketplace feature gains a standard protocol. Shared agent templates can include A2A endpoint URLs. The Arena competitive evaluation system can benchmark external A2A agents alongside local ones.

**Korai chain integration:**

6. **On-chain agent discovery** — Agent Cards stored on Korai become a decentralized registry. The GolemRegistry precompile (doc 05) already stores agent identity — extending it to serve A2A cards means any agent on the chain is discoverable via A2A.

7. **Cross-chain agent coordination** — Golems on different Korai instances communicate via A2A, with on-chain attestation of task outcomes. The InsightLedger precompile records task results; A2A provides the communication protocol.

8. **Paid agent services** — The GNOS token economy (doc 05) enables paid A2A services. An external client discovers a Roko agent's A2A card, negotiates a task, pays in GNOS, and receives results. The OaaSRegistry precompile (Orchestration-as-a-Service) was designed exactly for this.

### 2.2 Dependency on Existing Rust A2A Libraries

Multiple Rust A2A implementations exist. Recommendation: use `a2a-types` from crates.io for protocol types (pure types crate, zero I/O, serde-based), and build the server/client on top of Roko's existing `axum` + `tokio` + `hyper` stack rather than pulling in a full SDK. This avoids dependency conflicts with Roko's existing HTTP infrastructure and keeps the implementation thin.

```toml
[dependencies]
a2a-types = "0.1"           # Protocol type definitions only
serde = { version = "1", features = ["derive"] }
serde_json = "1"
axum = "0.7"                # Already in roko-serve
tokio = { version = "1", features = ["full"] }  # Already in workspace
tokio-stream = "0.1"        # For SSE
uuid = { version = "1", features = ["v4"] }
```

If `a2a-types` doesn't cover v1.0 yet, generate types from the official A2A JSON Schema protobuf definitions.

### 2.3 Layer Placement

A2A support spans two layers and one cross-cut:

- **L3 Harness** — `roko-a2a` server and client live here alongside `roko-agent`. The A2A server wraps agent execution behind a protocol boundary. The A2A client (`A2AAgent`) is a new agent backend alongside `ClaudeCliAgent` and `ExecAgent`.

- **L4 Orchestration** — The `ParallelExecutor` and `PlanRunner` gain a new `ExecutorAction::DelegateA2A` variant for dispatching tasks to remote A2A agents. The `UnifiedTaskDag` can include A2A delegation nodes.

- **Cross-cut: Safety** — A2A introduces an external trust boundary. Every incoming A2A request passes through `SafetyLayer::check_pre_execution()`. Every Agent Card from an external agent is treated as untrusted input (per the A2A security guidance about prompt injection via crafted card fields).

## 3. Implementation Details

### 3.1 Agent Card Generation (`card.rs`)

Every Roko agent gets an automatically generated A2A Agent Card. The card is derived from the agent's existing `RokoConfig`, on-chain Passport (if available), and runtime capabilities.

```rust
use a2a_types::{AgentCard, AgentSkill, AgentCapabilities, SecurityScheme};

pub struct CardGenerator {
    base_url: String,
    version: String,
}

impl CardGenerator {
    /// Generate an A2A AgentCard from Roko agent configuration
    pub fn generate(
        &self,
        agent_id: &str,
        config: &AgentConfig,
        passport: Option<&OnChainPassport>,
        capabilities: &RuntimeCapabilities,
    ) -> AgentCard {
        AgentCard {
            name: config.name.clone(),
            description: config.system_prompt_summary(),
            url: format!("{}/a2a/agents/{}/jsonrpc", self.base_url, agent_id),
            provider: Some(AgentProvider {
                organization: "Roko".to_string(),
                url: self.base_url.clone(),
            }),
            version: self.version.clone(),
            documentation_url: Some(format!("{}/docs/agents/{}", self.base_url, agent_id)),
            capabilities: AgentCapabilities {
                streaming: Some(true),       // Always — Bus + SSE
                push_notifications: Some(true), // Via WebSocket
                state_transition_history: Some(true),
            },
            security_schemes: self.derive_security_schemes(passport),
            default_input_modes: vec!["text/plain".into(), "application/json".into()],
            default_output_modes: self.derive_output_modes(config),
            skills: self.derive_skills(config, capabilities),
            // Chain-specific extensions
            extensions: self.chain_extensions(passport),
        }
    }

    fn derive_skills(
        &self,
        config: &AgentConfig,
        caps: &RuntimeCapabilities,
    ) -> Vec<AgentSkill> {
        let mut skills = vec![];

        // Map Roko domain to A2A skills
        match config.domain {
            Domain::Coding => {
                skills.push(AgentSkill {
                    id: "code-generation".into(),
                    name: "Code Generation".into(),
                    description: "Generate, modify, and review code".into(),
                    tags: vec!["coding".into(), config.language_hint().into()],
                    input_modes: None,
                    output_modes: Some(vec!["text/plain".into(), "application/json".into()]),
                    examples: None,
                });
                skills.push(AgentSkill {
                    id: "code-review".into(),
                    name: "Code Review".into(),
                    description: "Review code for bugs, security, and quality".into(),
                    tags: vec!["review".into(), "security".into()],
                    ..Default::default()
                });
            }
            Domain::Research => {
                skills.push(AgentSkill {
                    id: "research".into(),
                    name: "Deep Research".into(),
                    description: "Multi-source research with synthesis".into(),
                    tags: vec!["research".into(), "analysis".into()],
                    ..Default::default()
                });
            }
            Domain::Blockchain => {
                skills.push(AgentSkill {
                    id: "smart-contract-audit".into(),
                    name: "Smart Contract Audit".into(),
                    description: "Audit Solidity/EVM contracts for vulnerabilities".into(),
                    tags: vec!["blockchain".into(), "security".into(), "solidity".into()],
                    ..Default::default()
                });
            }
            // ... other domains
            _ => {}
        }

        // Add gate pipeline as a verifiable skill
        if caps.has_gate_pipeline {
            skills.push(AgentSkill {
                id: "verified-execution".into(),
                name: "Gate-Verified Execution".into(),
                description: "Execute with compile/test/clippy gate verification".into(),
                tags: vec!["verified".into(), "deterministic".into()],
                ..Default::default()
            });
        }

        skills
    }
}
```

### 3.2 A2A Server Handler (`server.rs`)

The server implements the A2A JSON-RPC 2.0 methods:

| A2A Method | Roko Implementation |
|---|---|
| `message/send` | Create task → dispatch to PlanRunner → return result or task reference |
| `message/stream` | Same as send, but return SSE stream from Bus events |
| `tasks/get` | Look up PlanState by task ID |
| `tasks/cancel` | Cancel via PlanRunner with CancelToken |
| `tasks/resubscribe` | Reconnect SSE stream to existing task |

```rust
use axum::{routing::post, Router};

pub fn a2a_router() -> Router<AppState> {
    Router::new()
        .route("/a2a/agents/:agent_id/jsonrpc", post(handle_jsonrpc))
        .route("/a2a/agents/:agent_id/sse/:task_id", 
               axum::routing::get(handle_sse_stream))
}

async fn handle_jsonrpc(
    State(state): State<AppState>,
    Path(agent_id): Path<String>,
    Json(request): Json<JsonRpcRequest>,
) -> Json<JsonRpcResponse> {
    // Route safety check — untrusted external input
    let safety_result = state.safety_layer
        .check_a2a_request(&agent_id, &request).await;
    if let Err(denial) = safety_result {
        return Json(JsonRpcResponse::error(denial.into()));
    }

    match request.method.as_str() {
        "message/send" => handle_send_message(state, agent_id, request).await,
        "message/stream" => handle_stream_message(state, agent_id, request).await,
        "tasks/get" => handle_get_task(state, request).await,
        "tasks/cancel" => handle_cancel_task(state, request).await,
        _ => Json(JsonRpcResponse::method_not_found()),
    }
}

async fn handle_send_message(
    state: AppState,
    agent_id: String,
    request: JsonRpcRequest,
) -> Json<JsonRpcResponse> {
    let params: SendMessageParams = serde_json::from_value(request.params)
        .map_err(|e| JsonRpcResponse::invalid_params(e))?;

    // Bridge: A2A Message → Roko task
    let task_spec = TaskBridge::from_a2a_message(&params.message, &agent_id);

    // Dispatch through PlanRunner
    let result = state.plan_runner
        .execute_single_task(task_spec)
        .await;

    // Bridge: Roko result → A2A Task/Message response
    match result {
        Ok(outcome) => {
            let a2a_response = TaskBridge::to_a2a_response(outcome);
            Json(JsonRpcResponse::success(a2a_response))
        }
        Err(e) => Json(JsonRpcResponse::internal_error(e)),
    }
}
```

### 3.3 A2A Client Agent (`a2a_agent.rs`)

The A2A client is a new agent backend that the PlanRunner can dispatch to, sitting alongside `ClaudeCliAgent` and `ExecAgent`:

```rust
use roko_a2a::client::A2AClient;
use roko_a2a::discovery::AgentDiscovery;

/// An agent backend that delegates work to a remote A2A server
pub struct A2AAgent {
    client: A2AClient,
    remote_card: AgentCard,
    /// Map A2A task states to Roko episode events
    task_bridge: TaskBridge,
}

impl A2AAgent {
    pub async fn from_url(url: &str) -> Result<Self> {
        let client = A2AClient::new(url)?;
        let remote_card = client.discover_agent_card().await?;
        Ok(Self {
            client,
            remote_card,
            task_bridge: TaskBridge::new(),
        })
    }

    pub async fn from_registry(
        registry: &AgentDiscovery,
        skill_query: &str,
    ) -> Result<Vec<Self>> {
        // Discover agents with matching skills
        let cards = registry.find_by_skill(skill_query).await?;
        let mut agents = vec![];
        for card in cards {
            let client = A2AClient::new(&card.url)?;
            agents.push(Self {
                client,
                remote_card: card,
                task_bridge: TaskBridge::new(),
            });
        }
        Ok(agents)
    }
}

#[async_trait]
impl AgentBackend for A2AAgent {
    async fn execute(
        &self,
        task: &TaskSpec,
        cancel: CancelToken,
        tx: mpsc::Sender<AgentEvent>,
    ) -> Result<AgentOutcome> {
        // Convert Roko task to A2A message
        let message = self.task_bridge.to_a2a_message(task);

        // Stream execution from remote agent
        let mut stream = self.client.stream_message(message).await?;

        while let Some(event) = stream.next().await {
            if cancel.is_cancelled() {
                self.client.cancel_task(&task.id).await?;
                return Ok(AgentOutcome::Cancelled);
            }

            match event? {
                A2AEvent::StatusUpdate(status) => {
                    // Map A2A status to Bus Pulse
                    let pulse = self.task_bridge.status_to_pulse(&status);
                    tx.send(AgentEvent::Progress(pulse)).await?;
                }
                A2AEvent::ArtifactUpdate(artifact) => {
                    // Map A2A artifact to Roko output
                    let output = self.task_bridge.artifact_to_output(&artifact);
                    tx.send(AgentEvent::Output(output)).await?;
                }
                A2AEvent::TaskComplete(task) => {
                    return Ok(self.task_bridge.task_to_outcome(&task));
                }
                A2AEvent::InputRequired(req) => {
                    // Escalate to human or to orchestrator
                    tx.send(AgentEvent::InputRequired(req.into())).await?;
                }
            }
        }

        Ok(AgentOutcome::Completed)
    }

    fn capabilities(&self) -> AgentCapabilities {
        // Report remote agent's capabilities
        AgentCapabilities::from_a2a_card(&self.remote_card)
    }
}
```

### 3.4 Task Bridge (`task_bridge.rs`)

The bridge maps between Roko's internal task/episode model and A2A's task lifecycle:

```
Roko PlanPhase          ←→  A2A TaskState
─────────────────────────────────────────
Queued                  ←→  submitted
Enriching               ←→  working
Implementing            ←→  working
Gating                  ←→  working (with progress artifacts)
Verifying               ←→  working
Reviewing               ←→  input-required (if human review needed)
DocRevision             ←→  working
Merging                 ←→  working
Complete                ←→  completed
Failed (any phase)      ←→  failed
Cancelled               ←→  canceled
```

**Safety enforcement for incoming A2A requests:**

```rust
impl SafetyLayer {
    pub async fn check_a2a_request(
        &self,
        agent_id: &str,
        request: &JsonRpcRequest,
    ) -> Result<(), AuthzDenial> {
        // 1. Verify authentication
        let principal = self.authenticate_a2a(&request.auth_header)?;

        // 2. Check rate limits
        self.rate_limiter.check(&principal)?;

        // 3. Sanitize input — Agent Card prompt injection defense
        //    (per A2A security guidance: treat all external input as untrusted)
        self.sanitize_a2a_input(&request.params)?;

        // 4. Check agent-specific permissions
        let agent = self.agents.get(agent_id)?;
        let action = Action::from_a2a_method(&request.method);
        self.authorize(principal, action, Target::Agent(agent_id))?;

        Ok(())
    }
}
```

### 3.6 Agent Discovery (`discovery.rs`)

Three discovery strategies, in order of priority:

```rust
pub enum DiscoveryStrategy {
    /// Well-known URI: GET https://<host>/.well-known/agent.json
    WellKnown { url: String },
    
    /// Static configuration in roko.toml
    Config { cards: Vec<AgentCard> },
    
    /// On-chain registry via GolemRegistry precompile
    Chain { chain_rpc: String, registry_address: Address },
    
    /// HTTP registry (future: Linux Foundation registry)
    Registry { registry_url: String, query: SkillQuery },
}

pub struct AgentDiscovery {
    strategies: Vec<DiscoveryStrategy>,
    cache: DashMap<String, CachedCard>,
    cache_ttl: Duration,
}

impl AgentDiscovery {
    /// Find agents that can perform a specific skill
    pub async fn find_by_skill(&self, skill_id: &str) -> Result<Vec<AgentCard>> {
        let mut results = vec![];
        for strategy in &self.strategies {
            let cards = strategy.discover().await?;
            for card in cards {
                if card.skills.iter().any(|s| s.id == skill_id || s.tags.contains(&skill_id.to_string())) {
                    results.push(card);
                }
            }
        }
        // Sort by trust level (chain-registered > config > discovered)
        results.sort_by_key(|c| c.trust_level());
        Ok(results)
    }
}
```

### 5.1 Agent Card as On-Chain Data

```solidity
// Extension to GolemRegistry precompile
```

On-chain cards provide decentralized, trustless discovery: any agent registered on Korai is discoverable via A2A, and its identity is cryptographically verified by the chain.

### 5.2 Task Attestation

When a task completes via A2A, the outcome can be attested on-chain via the InsightLedger:

```solidity
struct TaskAttestation {
    bytes32 taskId;
    bytes32 clientAgentId;
    bytes32 serverAgentId;
    bool success;
    uint256 gasUsed;
    uint256 gnosPayment;
    bytes32 artifactHash;  // IPFS or content hash of task artifacts
    uint256 timestamp;
}
```

This feeds into the Predictive Foraging system (doc 10, doc 17): A2A task outcomes become PF residuals, contributing to the autonomous eval generation pipeline.

### 5.3 GNOS Billing for A2A Tasks

The OaaSRegistry precompile handles payment:

1. Client agent discovers server agent's card (includes pricing extension)
2. Client agent locks GNOS tokens in escrow via OaaSRegistry
3. Server agent executes the task
4. On completion, GNOS released from escrow to server agent
5. On failure, GNOS returned to client agent (minus gas)

## 6. IDE & External Product Integration

### 6.1 VS Code Integration Path

VS Code already supports MCP natively. For A2A:

**Near-term (using existing MCP bridge):** Create an MCP server that proxies to Roko's A2A server. This lets VS Code users invoke Roko agents as MCP tools. The MCP server discovers Roko agents via their A2A cards and exposes each skill as an MCP tool.

```
VS Code → MCP → roko-mcp-bridge → A2A → Roko Agent
```

**Medium-term (native A2A in VS Code):** As A2A adoption grows, VS Code will likely add native A2A client support (similar to how it added MCP). Once available, Roko agents are discoverable directly from the IDE without a bridge.

**Implementation: `roko-mcp-a2a-bridge`** — A standalone MCP server that:
1. Takes a Roko A2A base URL as configuration
2. Discovers all exposed agents via `/.well-known/agent.json`
3. Exposes each agent skill as an MCP tool
4. Translates MCP tool calls to A2A `message/send`
5. Streams A2A SSE events back as MCP streaming responses

This is ~500 lines of Rust. It runs as `roko connect mcp-bridge` and auto-registers with VS Code's MCP configuration.

### 6.2 JetBrains Integration

Same approach as VS Code — MCP bridge in the near term, native A2A when available. JetBrains already supports MCP.

### 6.3 Cursor / Windsurf / Other AI-Native IDEs

These IDEs are heavily MCP-integrated. The MCP-A2A bridge makes Roko agents immediately usable. As A2A becomes native in these tools, the bridge becomes unnecessary.

## 7. Connection to Autonomous Eval Generation (Doc 17)

A2A integration strengthens the autonomous eval generation system in three ways:

### 7.1 Red Team Golems via A2A

Red team golems (Mechanism 4, doc 17) can be external A2A agents. This prevents the self-improvement chicken-and-egg problem from a different angle: external red team agents have independent knowledge bases and can generate truly adversarial edge cases that internal agents would never produce.

### 7.2 Cross-Instance Eval Sharing

Multiple Roko instances share eval scenarios via A2A:
- Instance A discovers a new DeFi exploit (Mechanism 2: scenario mining)
- Instance A publishes the exploit as an A2A artifact
- Instance B receives it, adds it to its eval suite, runs it against its agents
- Results flow back via PF residuals

### 7.3 Eval-as-a-Service

A Roko instance specializing in evaluation exposes an A2A skill `run-eval-suite`. Other instances delegate their agents to be evaluated:

1. Client sends agent config + test task to eval service
2. Eval service runs the agent against 500+ scenarios
3. Returns structured results: pass rates, PF residuals, calibration curves
4. Client uses results to optimize their agent's context assembly config

## 9. Risk Assessment & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| A2A spec instability (v1.0 just released) | Breaking changes require SDK updates | Pin to `a2a-types` with version lock; abstract behind internal trait boundary |
| Prompt injection via Agent Card fields | Crafted `description` or `skill.description` injected into LLM prompts | Never use external card text in system prompts; sanitize all fields; treat as display-only |
| Authorization creep across A2A boundaries | Agent accumulates cross-system permissions | Enforce per-task scoping; no persistent authorization from A2A; require re-auth per sensitive action |
| Latency from A2A network calls vs local dispatch | Slower task execution | Use A2A only for skills unavailable locally; connection pooling; SSE for async progress |
| Dependency on external agent availability | Task failures if remote agent is down | Timeout + fallback to local agent; circuit breaker pattern; retry with exponential backoff |
| GNOS payment disputes for A2A tasks | Economic game theory edge cases | Escrow with timeout release; on-chain arbitration via governance; minimum reputation threshold |
