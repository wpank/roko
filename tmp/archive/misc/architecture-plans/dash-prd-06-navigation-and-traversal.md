````markdown
## Why this document matters

A dashboard with good information architecture and good pages can still be hard to use if users cannot get where they need to go. Navigation is the connective tissue. A user in the middle of investigating an issue should be able to follow it from symptom to cause to fix without getting lost.

This document specifies: how users enter the dashboard, how they navigate the top level (sidebar, header, search), how they navigate inside a section (local navigation, tabs, breadcrumbs), how they follow threads between objects (click-through patterns, back-forward, deep links), and how the dashboard supports power users (command palette, keyboard shortcuts, URL state).

Navigation in the Nunchi dashboard is shaped by one principle above all: **every object is a starting point**. A user should be able to arrive at the dashboard with a URL pointing at an agent, an arena, a knowledge entry, an eval, or a measurement, and start working from there. The dashboard should not require users to drill down from the home page to reach anything.

---
````

````markdown
## Navigation state — back, forward, history

The dashboard respects the browser's history stack. Navigating between pages pushes a new history entry. Back and forward traverse it.

Some patterns:

- Opening a side panel or modal does not push history. Closing them does not go back.
- Applying a filter on a list page pushes history (so back reverses the filter).
- Switching tabs on a detail page pushes history.
- Switching lens pushes history.

This is consistent with how users expect browser navigation to work.

In-page state (scroll position, expanded items) is restored when navigating back to a page. This matters: a user who scrolled deep into a list, clicked an item, inspected it, then clicked back, should find their scroll position restored.
````
