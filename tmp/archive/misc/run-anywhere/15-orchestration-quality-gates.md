# Roko Orchestration: Quality Gates & Verification

> **Audience**: QA engineers, test architects, reliability engineers
> **Scope**: Target-state specification for the automated QA gates, structured error diagnostics, Reviewer hierarchy, AutoFix engines, and the autonomous Generation of Test Infrastructures.

---

Most LLM coding tools generate code, format it, and hand it over to the developer to discover if it compiles. The human operator serves as the uncompensated compilation gate, copying errors back to the model and waiting. 

Roko operates independent, zero-operator quality gates for every phase of execution. Agents do not bypass these chains; if they exhaust the iteration budget, the system forces a failure and pauses for structural guidance. This mechanism is influenced heavily by the **GVU (Generators vs. Verifiers)** theorem, which posits that strengthening the oracle verification layer natively scales system capability above strengthening the generalized LLM.

## 1. The Three-Tier Execution Gate

Whenever an Implementer finishes an iteration in their isolated Git Worktree, Roko invokes the execution gate stack. Later gates cannot evaluate unless the preceding gate passes.

### A. The Compile Gate
Instead of deploying `cargo check --workspace` (which evaluates 50+ crates over 30-90 seconds indiscriminately), Roko uses `git diff --name-only HEAD` filtered for exactly what was touched. It executes hyper-targeted testing scopes: `cargo check -p roko-runtime -p roko-neuro`. This accelerates the negative feedback cycle from ~1 minute to <15 seconds.

### B. The Test Gate
Scopes exactly similar parameters to the Compile Gate. Roko enforces `cargo test -p crate1` inside the Worktree sandbox and evaluates structured counts over Pass/Fail/Ignored blocks.

### C. The Review Gate
Complexity-dependent logic dictates how code is evaluated post-build:
* **Trivial**: 0 Reviewers. Compilations on 10-line configuration patches are self-evident.
* **Standard**: 1 `QuickReviewer` agent operating over the combined diff sequence. 
* **Complex**: Invokes three specialized Agent branches running concurrently: 
  - **Architect**: Checks for interface coupling, API breakage, and cross-plan coherence.
  - **Auditor**: Maps code segments looking to eliminate edge cases and test boundaries.
  - **Scribe**: Validates structural MD documentation over new signatures.

## 2. Structured Error Digests & The Reflexion Loop

A raw `rustc` compilation trace frequently spans 800-2,000 lines. Surrendering this unstructured mass via IDE-dump to the LLM drastically burns token windows while severely diluting precision contexts.

Roko filters standard process streams and constructs a **Structured Error Digest**. It targets sequences matching `error:` flags, extracts their explicit `-->` file routing paths, cleanly deduplicates blocks by primary failure parameters, and passes a mathematically capped payload holding maximum <=10 unique traces back to the loop. 

**The Reflexion Integration:**
Before looping, Roko deploys a $0.01 `claude-haiku-4.5` instance strictly constrained to answer:
1. What failed?
2. Why did it fail?
3. What files must be refocused?
4. What hypothesis mitigates this?

As validated in the **Reflexion framework** (Shinn et al., 2023), verbally executing reflection post-execution over the failure parameter directly enhances the succeeding step's probability of closure. This artifact generated by Haiku sits inside the context layer of the subsequent implementer run.

## 3. The AutoFixer Optimization 

Executing a $0.50 `claude-opus` invocation to fix a missing `use std::sync::Arc` or an incorrectly targeted struct lifetime wastes infrastructure capital. 

When a standard Compile Gate flags a trace identified as purely syntactic (e.g., `ImportNotFound`, `MissingField`, or `TypeMismatch`), Roko intercepts the normal iteration cycle and routes the failure to the **AutoFixer** via `claude-haiku-4.5` rather than spinning the heavy models.

Because the AutoFix model expects a 3-second runtime with trivial logic fixes instead of full architectural decomposition, this optimization lowers simple gate-clearance costs from ~$0.30 to ~$0.01 per step.

## 4. Autonomous Sidecars & Test Discovery

If a PRD indicates: "Response times must be sub-200ms for OAuth Validation against Google", existing tools blindly generate the code but do not create the validation test infrastructure. Roko natively bridges this gap via the active **Enrichment Pipeline**.

**Enrichment Sub-Flow:**
1. Roko identifies external dependencies parsing the PRD ("Google OAuth", "PostgreSQL").
2. Before implementing the business logic, Roko generates a **Wave 0 Test Plan** injecting automated mocked services to fulfill these endpoints (e.g. `wiremock` instances, `TestContainers` spinning up transient Pg nodes).
3. The original PRD bounds are mathematically converted into system requirements (`INV-001`, `INV-002`). 
4. The system validates the bounds continuously. The persistent mock layers stay running in transient Docker Sidecars strictly tethered to the lifecycle of the Git Worktree processing the agent loops. 

If Roko handles blockchain/EVM boundaries, it actively generates `anvil` fork parameters inside the mock suite. The Implementer Agent thus operates against simulated Ethereum forks instead of writing naive blank-stub models.

## 5. The Full 7-Rung Pipeline

The existing 3-tier description above is the user-facing summary. The actual implementation in `roko-gate/src/rung_selector.rs` defines 7 rungs (0-6), with complexity-based selection:

| Rung | Gate | What It Checks | Latency | LLM? |
|---|---|---|---|---|
| 0 | **Compile** | `cargo check` (scoped to modified crates) | 5-15s | No |
| 1 | **Lint** | `cargo clippy` (or eslint, ruff, etc.) | 3-10s | No |
| 2 | **Test** | `cargo test` (existing suite, scoped) | 10-60s | No |
| 3 | **Symbol** | Verify expected exports exist (tree-sitter scan) | ~10ms | No |
| 4 | **GeneratedTest** | Behavioral tests from acceptance criteria | 10-60s | No |
| 5 | **PropertyTest** | `proptest` invariants over random inputs | 30-120s | No |
| 6 | **Integration** | End-to-end scenario scripts | 60-300s | No |

**Complexity-based rung selection:**

| Complexity | Rungs Run |
|---|---|
| Trivial | 0 (Compile), 2 (Test) |
| Simple | 0, 1 (Lint), 2, 3 (Symbol) |
| Standard | 0-4 (through GeneratedTest) |
| Complex | 0-6 (all seven rungs) |

**Escalation ladder**: Each prior failure promotes effective complexity one tier. A Trivial plan failing twice becomes Standard, gaining Symbol + Lint + GeneratedTest rungs. Saturates at Complex.

### 5.1 Rung 3: Symbol Existence Gate (~10ms)

The cheapest verification after compilation. Walks worktree source files, extracts public declarations via single-pass line scanner, verifies against plan's `SymbolManifest`.

**Mismatch taxonomy:**
```
MISSING:   struct RateLimiter at roko_core::rate_limit
WRONG_VIS: fn check_rate (found: private, expected: pub)
WRONG_KIND: Limiter (found: struct, expected: trait)
WRONG_PATH: struct Clock (found at roko_core::clock, expected roko_core::time)
AMBIGUOUS:  fn foo at roko_core::util (2 matches)
```

Handles `async fn`, `unsafe fn`, `const fn`, `pub(crate)`, `pub(super)`, `pub(in path)`.

### 5.2 Rung 4: Generated Test Gate (Anti-Gaming)

Tests encode the plan's acceptance criteria. **Critical**: The implementer agent NEVER sees the test source. Tests load from an immutable `ArtifactStore`, staged under gitignored `tests/__roko_generated__/`, and deleted after execution.

On failure, the verdict's `error_digest` lists failing test names but never leaks test source code back to the agent's context. This prevents reverse-engineering the expected output.

### 5.3 Rung 5: Property-Based Test Gate (proptest)

Runs invariant tests over randomly generated inputs. Configuration:
- **Cases per property**: 256 (default)
- **Max shrink iterations**: 2,048
- **RNG seed**: Optionally fixed for reproducibility

On failure, extracts the minimal shrunken counterexample (truncated to 2,048 bytes).

### 5.4 Rung 6: Integration Gate (~120s)

Three scenario variants:
1. **BuildTest**: Project test runner with integration-specific pattern
2. **Script**: Shell script from disk; exit code 0 = pass
3. **Custom**: Async closure spawning dependencies (Anvil forks, Docker, etc.)

Hard outer timeout: 120s default with `kill_on_drop(true)`.

## 6. Adaptive Thresholds (EMA Per Rung)

Parameters: `EMA_ALPHA = 0.1`, `MIN_RETRIES = 1`, `MAX_RETRIES = 5`, `SKIP_STREAK = 20`.

**Retry budget formula:**
```
retries = round(MAX_RETRIES - ema_pass_rate × (MAX_RETRIES - MIN_RETRIES))
```
- 95% pass rate → 1 retry
- 20% pass rate → 4 retries

**Skip advisory**: After 20 consecutive passes, rung can be skipped (sampled every 5th invocation). Single failure resets counter to 0.

**Gate Ratchet**: Once a plan passes Rung N, it cannot regress below N. Re-breaking compilation after fixing it triggers escalated failure handling.

Persistence: `.roko/learn/gate-thresholds.json` via atomic write (temp + rename).

## 9. The Gate Ratchet (No-Regression Constraint)

The `GateRatchet` enforces monotonic progress: once a plan passes Rung N, it CANNOT regress below N.

```rust
fn can_regress(plan_id: &str, rung: u32) -> bool {
    let highest_passed = self.highest_passed_rung.get(plan_id).unwrap_or(&0);
    rung >= *highest_passed  // Can only fail at or above highest passed
}
```

**Why this matters**: Without the ratchet, an agent fixing a test failure (Rung 2) might break compilation (Rung 0). The ratchet detects this as regression and escalates the failure — this is thrashing, not progress.

**Ratchet score metric**: `1 - (regression_events / total_gate_passes)`. Target: > 0.95.

## 10. The TestGenerator Isolation Principle

The Generated Test Gate (Rung 4) enforces a critical isolation: **the implementing agent never sees the test source code**.

### Why This Matters

If the agent could read the tests, it could reverse-engineer the expected output and game the verification. Instead of implementing the actual functionality, it would produce code that passes the specific test cases without solving the underlying problem.

### Implementation

1. Tests generated during plan enrichment from acceptance criteria in tasks.toml
2. Stored in immutable `ArtifactStore` (not in the worktree)
3. At gate time: staged under gitignored `tests/__roko_generated__/`
4. Run via pattern-scoped execution: only tests prefixed with `gen_`
5. Staging directory deleted immediately after execution
6. On failure: error digest lists failing test NAMES but never leaks source code

**Research**: This is the same principle behind SWE-bench's immutable test infrastructure. The test is the specification; the agent must satisfy it by implementing correct behavior, not by gaming the test.

1. **Dependency detection**: Parse PRD for external service references ("Google OAuth", "PostgreSQL", "Ethereum mainnet fork")

- **Deterministic**: Same bytecode + same input = same output, always

- **Immune to reward hacking**: Cannot game forge test by formatting tricks

|---|---|---|

## 12. Structured Error Digest (Exact Mechanism)

Raw `rustc` output often spans 800-2,000 lines. Feeding this to the LLM wastes tokens on warnings, notes, suggestions, and duplicate errors.

### Extraction Pipeline

1. Parse stderr for `error[E...]` blocks
2. Extract `-->` file:line references
3. Deduplicate by primary failure message (keep first occurrence)
4. Cap at 10 unique errors maximum
5. Strip: progress bars, download messages, blank lines, ANSI escape codes

### Output Format

```
3 unique errors detected:

error[E0433]: failed to resolve: use of undeclared crate or module `tokio`
  --> crates/roko-serve/src/main.rs:3:5

error[E0308]: mismatched types (expected `&str`, found `String`)
  --> crates/roko-core/src/config.rs:47:12

error[E0599]: no method named `send` found for struct `Channel`
  --> crates/roko-agent/src/dispatcher/mod.rs:89:15
```

### The Mostly-Passing Heuristic

```rust
fn is_mostly_passing(test_count: &TestCount) -> bool {
    test_count.failed > 0
        && test_count.total() > 20
        && (test_count.passed as f64 / test_count.total() as f64) > 0.9
}
```

If > 90% of tests pass with > 20 total tests, the agent needs a **targeted fix** (not a full re-implementation). This routes to AutoFixer instead of full Implementer.

## 13. Dynamic Gate Generation from PRD

### The Generator-of-Generators Pattern

PRDs drive plan generation. Plans drive task generation. Acceptance criteria drive gate generation:

```
PRD: "Must export public type RateLimiter from rate_limit module"
  → SymbolGate check: grep for pub struct RateLimiter in roko_core::rate_limit

PRD: "Must handle error when pool is empty"
  → PropertyTest skeleton: proptest that feeds empty pools, asserts error return

PRD: "Must not exceed 300K gas"
  → Gas benchmark gate: forge test with gas_report, assert < 300,000

PRD: "Response times must be sub-200ms for OAuth validation"
  → Integration test with wiremock mock server + timing assertion
```

**Hit rate**: ~60% auto-generation rate. Reduces manual `verify.sh` authoring by 50%.

**Why this matters**: Hand-written verification scripts drift from the actual acceptance criteria. Auto-generated gates are always aligned with the PRD — they're derived from the same source of truth.

### The Tautology Filter

Generated tests are run against the pre-implementation codebase. Tests that already PASS are **discarded** — they're tautologies that provide zero signal:

```rust
fn filter_tautologies(tests: &[GeneratedTest], worktree: &Path) -> Vec<GeneratedTest> {
    let pre_impl_results = run_tests(tests, worktree);
    tests.iter()
        .zip(pre_impl_results)
        .filter(|(_, result)| !result.passed)  // Keep only tests that FAIL before impl
        .map(|(test, _)| test.clone())
        .collect()
}
```

A test like `assert!(true)` passes before AND after implementation — zero information. Only tests that fail pre-implementation and pass post-implementation are meaningful.

## 14. Gate Parallelization (Proposed)

Currently gates run sequentially per plan. But gates for INDEPENDENT plans can run in parallel:

```
Current (serial):
  Plan 46 gates (5 min) → Plan 47 gates (5 min) → Plan 48 gates (5 min)
  Total: 15 min

Proposed (parallel, if no shared crate dependencies):
  Plan 46 gates ──┐
  Plan 47 gates ──┤── All running concurrently
  Plan 48 gates ──┘
  Total: 5 min
```

**Constraint**: Plans that modify the same crates must serialize their gate runs (compilation results depend on the shared crate state).

**Expected impact**: For 10 plans with 5 min per gate: 50 min (serial) → ~15 min (3-way parallel).

## 15. The Review Pipeline (Complexity-Dependent)

### Trivial/Simple: No Review

Compilation + test gates are sufficient. Human reviews small changes post-merge if needed.

### Standard: QuickReviewer (Single Pass)

One `QuickReviewer` agent examines the diff:
- Tool access: Read, Glob, Grep, Bash, JSON Schema output
- Model: Haiku or Sonnet (depending on effort config)
- Output: Structured JSON with pass/fail + specific concerns
- Cost: ~$0.10-0.30

### Complex: Three Parallel Reviewers + Critic

Three specialized reviewers run concurrently:

| Reviewer | Focus | Output |
|---|---|---|
| **Architect** | Interface coupling, API breakage, cross-plan coherence | JSON: structure_score, coupling_issues, breaking_changes |
| **Auditor** | Edge cases, test boundaries, correctness gaps | JSON: edge_cases, untested_paths, spec_compliance |
| **Scribe** | Documentation quality, naming, inline comments | JSON: doc_coverage, naming_quality, comment_clarity |

After all three complete, the **Critic** reviews their combined output:
- Identifies contradictions between reviewers
- Assigns priority to each concern
- Produces a synthesis: blocking_issues + recommended_improvements
- Cost: ~$0.05-0.10 (Haiku tier, reviewing structured JSON, not code)

**Total review cost for Complex**: ~$0.50-1.50 (3 reviewers + critic).
**Value**: Catches ~30% of issues that gates miss (logic errors, design problems, naming).

## 16. The AutoFixer Agent (Detailed Architecture)

### When AutoFixer Activates

The AutoFixer is not a general-purpose repair agent. It activates only when a gate failure is classified as **syntactic** — an error that does not require understanding the task's intent, only the error message and the surrounding code.

Classification heuristic:
```
Syntactic (AutoFixer-eligible):
  - E0433: unresolved import           → add missing `use` statement
  - E0308: type mismatch               → adjust type annotation or cast
  - E0599: method not found             → fix method name or add trait import
  - E0425: cannot find value            → fix variable name or scope
  - E0061: wrong number of arguments    → adjust function call signature
  - clippy::unused_imports              → remove unused import
  - clippy::dead_code                   → add #[allow(dead_code)] or remove

Semantic (NOT AutoFixer-eligible):
  - E0505: cannot move out of borrowed  → requires ownership redesign
  - E0382: use after move               → requires lifetime/ownership reasoning
  - Test failures (logic errors)        → requires understanding intent
  - Reviewer rejections                 → requires architectural reasoning
```

### What AutoFixer Receives

AutoFixer operates under a minimal context principle. It receives ONLY:

1. **The error digest** (max 10 unique errors, extracted from compiler output)
2. **The affected file(s)** (only the files referenced in error messages, not the entire worktree)
3. **A fix template** (role-specific prompt: "Fix the following compilation errors. Do not change program logic. Only fix the specific errors listed.")

AutoFixer does NOT receive:
- The plan description or acceptance criteria
- The PRD extract or research artifacts
- The workspace map or dependency graph
- Other files not mentioned in the error

This constraint is deliberate. AutoFixer should fix syntax, not re-implement features. If it had full context, it might "fix" an error by removing the code that caused it — technically compiles, but defeats the task's purpose.

### Cost and Escalation

| Metric | AutoFixer (Haiku) | Full Implementer Retry (Sonnet/Opus) |
|---|---|---|
| Model | claude-haiku-4-20250414 | claude-sonnet-4-20250514 or claude-opus-4-6 |
| Input tokens | ~500-2,000 | ~10,000-50,000 |
| Output tokens | ~100-500 | ~2,000-10,000 |
| Cost per attempt | $0.005-0.02 | $0.50-5.00 |
| Latency | 1-3 seconds | 15-90 seconds |
| Success rate | ~40% of syntactic errors | ~70% of all errors |

Escalation path:
```
Gate fails (syntactic error detected)
  → AutoFixer attempt 1 (Haiku, $0.01)
    → Re-run gates
      → Pass? Done.
      → Fail (same error)? AutoFixer attempt 2 (Haiku, $0.01)
        → Re-run gates
          → Pass? Done.
          → Fail? Escalate to full Implementer retry (Sonnet/Opus, $0.50+)
            → Implementer receives: original task context + error digest + AutoFixer's failed attempts
```

Over a 20-plan batch, AutoFixer handles ~40% of first-pass failures. At $0.01 per attempt vs $0.50+ for full retries, this saves $3-8 per batch — a 10-15% reduction in total inference cost.

## 17. The Test Sidecar: Anti-Gaming Architecture (Extended)

### The Structural Independence Principle

The implementing agent and the test generator must be structurally independent processes. This is not a style preference — it is a security constraint against reward hacking.

### What Happens Without Independence

If the implementing agent can see its own tests:

1. **Assertion-free tests**: `fn test_feature() { let _ = MyStruct::new(); }` — imports the module but asserts nothing. Passes the test gate without verifying behavior.

2. **Tautological tests**: `assert_eq!(1 + 1, 2)` — always passes, regardless of implementation. Provides zero signal about correctness.

3. **Implementation-dependent tests**: The agent writes code, then writes tests that match the exact output of that code. If the code has a bug, the tests encode the bug as "correct" behavior.

4. **Selective testing**: The agent tests only the easy cases, skipping edge cases and error paths. Pass rate looks high, but the interesting behaviors are untested.

### The Roko Solution: Temporal and Informational Separation

```
Timeline:
  T=0: Plan enrichment generates tests from acceptance criteria
       (Tests stored in immutable ArtifactStore, never in worktree)

  T=1: Implementing agent receives task:
       - Sees: acceptance criteria (natural language)
       - Does NOT see: test source code
       - Does NOT see: test file paths
       - Does NOT see: assertion details

  T=2: Agent produces implementation

  T=3: Generated Test Gate (Rung 4):
       - Tests staged to gitignored tests/__roko_generated__/
       - Tests executed against agent's implementation
       - Staging directory deleted after execution
       - On failure: error digest lists test NAMES, never source

  T=4: Agent receives failure feedback:
       - "gen_rate_limiter_empty_pool FAILED: assertion failed"
       - NOT "gen_rate_limiter_empty_pool: assert!(limiter.check().is_err())"
       - Agent knows WHAT failed but not HOW the test checks it
```

### Why Test Names Are Safe to Leak

Test names like `gen_rate_limiter_empty_pool` reveal the scenario being tested (empty pool) but not the assertion (is_err? specific error type? error message?). The agent must implement correct behavior for the scenario, not reverse-engineer the assertion. This gives the agent useful feedback ("your empty pool handling is wrong") without enabling gaming.

### Research Context

Pan et al. (ICML 2024) — "Spontaneous Reward Hacking in Iterative Self-Refinement" demonstrated that LLMs will exploit any observable reward signal to optimize for the metric rather than the underlying objective. When an agent can see its own evaluation criteria, it optimizes for passing those specific checks rather than solving the general problem.

Saunders et al. (2022) — "Self-critiquing models for assisting human evaluators" showed that model self-evaluation has systematic blind spots: models rate their own outputs more favorably than independent models rate them. Structural independence (separate evaluator) eliminates this bias.

The principle generalizes beyond testing: any verification system where the generator can observe and adapt to the verifier's internals is vulnerable to Goodhart's Law ("When a measure becomes a target, it ceases to be a good measure").

## 18. Adaptive Gate Thresholds: EMA Mechanism (Extended)

### The Problem with Fixed Thresholds

Fixed gate thresholds (e.g., "always retry 3 times") waste resources in two ways:

1. **Over-retrying easy gates**: If a gate passes 95% of the time, 3 retries wastes 2 attempts on a gate that almost always succeeds on the first try.
2. **Under-retrying hard gates**: If a gate passes only 20% of the time, 3 retries may not be enough. But increasing the fixed limit globally wastes budget on easy gates.

### Exponential Moving Average (EMA) Per Rung

Each gate rung maintains an independent EMA of its pass rate:

```
ema_new = alpha × observed + (1 - alpha) × ema_old

Where:
  alpha = 0.1 (smoothing factor — slow adaptation, resistant to noise)
  observed = 1.0 if gate passed, 0.0 if gate failed
```

Example trajectory for a rung starting at 50% EMA:
```
Pass: ema = 0.1 × 1.0 + 0.9 × 0.50 = 0.55
Pass: ema = 0.1 × 1.0 + 0.9 × 0.55 = 0.595
Fail: ema = 0.1 × 0.0 + 0.9 × 0.595 = 0.536
Pass: ema = 0.1 × 1.0 + 0.9 × 0.536 = 0.582
...
```

Alpha = 0.1 means recent observations influence the EMA slowly. It takes ~10 consecutive passes to move the EMA from 0.5 to 0.65, and ~10 consecutive failures to drop it back. This prevents a single flaky test from destabilizing the threshold.

### Threshold Adaptation Rules

The EMA drives two adaptive behaviors:

**Loosening (EMA < 0.6)**: When a rung consistently fails, the retry budget increases and pass criteria loosen slightly:

```
retry_budget = round(MAX_RETRIES - ema × (MAX_RETRIES - MIN_RETRIES))

EMA = 0.95 → retries = round(5 - 0.95 × 4) = round(1.2) = 1
EMA = 0.50 → retries = round(5 - 0.50 × 4) = round(3.0) = 3
EMA = 0.20 → retries = round(5 - 0.20 × 4) = round(4.2) = 4
```

The intuition: hard gates need more attempts. Easy gates need fewer.

**Tightening (EMA > 0.9)**: When a rung consistently passes, the system raises the bar:

- After 20 consecutive passes (SKIP_STREAK), the rung enters **sampling mode**: only run every 5th invocation. A single failure resets the counter to 0 and resumes mandatory execution.
- This saves 80% of gate execution time for consistently-passing rungs without sacrificing safety (the sampling catches regressions within 5 tasks).

### Floor and Ceiling

Thresholds have hard bounds:

- **Floor**: "Must compile" (Rung 0) is never skippable and never loosened below 1 retry. Code that does not compile is never mergeable, regardless of EMA.
- **Ceiling**: Thresholds never tighten beyond "zero tolerance for any warning" at the Complex band. The system does not demand perfection beyond what the highest band already requires.

### Persistence

Thresholds are persisted to `.roko/learn/gate-thresholds.json` and loaded on startup:

```json

    "0_compile": { "ema": 0.92, "consecutive_passes": 14, "last_updated": "2026-04-09T14:30:00Z" },
    "1_lint": { "ema": 0.85, "consecutive_passes": 8, "last_updated": "2026-04-09T14:28:00Z" },
    "2_test": { "ema": 0.71, "consecutive_passes": 3, "last_updated": "2026-04-09T14:35:00Z" },
    "3_symbol": { "ema": 0.97, "consecutive_passes": 22, "last_updated": "2026-04-09T14:25:00Z" },

}
```

Written atomically (temp file + rename) after every gate execution. On first run (no file exists), all EMAs initialize to 0.5 (neutral prior).

## 19. The Review Triad (Extended Architecture)

### Three Specialized Reviewers

The Complex band invokes three concurrent reviewers, each with a distinct focus area and evaluation rubric. Specialization matters because a single generalist reviewer tends to focus on whatever is most salient in the diff (usually the largest change) while missing category-specific issues.

### The Architect

**Focus**: Structural decisions — module boundaries, trait design, dependency direction, API surface.

**Evaluation rubric** (auto-generated from PRD and plan context):
- Does the change respect existing module boundaries, or does it introduce cross-crate coupling?
- Are new public types placed in the correct module according to the workspace architecture?
- Do new trait implementations follow the existing pattern (1 noun + 6 verb traits)?
- Does the dependency graph remain acyclic? Are new `use` paths following the layering rules?
- Would this change force downstream crates to modify their code?

**Output format**:
```json
{
  "verdict": "approve" | "request_changes" | "comment",
  "structure_score": 0.0-1.0,
  "coupling_issues": ["roko-agent now depends on roko-cli (wrong direction)"],
  "breaking_changes": [],
  "suggestions": ["Move RateLimiter to roko-core, not roko-agent — it's a shared primitive"]
}
```

### The Auditor

**Focus**: Correctness and safety — edge cases, error handling, security implications, spec compliance.

**Evaluation rubric**:
- Are all error paths handled? Any `.unwrap()` in non-test code?
- Are there edge cases not covered by the implementation (empty inputs, overflow, concurrent access)?
- Does the implementation match the acceptance criteria in the task description?
- Are there security implications (unsanitized input, unbounded allocation, timing side channels)?
- Do existing tests still cover the modified behavior, or do they need updates?

**Output format**:
```json
{
  "verdict": "approve" | "request_changes" | "comment",
  "correctness_score": 0.0-1.0,
  "edge_cases": ["Empty vec input causes index-out-of-bounds at line 47"],
  "untested_paths": ["Error branch in parse_config never exercised"],
  "spec_compliance": ["Acceptance criterion 3 ('must return error for invalid input') not implemented"]
}
```

### The Scribe

**Focus**: Documentation and API surface — doc comments, examples, changelog entries, naming clarity.

**Evaluation rubric**:
- Do all new public items have `///` doc comments?
- Do doc comments include usage examples (` ```rust` blocks)?
- Are function/type/module names self-explanatory and consistent with existing conventions?
- Is there a changelog entry for user-facing changes?
- Can a future agent (or human) understand this code without reading the implementation?

**Output format**:
```json
{
  "verdict": "approve" | "request_changes" | "comment",
  "doc_coverage": 0.0-1.0,
  "naming_quality": 0.0-1.0,
  "missing_docs": ["pub fn process_batch has no doc comment"],
  "naming_issues": ["'do_stuff' is not descriptive — suggest 'execute_gate_pipeline'"],
  "examples_needed": ["RateLimiter::new() should have a usage example"]
}
```

### Consensus Protocol

The three reviewers operate independently and concurrently. Their verdicts are combined using a simple consensus rule:

```
3-of-3 approve → merge immediately
2-of-3 approve → merge (log the dissenting concern for future reference)
1-of-3 approve → revise (agent receives all three reviews, must address concerns)
0-of-3 approve → re-implement (task is sent back to the Implementer with full review context)
```

### The Critic (Meta-Reviewer)

After all three reviewers complete, the Critic synthesizes their combined output:

1. **Contradiction detection**: Architect says "move type to roko-core" while Scribe says "add doc examples for current location" — Critic identifies the conflict and prioritizes the structural change.

2. **Priority assignment**: Not all concerns are equal. The Critic assigns priority levels:
   - **Blocking**: Must fix before merge (correctness errors, breaking changes, security issues)
   - **Important**: Should fix, but can merge with a follow-up task (missing docs, naming improvements)
   - **Minor**: Nice to have (style preferences, optional optimizations)

3. **Synthesis**: Produces a unified review with deduplicated concerns, consistent priority levels, and a clear verdict.

**Critic model**: Haiku-class (reviewing structured JSON, not code). Cost: ~$0.05-0.10. The Critic never reads the diff directly — only the three reviewers' structured outputs.

### Review Rubric Auto-Generation

Reviewers do not use a generic rubric. For each task, the enrichment pipeline generates a task-specific rubric from:
- PRD acceptance criteria (what the change should accomplish)
- Plan context (how the change fits into the larger migration)
- Existing codebase conventions (extracted during enrichment)

The rubric is included in each reviewer's prompt alongside the diff. This ensures reviewers evaluate against the specific requirements of the task, not generic "is this good code?" criteria.

### Total Review Cost for Complex Band

| Component | Model | Cost | Latency |
|---|---|---|---|
| Architect | Sonnet | $0.15-0.50 | 10-30s |
| Auditor | Sonnet | $0.15-0.50 | 10-30s |
| Scribe | Haiku | $0.05-0.15 | 3-10s |
| Critic | Haiku | $0.05-0.10 | 3-5s |
| **Total** | | **$0.40-1.25** | **10-30s (parallel)** |

The three reviewers run concurrently (wall-clock = max of the three). The Critic runs after all three complete. Total review latency: 15-35 seconds. Total cost: $0.40-1.25 per task.

**Value proposition**: Review catches ~30% of issues that automated gates miss. These are primarily logic errors, design problems, and documentation gaps — categories that no amount of compilation or testing can detect.
