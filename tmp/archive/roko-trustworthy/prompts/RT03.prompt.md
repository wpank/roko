# RT03 - Gate Failure Memory and Cross-Agent Error Pattern Sharing

## Read First

- `tmp/architecture-plans/arch-20-orchestrator-gaps.md`
- `tmp/architecture-plans/05-self-hosting.md`
- RT01 and RT02 implementations
- failure/retry paths in `crates/roko-orchestrator`, `crates/roko-gate`, and `crates/roko-learn`

## Intent

Parallel or sequential agents should not rediscover the same gate failures. Add a structured failure memory surface that can be shared across retries and sibling agents.

## Implementation Requirements

- Identify where gate failures are emitted today.
- Add or extend a `FailurePattern`/`GateFailureObservation` style record.
- Normalize failure keys from verdict parsing and compile classification.
- Persist or expose failure patterns through the existing learning/substrate surface where appropriate.
- Make prompt/context assembly able to receive a bounded failure summary later, without stuffing unbounded logs.
- Add dedupe behavior so repeated identical failures increase evidence instead of creating noise.
- Add tests for dedupe, bounded summaries, and propagation into retry context.

## Acceptance

- Failed gates create structured observations.
- Later retries can access concise, relevant failure patterns.
- The system distinguishes one-off failures from repeated patterns.
- Pattern sharing does not leak unrelated large logs into prompts.
