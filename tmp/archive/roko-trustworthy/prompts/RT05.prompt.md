# RT05 - Context Injection Scoping and Knowledge Controls

## Read First

- `tmp/architecture-plans/generalized-cybernetic-agent-architecture-gaps.md`
- `tmp/architecture-plans/arch-20-orchestrator-gaps.md`
- context/prompt composition code in `crates/roko-compose`
- orchestration context assembly in `crates/roko-orchestrator`
- core signal/config types in `crates/roko-core`

## Intent

Move toward scoped, auditable context injection. The orchestrator should not be the place where every context detail is hardcoded and stuffed into prompts.

## Implementation Requirements

- Find current context assembly/injection points.
- Introduce or refine a scoped context section type with provenance, purpose, token/cost budget, and inclusion reason.
- Add bounded context injection behavior that can reject oversized or irrelevant sections.
- Ensure sensitive or cross-task context has explicit scope and reason.
- Create a seam where future context bidders can provide candidates without direct orchestrator coupling.
- Add tests for budget enforcement, provenance retention, and rejection behavior.

## Acceptance

- Context sections are structured and traceable.
- Orchestration can request context without owning every injection rule.
- Oversized/unscoped context fails or is rejected visibly.
- The design is compatible with later `CognitiveWorkspace` and bidder registry batches.
