  What's missing for a continuously-running research agent:

  1. No heartbeat loop (the big one). PRD-02 describes a
  HeartbeatPipeline with gamma/theta/delta timescales. Today, agents
  are spawn-execute-die. There's no persistent process that ticks,
  observes, decides whether to act, and sleeps. The dispatch_loop
  reacts to events but doesn't drive agent behavior on a schedule.

  2. SourceWatcherExt doesn't exist. PRD-06 describes an extension
  that polls Arxiv, Semantic Scholar, web feeds, and local directories
   for new papers. This is the perception layer for a research agent.
  Nothing in the codebase fetches papers on a schedule.

  3. Domain profiles aren't loaded at runtime. DomainProfile::Research
   exists as an enum with defaults, but roko serve and orchestrate.rs
  don't read it to configure clock rates, gate sets, or context
  weights. Everything gets the same treatment.

  4. SynthesisExt doesn't exist. The module that evaluates whether
  accumulated observations support/contradict/extend hypotheses — the
  core research reasoning loop — isn't built.

  5. KnowledgeGraphExt isn't used for research. The neuro store exists
   but isn't queried during research dispatch to build cumulative
  context.

  6. No T0/T1/T2 cognitive gating in dispatch. Every agent call hits
  the LLM. A continuously-running agent without gating would cost
  ~$72/day (research profile) instead of the ~$2-5/day target.

  7. Agent sidecar has no internal scheduler. roko agent serve is
  purely reactive to HTTP. A long-running research agent needs to
  self-drive on its theta cycle.

  How to set up continuous research paper synthesis today

  You don't need to build the full heartbeat pipeline to get something
   useful running. Here's what works with what you already have:

  Option A: Cron-driven research via roko serve

  Wire the existing cron scheduler to trigger research dispatches:

  # roko.toml additions

  # Poll arxiv every 6 hours
  [[scheduler.cron]]
  name = "research-arxiv-sweep"
  expression = "0 */6 * * *"
  signal_kind = "scheduler:cron:research-arxiv-sweep"

  # Nightly synthesis pass
  [[scheduler.cron]]
  name = "research-nightly-synthesis"
  expression = "0 2 * * *"
  signal_kind = "scheduler:cron:research-synthesis"

  # Wire cron signals to research dispatch
  [[subscription]]
  name = "arxiv-sweep"
  event_filter = "scheduler:cron:research-arxiv-sweep"
  agent_role = "researcher"
  prompt_template = """Search arxiv for recent papers on {topics}.
  For each relevant paper: extract key findings, update the knowledge
  base,
  note contradictions with existing research."""
  cooldown_secs = 21600

  [[subscription]]
  name = "synthesis"
  event_filter = "scheduler:cron:research-synthesis"
  agent_role = "synthesizer"
  prompt_template = """Review all research artifacts from the past 7
  days in .roko/research/.
  Identify emerging themes, contradictions, and novel connections.
  Produce a synthesis report."""
  cooldown_secs = 86400

  Then deploy roko serve to Railway as described earlier. The cron
  events fire into the dispatch loop, which spawns one-shot research
  agents per tick.

  Option B: Job-based research pipeline

  Drop JSON files into .roko/jobs/ (or POST them via the HTTP API).
  The job_runner picks them up every 5 seconds:

  {
    "id": "research-mev-papers",
    "type": "research",
    "auto_execute": true,
    "params": {
      "topic": "MEV mitigation strategies 2026",
      "deep": true
    }
  }

  You can combine this with an external scheduler (Railway cron,
  GitHub Actions) that POSTs new research jobs to POST /api/jobs on
  your deployed instance.

  What you'd need to build for a real continuous research agent

  The minimum viable heartbeat for a research domain agent, using your
   existing infrastructure:

  1. A ResearchHeartbeat struct that wraps ProcessSupervisor + a
  tokio::time::interval at the theta rate (300s for research). On each
   tick: check for new papers (Perplexity/Arxiv API), run T0 triage
  (keyword match, no LLM), dispatch T1/T2 only when novel.
  2. Wire DomainProfile::Research into daemon_start() so when you run
  roko daemon start --profile research, it configures the right clock
  rates, gates, and context weights.
  3. Add source polling to the existing CronEventSource — a new
  ArxivEventSource and SemanticScholarEventSource that emit signals
  when new papers match configured queries.
  4. Accumulate context across ticks — query KnowledgeStore at
  dispatch time (the neuro store already exists, it just isn't
  consulted). This is item #13 on your CLAUDE.md roadmap.

  The cron approach (Option A) gets you 80% of the value with zero new
   code — just config. The heartbeat approach is the PRD-02 vision but
   requires building the tick loop and T0 gating.
