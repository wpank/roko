# Resolved Dogfood Issues — 2026-04-26

Issues from the original 17 that were fixed in the `wp-arch2` branch.

## Fixed: TUI crash on plan-approval thread (NEW — not in original 17)
**Root cause**: `AgentStreamClient::connect()` called `tokio::spawn()` from the
plan-approval TUI thread, which is a standard `std::thread` without a tokio runtime.
Panicked at `ws_client.rs:60`.

**Fix**: Changed `connect()` and `connect_direct()` to return `Option<Self>`, guarded
by `tokio::runtime::Handle::try_current()`. No-op when no runtime is available.

**Files**: `crates/roko-cli/src/tui/ws_client.rs`, `crates/roko-cli/src/tui/app.rs`

## Fixed: #1 — Running plan invisible to TUI/HTTP
**Resolution**: `roko plan run <dir> --approval` already runs the TUI in the same
process, sharing a `StateHub`. No HTTP bridge needed — this was a misconception.
The TUI sees `DashboardEvent`s in real time via the shared `watch::Receiver`.

## Fixed: #3 — Enrichment too aggressive for pre-authored plans
**Fix**: Added `skip_enrichment: bool` to `TaskMeta` in `task_parser.rs`. When
`[meta] skip_enrichment = true` is set in `tasks.toml`, `handle_enriching()` skips
the 13-step pipeline and transitions directly to implementing.

**Verified**: Current run shows NO enrichment artifacts written after 09:32 start time.
Agent went straight to implementation.

**Files**: `crates/roko-cli/src/task_parser.rs`, `crates/roko-cli/src/orchestrate.rs`

## Fixed: #7 — StateHub not exposed via HTTP
**Fix**: Added `GET /api/statehub/snapshot` returning the current `DashboardSnapshot`
as JSON.

**File**: `crates/roko-serve/src/routes/status.rs`

## Fixed: #10 — No health endpoint
**Fix**: Added `GET /health` at the top level (outside `/api` prefix and auth middleware)
returning `{"status": "ok"}`. The existing `GET /api/health` already provides richer
telemetry.

**File**: `crates/roko-serve/src/routes/mod.rs`

## Fixed: #14 — Config v1 warnings spam every agent spawn
**Fix**: Wrapped the `tracing::warn!` in `from_toml()` with `std::sync::Once` so the
warning is emitted at most once per process.

**File**: `crates/roko-core/src/config/schema.rs`

## Fixed: Ctrl+C leaves zombie processes (from #3 resource management)
**Fix**: Reduced `SHUTDOWN_DRAIN_GRACE_SECS` from 30 to 3. After `supervisor.kill_all()`,
`force_shutdown()` now also sends `SIGTERM` to the entire process group via
`libc::kill(0, SIGTERM)` to catch grandchild processes.

**File**: `crates/roko-cli/src/orchestrate.rs`

## Fixed: F1 — plans_dir resolution bug (skip_enrichment didn't actually work)
**Root cause**: `ensure_task_tracker()` and `dispatch_agent_with()` both used
`plans_dir()` which resolves to `plans/` (top-level) when that directory exists.
The plan was in `.roko/plans/unified-migration-phase0/tasks.toml`. The tracker was
never loaded, so `skip_enrichment = true` was never checked, and task definitions
were never found for dispatch.

**Fix**: Both functions now check `plans_dir()` first, then fall back to
`.roko/plans/<plan_id>/` if `tasks.toml` not found in the primary location.

**File**: `crates/roko-cli/src/orchestrate.rs` — `ensure_task_tracker()` and
`dispatch_agent_with()`

## Fixed: F3 — AgentOutput event never emitted
**Root cause**: `ServerEvent::AgentOutput` type defined, conversion to
`DashboardEvent::AgentOutput` implemented, TUI handler ready — but the actual
`emit_server_event()` call was missing. Dead code pipeline.

**Fix**: Added `emit_server_event(ServerEvent::AgentOutput { ... })` at the end of
`dispatch_agent_with()`, emitting the final output text with model, tokens, and
cost metadata.

**File**: `crates/roko-cli/src/orchestrate.rs`

## Fixed: F4 — TaskState lacks title field
**Root cause**: `TaskState` in `DashboardSnapshot` only had `task_id`, no
human-readable `title`. `DashboardEvent::TaskStarted` also lacked a title field.
The TUI fell back to displaying task_id, resulting in "plan plan" repeated.

**Fix**: Added `title: String` to both `TaskState` and `DashboardEvent::TaskStarted`.
Updated all construction sites (10 files). TUI now displays title when non-empty.
