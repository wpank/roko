# Building UiGate: a deterministic verifier loop for agent-generated UI

The verdict up front: **build UiGate around a vector of orthogonal external verifiers, not a single LLM judge.** Every load-bearing signal — accessibility, performance, design-token adherence, layout-metric thresholds, visual regression — is a deterministic, machine-readable JSON oracle that already exists. Wire those first, treat the multimodal LLM judge as one signal among eight, gate on conjunctive hard checks plus a Pareto frontier on soft ones, and feed every run into a private preference dataset that compounds. The literature is unambiguous: Goodhart on visual eval is the default failure mode unless you enforce regulator variety (Ashby), constrained reward composition (Moskovitz 2024), and rotating held-out canaries. Pairwise judging beats absolute scoring (Chen et al. ICML 2024, MLLM-as-a-Judge); a panel of disjoint-family judges with trimmed-mean aggregation beats any single frontier model. The closest published self-improving loop you should literally copy is **WebRL** (Qi et al., arXiv 2411.02337) for curriculum-from-failures plus **WebGen-Agent** (arXiv 2509.22644) for step-level screenshot feedback. The closest production iteration loop worth stealing is **v0's composite stack** — frontier base model plus RAG over a curated component library plus a small RFT-trained post-processor (`vercel-autofixer-01`). Everything else in this report is a refinement of that core.

Roko's UiGate sits at a near-perfect intersection: external verifiers are abundant and high-quality for accessibility and performance, mid-quality for design-token adherence, and weak-but-improvable for aesthetic judgment. The right architecture treats each as a separate disjoint loop with its own sensor, comparator, controller, and gate. The flywheel is built from preference data, AST-mined patterns, and a small fine-tuned repair model, not from a custom UI judge — that's the wrong asset to compound on for the first year.

## The runtime substrate: chromiumoxide, Browserless, odiff, axe

For the inner runner, **chromiumoxide is the pick** (mattsse/chromiumoxide, MIT/Apache, ~0.7.x). Direct CDP gives you deterministic screenshots via `Page.captureScreenshot { fromSurface:true, captureBeyondViewport:true }`, animation freeze via `Animation.setPlaybackRate(0)`, request mocking via `Fetch.enable`, and DPR/viewport pinning via `Emulation.setDeviceMetricsOverride`. No Node runtime, no version-skew nightmares, ~5–15 ms CDP round-trip locally vs Playwright Node's 30–50 ms tax on chatty interaction-heavy flows. Pin to a single `chromium-headless-shell` major; the new `--headless=new` mode (now the only Chromium headless after Playwright v1.49) renders closer to headed but explicitly breaks old screenshot baselines. Re-baseline once and lock.

Avoid `playwright-rust` (lagging shim over Node subprocess), `headless_chrome` (sync, smaller community), and Cypress (wrong execution model — same-origin/iframe pain). `fantoccini` and `thirtyfour` only matter if Firefox/Safari parity is required; WebDriver lacks fine-grained CDP semantics like JS coverage, request-body interception, and isolated worlds. The agent-oriented harnesses — **Stagehand v3** (Browserbase, MIT, ~44% faster than v2 on iframes), **browser-use** (~90k★, MIT, claimed 89.1% WebVoyager), **Skyvern 2.0** (planner/actor/validator, 85.85% WebVoyager) — are competitors and baselines, not dependencies. Steel.dev's leaderboard puts H Company's Surfer 2 at 97.1% WebVoyager (Feb 2026), but the numbers are largely self-reported on non-uniform subsets; treat as directional. Roko's value is the deterministic verifier loop; consume the act/observe primitive pattern and the planner/validator split, but don't import the runtime.

For the browser pool, a **self-hosted Browserless `multi` Docker image behind a token, with chromiumoxide connecting via raw CDP WebSocket** is the cleanest control plane. Browserless flipped to **SSPL** — verify before shipping commercial Roko or swap to the official `mcr.microsoft.com/playwright:vX-noble` image with `playwright run-server` (Apache-2.0). Either works; Browserless has better operational defaults (queueing, reconnect, `/sessions`). Bake five non-negotiables into every capture: stable selectors (`getByRole`, `data-testid`), deterministic waits (sentinels, never `sleep`), CDP request mocking against a per-task fixture manifest, fixed (1280×720, DPR=1) viewport, and clock freezing via `Emulation.setVirtualTimePolicy`. Container fonts must be pinned via `fontconfig` + Noto/Liberation; without this even no-change runs show 0.5–1% pixel drift.

For visual regression, **odiff is the differ** (MIT, Zig+SIMD, ~6.6× pixelmatch via `npm odiff-bin`), emitting `{match, reason, diffCount, diffPercentage}`. Skip Applitools, Percy, and Chromatic — all SaaS-locked, ML-opaque, antithetical to the external-deterministic-verifier philosophy. Their Layout-mode and Ultrafast-Grid ideas are worth implementing yourself in <500 LoC of Rust on top of CDP `DOM.getBoxModel` + `Accessibility.getFullAXTree`. Emit reg-suit-compatible JSON shape (`{failedItems, newItems, deletedItems, passedItems, diffItems}`) so existing OSS tooling consumes UiGate verdicts. Add **dssim** (Rust, AGPL-or-commercial) as a tiebreaker only — if odiff says `diffPercentage > 0.1%` AND dssim score > 0.05, fail; if pixel diff but dssim < 0.01, mark as anti-aliasing noise and pass. Honeydiff (Vizzly, proprietary) claims 9–16× over odiff but reproductions are unpublished; ignore until verified.

The accessibility verifier is **axe-core** (MPL-2.0) injected as a script via CDP and harvested as JSON. Configure it maximally — all WCAG tags through `wcag22aa`, plus `wcag2aaa`, `wcag21aaa`, `best-practice`, `ACT`, and `experimental` enabled — and accept the extra noise for higher recall. axe is the spine; layer **IBM Equal Access (`achecker`)** on top because its independent rule engine has a different false-positive surface and adds 5–10% recall on real audits. Lighthouse a11y is a lossy score over an axe subset — assert on the underlying axe violations, not the score. Pa11y's `htmlcs` runner is noisier; if you use Pa11y at all, set `--runner=axe` or skip it. WAVE/Tenon/Stark add nothing deterministic beyond axe + IBM. Realistic automation coverage is **29.5% fully automated, 10.3% partial, 60.2% manual** of WCAG 2.2 SCs (Rushi 2024); axe's "57%" marketing counts violation instances, not SCs. Plan for the manual layer; don't pretend automation is sufficient.

For component-level eval, target a built `storybook-static`, but **reimplement the Storybook test-runner pattern in Rust** against chromiumoxide rather than importing it. Each story is the unit of UI contract; the `play()` function pattern is your per-task verification DSL. Pair with axe per story, screenshot per story, and odiff per story. Component-level diffs are 5–10× less flaky than full-page; do component-level on every PR and full-page nightly on canonical user journeys.

## The judging methodology: pairwise Bradley–Terry with disjoint panels

The single most important methodological commitment is **pairwise comparison aggregated via Bradley–Terry, not absolute Likert scoring**. Chen et al.'s MLLM-as-a-Judge benchmark (ICML 2024 Oral, arXiv 2402.04788) is decisive: GPT-4V/Gemini/Qwen-VL/LLaVA-1.6 align with humans on pairwise (~0.6–0.7+ agreement) but diverge significantly on absolute scoring (Pearson ~0.49) and batch ranking. Pairwise has lower variance per comparison, no fixed scale to anchor on (harder to Goodhart), and composes natively with TensorZero/LinUCB which fundamentally consume preference signals. The cost is O(N²); mitigate via active sampling à la Chiang et al. (Chatbot Arena, arXiv 2403.04132) where information-gain sampling concentrates comparisons among closely-rated candidates, and by using cheap absolute scores as a coarse prior to pre-filter to top-K before pairwise.

Aggregate via **Bradley–Terry MLE with bootstrap CIs**, not online Elo. BT is the MLE under static-skill assumption; Chatbot Arena switched in 2024. Implementation is a logistic regression over `(model_a, model_b, win)` triples with no intercept and high C; map coefficients to Elo scale via `* 400/log(10)`. Use the Davidson model when judges return ties (arXiv 2412.18407). Always evaluate `(A,B)` and `(B,A)` and discard inconsistent verdicts — position bias is the single biggest reliability killer (Wang et al. ACL 2024, arXiv 2305.17926; Shi et al. 2024 show position consistency drops below 0.5 with 3–4 candidates when quality gaps are small). The 2× cost is non-negotiable. Always evaluate `new_candidate vs prev_best_release` against a fixed anchor; never score in isolation, because BT against an anchor is harder to inflate than an absolute score and gives progress in interpretable Elo points.

For the panel itself, run **3–4 judges from disjoint families with trimmed-mean (10–20%) aggregation**. Verga et al. PoLL (arXiv 2404.18796) shows a panel of smaller diverse-family models outperforms a single large judge at 7× lower cost. The mandatory composition: one closed frontier (Claude Opus 4.5 or GPT-5), one open multimodal critic (LLaVA-Critic-72B, arXiv 2410.02712, fine-tuned on 113k pointwise+pairwise judgments and matching GPT-4o on judgment alignment), one rubric-conditioned specialist (Prometheus-Vision, arXiv 2401.06591, Pearson ~0.5–0.6 with humans at a fraction of API cost). **Never use the same model family as both generator and judge** — self-preference is well-documented (Wataoka et al. NeurIPS Safe-GenAI WS 2024, arXiv 2410.21819, correlates with low perplexity-on-self). Trimmed mean is robust to one judge going off-rails while using more information than median; learned weights via stacking on held-out human labels gain another 3–8 correlation points once you have ≥500 human-rated examples.

Treat **G-Eval as a structural template, not a literal recipe** — modern frontier APIs no longer reliably expose token logprobs at the scale G-Eval needs, killing the elegant probability-weighted variant. Fall back to N=5 samples per judgment at T=0 (captures ~80% of N=20's variance reduction; Chiang & Lee arXiv 2310.05657 and Chen et al. 2024) with majority vote or median, and always force analyze-before-rate rather than score-only output. UICrit (Duan et al., UIST 2024, arXiv 2407.08850) is the single best academic resource for "is this UI good": 1,000 RICO mobile UIs with 3,059 critiques, bounding boxes, and ratings on aesthetics/learnability/usability/overall by 7 professional designers, three annotators per UI, CC-BY 4.0. Their few-shot + visual prompting achieved a +55% improvement in LLM feedback quality. **Use UICrit's annotation schema verbatim** for UiGate's judge prompts and as a fine-tuning corpus. Their critical finding: app-store rating correlates r=0.007–0.023 with expert aesthetic ratings — aggregate user ratings are useless as quality signal. UIClip (arXiv 2404.12500) gets 67.65% on synthetic UI quality but only 57.89% on real apps; off-the-shelf MLLM judges underperform on real UIs and need fine-tuning or ensembling.

For statistical rigor, **Krippendorff's α ≥ 0.8 on a held-out human canary is the eval-quality gate** (Krippendorff 2004 thresholds, stricter than Landis-Koch κ which most LLM-judge papers cite to over-claim). Below 0.667, the judge is broken. Use **BCa bootstrap intervals** (Efron 1987, second-order accurate) for skewed aesthetic scores rather than percentile bootstrap. Adopt the arXiv 2511.19794 release-gate protocol: paired-by-prompt design + BCa + sign-flip permutation; claim significance only if BCa excludes 0 AND permutation p<0.05. For multi-dimensional rubric analysis, **Benjamini–Hochberg FDR for exploratory dim-by-dim, Bonferroni for the headline metric**. Sample-size guidance from Chatbot Arena empirics: ~400 votes per model gives ±10 Elo at typical Arena dispersions; for absolute 5-pt Likert at σ≈1.0 detecting Δ=0.2 at α=0.05/power=0.8, n≈393 per arm unpaired or n≈199 paired with ρ=0.5. Run each judgment ≥3× even at T=0 — API non-determinism produces 1–3% disagreement on identical inputs.

## Operationalizing design theory: 15 metrics every render must compute

The literature gap UiGate must close: there is **no peer-reviewed reference-free benchmark for "good design" that has high inter-rater reliability and large scale**. WebDev Arena (lmarena, 80k+ votes by early 2025, with the openly-licensed `lmarena-ai/webdev-arena-preference-10k` dataset of 10k pairwise battles) confounds aesthetic with functionality and shows 26% tie ratio. Design Arena has selection bias. UICrit is excellent but 1,000-UI scale and mobile-only. DesignBench (arXiv 2506.06251) showed MLLMs achieve only **0.27 accuracy on UI issue identification** — frontier judges systematically miss real defects. The pragmatic stack is therefore composite: cheap automatic floor + rubric-conditioned VLM judges + preference reward model + computational aesthetic priors. The 15 metrics below give you the floor, every one computable in <2 s on a screenshot+DOM with a numerical formula.

The **hard-gate metrics, in order of ROI**: token coverage (≥0.9 of CSS values resolve to design tokens), WCAG 2.1 contrast pass rate (1.00, hard gate), APCA Lc per text node (≥95% of body text Lc≥60, ≥75 preferred via `Myndex/apca-w3`), grid adherence (≥0.95 of spacing values divisible by 4 or 8), alignment score (1 − unique_x_coords/element_count along major axis, ≥0.7), modular-scale conformity (≥0.9 of font sizes fitting one ratio in {1.125, 1.2, 1.25, 1.333, 1.414, 1.5, 1.618}), and color-palette ΔE compactness (k-means k=5–8 in OKLCH, max ΔE2000 to nearest token ≤2.3 for >0.95 of pixels excluding images). The **soft-gate metrics, contributing to a Pareto frontier**: Hasler-Süsstrunk colorfulness M in [15, 35] (Reinecke et al. CHI 2013 first-impression band), element density (≤30 mobile / ≤50 desktop per Miniukovich AVI 2014), text/whitespace ratio in [0.15, 0.40], visual balance |BM_v|<0.15 unless explicit asymmetry, saliency-on-CTA score from DeepGaze IIE+UMSI++ ensemble (S_cta>0.7 with primary CTA in top-3 saliency peaks), F-pattern/layer-cake template correlation r>0.4, LPIPS<0.15 vs golden reference, and AIM Feature Congestion / Grid Quality from `aalto-ui/aim`.

The foundational reference is **AIM (Aalto Interface Metrics)** — Oulasvirta et al. UIST 2018, twenty-one computational metrics including Feature Congestion (Rosenholtz), Grid Quality, color-blindness simulation, JPEG-size complexity proxy. AIM is the load-bearing externals layer for layout health; vendor it. For Gestalt operationalization, follow Koch & Oulasvirta CHI 2016: proximity via DBSCAN silhouette on element centroids, similarity via cosine on style vectors per role-class, continuity via shared-axis alignment runs ≥3, closure via bounded-region detection (`border|background-color|box-shadow`), figure/ground via APCA on primary surfaces. For visual hierarchy, compute saliency-weighted center of mass plus symmetry score `BM_v = |W_L − W_R| / max(W_L, W_R)`. For F-pattern detection on text-heavy pages, run DeepGaze IIE (arXiv 2105.12441, AUC 88.3) and UMSI++ (UEyes, CHI 2023, UI-specific SOTA, Zenodo 8010312) ensemble-averaged, then correlate against an F-template kernel. Banner blindness is real (Pernice/NN/g): penalize CTAs styled like ads. Left-side bias means primary nav and content go left on desktop (~80% of fixation time per Fessenden). For motion compliance, durations ∈ Material 3 bands (100–150ms small, 200–300ms medium, 400–500ms large), easing ∈ allowlist of `cubic-bezier(0.2, 0, 0, 1)` and siblings, and **mandatory `prefers-reduced-motion` differential testing** — the most common failure is sites that ignore the user preference entirely.

For Nielsen's heuristics, collapse to Shneiderman's overlap and operationalize as DOM/AST + screenshot probes: visibility-of-status (loading indicator within 200ms of any async dispatch), error-prevention (every required input has aria-describedby and confirmation modal for destructive ops), recognition-over-recall (icon-only buttons require aria-label and tooltip), aesthetic-and-minimalist (element count ≤30 per viewport, density 0.2–0.5). Each probe emits a 0/1 plus continuous severity. Material Design 3 compliance reduces to token adherence: `--md-sys-color-*` ΔE<2 to nearest token, elevation matches dp 0–5 shadow specs, motion duration and easing in the M3 token set, border-radius ∈ {0, 4, 8, 12, 16, 28, full}.

## The cybernetic architecture: eight disjoint loops, conjunctive hard gates, Pareto soft

The right control-theoretic frame is **Ashby's Law of Requisite Variety**: V(R) ≥ V(D), the regulator's variety must equal or exceed the disturbance's. Single-signal eval has insufficient regulator variety against the manifold of bad UIs an LLM agent can emit. UiGate must operate **a vector of orthogonal sensors, each catching a different failure mode the others miss** — Conant–Ashby's good-regulator theorem says every good regulator must be a model of the system, and ensembling partial models approximates a sufficient one.

Architect **eight disjoint loops in parallel**, each with its own sensor, comparator, controller, and gate, with no signal sharing between loops:

- Loop A — Tokens/Static: DOM + tokens.json adherence; **hard gate**.
- Loop B — Accessibility: axe-core + IBM achecker + APCA Lc; **hard gate** (Tier 1+2).
- Loop C — Layout metrics: AIM (grid, balance, symmetry, congestion); **soft, threshold band**.
- Loop D — Visual regression: odiff + LPIPS vs golden; **soft, ref-anchored**.
- Loop E — Saliency intent: DeepGaze IIE + UMSI++ ensemble; **soft**.
- Loop F — Heuristic LLM judge: rubric-prompted disjoint-family panel; **soft, ensemble vote**.
- Loop G — Behavioral probe: Playwright task script (click flow, form completion); **hard gate**.
- Loop H — Performance: Lighthouse-CI + LoAF-instrumented INP + reduced-motion diff; **hard gate** (Tier 3+5).

Each loop has its own controller policy. Hard-gate failures block and trigger localized retry with the violation context fed back as actionable feedback. Soft-gate scores feed a Pareto frontier; the controller is a multi-armed bandit (Thompson sampling over `(loop, fix-prompt-template)` posteriors). Critically, **gate composition is conjunctive on hard gates and Pareto on soft, never weighted-sum**. Moskovitz et al. ICLR 2024 ("Confronting Reward Model Overoptimization with Constrained RLHF") shows constrained-RLHF with per-RM thresholds outperforms weighted-sum because correlated proxy RMs amplify Goodhart. Translation: a strong saliency score must not compensate for an APCA failure. Each loop must independently pass its threshold.

Apply control-theory primitives to the retry policy: **dead-band** (don't retry if all soft scores within ε of target — ε=2 ΔE for color, 0.05 for density, 0.03 for S_cta — prevents oscillation on marginal improvements), **hysteresis** (require 2 consecutive passes to accept, 1 strong fail to reject — prevents flapping under stochastic judge noise), **anti-windup** (cap retries to 4 per loop and 12 per session; if cumulative error fails to decrease over 2 retries on the same loop, switch controller — escalate prompt template, swap models, or accept-with-warning), and **derivative term** (track Δscore between renders; if Δ<0, revert to previous candidate). Add **feedforward** lints on generated source pre-render (token usage, contrast on specified colors) to short-circuit obvious failures cheaply.

For **Goodhart-resistance specifically on visual metrics**, apply Manheim & Garrabrant's taxonomy (arXiv 1803.04585). Regressional Goodhart on AIM "balance" selects for noise; counter by requiring improvements on ≥3 orthogonal metrics. Extremal Goodhart hits when AIM metrics validated in [normal, normal-bad] are pushed to perfect-score regime (perfect symmetry = sterile UIs, max contrast everywhere = harsh); counter by clipping rewards into a band rather than maximizing. Causal Goodhart: high WCAG ratio doesn't cause readability if font is too thin — use APCA, which models font-weight, alongside not instead. Adversarial Goodhart: the agent itself is the adversary against any single judge it learns to imitate; counter via rotating eval suites, holdouts, and meta-eval. Six concrete operational tactics: a **panel of disjoint-family judges with trimmed-mean aggregation** so the inner-loop bandit reward and held-out validation never share a judge; a **frozen human-rated Canary Set of 200–500 prompts** with UICrit-style ratings (α≥0.8) re-evaluated every release — if panel score rises but Canary doesn't, you're Goodharting; **eval diversity with quarterly rubric rotation** because Goodhart relies on a stable target; **adversarial red-team eval** running UIClip + LAION + a "trivially gameable patterns" detector (gradient bombs, oversaturated hero, blur-everything) on the bandit's top-K outputs; **reference-conditioned BT against a strong baseline** rather than absolute scoring; and a **canary correlation monitor** — if Spearman between cheap inner-loop judge and expensive Canary drops below ~0.6, the inner judge has drifted, retrain or replace. Maintain Krakovna's specification gaming list as a continuously-updated red-team prompt set; every quarter, run agents adversarially against current eval and add new gaming patterns.

## Three-fidelity design extraction: reference URL → W3C tokens

The W3C Design Tokens Community Group **2025.10 spec is the stable target** (released October 28, 2025). It's still a Community Group Report not a W3C Standard, but ten-plus tools have shipped or are implementing it: Penpot, Figma, Sketch, Framer, Knapsack, Supernova, zeroheight, Tokens Studio, Specify, Style Dictionary v4 (with v5 in progress for full 2025.10). File extension `.tokens` or `.tokens.json`, MIME `application/design-tokens+json`. Token shape uses `$type`/`$value` with composite types (`typography`, `border`, `shadow`, `transition`, `gradient`) taking object values whose properties are themselves typed. Aliases use `{dot.path}` syntax with recursive resolution. **Adopt the three-tier taxonomy directly**: primitive (`color.blue.500 = #3B82F6`), semantic (`color.action.primary → {color.blue.500}`), component (`button.primary.bg → {color.action.primary}`).

The pipeline: **Rust orchestrator → Playwright/chromiumoxide → Wallace + custom analyzer → token assembler → Style Dictionary**. **Project Wallace** (`@projectwallace/css-analyzer` MIT, `@projectwallace/css-design-tokens` EUPL-1.2 — review the copyleft before vendoring) is the primary primitive-extractor: it ingests raw CSS and emits near-DTCG-shaped JSON for color, fontSize, fontFamily, lineHeight, gradient, boxShadow, radius, duration, easing. It analyzes what's literally written in CSS, not what's computed at render time, so pair it with a **custom Rust analyzer** that walks `getComputedStyle()` over every visible element, harvests area-weighted color values, runs k-means in OKLCH (using the Rust `palette` crate) for the rendered color picture, and applies modal extraction on spacing/radius/font-size. Cross-check palette against `node-vibrant`/`colorthief` on the screenshot — if cluster centroid is within ΔE2000<4 of a Vibrant swatch, mark as confirmed. Auto-name primitives by hue family + lightness bucket. Promote to semantic aliases by role inference (dominant text color → `color.text.primary`; highest-area accent on `<button>` and CTA links → `color.action.primary`). Style Dictionary v4 (Apache-2.0) is the canonical OSS sink for multi-platform output; emit CSS variables, Tailwind config, Swift, Kotlin, Compose.

**Three fidelities map cleanly**: Tier 1 qualitative (vision LLM with structured-output schema covering voice/material/density/palette-character/typography-character/shadow-character/radius-character/imagery/notable-patterns); Tier 2 mid-level statistical extraction (Wallace + custom analyzer + Vibrant); Tier 3 reproducible W3C tokens (assembled DTCG file). Treat each as an independent judging axis; composite UI score is weighted across tiers, not a single number. Skip Diez (effectively dead since 2020), Specify (wrong shape — assumes designer-curated Figma source), html-to-figma (layout snapshotter, not a token source), and Tokens Studio (Figma-only). **Magicast** is the right tool only for emitting `tailwind.config.ts` or `style-dictionary.config.ts` patches into a generated repo while preserving formatting — it doesn't extract tokens, it edits source. Watch `Manavarya09/design-extract` on GitHub: a one-person OSS that's already 70% of UiGate's Tier-2/3 ambition, MIT-licensed.

The **token adherence score** for a generated UI is the load-bearing per-render metric. Algorithm: render the UI in headless Chrome, walk every visible element, harvest style properties per token category, compute per-occurrence minimum perceptual distance to nearest token in the reference set (ΔE2000 in OKLCH for color with threshold ε=2.0; px for dimension with ε=0.5; weighted L2 over (offset, blur, spread, ΔE_color) for shadow with ε=4.0; exact match for font family / weight bucket). Each occurrence is **area-weighted** (large hero counts ≫ tiny footnote). Per-category score = Σ(hit·area) / Σ(area). Overall adherence is a category-weighted sum with default weights {color: 0.30, spacing: 0.25, fontSize: 0.15, radius: 0.10, shadow: 0.10, fontFamily: 0.10}. Resolve `{...}` aliases first; the leaf value is what you compare. Skip `transparent`/`currentColor` to avoid bias. Output a violations list sorted by area — that's the actionable diff fed back to the agent. Provide a `--strict` mode with ε=0 for tokens-only audits.

On the question of **screenshot-only extraction**: feasible for Tier 1 and partial Tier 2, **not adequate for Tier 3 reproducible tokens**. Pixel values lie about CSS values (subpixel AA produces hundreds of slightly-off RGB values; you can recover clusters of perceptually similar colors but not discrete authoring decisions). Spacing is fundamentally ambiguous from raster — was that 24-px gap `space.6 = 24px` or `padding-y: 1.5rem` on a parent with `margin: 1rem`? Composite tokens (layered shadows, gradients, transitions) are largely lost. Aliases are unrecoverable. Rico, WebUI, and all academic baselines rely on view hierarchies precisely because screenshot-only is noisy. Reserve screenshot-only paths for image-only references (Dribbble, brand decks) and tag those Tier-3 outputs with `$extensions: { "ai.uigate.confidence": "low" }`.

For static linting of the generator's own source code, vendor patterns from `stylelint-plugin-carbon-tokens` (Carbon Design System) and `@kong/design-tokens/stylelint-plugin` — the latter is best-in-class for type-correctness, checking that the *correct* token is used per-property (`kui-color-text-primary` on `color`, not `background-color`). Run as a pre-render feedforward check; cheaper than runtime adherence and catches "literal `#3B82F6`" violations instantly.

## Benchmarks: what to copy, what to ignore

The field has bifurcated into three traditions Will needs to keep mentally separated. **Reference-matching benchmarks** (Pix2Code, WebSight, Design2Code, WebCode2M, DesignBench) measure "does the rendered HTML match a target screenshot." **Agentic/functional benchmarks** (WebShop, Mind2Web, WebArena, VisualWebArena, WebLINX, ScreenSpot-V2, VisualWebBench, AITW, WebGen-Bench) measure "did the agent click the right button." **Aesthetic/preference benchmarks** (WebDev Arena, Design Arena, UICrit) measure "do humans prefer this UI." UiGate cares about all three but especially the third, where rigorous benchmarks barely exist.

**Three papers to literally copy methodology from.** First, **Design2Code** (Si et al. Stanford, arXiv 2403.03163, 484 real C4 webpages + 80 HARD subset) for the cheap automatic floor. Their metric stack: Block-Match (visual block detection + Jonker-Volgenant assignment + area-overlap), text similarity per matched block, position similarity (1 − normalized centroid distance), color similarity (CIEDE2000 mean color), CLIP cosine. Critical gameability analysis: **CLIP is the most gameable** (right gestalt color/density wins with wrong content; sanity check only, never primary reward), **Color is gameable** by emitting one dominant palette color globally (counter by weighting blocks by area variance), **Position is the most robust signal** (hard to fake without actually placing elements correctly), **Block-Match recall is robust but penalizes valid alternative decompositions** (don't use as ranking signal between two valid generations), **Text similarity is robust if you OCR rather than parse DOM**. Use geometric mean of Block-Match + Position + Text as defensible auto-score. Their human eval finding that **GPT-4V was rated better than original in 64%** is alarming — references aren't a quality ceiling, they're a similarity target. Use Design2Code-HARD as held-out eval.

Second, **UICrit** (Duan et al. UIST 2024, arXiv 2407.08850) for the annotation schema — bounding-box-grounded critiques + multi-axis ratings (aesthetics, learnability, usability, overall) — and the few-shot critique prompt structure. Their +55% gain methodology is the template; the open dataset (1,000 mobile UIs, 3,059 critiques, CC-BY 4.0) is your judge fine-tuning corpus. Third, **WebGen-Agent** (Lu et al. Sep 2025, arXiv 2509.22644) for the **Step-GRPO with Screenshot and GUI-agent Feedback loop**. They lift Claude-3.5-Sonnet from 26.4%→51.9% functional accuracy and 3.0→3.9 appearance via screenshot-VLM scoring at every step. This is the closest extant work to a self-improving UI generation loop, published with code. Honorable mentions: **DigiRL** (arXiv 2406.11896) lifts a 1.3B VLM from 17.7%→67.2% on AITW via autonomous RL with VLM-evaluator — methodological blueprint for self-improving loops; **Online-Mind2Web's WebJudge-7B** for the agent-as-judge reward model approach; **WebSight's hard lesson** that validation loss ≠ generation quality (always rely on rendered-screenshot eval).

For agentic baselines, **VisualWebArena** (910 vision-required tasks, human 88.7%, current SOTA tree-search GPT-4o = 26.4%), **WebArena** (812 tasks, ~58% Operator/CUA), and **Mind2Web/Online-Mind2Web** (2,350 → 300 live tasks) are the canonical agent benchmarks. **VisualWebBench** (NAACL 2024, arXiv 2404.05955, 1,500 instances, 7 tasks) decouples understanding from end-to-end navigation. Treat all of these as adversarial stressors against UiGate's generated UIs — if your generated UI is consumed by a Mind2Web-class agent, ScreenSpot-V2 grounding accuracy (current SOTA GUI-CURSOR 93.9%) tells you whether the agent can find your buttons. **WebDev Arena's `lmarena-ai/webdev-arena-preference-10k`** dataset is your free pretraining corpus for a UiGate judge: 10k pairwise battles, commercial-use licensed, real preference data.

## The flywheel: seven steps that turn every run into permanent capital

Every UiGate run must emit labeled, structured, retrievable artifacts, or the system doesn't compound. The seven steps: **(1) Trace capture** via TensorZero — store prompt, retrieved components, all rollout variants, bandit-arm chosen, AST of final output, screenshots, compile/render telemetry, user edits/keeps/discards, time-to-accept. **(2) Auto-grade with the disjoint-family panel plus deterministic linters plus screenshot-diff vs nearest neighbor in the success bank**; disagreement triggers human review. Pairwise vs previous-best, never absolute. **(3) Pairwise preference mining**: every time the user picks variant A over B (à la 21st.dev's 4-variant UI), log the preference triple and append to a Bradley-Terry pool; user edits become positive/negative pairs (post-edit positive, pre-edit near-positive negative). This is your private WebDev Arena. **(4) Pattern extraction via AST diffing nightly**: cluster successful components by (Tailwind-class bag, DOM shape, prop signature) and emit cluster centroids as templated entries in a private pattern library; diff successful-vs-rejected pairs to extract anti-patterns; feed both as positive/negative few-shots through DSPy `BootstrapFewShotWithRandomSearch`. **(5) Curriculum-from-failures, the WebRL move**: cluster failed runs by judge-rationale embedding; each cluster spawns synthetic training tasks ("pricing card with 3 tiers and footnote disclaimer," "dashboard with 4 KPI cards + sparkline"); schedule retries on improved pipeline; promote successes into the regression eval set. **(6) MIPROv2 over the right surface** — optimize retrieval queries used to fetch components, the clarifying-question prompt, judge rubrics, and AutoFix repair prompts; **do not** optimize the base generator prompt directly (too much capacity, low signal-per-call); bandits over discrete pipeline arms (model choice, retrieval-k, with/without clarifying turn, with/without post-fixer). **(7) Periodic RFT of a small post-processor** — once you have ≥10k labeled (broken-output, fixed-output) pairs from steps 2 and 4, RFT a Llama-3.1-8B or Qwen2.5-Coder-7B on Fireworks/Together as `uigate-autofixer-01`. This is the **single asset that actually compounds** — every fix it learns reduces frontier API spend forever and survives model swaps.

The position on **custom UI judge vs frontier APIs**: frontier-only for the first year. APIs already hit ~70–80% human agreement on pairwise UI judgments; cost is real (~$0.05–0.20/judgment, $500–2000/day at 10k judgments) but judging is parallelizable and infrequent compared to generation. Lock-in is overstated — judges are commodity, generators less so; swap Claude for Gemini for GPT-5 if pricing shifts. The custom judge wins only on three conditions, all of which take time to satisfy: ≥5k high-quality labeled pairs from steps 2–3, a domain quirk frontier judges miss (specific brand-token compliance), or per-judgment latency materially blocking the inner loop. **Spend fine-tune budget on the post-fixer first** — repair tasks have crisper signal (broken→working is binary-ish), more abundant data (every error log is training), and savings compound on every subsequent generation. v0's `vercel-autofixer-01` (Fireworks RFT, replacing Gemini Flash 2.0) is the right pattern. The judge can stay frontier-API for 12+ months. Train one tiny exception: a DistilBERT-class router classifier on ~1k UICrit + own-data pairs to gate "is this prompt likely to need clarification?" or "is this likely to need a chart library?" — cheap, high-signal, low-stakes.

For **products to specifically steal from**: **v0's composite stack** (frontier base + RAG + RFT post-fixer) is the architectural template; **WebRL's curriculum-from-failures + outcome reward model + KL-constrained policy update** is the academic prototype; **21st.dev's curated community library** is the cleanest compounding-asset pattern (RAG over juried inventory that grows with submissions); **Subframe's clarifying-question step before generation** is the cheapest MIPROv2-style improvement (DSPy signature with conditional emission); **Builder.io's Mitosis IR layer** is worth studying as a way to optimize once and emit many target frameworks; **Locofy's Large Design Model** is proof that heuristics beat LLMs on deterministic parts of the pipeline. Ignore Lovable/Bolt's "agent" framing — public reports of regression-loops suggest no compounding asset is being created. Builder.io's component-mapping feature (RAG over your repo's actual components) is the right architecture for a private design-system enforcement layer — clone the pattern.

## Hard gates ordered by ROI

The 5-tier hard-gate pipeline, ordered highest ROI first; production conformance requires passing all five but UiGate should attempt them in this order:

**Tier 1 — Structural integrity** (cheapest, ~2s, FP rate ~0%, ROI ★★★★★): `axe-core(wcag2a, wcag2aa, wcag21a, wcag21aa, wcag22aa, best-practice).violations.filter(impact ∈ {critical, serious}) == [] AND parses_as_valid_html5 AND no_console_errors`. Catches ~90% of real-world a11y bug reports. **Tier 2 — Full WCAG 2.2 AA conformance**: full axe + ACT + experimental tags + IBM achecker (different rule engine adds 5–10% recall) + tab-order graph capture (Playwright Tab walk asserting completeness, no traps, DOM order ≈ visual order via Levenshtein) + focus-visible screenshot diff with ≥3:1 contrast change. **Tier 3 — Core Web Vitals all good**: LHCI median-of-5 with `LCP≤2500ms, CLS≤0.10, TBT≤200ms, FCP≤1800ms` plus scripted-INP-p99≤200ms via `web-vitals` attribution build plus LoAF `blockingDuration` p95≤100ms plus `non-composited-animations.score==1`. **Tier 4 — APCA perceptual contrast**: ∀ text element, |APCA.Lc| ≥ thresholdFor(fontSize, fontWeight) per APCA lookup table (Lc 75 body preferred, Lc 60 minimum body, Lc 45 large, Lc 30 floor, Lc 15 non-text), polarity-corrected on composited pixel colors (handle alpha + stacking contexts; `Myndex/apca-w3` Apache-2.0). Stricter than axe color-contrast and catches the orange-button / thin-font failures that pass AA but are perceptually weak. **Tier 5 — Motion + reduced-motion compliance**: two Playwright runs, `emulateMedia({reducedMotion:'reduce'})` and `'no-preference'`; reject any animation where reduced-motion variant is visually identical to default; under no-preference, durations ∈ Material bands, easing ∈ allowlist, no flash >3Hz, no autoplay >5s, frame-jank-rate <2%.

The bottom-line opinionated pipeline: **axe-core (max config) + IBM achecker (differential coverage) + Playwright tab-walk + LHCI 5-run-median + LoAF-instrumented INP + APCA Lc≥60 body / ≥45 large + reduced-motion diff**, all driven from Rust via chromiumoxide so there's no Node runtime in the gate. Reject builds that fail Tier 1–3; warn on Tier 4–5 for two weeks then promote to hard fail. Skip Lighthouse's accessibility *score* (lossy aggregation over an axe subset) — assert on underlying axe violations directly. Skip WAVE/Tenon/Stark — they don't add deterministic signal beyond axe + IBM. Run 5 runs minimum and take median; single-run LCP/CLS variance is ±5%, pin Chrome version, block third-party domains in test profile. INP in lab is synthetic — real INP requires field RUM via CrUX or self-hosted `web-vitals`; use TBT as lab proxy and synthetic interaction script for the gate.

Normalize all tool outputs to **EARL+JSON-LD** (Evaluation and Report Language) for cross-tool aggregation. axe-reporter-earl, IBM Equal Access, and the W3C report tool all consume/emit it. UiGate's unified report shape: `UiGateReport { tier, passed, violations: Vec<Finding> }` with `Finding { rule_id, wcag_sc, impact, selector, snippet, fix_hint, source_tool }` — stable JSON schema enables deterministic diffing across runs and feeds the agent's retry context.

## Synthesis: what to build first, what to defer

Build in this order. **Months 1–2**: chromiumoxide + Browserless multi pool + the eight-loop sensor harness emitting unified JSON, with Tiers 1–3 hard gates (axe + IBM + LHCI), odiff visual regression with reg-suit-compatible output, and the Wallace + custom analyzer extraction pipeline producing W3C 2025.10 tokens plus the area-weighted token adherence score. This gives you the deterministic verifier core before any LLM judge. **Months 3–4**: the disjoint-family judge panel (Claude Opus 4.5 + LLaVA-Critic-72B + Prometheus-Vision) with pairwise BT aggregation, Design2Code-style Block-Match/Position/Text floor metrics, the AIM layout-metric soft loop, DeepGaze IIE + UMSI++ saliency loop, the held-out human canary set with Krippendorff α≥0.8 monitoring, and the Tier 4–5 APCA + motion gates as warnings. **Months 5–6**: the flywheel — TensorZero traces feeding pairwise preference mining and AST pattern extraction, MIPROv2 over retrieval queries and judge rubrics and clarifying prompts, bandit routing over pipeline arms, curriculum-from-failures synthesis. **Month 7+**: once ≥10k repair pairs accumulate, RFT a 7B-class post-processor on Fireworks. Defer the custom UI judge until ≥10k human-verified preferences exist and frontier-judge cost is the binding constraint.

The **single highest-leverage architectural commitment**: pairwise comparison aggregated via Bradley–Terry against a fixed anchor, with constrained per-loop thresholds rather than weighted sums, run by a panel of disjoint-family judges with mandatory `(A,B)`/`(B,A)` swap-and-discard, evaluated against a frozen UICrit-style human canary with Krippendorff monitoring and a quarterly rubric rotation. Everything else is plumbing on top of that backbone. The literature and the production stacks converge on this architecture from independent angles — academic alignment work (PoLL, MLLM-as-a-Judge, Constrained-RLHF), Chatbot Arena methodology, Vercel's composite stack, and the basic Ashby-cybernetic argument all point to the same shape. The one failure mode this architecture cannot prevent is the agent learning to imitate the judge panel itself; that's why the rotating held-out canary and the meta-eval are non-optional, and why Krakovna's specification-gaming list belongs in your CI as a continuously-updated red-team prompt set.

The deepest open problem the field hasn't solved and UiGate likely will not solve alone: **there is no peer-reviewed reference-free benchmark for "good design" with high inter-rater reliability and large scale**. WebDev Arena confounds aesthetic with functional, UICrit is too small and mobile-only, DesignBench shows MLLMs miss 73% of UI defects. The pragmatic answer is the composite stack above; the strategic answer is that UiGate's private preference dataset, accumulated over thousands of runs with user edits as labeled examples, is potentially a more rigorous "good design" benchmark than anything currently published. That is the asset to compound on.
