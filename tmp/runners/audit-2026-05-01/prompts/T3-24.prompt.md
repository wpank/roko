# T3-24: Lower global body limit to 4 MiB; per-route overrides

## Task
Lower the global request body limit from 32 MiB to 4 MiB in `roko-serve`. Add per-endpoint overrides for routes that legitimately accept larger bodies (uploads, log shippers).

## Runner Context
Runner audit-2026-05-01, group T3. No dependencies. Wave 1.

## Source plan
`tmp/subsystem-audits/implementation-plans/13-tier3-security-hardening.md` § T3-24.

## Why
32 MiB is too generous for general API traffic. Most prompts are <200 KiB; 32 MiB enables a memory-pressure DoS by spamming oversized bodies. Lower the global; raise specific endpoints that need more.

## Current code

`crates/roko-serve/src/routes/mod.rs:183`:

```rust
router
    .layer(RequestBodyLimitLayer::new(32 * 1024 * 1024))
```

## Exact changes

### 1. Lower global limit

`crates/roko-serve/src/routes/mod.rs:183`:

```rust
router
    .layer(RequestBodyLimitLayer::new(4 * 1024 * 1024))   // 4 MiB global
```

### 2. Per-route overrides

For routes that need more than 4 MiB, add a `RequestBodyLimitLayer::new(N)` to the route. Note: per-route limits **override** the global only if the route layer is applied **inner-to-outer** correctly. In axum, `route_layer` or `.route(... .layer(...))` works.

Apply these overrides:

#### Webhooks (smaller, not larger)

`crates/roko-serve/src/routes/webhooks.rs::public_routes()`:

```rust
.route(
    "/webhooks/github",
    post(github_webhook).layer(RequestBodyLimitLayer::new(1 * 1024 * 1024)), // 1 MiB
)
.route(
    "/webhooks/slack",
    post(slack_webhook).layer(RequestBodyLimitLayer::new(256 * 1024)), // 256 KiB
)
```

`crates/roko-serve/src/routes/webhooks.rs::authenticated_routes()`:

```rust
.route(
    "/api/webhooks/generic",
    post(generic_webhook).layer(RequestBodyLimitLayer::new(64 * 1024)), // 64 KiB
)
```

#### Inference / agent register (smaller)

`crates/roko-serve/src/routes/inference.rs::routes()`:

```rust
.route(
    "/inference/complete",
    post(complete).layer(RequestBodyLimitLayer::new(4 * 1024 * 1024)), // explicit 4 MiB
)
```

(Same as global; explicit for documentation.)

`crates/roko-serve/src/routes/agents.rs::routes()`:

```rust
.route(
    "/agents/register",
    post(register_agent).layer(RequestBodyLimitLayer::new(256 * 1024)), // 256 KiB
)
```

#### Larger-body routes (raise above global)

Search:

```bash
rg 'fn (upload|put_(plan|prd|workspace)|post_workspace_files)' crates/roko-serve/src/routes/ -n
```

For each upload-like route, set a higher limit (e.g. 16 or 32 MiB) explicitly. Example for a hypothetical workspace upload:

```rust
.route(
    "/workspaces/:id/files/upload",
    post(upload_workspace_file).layer(RequestBodyLimitLayer::new(32 * 1024 * 1024)),
)
```

If no upload routes exist, that's fine — skip this step and log "no upload routes found."

### 3. Add a regression test

`crates/roko-serve/src/routes/mod.rs` (in a `#[cfg(test)]` module) or in
`crates/roko-serve/tests/body_limit.rs`:

```rust
#[tokio::test]
async fn rejects_oversize_body_against_global_limit() {
    let app = build_test_app().await;  // existing helper
    let body = vec![b'x'; 5 * 1024 * 1024]; // 5 MiB > 4 MiB global
    let req = axum::http::Request::builder()
        .method("POST")
        .uri("/api/health")
        .body(axum::body::Body::from(body))
        .unwrap();
    let resp = app.oneshot(req).await.unwrap();
    assert_eq!(resp.status(), axum::http::StatusCode::PAYLOAD_TOO_LARGE);
}
```

Use whatever `build_test_app` helper already exists; do not add a new one.

## Acceptance Criteria

- Global `RequestBodyLimitLayer::new(...)` is `4 MiB`.
- Per-route caps applied to webhooks (github 1 MiB, slack 256 KiB, generic 64 KiB).
- Per-route caps applied to inference complete (explicit 4 MiB) and agent register (256 KiB).
- Upload routes (if any) raise above 4 MiB explicitly.
- Regression test asserts 413 for >4 MiB body.

## Do NOT

- Do NOT raise the global limit to accommodate one upload route. Override that one route.
- Do NOT remove `RequestBodyLimitLayer` entirely.
- Do NOT add per-route caps to `GET` routes (body limits are about request bodies; GETs typically have empty bodies).
- Do NOT skip the test.
- Do NOT couple this with T3-23 (rate limiting); they are separate batches.
- Do NOT change the body-extractor type used by handlers (`Json`, `Bytes`, `Form`, etc.).
