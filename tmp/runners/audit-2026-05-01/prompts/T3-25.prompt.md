# T3-25: PORT env changes port only; bind stays from config

## Task
Stop coercing the bind to `0.0.0.0` when `PORT` is set. The bind address should come from `serve.bind` (or the constructor's `addr`), and `PORT` should only override the **port** portion.

## Runner Context
Runner audit-2026-05-01, group T3. No dependencies. Wave 1.

## Source plan
`tmp/subsystem-audits/implementation-plans/13-tier3-security-hardening.md` § T3-25.

## Why
Cloud platforms (Railway, Fly) set `PORT` and expect the server to listen on `0.0.0.0:$PORT`. But `PORT` can also leak into local-dev sessions; today, that silently exposes the server publicly without `validate_bind_safety` having any way to tell the difference.

The fix: `PORT` overrides the port only. If a deployment really needs `0.0.0.0`, set `serve.bind = "0.0.0.0"` explicitly in config. `validate_bind_safety` then catches it correctly.

## Exact changes

### 1. Replace the `PORT` handling

`crates/roko-serve/src/lib.rs:230-240`:

```rust
let addr = if let Ok(env_port) = std::env::var("PORT") {
    let p: u16 = env_port
        .parse()
        .context("PORT env var must be a valid u16")?;
    // PORT overrides the port portion of self.addr only.
    // To bind publicly, set `serve.bind = "0.0.0.0"` (or pass --bind 0.0.0.0)
    // explicitly. This is gated by `validate_bind_safety` further down.
    let bind_host = self
        .addr
        .rsplit_once(':')
        .map(|(host, _)| host.to_string())
        .unwrap_or_else(|| "127.0.0.1".to_string());
    info!("PORT env var detected ({p}); binding to {bind_host}:{p}");
    format!("{bind_host}:{p}")
} else {
    self.addr.clone()
};
```

### 2. Update the `roko init` template

`crates/roko-cli/src/init.rs`: ensure the rendered `[server]` block has an explicit `bind`:

```toml
[server]
# Bind address. "127.0.0.1" for local-only.
# Set "0.0.0.0" for cloud platforms (Railway / Fly / etc.); requires
# `serve.auth.enabled = true` or `serve.acknowledge_public_risk = true`.
bind = "127.0.0.1"
```

If the template generates this from a struct, override `server.bind = "127.0.0.1"` explicitly before serialization.

### 3. Audit deploy / Dockerfile assets

If there's a Dockerfile or Railway config that sets `PORT` without setting `bind`, update it. Search:

```bash
rg 'ENV PORT|RAILWAY_|FLY_|^bind ?=' Dockerfile* railway.toml fly.toml docker/ 2>/dev/null
```

For each cloud-deploy artifact, ensure either:
- `bind = "0.0.0.0"` is set in the deployed `roko.toml`, or
- An env var override (`ROKO_SERVER_BIND=0.0.0.0`) is set alongside `PORT`.

If you find a discrepancy, document in the batch journal but **do not** fix the deploy artifacts in this batch (separate concern; D-track territory).

### 4. Test

Add to `crates/roko-serve/tests/port_env.rs` (or to an existing `lib.rs` test module):

```rust
#[test]
fn port_env_does_not_force_public_bind() {
    use roko_serve::ServeServer; // or whichever public type owns `addr`
    std::env::set_var("PORT", "4444");

    let server = ServeServer::default(); // assume default addr is 127.0.0.1:6677
    let addr = server.compute_addr();   // helper to compute addr without binding
    assert!(addr.starts_with("127.0.0.1:"), "got {addr}");
    assert!(addr.ends_with(":4444"), "got {addr}");

    std::env::remove_var("PORT");
}

#[test]
fn explicit_zero_bind_kept() {
    std::env::set_var("PORT", "4444");
    let server = ServeServer::with_addr("0.0.0.0:6677");
    let addr = server.compute_addr();
    assert_eq!(addr, "0.0.0.0:4444");
    std::env::remove_var("PORT");
}
```

If `ServeServer` doesn't have a `compute_addr` helper, extract the addr-computation logic into a private fn that takes `(self_addr: &str, port_env: Option<u16>) -> String`, and test that fn directly.

- `validate_bind_safety` (line 641) is unchanged.

## Do NOT

- Do NOT remove the `PORT` env var handling. Cloud platforms need it.
- Do NOT introduce a new env var (e.g. `BIND`) in this batch — out of scope.
- Do NOT change `validate_bind_safety`.
- Do NOT silently default to `0.0.0.0` if `bind` is unset; the existing `127.0.0.1` default in `serve.bind` is correct.
- Do NOT modify Dockerfiles or `railway.toml` in this batch unless the file already mishandles `PORT`. Log instead.
- Do NOT add a "deprecated" warning if PORT is set without explicit bind.
