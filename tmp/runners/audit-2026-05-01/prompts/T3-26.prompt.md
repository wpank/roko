# T3-26: Add WebSocket message size limits

## Task
Add `max_message_size` and `max_frame_size` to every `WebSocketUpgrade` call in `roko-serve`. WebSocket messages are not subject to the HTTP body limit; an unbounded WS message can exhaust memory.

## Runner Context
Runner audit-2026-05-01, group T3. No dependencies. Wave 1.

## Source plan
`tmp/subsystem-audits/implementation-plans/13-tier3-security-hardening.md` § T3-26.

## Why
HTTP body limits don't apply to WebSocket frames. Without explicit caps, a client can send a multi-GB binary frame and OOM the server. Set sensible per-endpoint caps.

## Discover upgrade points

```bash
rg 'WebSocketUpgrade|on_upgrade|axum::extract::ws' crates/roko-serve/src/ -n
```

Expected sites (verify):

| Endpoint | Source | Recommended caps |
|---|---|---|
| `/ws` (general) | `routes/ws.rs` | message 1 MiB, frame 256 KiB |
| `/api/terminal/sessions/:id/io` | `terminal.rs` | message 256 KiB, frame 64 KiB |
| `/workflow/events` (if WS) | `routes/mod.rs` (workflow_sse_handler) | only if WS |
| `/api/terminal/sessions/:id/events` | `terminal/events_ws.rs` (created in S-term-1) | message 64 KiB, frame 16 KiB |

## Exact changes (for each upgrade site)

### Pattern

Today (before):

```rust
async fn handler(ws: WebSocketUpgrade, State(state): State<Arc<AppState>>) -> Response {
    ws.on_upgrade(|socket| handle_socket(socket, state))
}
```

After:

```rust
async fn handler(ws: WebSocketUpgrade, State(state): State<Arc<AppState>>) -> Response {
    ws.max_message_size(1 * 1024 * 1024)   // 1 MiB
        .max_frame_size(256 * 1024)        // 256 KiB
        .on_upgrade(|socket| handle_socket(socket, state))
}
```

### General `/ws` route

`crates/roko-serve/src/routes/ws.rs`:

```rust
ws.max_message_size(1 * 1024 * 1024)
    .max_frame_size(256 * 1024)
    .on_upgrade(...)
```

### Terminal IO socket

`crates/roko-serve/src/terminal.rs` (or `terminal/io_ws.rs` if split):

```rust
ws.max_message_size(256 * 1024)
    .max_frame_size(64 * 1024)
    .on_upgrade(...)
```

(Smaller because terminal IO is chunked anyway; legitimate frames stay under 4 KiB.)

### Workflow events (if it's a WS endpoint)

```bash
rg 'workflow_sse_handler|workflow.*events' crates/roko-serve/src/routes/ -n
```

If it's actually SSE (Server-Sent Events, not WebSocket), **skip it** — SSE is one-way text and not subject to this issue. Note in journal.

If it's WS:

```rust
ws.max_message_size(64 * 1024)
    .max_frame_size(16 * 1024)
    .on_upgrade(...)
```

### Don't forget the new `/events` endpoint (S-term-1)

If `S-term-1` has not landed yet, this batch can skip. If it has, the `/events` WebSocket also needs caps. Look at `crates/roko-serve/src/terminal/events_ws.rs`.

## Acceptance Criteria

- Every `WebSocketUpgrade` `on_upgrade(...)` call has `.max_message_size(...)` and `.max_frame_size(...)` immediately before it.
- General `/ws` caps: 1 MiB / 256 KiB.
- Terminal IO caps: 256 KiB / 64 KiB.
- Terminal events caps (if endpoint exists): 64 KiB / 16 KiB.
- Workflow events caps (if WS): 64 KiB / 16 KiB.
- Test added or "manual verification only" logged.

## Do NOT

- Do NOT set caps below 16 KiB (legitimate streaming output gets truncated).
- Do NOT use `usize::MAX` "for now."
- Do NOT add a global WS limit middleware (no such thing in axum; per-upgrade is the chokepoint).
- Do NOT couple this with HTTP body limits (T3-24).
- Do NOT mix this with auth/rate-limit changes (T3-22, T3-23).
- Do NOT skip the IO socket; that's the highest-traffic WS endpoint.
