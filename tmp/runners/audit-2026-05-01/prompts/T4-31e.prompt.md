## Exact changes

### Case A: Cursor uses shared OpenAI-compatible parser

Add tests that document the contract:

```rust
#[test]
fn cursor_response_uses_openai_compatible_parser() {
    let body = r#"{
        "id": "x",
        "choices": [{"message": {"role": "assistant", "content": "hi"}}],
        "usage": {"prompt_tokens": 0, "completion_tokens": 0, "total_tokens": 0}
    }"#;
    let parsed = roko_agent::translate::openai::parse_response(body).unwrap();
    let u = parsed.usage.expect("usage block present");
    assert_eq!(u.input_tokens, Some(0));
    assert_eq!(u.output_tokens, Some(0));
}

#[test]
fn cursor_response_no_usage_block_yields_none() {
    let body = r#"{
        "id": "x",
        "choices": [{"message": {"role": "assistant", "content": "hi"}}]
    }"#;
    let parsed = roko_agent::translate::openai::parse_response(body).unwrap();
    assert!(parsed.usage.is_none());
}
```

### Case B: Cursor has its own parser

Delete the duplicate and route through `translate::openai::parse_response`. Cursor returns OpenAI-compatible bodies; the only adapter-specific concerns are auth (Cursor session token vs API key) and base URL.

If Cursor's response carries any extension fields not in the OpenAI shape (e.g. custom rate-limit headers, billing notes), keep parsing them in the adapter but **do not** duplicate usage parsing.
