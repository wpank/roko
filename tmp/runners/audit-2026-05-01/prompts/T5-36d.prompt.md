# T5-36d: Migrate diagnosis routes to ModelCallService

## Task
Replace route-local HTTP construction in `crates/roko-serve/src/routes/diagnosis.rs` with `state.model_call_service`. Diagnosis runs LLM-driven analysis (config-doctor extensions, error explanation, health check); migrate the LLM dispatch parts.

## Runner Context
Runner audit-2026-05-01, group T5. No dependencies. Wave 1.

## Source plan
`tmp/subsystem-audits/implementation-plans/22-dispatch-streaming-completion.md` § Phase 1.

## Read first

```bash
rg 'fn |reqwest::Client|model_call_service' crates/roko-serve/src/routes/diagnosis.rs -n
```

Diagnosis routes typically:
- `POST /api/diagnosis/explain-error` — given an error, ask the model for an explanation.
- `POST /api/diagnosis/health-llm` — LLM-based health/regression analysis.
- `POST /api/diagnosis/...` — others.

Only the LLM-dispatch parts migrate. Diagnostics that read local files / state without invoking a model don't change.

## Exact changes

For each LLM-invoking handler, follow the T5-36a pattern:

1. Build `ModelCallRequest`.
2. Call `state.model_call_service.call(...)`.
3. Map the response into the route's response type.

If the route currently has hardcoded model selection (e.g. `"claude-haiku"` for cheap diagnosis), keep that hardcode in the route as a default but allow the request body to override:

```rust
let model = req.model.unwrap_or_else(|| state.config.agent.diagnosis_model.clone()
    .unwrap_or_else(|| "claude-haiku".to_string()));
```

If `cfg.agent.diagnosis_model` doesn't exist, just use the default.
