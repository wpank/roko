# T5-37: Feature-gate dispatch_direct + migrate production callers

## Task
Move `crates/roko-cli/src/dispatch_direct.rs` behind a `legacy-direct-dispatch` cargo feature (off by default). Migrate production callers (`chat_inline.rs`, `unified.rs`, `marketplace.rs`, `identity_economy_markets.rs`) to `ModelCallService`.

## Runner Context
Runner audit-2026-05-01, group T5. No dependencies. Wave 1.

## Source plan
`tmp/subsystem-audits/implementation-plans/22-dispatch-streaming-completion.md` § Phase 2.

## Why
`dispatch_direct.rs` bypasses `ModelCallService`, the safety layer, and the streaming contract. It's the canonical example of a "fifth dispatch path" anti-pattern. Production must not use it; tests may.

**Important**: this batch does NOT delete `dispatch_direct.rs`. Deletion is a follow-up after the static check is green for 30 days. This batch:
1. Feature-gates the module.
2. Migrates production callers.
3. Adds the static check.

## Step 1: Feature-gate the module

`crates/roko-cli/Cargo.toml`:

```toml
[features]
default = []
legacy-direct-dispatch = []
```

`crates/roko-cli/src/lib.rs`:

```rust
#[cfg(feature = "legacy-direct-dispatch")]
pub mod dispatch_direct;
```

(Same gate on the re-export if any: `#[cfg(feature = "legacy-direct-dispatch")] pub use dispatch_direct::*;`.)

## Step 3: Migrate `chat_inline.rs`

```bash
rg 'dispatch_direct' crates/roko-cli/src/chat_inline.rs -n
```

For each call site, replace `dispatch_direct::run(...)` (or whatever the entry is) with `self.model_call_service.call(...)` or `.stream(...)`.

If `chat_inline.rs` is mostly migrated already (R3_D02 partly addressed this), some `dispatch_direct` references may already be gated by `#[cfg(feature = "legacy-direct-dispatch")]`. Verify each remaining site.

If a call site is reached only on a fallback path (e.g. when `model_call_service` errored), **delete the fallback**. That's the silent-fallback anti-pattern.

## Step 4: Migrate `unified.rs`

Same pattern. The single-prompt entry point should dispatch through `ChatAgentSession` (which uses `ModelCallService` internally) per R3_D02.

## Step 5: Migrate `marketplace.rs` and `identity_economy_markets.rs`

These are in `roko-chain`, not `roko-cli`. The chain crate calls a model to judge marketplace decisions / identity changes.

Two options:

(a) **Pass `ModelCallService` in via dependency injection.** The chain crate doesn't depend on `roko-agent` today; if adding the dep is acceptable, the call sites become:

```rust
pub struct Marketplace {
    // ...
    model_call_service: Arc<dyn ModelCallService>,
}

impl Marketplace {
    pub async fn evaluate(&self, ...) -> Result<...> {
        let req = ModelCallRequest { ... };
        let resp = self.model_call_service.call(req).await?;
        // ...
    }
}
```

(b) **Lift the LLM call out of the chain crate.** The caller (in `roko-cli` or `roko-serve`) does the LLM call and passes the result into a pure `Marketplace::evaluate(decision_input)`.

Pick **(a)** if the chain crate has many model-using methods (architectural shift would be huge). Pick **(b)** if there are only 1-2 sites and the chain crate is supposed to remain LLM-free.

If neither pattern fits cleanly, **feature-gate the chain crate's caller** behind `legacy-direct-dispatch` too — this defers the chain migration to a follow-up:

```rust
#[cfg(feature = "legacy-direct-dispatch")]
mod legacy_dispatch_direct {
    use roko_cli::dispatch_direct;
    // existing code
}
```

Document the decision in the batch journal.
