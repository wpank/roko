# T5-39: Add RunnerBudgetGuardrail to Ollama dispatch path

## Task
Wrap the Ollama dispatch loop (in `orchestrate.rs`, around line 15910-16011) in `RunnerBudgetGuardrail` (or `TaskRunner`) so it can't exceed iteration / time / cost caps. Other dispatch paths already have this; Ollama doesn't.

## Runner Context
Runner audit-2026-05-01, group T5. No dependencies. Wave 1.

## Source plan
`tmp/subsystem-audits/implementation-plans/15-tier5-architectural.md` § T5-39.

## Why
A misconfigured local Ollama agent can loop indefinitely. Local GPU OOMs. Most other dispatch paths wrap their tool loop in `TaskRunner` / `RunnerBudgetGuardrail`. The Ollama path was added separately and missed this.

## Exact changes

### Option A: Wrap in `TaskRunner`

If Ollama's loop matches the `ToolLoop` trait `TaskRunner` expects:

```rust
let runner = TaskRunner::new(
    OllamaToolLoop::new(...),
    RunnerBudgetGuardrail::from_config(&task_runner_config),
);
let outcome = runner.run().await?;
```

Replace the inline loop body with this two-line construction.

### Option B: Adapter for mismatched signatures

If `OllamaToolLoop`'s API doesn't match `ToolLoop` exactly:

```rust
// crates/roko-cli/src/dispatch/ollama_adapter.rs (new)
pub struct OllamaToolLoopAdapter {
    inner: OllamaToolLoop,
}

#[async_trait]
impl ToolLoop for OllamaToolLoopAdapter {
    type Outcome = OllamaOutcome;
    async fn step(&mut self) -> Result<StepOutcome<Self::Outcome>, ToolLoopError> {
        self.inner.iterate().await
            .map_err(ToolLoopError::from)
    }
    fn is_done(&self) -> bool { self.inner.is_done() }
}

let runner = TaskRunner::new(
    OllamaToolLoopAdapter::new(/* ... */),
    RunnerBudgetGuardrail::from_config(&cfg.runner),
);
let outcome = runner.run().await?;
```

### Option C: Inline guardrail (minimal change)

If `TaskRunner` integration is too invasive (e.g. Ollama's loop has a fundamentally different shape), add the guardrail directly:

```rust
let mut guardrail = RunnerBudgetGuardrail::from_config(&cfg.runner);
loop {
    guardrail.tick()?;   // returns BudgetExceeded if over caps
    let step = ollama_iterate(...).await?;
    if step.is_done() { break; }
}
```

`tick()` increments iteration count, checks elapsed time, checks cost (if cost is plumbed). Returns a typed error when any cap is exceeded.

## Pick one

Read the existing Ollama code carefully. If `TaskRunner` integration is a 5-line change, do A. If A would require modifying `TaskRunner` or a major refactor, do C. **Don't fork `TaskRunner`.**

## Acceptance Criteria

- Ollama dispatch path wrapped in `TaskRunner` or has inline `RunnerBudgetGuardrail`.
- Iteration / time caps enforced.
- Test verifies cap is honored.
- No `TaskRunner` fork.

## Do NOT

- Do NOT fork `TaskRunner` to fit Ollama's signature. Use an adapter.
- Do NOT crank the cap to "long enough." The point is bounded execution.
- Do NOT skip this for "local-only" runs. Local Ollama can still consume the whole machine.
- Do NOT wrap in a separate config struct just for Ollama; use `cfg.runner.budget`.
- Do NOT change `RunnerBudgetGuardrail` API.
