# T5-40c: RunLedger — command events for workflow-bound terminal sessions

## Task
Add `RunLedgerEntry::Command { session_id, event, ts_ms }` for terminal sessions that are part of a workflow run. Append on each `CommandEvent` emission. Used by resume to replay terminal scroll position.

## Runner Context
Runner audit-2026-05-01, group T5. Depends on T5-40a. Wave 2.

## Source plan
`tmp/subsystem-audits/implementation-plans/24-runtime-ledger-migration.md` § Slice 3.

## Why
Plan 26 / S-term-1 emits typed `CommandEvent`s on a `/events` WebSocket. For workflow-bound sessions, these events should also be durable in the run ledger so resume can replay them.

A terminal session is "workflow-bound" if it was created with a workflow context (e.g. `POST /api/terminal/sessions { workflow_id: "..." }`). If the field doesn't exist, add it as optional.

## Exact changes

### 1. Add ledger variant

```rust
// crates/roko-runtime/src/run_ledger.rs
use roko_serve::command_events::CommandEvent;

RunLedgerEntry::Command {
    session_id: String,
    event: CommandEvent,
    ts_ms: i64,
}

impl RunLedger {
    pub fn record_command(&mut self, session_id: String, event: CommandEvent) {
        self.append(RunLedgerEntry::Command {
            session_id,
            event,
            ts_ms: chrono::Utc::now().timestamp_millis(),
        });
    }
}
```

(If `RunLedger` taking a dep on `roko-serve::command_events` introduces a cycle, move `CommandEvent` to `roko-core` and re-export from `roko-serve`. Either way, the typed event is shared, not duplicated.)

### 2. Wire from terminal session

`crates/roko-serve/src/terminal.rs`: where `CommandEvent`s are emitted to the WebSocket subscribers, also emit to the ledger if the session is workflow-bound:

```rust
async fn emit_event(&self, event: CommandEvent) {
    // existing: broadcast to subscribers
    self.events_tx.send(event.clone());

    // new: persist in run ledger if workflow-bound
    if let Some(workflow_id) = self.workflow_id.as_ref() {
        if let Some(ledger) = self.app_state.run_ledger_for(workflow_id) {
            ledger.lock().await.record_command(self.session_id.clone(), event);
        }
    }
}
```

`run_ledger_for(workflow_id)` returns `Option<Arc<Mutex<RunLedger>>>`. If the lookup helper doesn't exist, add it.

### 3. Resume replay

`crates/roko-cli/src/runner/resume.rs` (when this batch lands T5-40d will fold in checkpoint+command replay together; T5-40c just emits the entries):

For now, just confirm `RunLedger::commands_for_session(session_id)` returns the recorded events:

```rust
impl RunLedger {
    pub fn commands_for_session(&self, session_id: &str) -> Vec<&CommandEvent> {
        self.entries()
            .filter_map(|e| match e {
                RunLedgerEntry::Command { session_id: s, event, .. } if s == session_id => Some(event),
                _ => None,
            })
            .collect()
    }
}
```

## Do NOT

- Do NOT bundle with T5-40a/b/d.
- Do NOT record command events for non-workflow sessions (would balloon ledger size).
- Do NOT change the WebSocket subscriber broadcast (still emit to `/events` clients).
- Do NOT make the ledger write block the WebSocket emit; if the lock is contended, log and continue. (Plan 24 § Phase 5 / S-ledger-1 may make this fail-closed for correctness-critical writes; commands aren't critical.)
- Do NOT replicate event payloads through both ACP and ledger; ledger is the durable record.
