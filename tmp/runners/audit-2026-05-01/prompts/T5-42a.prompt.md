# T5-42a: Anthropic provider — native structured message history

## Task
Anthropic adapter must accept `Vec<Message>` and translate to native messages array `[{role, content: [...blocks]}]` with proper tool-use / tool-result blocks. Replace any flat-string prompt rendering with structured.

## Runner Context
Runner audit-2026-05-01, group T5. No dependencies. Wave 1.

## Source plan
`tmp/subsystem-audits/implementation-plans/15-tier5-architectural.md` § T5-42.

## Why
Some adapters still build a flat string ("user: foo\nassistant: bar\n...") instead of using the provider's structured message format. This loses role boundaries and tool-call structure on multi-turn conversations.

## Reference

`crates/roko-agent/src/translate/openai.rs` — already does this for OpenAI-compatible. Read first.

## Anthropic message shape

```json
{
  "model": "claude-sonnet-4-6",
  "max_tokens": 1024,
  "messages": [
    {"role": "user", "content": [{"type": "text", "text": "hello"}]},
    {"role": "assistant", "content": [{"type": "text", "text": "hi"}]},
    {"role": "user", "content": [
      {"type": "text", "text": "use the calc tool"},
    ]},
    {"role": "assistant", "content": [
      {"type": "tool_use", "id": "calc-1", "name": "calc", "input": {"x": 1}}
    ]},
    {"role": "user", "content": [
      {"type": "tool_result", "tool_use_id": "calc-1", "content": "2"}
    ]}
  ]
}
```

## Exact changes

### 1. Define translation

`crates/roko-agent/src/providers/anthropic.rs` (or sibling):

```rust
use serde_json::{json, Value};
use roko_agent::{Message, Role, ContentBlock};

pub(crate) fn translate_messages(msgs: &[Message]) -> Vec<Value> {
    msgs.iter().map(translate_message).collect()
}

fn translate_message(m: &Message) -> Value {
    let role = match m.role {
        Role::User => "user",
        Role::Assistant => "assistant",
        Role::System => panic!("system messages must be passed via top-level `system` field"),
        Role::Tool => "user",   // tool results are user-side in Anthropic's API
    };
    let content: Vec<Value> = m.content.iter().map(translate_block).collect();
    json!({"role": role, "content": content})
}

fn translate_block(b: &ContentBlock) -> Value {
    match b {
        ContentBlock::Text(text) => json!({"type": "text", "text": text}),
        ContentBlock::ToolUse { id, name, input } => json!({
            "type": "tool_use",
            "id": id,
            "name": name,
            "input": input,
        }),
        ContentBlock::ToolResult { tool_use_id, content } => json!({
            "type": "tool_result",
            "tool_use_id": tool_use_id,
            "content": content,
        }),
    }
}
```

If `ContentBlock` doesn't exist or has a different shape, **read** `roko-agent`'s actual definition first. Match exactly what's in the codebase.

### 2. Use in request builder

```rust
let body = json!({
    "model": req.model,
    "max_tokens": req.max_tokens,
    "system": req.system_prompt,    // top-level field, not inside messages
    "messages": translate_messages(&req.messages),
});
```

If the existing builder used a flat string, **replace** with the structured form. Don't keep both.

### 3. Tests

```rust
#[test]
fn anthropic_translates_multi_turn_history() {
    let msgs = vec![
        Message::user_text("hello"),
        Message::assistant_text("hi back"),
        Message::user_text("how are you"),
    ];
    let translated = translate_messages(&msgs);
    assert_eq!(translated.len(), 3);
    assert_eq!(translated[0]["role"], "user");
    assert_eq!(translated[1]["role"], "assistant");
    assert_eq!(translated[2]["role"], "user");
    assert_eq!(translated[0]["content"][0]["type"], "text");
    assert_eq!(translated[0]["content"][0]["text"], "hello");
}

#[test]
fn anthropic_translates_tool_call_round_trip() {
    let msgs = vec![
        Message::user_text("call the calc"),
        Message::assistant_tool_use("calc-1", "calc", serde_json::json!({"x": 1})),
        Message::user_tool_result("calc-1", "2"),
    ];
    let translated = translate_messages(&msgs);
    assert_eq!(translated[1]["content"][0]["type"], "tool_use");
    assert_eq!(translated[1]["content"][0]["id"], "calc-1");
    assert_eq!(translated[2]["content"][0]["type"], "tool_result");
    assert_eq!(translated[2]["content"][0]["tool_use_id"], "calc-1");
}
```

Constructor helpers (`Message::user_text`, etc.) live in `roko-agent`. If they don't exist for the test's needs, build them in a `#[cfg(test)]` helper module — don't add public API for tests only.

## Acceptance Criteria

- `translate_messages` produces native Anthropic messages array.
- Tool use + tool result blocks structured (not text-encoded).
- System prompt goes in top-level `system` field, not in messages.
- 2 tests cover multi-turn and tool-call round-trip.

## Do NOT

- Do NOT mix system messages into the messages array. Anthropic's `system` field is top-level.
- Do NOT bundle with T5-42b/c.
- Do NOT change `Message` / `ContentBlock` / `Role` types in `roko-agent`. If they don't expose what the translator needs, that's a separate batch.
- Do NOT keep the flat-string fallback "as defense in depth." Native structured is the only path.
- Do NOT change the streaming parser (T4-31a owns usage; T5-42a is request-side only).
