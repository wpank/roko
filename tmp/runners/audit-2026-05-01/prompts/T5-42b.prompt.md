## Gemini message shape

```json
{
  "model": "gemini-1.5-pro",
  "contents": [
    {"role": "user", "parts": [{"text": "hello"}]},
    {"role": "model", "parts": [{"text": "hi"}]},
    {"role": "user", "parts": [
      {"text": "use the calc tool"}
    ]},
    {"role": "model", "parts": [
      {"functionCall": {"name": "calc", "args": {"x": 1}}}
    ]},
    {"role": "user", "parts": [
      {"functionResponse": {"name": "calc", "response": {"result": "2"}}}
    ]}
  ],
  "systemInstruction": {"parts": [{"text": "You are helpful."}]}
}
```

Note the role convention: Gemini uses `"model"` (not `"assistant"`), and tool roles are encoded as user-side `functionResponse`.

## Exact changes

### 1. Translation

```rust
fn translate_message_to_content(m: &Message) -> serde_json::Value {
    let role = match m.role {
        Role::User => "user",
        Role::Assistant => "model",
        Role::Tool => "user",
        Role::System => panic!("system goes in systemInstruction"),
    };
    let parts: Vec<serde_json::Value> = m.content.iter().map(translate_block).collect();
    serde_json::json!({"role": role, "parts": parts})
}

fn translate_block(b: &ContentBlock) -> serde_json::Value {
    use serde_json::json;
    match b {
        ContentBlock::Text(text) => json!({"text": text}),
        ContentBlock::ToolUse { name, input, .. } => json!({
            "functionCall": {"name": name, "args": input}
        }),
        ContentBlock::ToolResult { tool_use_id: _, content } => {
            // `tool_use_id` is implicit in Gemini (matched by name + order).
            // Wrap result in {"response": ...} per the API.
            json!({
                "functionResponse": {
                    "name": "tool",  // ideally from a paired ToolUse; otherwise omit
                    "response": {"content": content},
                }
            })
        }
    }
}
```

If `ContentBlock::ToolResult` doesn't carry a `name` field (it might only have `tool_use_id`), pair it with the most recent `ToolUse` in the message list to recover the function name. If pairing is fragile, document the limitation and either: (a) require the caller to provide a name, or (b) fall back to `"tool"` literal.

### 2. System prompt

Gemini's system prompt is `systemInstruction`, top-level. Don't put it in `contents`.

```rust
let body = json!({
    "contents": req.messages.iter().map(translate_message_to_content).collect::<Vec<_>>(),
    "systemInstruction": {"parts": [{"text": req.system_prompt}]},
    "generationConfig": {"maxOutputTokens": req.max_tokens, "temperature": req.temperature},
});
```

### 3. Tests

```rust
#[test]
fn gemini_translates_multi_turn_history() {
    let msgs = vec![Message::user_text("hi"), Message::assistant_text("hello")];
    let parts: Vec<_> = msgs.iter().map(translate_message_to_content).collect();
    assert_eq!(parts[0]["role"], "user");
    assert_eq!(parts[1]["role"], "model");
    assert_eq!(parts[0]["parts"][0]["text"], "hi");
}

#[test]
fn gemini_translates_function_call_and_response() {
    let msgs = vec![
        Message::assistant_tool_use("calc-1", "calc", serde_json::json!({"x": 1})),
        Message::user_tool_result("calc-1", "2"),
    ];
    let parts: Vec<_> = msgs.iter().map(translate_message_to_content).collect();
    assert_eq!(parts[0]["parts"][0]["functionCall"]["name"], "calc");
    assert!(parts[1]["parts"][0]["functionResponse"].is_object());
}
```

## Acceptance Criteria

- Gemini request body uses native `contents` + `systemInstruction`.
- Role mapping: User → "user", Assistant → "model", Tool → "user".
- 2 tests cover multi-turn and function-call round-trip.

## Do NOT

- Do NOT use role `"assistant"` in Gemini bodies (it's `"model"`).
- Do NOT put system prompt in `contents`. Use `systemInstruction`.
- Do NOT bundle with T5-42a/c.
- Do NOT change `ContentBlock` types.
- Do NOT touch the response parser (T4-31c owns usage parsing).
