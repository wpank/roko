# T5-42c: Ollama chat endpoint — structured messages array

## Task
Ollama's `/api/chat` endpoint accepts a `messages: [{role, content}]` array. The adapter must use that, not flatten messages into a single `prompt` string.

## Runner Context
Runner audit-2026-05-01, group T5. No dependencies. Wave 1.

## Source plan
`tmp/subsystem-audits/implementation-plans/15-tier5-architectural.md` § T5-42.

## Why
Some adapters use Ollama's older `/api/generate` endpoint (single `prompt` field). The newer `/api/chat` endpoint preserves role boundaries with `messages: [...]`. Switch.

## Ollama chat shape

```json
{
  "model": "llama3.2",
  "messages": [
    {"role": "system", "content": "You are helpful."},
    {"role": "user", "content": "hello"},
    {"role": "assistant", "content": "hi"}
  ],
  "stream": true,
  "options": {"num_predict": 1024, "temperature": 0.7}
}
```

Ollama's chat API accepts roles `system`, `user`, `assistant`, `tool`. Tool calling support varies by model; if a model doesn't support tools, the adapter should reject `ToolUse` / `ToolResult` blocks early.

## Exact changes

### 1. Switch endpoint to `/api/chat`

If the adapter uses `/api/generate`:

```rust
// Before
let url = format!("{base}/api/generate");
let body = json!({"model": req.model, "prompt": flat_prompt(&req)});

// After
let url = format!("{base}/api/chat");
let body = json!({
    "model": req.model,
    "messages": req.messages.iter().map(translate_message).collect::<Vec<_>>(),
    "stream": req.streaming,
    "options": ollama_options(&req),
});
```

### 2. Translation

```rust
fn translate_message(m: &Message) -> serde_json::Value {
    let role = match m.role {
        Role::User => "user",
        Role::Assistant => "assistant",
        Role::System => "system",
        Role::Tool => "tool",
    };
    // Ollama's chat format is {role, content: <string>}; multimodal is
    // a separate `images` field. For text-only, flatten ContentBlocks
    // into a single string.
    let content = m.content.iter()
        .filter_map(|b| match b {
            ContentBlock::Text(t) => Some(t.as_str()),
            ContentBlock::ToolUse { .. } | ContentBlock::ToolResult { .. } => {
                // Ollama tool-call support is model-specific; for now,
                // text-encode the structure for adapter compatibility.
                None  // or serialize via a fallback path
            }
        })
        .collect::<Vec<_>>()
        .join("");
    serde_json::json!({"role": role, "content": content})
}
```

If the adapter needs to support tool-using Ollama models (e.g. llama3.1+ with tool calls), the chat API has a `tool_calls` array on assistant messages. Match the API exactly per Ollama docs.

### 3. System prompt

System messages can be the first element of the `messages` array (role `"system"`), or — if Ollama is treated as system-prompt-as-config — pulled into a top-level `system` field. Either is acceptable; pick what `req.system_prompt` already mandates.

### 4. Tests

```rust
#[test]
fn ollama_chat_request_uses_messages_array() {
    let req = test_request(vec![
        Message::user_text("hello"),
        Message::assistant_text("hi"),
    ]);
    let body = build_ollama_chat_body(&req);
    let messages = body["messages"].as_array().unwrap();
    assert_eq!(messages.len(), 2);
    assert_eq!(messages[0]["role"], "user");
    assert_eq!(messages[0]["content"], "hello");
    assert_eq!(messages[1]["role"], "assistant");
}

#[test]
fn ollama_endpoint_is_api_chat() {
    let url = build_ollama_url("http://localhost:11434", "/chat");
    assert_eq!(url, "http://localhost:11434/api/chat");
}
```

- Adapter dispatches via `/api/chat`.
- Body has `messages: [...]` array.
- Roles map correctly (user/assistant/system/tool).
- 2 tests cover the new shape.

- Do NOT bundle with T5-42a/b.
- Do NOT change Ollama URL config (`base_url` from `[providers.ollama]`).
- Do NOT touch the response parser (T4-31b owns usage).
- Do NOT add a tokenizer for prompt-length estimation here.
