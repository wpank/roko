# BI_01: Default share to --secret gist with scrubbed payload

## Task
Switch the GitHub Gist upload in `share.rs` from `--public` to `--secret` by default and add an opt-in `public: bool` parameter to `share_run` / `upload_gist`. The `LogScrubber` pass already runs in `share_run` — verify it is the only path to `upload_gist` and update the in-file test caller to pass the new argument.

## Runner Context
Group **BI_SEC**, batch 1 of 56. Closes MASTER-INDEX § **S5.5**. Deps: none.

## Problem
`crates/roko-cli/src/share.rs` calls `gh gist create --public ...` at line 156. A user who runs `roko run … --share` against a workspace transcript (full prompt, agent output, tool I/O) gets a public, world-indexable URL. Even though `share_run` already scrubs the markdown via `roko_core::obs::LogScrubber` (see `share.rs:25` import and `share.rs:49-54` `scrub_share_text`) plus heuristic long-hex / long-base64 passes, the `--public` default violates universal rule **S2** in `context-pack/00-RULES.md` ("Default to `--secret` and run `LogScrubber` over the payload first").

The default build wires `share_run` from exactly two sites:

- `crates/roko-cli/src/commands/util.rs:363-370` — production caller behind `roko run ... --share`.
- `crates/roko-cli/src/share.rs:348-356` — in-file unit test that asserts secret scrubbing.

The function signature does not currently expose visibility, so the CLI cannot opt back in without code changes. The fix is mechanical: flip the default to `--secret`, plumb a `public: bool` parameter through `share_run` → `upload_gist`, and have the in-file test caller pass `false`.

Promoting `public` to a CLI flag and a `serve.share.public` config field is **left to a follow-up batch**, since `commands/util.rs` and `roko-core/src/config/serve.rs` are out of write scope here. The validation pipeline that runs after merge will pick up the resulting compile error in `commands/util.rs`; that is the signal for the follow-up batch to land.


```rust

/// Upload markdown content as a GitHub Gist via `gh` CLI.
```
