# BI_03: Cap PTY session count and apply idle TTL

## Task
Bound `SessionManager`'s in-memory maps with a hard cap (`MAX_TERMINAL_SESSIONS = 16`) and a per-session idle TTL (`TERMINAL_IDLE_TTL_SECS = 1800`). Track a `last_used` timestamp per session, reject `create_session` over the cap with HTTP 429, and spawn a single background reaper task at first-session creation that destroys sessions whose `last_used` is older than the TTL.

## Runner Context
Group **BI_SEC**, batch 3 of 56. Closes MASTER-INDEX § **S5.7**. Deps: none. (Independent of BI_02 but lives in the same file; if BI_02 lands first, the typed-error work composes cleanly.)

## Problem
`SessionManager` (`crates/roko-serve/src/terminal.rs:176-194`) holds two unbounded maps:

```176:194:crates/roko-serve/src/terminal.rs
pub struct SessionManager {
    pub(crate) sessions: Mutex<HashMap<String, PtySession>>,
    pub(crate) session_info: Mutex<HashMap<String, SessionInfo>>,
    workdir: std::path::PathBuf,
    sess_generation: AtomicU64,
    command_event_emitter: TerminalCommandEventEmitter,
}

impl SessionManager {
    pub fn new(workdir: std::path::PathBuf) -> Self {
        Self {
            sessions: Mutex::new(HashMap::new()),
            session_info: Mutex::new(HashMap::new()),
            workdir,
            sess_generation: AtomicU64::new(0),
            command_event_emitter: TerminalCommandEventEmitter::new(),
        }
    }
```

Every `POST /api/terminal/sessions` call (handler at `terminal.rs:457-482`) inserts a new `PtySession` (which holds a live child process plus PTY file descriptors) into both maps. Nothing reaps idle sessions — `destroy_session` is only called explicitly by the REST DELETE handler or by `ws_terminal` when a websocket disconnects (`terminal.rs:521, 602-604`). An authenticated client (or a buggy frontend on reconnect) can leak processes and FDs until the OS rejects new spawns.

Universal rule **S4** in `00-RULES.md`: "Anything that spawns a process, opens an FD, or holds memory needs a per-source cap and an idle TTL." This batch adds both.

The constants live as `pub const`s near the top of `terminal.rs` per rule **A3**; promoting them to `serve.terminal.{max_sessions,idle_ttl_secs}` config fields is left to a follow-up batch (the serve config struct in `roko-core/src/config/serve.rs` is out of scope here).

### Step 5: Spawn the background idle reaper

`SessionManager::new` currently constructs the struct synchronously and is called from `roko-serve` startup (search via `rg -n 'SessionManager::new' crates/roko-serve/`). To avoid lifecycle changes outside this file's scope, spawn the reaper lazily — on the first successful `create_session` — using a `std::sync::Once` guard.

Add to `SessionManager`:

```rust

    pub(crate) sessions: Mutex<HashMap<String, PtySession>>,
    pub(crate) session_info: Mutex<HashMap<String, SessionInfo>>,
    workdir: std::path::PathBuf,
    sess_generation: AtomicU64,
    command_event_emitter: TerminalCommandEventEmitter,
    reaper_started: std::sync::Once,
}
```

Initialise it in `new`:

    Self {
        sessions: Mutex::new(HashMap::new()),
        session_info: Mutex::new(HashMap::new()),
        workdir,

Add the reaper method on `SessionManager` (the manager is already wrapped in `Arc<SessionManager>` via `AppState::terminal_sessions` — confirm by searching `Arc<SessionManager>` in the crate). The reaper takes `Arc<Self>` so it owns a refcount for the lifetime of the task:

```rust
fn ensure_idle_reaper(self: &Arc<Self>) {
    let manager = Arc::clone(self);
    self.reaper_started.call_once(move || {
        tokio::spawn(async move {
            let interval = std::time::Duration::from_secs(60);
            let ttl = std::time::Duration::from_secs(TERMINAL_IDLE_TTL_SECS);
            loop {
                tokio::time::sleep(interval).await;
                manager.reap_idle(ttl);
            }
        });
    });
}

fn reap_idle(&self, ttl: std::time::Duration) {
    let now = std::time::Instant::now();
    let stale: Vec<String> = {
        let sessions = self.sessions.lock();
        sessions
            .iter()
            .filter_map(|(id, session)| {
                let last = *session.last_used.lock();
                if now.saturating_duration_since(last) >= ttl {
                    Some(id.clone())
                } else {
                    None
                }
            })
            .collect()
    };
    for id in stale {
        tracing::info!(id = %id, ttl_secs = TERMINAL_IDLE_TTL_SECS, "reaping idle PTY session");
        self.destroy_session(&id);
    }
}
```

Because `ensure_idle_reaper` requires `&Arc<Self>`, change `create_session` and `create_session_with_id` (the public entry points at lines 197-216) to take `self: &Arc<Self>` instead of `&self`, then add `self.ensure_idle_reaper();` at the top of each. The internal `create_session_inner` keeps `&self`.

The handler (`terminal.rs:457-482`) already calls `state.terminal_sessions.create_session(...)` where `terminal_sessions: Arc<SessionManager>`; the receiver type change is source-compatible. The same applies to the WS handler call at `terminal.rs:524-527`.
