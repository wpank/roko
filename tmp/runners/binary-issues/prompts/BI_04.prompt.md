# BI_04: Default dangerously_skip_permissions to false at every hardcoded site

## Task
Flip every hardcoded `dangerously_skip_permissions: true` (and the equivalent default-constructor literal) to `false` at the four production sites in scope. Document the propagation sites that read from `roko_config.runner.dangerously_skip_permissions` so the next reader can see the chain.

## Runner Context
Group **BI_SEC**, batch 4 of 56. Closes MASTER-INDEX § **S5.8**. Deps: none.

## Problem
Universal rule **S1** in `00-RULES.md`: "Any boolean named `dangerous*`, `skip_*`, `allow_*`, or `permissive*` defaults to the safe value. The unsafe value is opt-in only via explicit user config or a `#[cfg(test)]` constructor."

`dangerously_skip_permissions = true` makes the Claude CLI skip its tool-permission system entirely. Today it is hardcoded `true` in four production sites in the default build path:

| File | Line | Context |
|---|---|---|
| `crates/roko-agent/src/claude_cli_agent.rs` | 128 | `ClaudeCliAgent::new()` default constructor. Every fresh agent skips permissions. |
| `crates/roko-cli/src/runner/types.rs` | 1391 | `impl Default for RunConfig` |
| `crates/roko-cli/src/agent_exec.rs` | 147 | `agent_exec` direct spawn |
| `crates/roko-cli/src/commands/plan.rs` | 402 | `roko plan run` constructs `RunConfig { dangerously_skip_permissions: true, ... }` |

Two additional sites **propagate** the value from `roko_config.runner.dangerously_skip_permissions` and are intentionally left as-is:

- `crates/roko-cli/src/runner/types.rs:1344` — `RunConfig::from_roko_config` (forwards user config).
- `crates/roko-cli/src/serve_runtime.rs:552` — serve runtime (forwards user config).

A third site lives in `crates/roko-acp/src/runner.rs:1981` (`ClaudeCliAgent::new(...)` with no explicit `with_dangerously_skip_permissions(...)` chain), so it inherits whatever the constructor default is. Once Step 1 below flips the constructor default to `false`, the ACP path becomes safe automatically; **no edit to `roko-acp` is required** for this batch (and `roko-acp` is not in write scope).

There is one further hardcoded `true` in `crates/roko-serve/src/dispatch.rs:1824` and several in `crates/roko-cli/src/orchestrate.rs` (lines 1600/1661/1761/15913). Those live behind the `legacy-orchestrate` feature flag (see `context-pack/01-ARCHITECTURE.md`) — they are not in the default build and are out of scope for this batch.

The schema-level default in `crates/roko-core/src/config/schema.rs:1338-1340` (`default_dangerously_skip_permissions() -> true`) is also out of scope for this batch — flipping it would change the meaning of every existing user `roko.toml` and is tracked separately.
