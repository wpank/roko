# BI_05: Promote secret-leak / forbidden-write violations to Block severity

## Task
In `SafetyLayer::post_dispatch_check`, change the `ViolationSeverity::Warn` recorded for "secret leak" and "forbidden file write" violations to `ViolationSeverity::Block`. Path-escape violations stay at `Warn` (they are an ambiguity signal, not a known-bad pattern).

## Runner Context
Group **BI_SEC**, batch 5 of 56. Closes MASTER-INDEX § **S5.9**. Deps: none.

## Problem
Universal rule **S5** in `00-RULES.md`: "A safety violation that is 'secret leak' or 'forbidden write' is a `Block`. `Warn` is for ambiguity, not for known-bad patterns."

`crates/roko-agent/src/safety/mod.rs::post_dispatch_check` currently records both as `Warn`:

```705:756:crates/roko-agent/src/safety/mod.rs
pub fn post_dispatch_check(
    &self,
    plan_id: &str,
    task_id: &str,
    role: &str,
    agent_output: &str,
    changed_files: &[String],
) -> Vec<SafetyViolation> {
    let mut violations = Vec::new();

    // 1. Check for secret leaks in agent output.
    let scrubbed = scrub::scrub_secrets(agent_output, &self.scrub_policy);
    if scrubbed != agent_output {
        violations.push(SafetyViolation {
            plan_id: plan_id.to_string(),
            task_id: task_id.to_string(),
            violation_type: ViolationType::SecretLeak,
            message: "agent output contains secrets that were scrubbed".to_string(),
            severity: ViolationSeverity::Warn,
        });
    }

    // 2. Check for path escapes in changed files.
    if self.path_policy.prevent_escapes {
        for file in changed_files {
            if file.contains("..") || file.starts_with('/') {
                violations.push(SafetyViolation {
                    plan_id: plan_id.to_string(),
                    task_id: task_id.to_string(),
                    violation_type: ViolationType::PathEscape,
                    message: format!("agent modified file outside worktree: {file}"),
                    severity: ViolationSeverity::Warn,
                });
            }
        }
    }

    // 3. Check governance limits on changed files.
    for rule in &self.contract.governance {
        if let GovernanceRule::ForbiddenTools(tools) = rule {
            if (tools.contains(&"write_file".to_string())
                || tools.contains(&"edit_file".to_string()))
                && !changed_files.is_empty()
            {
                violations.push(SafetyViolation {
                    plan_id: plan_id.to_string(),
                    task_id: task_id.to_string(),
                    violation_type: ViolationType::ContractViolation,
                    message: format!(
                        "role `{role}` forbids file writes but {} files were changed",
                        changed_files.len()
                    ),
                    severity: ViolationSeverity::Warn,
                });
            }
        }
    }
    // ...
    violations
}
```

A `Warn` violation is logged but does not stop the run; a `Block` violation makes downstream policy enforcement reject the dispatch result. Today, an agent that writes an API key into its output (case 1) or a reviewer/researcher role that touches files despite a `ForbiddenTools: ["write_file"]` clause (case 3) only produces a `tracing::warn!` line and the orchestrator accepts the artefact. That contradicts both the safety contract and rule **S5**.

The path-escape branch (case 2) stays at `Warn`: a file path containing `..` or starting with `/` may legitimately be an absolute system header reference, a `..` inside a normalised relative path, or a worktree-rooted path that the safety scrubber is matching too eagerly. Promoting it would create false positives until the path policy is tightened separately.
