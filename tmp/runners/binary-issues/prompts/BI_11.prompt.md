# BI_11: Activate VCG strategy when budget pressure crosses threshold

## Task
`CompositionStrategy::Auto` only flips to VCG once every active bidder has
collected `vcg_warmup_observations` samples (default 10). In practice the
density-greedy path keeps winning indefinitely, even when the prompt is hard
up against its token budget — exactly the situation where VCG's payment-aware
allocation matters most. Add a budget-pressure short-circuit so `Auto` selects
`Vcg` when `used_tokens / max_tokens > 0.8`, regardless of warmup state.

## Runner Context
Group **BI_PHN**, batch 11 of 56. Closes MASTER-INDEX § **S8.4**. Deps: none.

## Problem
The decision point is `crates/roko-compose/src/prompt.rs:863-897`. After
critical sections reserve their tokens, the composer asks
`self.composition_strategy.resolve(...)` (line 866) for the strategy that
will allocate the optional pool. `resolve` lives in
`crates/roko-compose/src/strategy.rs:34-58` and currently returns:

- `Vcg` if all active bidders have ≥ `warmup_observations` samples
- `DensityGreedy` otherwise

The audit (S8.4) noted that the greedy path dominates because cold-start
bidders are common and `warmup_observations` is set to 10 by default.

`prompt.rs:817-822` already computes `remaining_tokens` and `remaining_signals`
(both `usize::MAX` when no budget cap is set). What we need is a side-channel
that tells `resolve` how full the budget is. We add a second boolean input —
"budget under pressure" — and route the decision through it.

The threshold must be configurable. Use a const at the top of `strategy.rs`
plus a builder on `PromptComposer` that mirrors `with_vcg_warmup_observations`
(`prompt.rs:664-667`). A new config field is **out of scope** for this batch
— the const becomes the production default; downstream batches can promote
it to `compose.vcg_pressure_threshold` if needed.

### Step 2: Compute budget pressure at the decision site

In `crates/roko-compose/src/prompt.rs:863-897`, replace the existing
`resolve(...)` call with a pressure-aware variant. Compute pressure from the
already-available `budget.max_tokens` and the running `token_total` plus the
optional candidates' estimated cost:

```rust
let bidder_observations = self.bidder_observation_counts(&optional);

let optional_estimated_tokens: usize = optional
    .iter()
    .map(|c| c.section.estimated_tokens())
    .sum();
let projected_tokens = token_total.saturating_add(optional_estimated_tokens);

let budget_pressure = match budget.max_tokens {
    Some(max) if max > 0 => (projected_tokens as f32) / (max as f32),
    _ => 0.0_f32,
};
let under_pressure = budget_pressure > self.vcg_pressure_threshold;

let selected_strategy = self.composition_strategy.resolve_with_pressure(
    &bidder_observations,
    self.vcg_warmup_observations,
    under_pressure,
);
```

### Step 3: Add `resolve_with_pressure` to `CompositionStrategy`

In `crates/roko-compose/src/strategy.rs` extend the `impl
CompositionStrategy` block:

```rust
/// Resolve `Auto` taking budget pressure into account.
///
/// When `under_pressure` is true, `Auto` selects `Vcg` regardless of
/// warmup state — VCG's payment-aware allocation matters most when the
/// budget is nearly exhausted. Otherwise this falls through to
/// [`Self::resolve`].
#[must_use]
pub fn resolve_with_pressure(
    self,
    bidder_observations: &HashMap<AttentionBidder, u32>,
    warmup_observations: u32,
    under_pressure: bool,
) -> Self {
    if matches!(self, Self::Auto) && under_pressure {
        return Self::Vcg;
    }
    self.resolve(bidder_observations, warmup_observations)
}
```

Keep the existing `resolve` (and `auto_select`) untouched — they still serve
as the "no pressure information available" entry point.
