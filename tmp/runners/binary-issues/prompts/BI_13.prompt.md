# BI_13: Share single SharedStateHub across TUI and serve

## Task
The unified chat path spawns a background `roko serve` whose `AppState` owns
its own `roko_core::SharedStateHub`. The TUI builds another. The
`TODO(converge)` at `crates/roko-cli/src/main.rs:2207-2210` is the closest
form of that drift. Now that `roko-core` exports `SharedStateHub` (verified
at `crates/roko-serve/src/state.rs:365`), wire the **same** hub through
`spawn_background_serve` and into the inline TUI so dashboard events flow
to one bus.

## Runner Context
Group **BI_PHN**, batch 13 of 56. Closes MASTER-INDEX § **S8.7**. Deps: none.

## Problem
Three call paths build state hubs independently today:

1. `crates/roko-serve/src/state.rs:516-519` — `AppState::new_with_daimon_strategy`
   constructs `roko_core::SharedStateHub::new(StateHub::with_event_log(...))`
   pinned to `<workdir>/.roko/events.jsonl`. The HTTP server's SSE / dashboard
   routes subscribe to this hub.
2. `crates/roko-cli/src/unified.rs:167-179` — `spawn_background_serve` calls
   `roko_serve::start_server_background(workdir, runtime, None, None)` which
   internally builds an `AppState` (and therefore its own hub). The TUI's
   inline runner (`chat_inline::run_unified_inline`) holds no reference to it.
3. `crates/roko-cli/src/main.rs:2207-2210` — same divergence in the
   `Command::Unified` arm:
   ```rust
   let (state, server_handle) = server_builder.start_background().await?;
   // TODO(converge): pass server's SharedStateHub once roko-core
   // re-exports StateHub to unify the duplicated #[path] types.
   let tui_result =
       commands::dashboard::cmd_dashboard(cli, Some(wd), None, false, false, None)
           .await;
   ```
4. `crates/roko-cli/src/commands/util.rs:270-275` — the `cmd_run`/`cmd_share`
   path explicitly comments out the bridge:
   ```rust
   // TODO(converge): When --serve is active we should share the server's StateHub …
   // Currently roko_serve::StateHub and roko_cli::state_hub::StateHub are distinct
   // types …
   let external_hub: Option<&roko_cli::state_hub::StateHub> = None;
   ```

`roko_core::SharedStateHub` (defined at
`crates/roko-core/src/state_hub.rs:295-353`) is `Clone` and wraps an
`Arc<StateHub>`. `AppState.state_hub` is **already** typed as
`roko_core::SharedStateHub`. So the only changes needed are:

- Plumb the hub *out* of `start_server_background` so callers can clone it.
- Pass the cloned hub into the TUI / inline chat dispatch.
- Replace the TODO at `main.rs:2207-2210` with a real handoff.

## Required Changes

### Step 1: Plumb the hub through `spawn_background_serve`

Update `crates/roko-cli/src/unified.rs:167-179`:

```rust
pub(crate) async fn spawn_background_serve(
    config: &crate::config::Config,
    workdir: &std::path::Path,
) -> Option<(
    Arc<roko_serve::state::AppState>,
    JoinHandle<Result<()>>,
    roko_core::SharedStateHub,
)> {
    let runtime = RokoCliRuntime::new(config.clone(), RepoRegistry::default()).into_arc();
    match roko_serve::start_server_background(workdir.to_path_buf(), runtime, None, None).await {
        Ok((state, handle)) => {
            let hub = state.state_hub.clone();
            Some((state, handle, hub))
        }
        Err(e) => {
            tracing::warn!("background serve failed to start: {e:#}");
            None
        }
    }
}
```

`SharedStateHub` is `Clone` (verify with
`rg -n 'impl Clone for SharedStateHub' crates/roko-core/src/state_hub.rs`),
and `state.state_hub` is the right field name (line 365).

### Step 2: Hand the hub to inline chat

`crates/roko-cli/src/unified.rs:74` calls
`chat_inline::run_unified_inline(&auth).await`. Add a sibling entry point
that accepts an optional hub:

```rust
let result = if let Some((_, _, hub)) = serve_state.as_ref() {
    chat_inline::run_unified_inline_with_hub(&auth, Some(hub.clone())).await
} else {
    chat_inline::run_unified_inline(&auth).await
};
```

In `crates/roko-cli/src/chat_inline.rs` (around the existing
`pub async fn run_unified_inline(auth: &AuthMethod) -> Result<()>` at line
1553), add a wrapper:

```rust
pub async fn run_unified_inline_with_hub(
    auth: &AuthMethod,
    hub: Option<roko_core::SharedStateHub>,
) -> Result<()> {
    // Existing body, but stash `hub` in whatever struct already holds
    // dashboard / event-publishing handles. If no current handle exists,
    // pass the hub down to `build_unified_inline_agent_session` so dispatch
    // can publish DashboardEvents through it.
    inline_run_inner(auth, hub).await
}

pub async fn run_unified_inline(auth: &AuthMethod) -> Result<()> {
    inline_run_inner(auth, None).await
}
```

The internal entry point (`inline_run_inner`) should be the existing body of
`run_unified_inline`, plus an extra `hub: Option<SharedStateHub>` argument.
Forward the hub to wherever `commands::dashboard` / projection events would
otherwise be dropped.

If there is no obvious downstream consumer for `hub` inside
`run_unified_inline` today (the audit suggests there is — see
`crates/roko-cli/src/runner/event_loop.rs` and `runner/tui_bridge.rs`),
store it on the inline session struct under a `dashboard_hub:
Option<SharedStateHub>` field and use it the next time
`SharedStateHub::sender()` is called in that file. This batch is about
plumbing — the consumer wiring lands in subsequent work.
