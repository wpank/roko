# BI_16: Query knowledge store and populate knowledge_ids in chat path

## Task
`ChatAgentSession::model_call_request` always sends
`knowledge_ids: Vec::new()` — every turn dispatches with no knowledge
context, regardless of what the user actually asked. Wire the chat path to
`roko_neuro::KnowledgeStore` and pass the top-K knowledge entry IDs whose
content matches the prompt. Top-K defaults to 5 and is overridden via a new
`learning.chat_knowledge_top_k: u32` config field.

Group **BI_PHN**, batch 16 of 56. Closes MASTER-INDEX § **S1.7**. Deps: none.

## Required Changes

### Step 1: Add `learning.chat_knowledge_top_k` config field

**Note**: The batch's `scope` does not include `roko-core/src/config/`.
The config knob must therefore land via a small read-only fallback rather
than a new schema field. Follow the H2 rule from `00-RULES.md` ("Hardcoded
as fallback only"): expose a const, accept an override at construction
time, fall back to the const if no override is provided.

In `crates/roko-cli/src/chat_session.rs`, near the top of the file (after
the imports), add:

```rust
/// Default number of knowledge entries to surface in each chat turn.
///
/// Wire to `learning.chat_knowledge_top_k` once that field lands in
/// `roko-core/src/config/learning.rs`. Until then this constant is the
/// production default.
pub const DEFAULT_CHAT_KNOWLEDGE_TOP_K: usize = 5;
```

If the writer prefers a schema-first change *and* the runner allows
batch-scope expansion to include `crates/roko-core/src/config/learning.rs`,
add a field on `LearningConfig`:

```rust
#[serde(default = "default_chat_knowledge_top_k")]
pub chat_knowledge_top_k: u32,
```

with a `const fn default_chat_knowledge_top_k() -> u32 { 5 }` and an
initializer in `Default::default()`. **Default to the const-only
implementation** to keep the batch self-contained inside `chat_session.rs`.

### Step 2: Lazy-load a knowledge store on the session

Add an `Option<Arc<roko_neuro::KnowledgeStore>>` field to
`ChatAgentSession`:

```rust
/// Optional knowledge store backing per-turn `knowledge_ids` lookup.
/// `None` means we couldn't locate `.roko/neuro/knowledge.jsonl` and the
/// chat path falls back to empty knowledge_ids.
pub knowledge_store: Option<std::sync::Arc<roko_neuro::KnowledgeStore>>,
/// Top-K cap applied to per-turn knowledge queries.
pub knowledge_top_k: usize,
```

Initialize both in `ChatAgentSession::new` (line ~363):

```rust
let knowledge_store = {
    let candidate = roko_neuro::KnowledgeStore::for_workdir(&workdir);
    let path = candidate.path().to_path_buf();
    if path.exists() {
        Some(std::sync::Arc::new(candidate))
    } else {
        None
    }
};
let knowledge_top_k = DEFAULT_CHAT_KNOWLEDGE_TOP_K;
```

Add `knowledge_store` and `knowledge_top_k` to the `Self { ... }` literal.
The store is cheap to clone (`Arc`) and the file existence check avoids
hammering an empty workspace.

### Step 3: Populate `knowledge_ids` inside `model_call_request`

Replace `knowledge_ids: Vec::new()` at line 585 with a query against the
store. Because `model_call_request` is `&self` and `KnowledgeStore::query`
is sync (`fn query(&self, topic: &str, limit: usize) -> Result<...>`), no
`async` plumbing is needed.

```rust
let knowledge_ids = if let Some(store) = self.knowledge_store.as_ref() {
    match store.query(prompt, self.knowledge_top_k) {
        Ok(entries) => entries.into_iter().map(|e| e.id).collect(),
        Err(err) => {
            tracing::warn!(
                error = %err,
                "ChatAgentSession: knowledge query failed; sending empty knowledge_ids"
            );
            Vec::new()
        }
    }
} else {
    Vec::new()
};

ModelCallRequest {
    // ...existing fields...
    knowledge_ids,
    // ...
}
```

`KnowledgeStore::query` already enforces a non-zero limit; passing
`self.knowledge_top_k` of 0 (caller deliberately disabled) returns an empty
vec without I/O — but to keep the contract honest add a guard:

```rust
if self.knowledge_top_k == 0 { return Vec::new(); }
```

inside the `Some(store)` branch before invoking `query`.

### Step 4: Trace bookkeeping

Inside the same branch, add a single `tracing::debug!` per turn so
operators can see the populated IDs without enabling trace-level logs:

```rust
tracing::debug!(
    count = knowledge_ids.len(),
    top_k = self.knowledge_top_k,
    "ChatAgentSession: populated knowledge_ids for turn"
);
```

Place this immediately before the `ModelCallRequest { ... }` construction.

### Step 5: Builder for tests / opt-out

Add a small builder so tests (and any future caller that wants to
override) can swap the store:

```rust
impl ChatAgentSession {
    /// Replace the per-session knowledge store. Use `None` to disable
    /// per-turn `knowledge_ids` population entirely.
    #[must_use]
    pub fn with_knowledge_store(
        mut self,
        store: Option<std::sync::Arc<roko_neuro::KnowledgeStore>>,
    ) -> Self {
        self.knowledge_store = store;
        self
    }

    /// Override the per-turn knowledge top-K cap.
    #[must_use]
    pub fn with_knowledge_top_k(mut self, top_k: usize) -> Self {
        self.knowledge_top_k = top_k;
        self
    }
}
```

Place these next to the existing `impl ChatAgentSession` block (around the
constructor at line 339).
