# BI_24: Forward WorkflowEngine lifecycle events to terminal during plan run

## Task
`run_workflow_engine_with_services` (`crates/roko-cli/src/run.rs:600-654`) registers a `JsonlWorkflowConsumer` and an optional `StateHubBridge` on the `WorkflowEngine`, then calls `engine.run(config).await` and finally `print_workflow_run_report` produces a post-hoc summary at lines 656-708. Lifecycle events flow into JSONL on disk and into the SSE/WS hub, but **nothing reaches stdout while the run is in flight**. Add a stdout/stderr `EventConsumer` that streams phase / agent / completion events as they arrive, plus a `_with_consumers` variant of the entry point so the chat-side `/run` and `/plan run` (BI_19, BI_20) can attach their own forwarder. Keep the post-hoc summary intact.

## Runner Context
Group **BI_STR**, batch 24 of 56. Closes MASTER-INDEX § **S6.5**. Deps: none. Blocks BI_19 / BI_20 / BI_25.

## Problem
The default `roko run <prompt>` and `roko plan run <dir>` paths print the resolved model + workflow template via `selection.print_stderr()`, then go silent until `print_workflow_run_report` runs at the end (`run.rs:570`). Internally, `WorkflowEngine::emit` (`crates/roko-runtime/src/workflow_engine.rs:538-554`) is firing `RuntimeEvent::WorkflowStarted`, `PhaseTransition`, `AgentSpawned`, `WorkflowCompleted` — but those events only land in `.roko/events.jsonl` and (when an external `StateHub` is supplied) in the dashboard SSE stream. A user running `roko run "fix the failing test"` sees nothing for 30+ seconds.

The clean place to fan out is the same `add_consumer` seam that `JsonlWorkflowConsumer` and `StateHubBridge` already use:

```106:124:crates/roko-runtime/src/workflow_engine.rs
pub struct WorkflowEngine {
    services: EffectServices,
    /// Optional event consumers to notify.
    consumers: Vec<Arc<dyn EventConsumer>>,
}

impl WorkflowEngine {
    pub fn new(services: EffectServices) -> Self {
        Self { services, consumers: Vec::new() }
    }

    pub fn add_consumer(&mut self, consumer: Arc<dyn EventConsumer>) {
        self.consumers.push(consumer);
    }
```

The fix: a `StdoutLifecycleStreamer: EventConsumer` that translates `RuntimeEvent` into one-line `output_format::step` / `output_format::branch` / `output_format::dim` lines, attached unconditionally inside `run_workflow_engine_with_services`. Plus a public `_with_consumers` variant of the two entry points so the chat code can attach its own.

## Required Changes

### Step 1: Add `StdoutLifecycleStreamer`

Below the `StateHubBridge` impl (around `run.rs:318`), add:

```rust
/// `EventConsumer` that prints WorkflowEngine lifecycle events to stdout
/// in real time. Used by the default `roko run` / `roko plan run` paths
/// so users see progress while the engine is working.
///
/// The post-hoc `print_workflow_run_report` summary is unchanged; this
/// streamer fills the silence between dispatch and final summary.
struct StdoutLifecycleStreamer;

impl WorkflowEventConsumer for StdoutLifecycleStreamer {
    fn consume(&self, event: &roko_core::RuntimeEvent) {
        match event {
            roko_core::RuntimeEvent::WorkflowStarted { run_id, template, prompt } => {
                output_format::step(
                    "engine",
                    &format!(
                        "{template} run {} ({})",
                        run_id,
                        truncate(prompt, 60),
                    ),
                );
            }
            roko_core::RuntimeEvent::PhaseTransition { from, to, .. } => {
                output_format::branch(&format!("phase {from} → {to}"));
            }
            roko_core::RuntimeEvent::AgentSpawned { agent_id, role, .. } => {
                output_format::branch(&output_format::dim(&format!(
                    "agent {role} ({agent_id})"
                )));
            }
            roko_core::RuntimeEvent::WorkflowCompleted { outcome, .. } => {
                let label = match outcome {
                    roko_core::WorkflowOutcome::Success { .. } => "success",
                    roko_core::WorkflowOutcome::Failure { .. } => "failure",
                    roko_core::WorkflowOutcome::Cancelled => "cancelled",
                    _ => "completed",
                };
                output_format::branch(&format!("workflow {label}"));
            }
            _ => {}
        }
    }
}
```

The `WorkflowOutcome` variants must be matched exactly as they exist in `roko_core` — verify with `rg -n 'pub enum WorkflowOutcome' crates/roko-core/src/`. If a variant we listed doesn't exist in the current enum, drop the arm (don't invent variants); a `_ => "completed"` default catches the rest.

`output_format::*` helpers are already in scope (the file uses them at line 661+ in `print_workflow_run_report`). No new imports needed.
