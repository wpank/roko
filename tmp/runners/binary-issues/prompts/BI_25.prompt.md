# BI_25: Stream incremental output for roko run v2

## Task
Builds on BI_24. `roko run <prompt>` invokes `run_with_workflow_engine_with_hub`, which now (after BI_24) streams `RuntimeEvent` lifecycle lines via `StdoutLifecycleStreamer`. This batch adds a **per-token** progress line: when the run is in an interactive TTY (`std::io::stdout().is_terminal()`), render a single carriage-return-rewriting status line that shows the active model, tokens-so-far, and cost-so-far. The final post-hoc summary from `print_workflow_run_report` still prints at end-of-run.

## Runner Context
Group **BI_STR**, batch 25 of 56. Closes MASTER-INDEX § **S6.6**. Deps: **BI_24**.

## Problem
Even after BI_24's lifecycle streamer lands, the user still has no signal that "the agent is producing tokens right now" between `phase strategy → execute` and `workflow success`. For a run that takes 30-90 seconds, the screen sits silent. The fix is a single status line, updated in place, that ticks with the model + tokens + cost as `WorkflowEngine` consumes them. We do this without touching `WorkflowEngine` itself: model deltas are reported by the dispatcher via `RuntimeEvent` variants the engine emits whenever a model call completes (search the variants for `Token` / `Usage` / `ModelCallCompleted`).

### Step 2: Add `LiveProgressLineStreamer`

Below `StdoutLifecycleStreamer` (added by BI_24), add:

```rust
use std::sync::Mutex;

/// `EventConsumer` that maintains a single-line live progress indicator
/// on stdout. Shows model + cumulative tokens + cumulative USD cost,
/// updated in place via `\r`. Only attached when stdout is a TTY.
struct LiveProgressLineStreamer {
    inner: Mutex<LiveProgressState>,
}

#[derive(Default)]
struct LiveProgressState {
    model: String,
    input_tokens: u64,
    output_tokens: u64,
    cost_usd: f64,
    last_render_ms: u64,
}

impl LiveProgressLineStreamer {
    fn new() -> Self {
        Self { inner: Mutex::new(LiveProgressState::default()) }
    }

    fn render(state: &LiveProgressState) {
        // Use eprint! so stdout-piping users (e.g. `roko run ... | tee log`)
        // still get a clean stdout. The line is written to stderr and erased
        // before the post-hoc summary lands on stdout.
        use std::io::Write as _;
        let mut stderr = std::io::stderr().lock();
        let _ = write!(
            stderr,
            "\r⏳ {model}  tokens {tin}→{tout}  cost ${cost:.4}    ",
            model = if state.model.is_empty() { "—" } else { &state.model },
            tin = state.input_tokens,
            tout = state.output_tokens,
            cost = state.cost_usd,
        );
        let _ = stderr.flush();
    }
}

impl WorkflowEventConsumer for LiveProgressLineStreamer {
    fn consume(&self, event: &roko_core::RuntimeEvent) {
        // Match the engine's per-call event for usage updates. The exact
        // variant + field names live in `roko_core::RuntimeEvent` — see
        // `rg -n 'pub enum RuntimeEvent' crates/roko-core/src/`. Adjust
        // the arms below to match what the runtime actually emits.
        let mut state = match self.inner.lock() {
            Ok(g) => g,
            Err(poisoned) => poisoned.into_inner(),
        };
        let mut updated = false;
        match event {
            roko_core::RuntimeEvent::AgentSpawned { agent_id, role, .. } => {
                if state.model.is_empty() {
                    state.model = format!("{role}:{agent_id}");
                    updated = true;
                }
            }
            // Add an arm for whatever variant carries cumulative usage in
            // `roko_core::RuntimeEvent`. Examples worth grepping for:
            //   - ModelCallCompleted { input_tokens, output_tokens, cost, .. }
            //   - TokenUsage { delta_in, delta_out, .. }
            // If no per-call usage event exists, leave this consumer
            // attached as a no-op for now and rely on the lifecycle
            // streamer (BI_24) for live signal — TODO(BI_25 follow-up):
            // emit per-call usage from `WorkflowEngine`.
            _ => {}
        }
        if updated {
            // Throttle redraws to ~10 Hz so noisy event streams don't
            // saturate the terminal.
            let now_ms = (std::time::SystemTime::now()
                .duration_since(std::time::UNIX_EPOCH)
                .map(|d| d.as_millis() as u64)
                .unwrap_or(0))
                .max(state.last_render_ms);
            if now_ms - state.last_render_ms >= 100 {
                state.last_render_ms = now_ms;
                Self::render(&state);
            }
        }
    }
}

impl Drop for LiveProgressLineStreamer {
    fn drop(&mut self) {
        // Clear the line on drop so the post-hoc summary lands on a clean row.
        use std::io::Write as _;
        let mut stderr = std::io::stderr().lock();
        let _ = write!(stderr, "\r{:80}\r", "");
        let _ = stderr.flush();
    }
}
```

The `_` arm is intentional: `RuntimeEvent` may not yet carry a per-token usage variant. Verify with `rg -n 'pub enum RuntimeEvent' crates/roko-core/src/` and add an arm for whichever variant currently carries cumulative usage — or, if none exists, leave the streamer as a near-no-op that updates `state.model` from `AgentSpawned` and document the gap in the function doc comment. Do NOT add a new `RuntimeEvent` variant from this batch.

### Step 4: Pre-final-summary cleanup

The `LiveProgressLineStreamer::Drop` impl already clears the live line. But the consumer is held inside the `engine.consumers: Vec<Arc<dyn EventConsumer>>` for the entire run, and `Arc::strong_count` may not drop to 0 until after `print_workflow_run_report` runs. To make sure the live line is cleared *before* `print_workflow_run_report`, add an explicit `eprint!("\r{:80}\r", "");` + `stderr.flush()` call right before `print_workflow_run_report(prompt, workflow_label, &report);`. Wrap it in `if is_tty {}`.
