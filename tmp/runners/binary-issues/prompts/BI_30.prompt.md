# BI_30: Non-blocking background-serve startup

## Task
Stop blocking inline chat boot on `roko serve` startup. Today `cmd_unified_chat` `await`s `spawn_background_serve(...)` before chat begins; if the bind, snapshot restore, or relay registration takes a few hundred ms, the user stares at a blank terminal. Refactor so the serve startup runs as its own `tokio::spawn`, returns a `Ready` signal (oneshot) the chat path can poll, and chat renders a "starting…" hint until the serve task signals ready (or fails).

## Runner Context
Group **BI_SUB**, batch 30 of 56. Closes MASTER-INDEX § **S9.7**. Deps: none.

## Problem
`crates/roko-cli/src/unified.rs:49-59` calls:
```rust
let serve_state = if no_serve {
    None
} else if load_auto_start_config(&config) {
    spawn_background_serve(&config, &workdir).await   // <-- blocks chat boot
} else {
    if !quiet {
        eprintln!("Tip: run `roko serve` to start the HTTP control plane");
    }
    None
};
```

`spawn_background_serve` (`unified.rs:167-179`) calls `roko_serve::start_server_background(...).await`, which itself does serial work: build state, validate bind, restore snapshot, build router, bind listener, etc. (`crates/roko-serve/src/lib.rs:264-405`). Until that future completes, `chat_inline::run_unified_inline` does not start.

The fix is to make the serve task a fire-and-forget, hand the chat path a `tokio::sync::oneshot::Receiver<Result<Arc<AppState>, ServeError>>` (or equivalent), and let the chat loop poll it on each frame.

## Required Changes

### Step 1: Add a non-blocking spawn helper in `unified.rs`
Replace the `async fn spawn_background_serve` with a synchronous spawner that returns a `BackgroundServeStartup` token:

```rust
/// State of a backgrounded `roko serve` startup.
pub(crate) struct BackgroundServeStartup {
    /// Resolves with the live state once the server has bound and registered.
    /// Resolves with `None` if startup failed (the warning was already logged).
    pub ready_rx: tokio::sync::oneshot::Receiver<
        Option<(Arc<roko_serve::state::AppState>, JoinHandle<Result<()>>)>,
    >,
}

/// Kick off a background `roko serve` task that does not block chat boot.
fn spawn_background_serve(
    config: &crate::config::Config,
    workdir: &std::path::Path,
) -> BackgroundServeStartup {
    let runtime = RokoCliRuntime::new(config.clone(), RepoRegistry::default()).into_arc();
    let workdir = workdir.to_path_buf();
    let (tx, ready_rx) = tokio::sync::oneshot::channel();
    tokio::spawn(async move {
        let result = match roko_serve::start_server_background(workdir, runtime, None, None).await {
            Ok(pair) => Some(pair),
            Err(e) => {
                tracing::warn!("background serve failed to start: {e:#}");
                None
            }
        };
        let _ = tx.send(result);
    });
    BackgroundServeStartup { ready_rx }
}
```

### Step 3: Add a sibling entry point in `chat_inline`
Edit `crates/roko-cli/src/chat_inline.rs`. Add a thin wrapper next to the existing `run_unified_inline` that accepts the optional serve startup token:

```rust
/// Drive the unified inline chat loop. While `serve_startup.is_some()`, the
/// loop polls the oneshot non-blockingly each frame and updates the status
/// bar from "starting…" to ":6677" once ready. Failures keep `serve = None`
/// and emit a one-time warning line.
pub async fn run_unified_inline_with_serve(
    auth: &AuthMethod,
    mut serve_startup: Option<crate::unified::BackgroundServeStartup>,
) -> anyhow::Result<()> {
    // Existing setup path lives in run_unified_inline; reuse it.
    let mut session = build_unified_inline_session(auth)?;
    session.serve_startup = serve_startup.take();
    drive_chat_loop(session).await
}
```

### Step 4: Poll the oneshot in the chat event loop
At the top of the per-frame loop (search for `session.tick += 1;` near line 1830), drain the startup signal non-blockingly:
```rust
if let Some(startup) = session.serve_startup.as_mut() {
    use tokio::sync::oneshot::error::TryRecvError;
    match startup.ready_rx.try_recv() {
        Ok(Some((state, handle))) => {
            session.serve_state = Some((state, handle));
            session.serve_startup = None;
            term.push_lines(&[styled::continuation(
                &theme,
                "serve",
                "ready on :6677",
                None,
            )])?;
        }
        Ok(None) => {
            session.serve_startup = None;
            term.push_lines(&[styled::continuation(
                &theme,
                "serve",
                "failed to start (continuing without HTTP plane)",
                None,
            )])?;
        }
        Err(TryRecvError::Empty) => { /* still booting */ }
        Err(TryRecvError::Closed) => {
            session.serve_startup = None;
        }
    }
}
```

Add the `serve_state: Option<(Arc<AppState>, JoinHandle<Result<()>>)>` companion field to `ChatSession`, and route shutdown through it at session end (in the existing tear-down site at the bottom of the chat loop).

### Step 5: Move shutdown out of `cmd_unified_chat`
Because chat now owns the serve handle, the shutdown block in `cmd_unified_chat` (`unified.rs:77-80`) is no longer needed. Remove that block; `run_unified_inline_with_serve` returns after it has called `state.shutdown().await; handle.abort();` itself.
