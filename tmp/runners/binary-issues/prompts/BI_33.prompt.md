# BI_33: Log AffectPolicy::persist failures (warn + counter)

## Task
Replace the silent `let _ = policy.persist().await` in `WorkflowEngine::persist_affect_policy` with an explicit `if let Err(e)` arm that emits `tracing::warn!` and increments a static counter so operators can observe persist health over time.

## Runner Context
Group **BI_ERR**, batch 33 of 56. Closes MASTER-INDEX § **S4.3**. Deps: none.

## Problem
`crates/roko-runtime/src/workflow_engine.rs:590-596`:
```rust
async fn persist_affect_policy(&self) {
    if let Some(ref affect) = self.services.affect_policy {
        let policy = affect.lock().await;
        let _ = policy.persist().await;
        drop(policy);
    }
}
```

Affect policy persistence is the only durable record of agent emotional / motivational state across runs. Silently dropping persist failures means a process restart can lose hours of online learning with zero operator signal. Per `00-RULES.md` A2, every `let _ =` needs either a justification comment or a logged failure path.

`metrics`/`opentelemetry` is **not** a dependency of `roko-runtime` (`Cargo.toml:19` only has `tracing = { workspace = true }`). Use a process-wide `AtomicU64` counter exposed via a public function so consumers (CLI status bar, serve metrics endpoint) can read it later.

## Required Changes

### Step 1: Add a process-wide failure counter near the top of `workflow_engine.rs`
Find the existing module-level constants/statics section (search for `static ` or just place after the module-level imports). Add:
```rust
use std::sync::atomic::{AtomicU64, Ordering};

/// Process-wide count of `AffectPolicy::persist` failures.
///
/// Exposed via [`affect_persist_failure_count`] so the CLI status bar and
/// `roko serve` `/metrics` can surface the value. Reset only on process
/// restart; this counter is intentionally monotonic to make trend analysis
/// straightforward.
static AFFECT_PERSIST_FAILURES: AtomicU64 = AtomicU64::new(0);

/// Read the current value of [`AFFECT_PERSIST_FAILURES`].
pub fn affect_persist_failure_count() -> u64 {
    AFFECT_PERSIST_FAILURES.load(Ordering::Relaxed)
}
```

### Step 2: Replace the silent persist with a logged + counted branch

Update `persist_affect_policy` (`workflow_engine.rs:590-596`) to:
```rust
async fn persist_affect_policy(&self) {
    if let Some(ref affect) = self.services.affect_policy {
        let policy = affect.lock().await;
        if let Err(e) = policy.persist().await {
            AFFECT_PERSIST_FAILURES.fetch_add(1, Ordering::Relaxed);
            tracing::warn!(
                target: "affect",
                error = %e,
                total_failures = AFFECT_PERSIST_FAILURES.load(Ordering::Relaxed),
                "AffectPolicy::persist failed",
            );
        }
        drop(policy);
    }
}
```

### Step 3: Re-export the counter accessor from `roko-runtime`

If `crates/roko-runtime/src/lib.rs` (or the parent `mod.rs`) re-exports other workflow_engine items, add:

If the file does not currently re-export anything, leave the `pub fn` accessible via `roko_runtime::workflow_engine::affect_persist_failure_count` and skip the re-export.

### Step 4: Smoke test the counter

Append to whichever `#[cfg(test)] mod tests { ... }` block lives at the bottom of `workflow_engine.rs` (or create one if absent):

fn affect_persist_failure_count_is_monotonic() {
    let before = super::affect_persist_failure_count();
