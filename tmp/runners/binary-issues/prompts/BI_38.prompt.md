## Problem
The string literal `"claude-opus-4-6"` appears in 30+ files across the workspace (run `rg -n "claude-opus-4-6" crates/ | wc -l` to confirm). Most occurrences are tests or documentation, but the production preset definitions, default routing helpers, and runtime-feedback paths all spell the slug out by hand. The next time the opus model name is bumped (e.g. `claude-opus-4-7`), every site has to be edited individually — exactly the failure mode universal rule **H1** in `00-RULES.md` warns against ("Single source of truth. This batch establishes that single source of truth. The write scope is intentionally narrow:

- `crates/roko-core/src/config/presets.rs` is the **owner** of the constant.
- `crates/roko-cli/src/runtime_feedback/routing.rs` is the **first consumer** (it pulls the slug in via `roko_core` and is one of the most-edited files in this group).

A follow-up sweep (out of scope for this batch) will migrate the remaining sites — `dispatch/model_routing.rs`, `model_selection.rs`, dreams, the TUI helpers, `lifecycle.rs`, etc. — to read from the same constant. The compile graph keeps working in the meantime because the literal `"claude-opus-4-6"` is unchanged; only the spelling of who owns it changes.

## Required Changes

### Step 1: Define the constant in `presets.rs`

In `crates/roko-core/src/config/presets.rs`, immediately after the `use super::schema::{...};` import block (currently lines 6-9) and before `pub enum Preset`, insert:

```rust
/// Canonical opus-tier Claude model slug.
///
/// This is the single source of truth for the opus model used by the
/// `thorough` preset, the default routing helpers, and the runtime
/// feedback sink. When Anthropic ships a new opus generation, bump this
/// constant and every consumer that reads it picks up the new slug.
///
/// Consumers MUST import this constant rather than spelling the slug
/// inline. See BI_HRD follow-up batches for the migration of the
/// remaining literal call sites.
pub const DEFAULT_OPUS_MODEL: &str = "claude-opus-4-6";
```

### Step 3: Re-export the constant from `roko_core::config`

`crates/roko-core/src/config/presets.rs` is already inside the `config` module tree. If `crates/roko-core/src/config/mod.rs` already re-exports preset items (`pub use presets::Preset;` or similar) extend that re-export to include `DEFAULT_OPUS_MODEL`. If it does (e.g. via `pub use presets::*;`), skip this step entirely and rely on the existing wildcard. ### Step 4: Wire the constant into `runtime_feedback/routing.rs`

In `crates/roko-cli/src/runtime_feedback/routing.rs`, add an import alongside the existing `roko_core` imports near the top of the file (after the existing `use roko_core::config::RewardWeights;` line):


The `#[allow(unused_imports)]` is intentional and temporary: this batch establishes the import path; the follow-up batch (BI_HRD wave 2) replaces the literal slug usages inside this file with `DEFAULT_OPUS_MODEL`. Document this in a `// SAFETY:`-style comment immediately above the import:

```rust

// literal opus slug references in this file without touching the
// import block. Removed once the migration completes.
```

If `roko_core::config::presets` is **not** publicly reachable (e.g. the `presets` submodule is `pub(crate)`), import via the re-export from Step 3 instead: `use roko_core::config::DEFAULT_OPUS_MODEL;`.
