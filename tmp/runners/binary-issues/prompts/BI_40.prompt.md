# BI_40: Anthropic API version through provider config

## Task
Add an `api_version: Option<String>` field to `ProviderConfig` in `crates/roko-core/src/config/provider.rs` and thread it through `ClaudeAgent::from_provider` (introduced by BI_39) so the `anthropic-version` HTTP header can be overridden via configuration. The existing `pub const DEFAULT_ANTHROPIC_VERSION` remains as the single fallback when the field is `None`.

## Runner Context
Group **BI_HRD**, batch 40 of 56. Closes MASTER-INDEX § **S7.5**. Deps: BI_39 (this batch extends `ClaudeAgent::from_provider`; if BI_39 has not landed, add the field and the constructor's version handling in the same edit and document the dependency in the commit message).

## Problem
`crates/roko-agent/src/claude_agent.rs:32` defines `pub const DEFAULT_ANTHROPIC_VERSION: &str = "2023-06-01"`, baked into `ClaudeAgent::new` at line 208. Anthropic occasionally ships new API-version headers (e.g. for streaming changes, prompt caching betas). Today the only override seam is the manual builder method `with_anthropic_version` (line 274). No production caller uses it — see `rg -n 'with_anthropic_version' crates/`, which returns only test sites in `claude_agent.rs:864` and `provider/anthropic_api/tool_loop.rs` constants.

The result: any operator who needs to opt into a newer API version must rebuild from source. Universal rule **H2** in `00-RULES.md` — "Hardcoded as fallback only" — wants the constant to remain *as a fallback* with a config-driven override on top.

This batch adds the override field to `ProviderConfig` (the natural home, alongside `base_url` and `timeout_ms`), wires it through `ClaudeAgent::from_provider`, and leaves `DEFAULT_ANTHROPIC_VERSION` alone so existing callers keep working.

            kind: roko_core::agent::ProviderKind::AnthropicApi,
            base_url: None,

            .iter()

    }
