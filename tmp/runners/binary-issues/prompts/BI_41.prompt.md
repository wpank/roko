# BI_41: Consolidate 8192 max_tokens into per-role / per-provider config

## Task
Add a `max_tokens: Option<u32>` field to `ProviderConfig` in `crates/roko-core/src/config/provider.rs` so providers can opt out of the hardcoded 8192 ceiling, and hoist the two in-scope `8192` literals (`crates/roko-compose/src/enrichment/pipeline.rs:367, 425` and `crates/roko-agent/src/lifecycle.rs:359`) into named constants in the file that owns them. Document that the long-term home of this knob is `ProviderConfig.max_tokens` and that the per-call literals are local fallbacks.

## Runner Context
Group **BI_HRD**, batch 41 of 56. Closes MASTER-INDEX § **S7.6**. Deps: none (BI_39 / BI_40 add other `ProviderConfig` fields but do not collide with this one).

## Problem
Two separate paths bake the magic number `8192` into Rust source:

1. `crates/roko-compose/src/enrichment/pipeline.rs:367` and `:425` pass `8192` as the `max_tokens` argument on `LlmClient::call(model, system, user, max_tokens)`. Both call sites use the same literal — bumping the cap means editing both.
2. `crates/roko-agent/src/lifecycle.rs:359` initializes `BudgetConfig.max_tokens_per_turn: 8192` inside `impl Default for BudgetConfig`. The field is `u64`; this is a budget cap (per-turn token ceiling), distinct from the per-call max-tokens in (1), but the spelling `8192` is identical and operators conflate the two.

Universal rule **H3** in `00-RULES.md`: "Per-role / per-provider knobs. `max_tokens` … values live on the provider/role config, not in the dispatch function body." This batch sets up the config field that future batches will read from, and pulls each existing literal into a named constant in its own module so the value is grep-discoverable and consumer code reads `LOCAL_DEFAULT` instead of an unexplained `8192`.

The downstream wiring — having `enrichment::pipeline` actually read `ProviderConfig.max_tokens` from the resolved provider, and having `lifecycle::BudgetConfig` consult it on a per-role basis — is **out of scope** for this batch (it requires plumbing `RokoConfig` into `EnrichmentConfig`, which is not in this batch's scope). The follow-up batch will pick up the new field and switch the constant to a `provider.max_tokens.unwrap_or(DEFAULT_*)` pattern.
