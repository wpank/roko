# BI_43: CostTable loaded from config with hardcoded fallbacks

## Task

## Problem
`crates/roko-learn/src/cost_table.rs` currently has two public constructors:

1. `CostTable::from_config(models: &HashMap<String, ModelProfile>)` — reads `cost_input_per_m` / `cost_output_per_m` / `cost_cache_*` from each profile and builds a `ModelPricing` row. Skips profiles that don't define both `input_per_m` and `output_per_m`.
2. `CostTable::with_defaults(self)` — fills in eight hardcoded fallback rows for the well-known model slugs (lines 137-156). Returns `0.0` for slugs that don't match any row.

Callers stitch these together by hand: every working caller has `CostTable::from_config(&model_config.models).with_defaults()` (see `crates/roko-cli/src/chat_inline.rs:1528, 5375`). That works, but:

- The fallback path is the same eight-line literal in every consumer. If a new caller forgets `.with_defaults()`, unknown models silently cost `0.0`.
- There is no place for an operator to *override* the per-model pricing without overwriting the `[models.<name>].cost_*` fields. A separate `[cost.<model>]` section (read after model profiles, before defaults) is the standard pattern (e.g. `LITELLM` and `langfuse` use the same shape).
- `with_defaults` is the most-grep'd entry point but it's a fallback — newcomers think it's the primary constructor.

This batch:

1. Adds `CostTable::from_roko_config(config: &RokoConfig)` as the new "one true constructor" — chains config-derived rows, optional `[cost.*]` overrides (read via a new helper that we leave forward-compatible if no section exists yet), and `with_defaults` in that order.
2. Documents the `[cost.<model>]` schema in the file's `//!` doc block so operators know how to override.
3. Decorates `with_defaults` with `#[doc(hidden)]` and a doc comment that points at `from_roko_config`. Keeps the `pub` visibility because `crates/roko-cli/src/chat_inline.rs` (out of scope) still calls it directly; downgrading visibility to `pub(crate)` is a follow-up batch once those call sites migrate.

//! Per-model pricing tables and cost normalization utilities.
//!
//! # Pricing resolution order
//!
//! [`CostTable::from_roko_config`] is the canonical constructor. It builds
//! the pricing table by layering three sources, last-write-wins:
//!
//! 1. Hardcoded fallbacks ([`CostTable::with_defaults`]).
//! 2. `[models.<slug>]` profile fields (`cost_input_per_m`,
//!    `cost_output_per_m`, `cost_cache_read_per_m`,
//!    `cost_cache_write_per_m`, `tokenizer_ratio`). These come from the
//!    project's `roko.toml` model registry and are the primary place to
//!    document a model's wire pricing.
//! 3. `[cost.<slug>]` override entries (forward-compatible). When present
//!    in `roko.toml`, they replace the model profile's pricing for the
//!    same slug. Use this section to override pricing without editing the
//!    model registry — handy for testing and for operators who do not own
//!    the registry. The schema (mirrors `ModelPricing`):
//!
//!    ```toml
//!    [cost.claude-opus-4-6]
//!    input_per_m = 15.0
//!    output_per_m = 75.0
//!    cache_read_per_m = 3.75
//!    cache_write_per_m = 18.75
//!    tokenizer_ratio = 1.0
//!    ```
//!
//! [`CostTable::from_config`] (raw model-profile constructor) and
//! [`CostTable::with_defaults`] (fallback layer) remain `pub` for legacy
//! callers but are no longer the recommended entry points. New callers
//! should use [`CostTable::from_roko_config`].
```

### Step 2: Add the new `from_roko_config` constructor

In `crates/roko-learn/src/cost_table.rs`, just below the existing `pub fn from_config(...)` (currently lines 109-132), add:

```rust
    /// Construct a `CostTable` from a fully-resolved [`RokoConfig`].
    ///
    /// Layering order (each layer overwrites the previous):
    ///
    /// 1. [`Self::with_defaults`] seeds the well-known fallback rows.
    /// 2. `config.effective_models()` profile pricing (via
    ///    [`Self::from_config`] semantics).
    /// 3. Optional `[cost.<slug>]` override section (currently
    ///    forward-compatible: when no `cost` section is present in the
    ///    schema this layer is a no-op).
    ///
    /// Use this constructor in production paths so future schema additions
    /// (e.g. the `[cost.<slug>]` overrides described in the module docs)
    /// land everywhere at once.
    #[must_use]
    pub fn from_roko_config(config: &roko_core::config::schema::RokoConfig) -> Self {
        // Layer 1: seeded defaults.
        let mut table = Self {
            models: HashMap::new(),
        }
        .with_defaults();

        // Layer 2: per-profile pricing from the model registry. We
        // intentionally do NOT call Self::from_config here; that helper
        // discards rows missing both input and output. We want partial
        // rows to override partial fields when explicitly set.
        for profile in config.effective_models().values() {
            if let (Some(input), Some(output)) =
                (profile.cost_input_per_m, profile.cost_output_per_m)
            {
                table.models.insert(
                    profile.slug.clone(),
                    ModelPricing {
                        input_per_m: input,
                        output_per_m: output,
                        cache_read_per_m: profile
                            .cost_cache_read_per_m
                            .unwrap_or(input * 0.5),
                        cache_write_per_m: profile
                            .cost_cache_write_per_m
                            .unwrap_or(input * 1.25),
                        tokenizer_ratio: profile.tokenizer_ratio.unwrap_or(1.0),
                    },
                );
            }
        }

        // Layer 3: forward-compatible `[cost.<slug>]` overrides. When the
        // `RokoConfig` schema gains a `cost: HashMap<String, ModelPricing>`
        // field this loop will pick up overrides automatically; until
        // then it is a guarded no-op so the constructor is stable across
        // schema bumps.
        if let Some(overrides) = cost_overrides_from_config(config) {
            for (slug, pricing) in overrides {
                table.models.insert(slug, pricing);
            }
        }

        table
    }
```

### Step 4: Add a unit test for the layered constructor

Append to the existing `#[cfg(test)] mod tests { ... }` block:

```rust
    #[test]
    fn from_roko_config_layers_defaults_then_models() {
        use roko_core::config::schema::{ModelProfile, RokoConfig};

        // Empty config -> just the defaults.
        let empty = RokoConfig::default();
        let table = CostTable::from_roko_config(&empty);
        let opus = table
            .lookup("claude-opus-4-6")
            .expect("default opus row");
        assert!((opus.input_per_m - 15.00).abs() < 1e-12);

        // Config with an overriding model profile -> override wins.
        let mut cfg = RokoConfig::default();
        let mut profile = ModelProfile::default();
        profile.slug = "claude-opus-4-6".to_string();
        profile.cost_input_per_m = Some(20.0);
        profile.cost_output_per_m = Some(99.0);
        cfg.models.insert("opus".to_string(), profile);

        let table = CostTable::from_roko_config(&cfg);
        let opus = table
            .lookup("claude-opus-4-6")
            .expect("overridden opus row");
        assert!((opus.input_per_m - 20.0).abs() < 1e-12);
        assert!((opus.output_per_m - 99.0).abs() < 1e-12);
    }
```
