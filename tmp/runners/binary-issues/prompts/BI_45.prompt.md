# BI_45: PID file path uses .roko discovery

## Task

## Problem
`crates/roko-agent/src/process/registry.rs:24-28`:

```rust
/// Path to the persistent PID file: `<cwd>/.roko/runtime/agent-pids.json`.
fn agent_pids_path() -> Option<PathBuf> {
    let cwd = std::env::current_dir().ok()?;
    Some(cwd.join(".roko/runtime/agent-pids.json"))
}
```

If `roko` is invoked from a subdirectory of a project (e.g. `cd crates/foo && roko chat`), the PID registry writes to `crates/foo/.roko/runtime/agent-pids.json` — a brand-new directory unrelated to the workspace's real `.roko/`. Cleanup on the next startup then fails because `agent_pids_path()` resolves to a different `.roko/` for each invocation. Symptom: zombie agent processes accumulate across restarts.

Every other Roko subsystem already discovers `.roko/` by walking up from `cwd` (the same way `git` finds its `.git/`). The CLI even has a helper at `crates/roko-cli/src/workspace_paths.rs:17`:

```rust
pub fn roko_dir(workdir: &Path) -> PathBuf {
    workdir.join(".roko")
}
```

…but that helper takes an explicit `workdir` argument and assumes the caller has already done the discovery. `roko-agent` lives below `roko-cli` in the dependency graph (verify: `crates/roko-agent/Cargo.toml` depends on `roko-core` but **not** `roko-cli`), so `registry.rs` cannot import that helper directly. The fix is to add a small private discovery helper inside `registry.rs` itself; the helper walks up from the current working directory looking for a `.roko/` child.

This is intentionally local: the long-term plan (out of scope here) is to move the upward-walk helper into `roko-core` so every crate can share one definition. The `also_read` entry in `batches.toml` for this batch (`crates/roko-core/src/paths.rs`) reflects that intent — but the file does not yet exist in the workspace (verify with `ls crates/roko-core/src/paths.rs` → "No such file or directory"). Do **not** create it in this batch; that crate-level move is a follow-up.

//!
//! Tracks every child PID spawned by Roko so that:
//! - A restarting instance can kill zombies from a crash.
//! - The reaper can detect orphans reparented to PID 1 (init/launchd).
//!
//! The registry is backed by a static `OnceLock<Mutex<HashSet<u32>>>` and persists

use std::sync::Mutex;

use std::collections::HashSet;
use std::path::PathBuf;
use std::sync::OnceLock;

#[allow(clippy::disallowed_types)]
fn spawned_pids() -> &'static Mutex<HashSet<u32>> {
    static PIDS: OnceLock<Mutex<HashSet<u32>>> = OnceLock::new();
    PIDS.get_or_init(|| Mutex::new(HashSet::new()))
}

/// Path to the persistent PID file: `<cwd>/.roko/runtime/agent-pids.json`.
fn agent_pids_path() -> Option<PathBuf> {
    let cwd = std::env::current_dir().ok()?;

}

```rust
/// Walk upward from `start` looking for the nearest enclosing `.roko/`
/// directory (the workspace root marker).
///
/// Returns the path to the `.roko/` directory itself (not its parent).
/// Returns `None` only when no ancestor of `start` contains a `.roko/`
/// child; callers are expected to fall back to `start.join(".roko")` in
/// that case so a fresh checkout still has a deterministic path.
///
/// The walk stops at the filesystem root. Symbolic links along the
/// ancestor chain are followed by `Path::ancestors`, which is the same
/// behavior other Roko discovery helpers (e.g. `git`'s own `.git`
/// search) use.
///
/// BI_45: this helper is intentionally duplicated inside
/// `roko-agent/process/registry.rs` because `roko-agent` does not depend
/// on `roko-cli` (which owns the existing `workspace_paths::roko_dir`
/// helper). A follow-up batch will move the helper into
/// `crates/roko-core/src/paths.rs` and have both crates import it from
/// there; this duplication is the intermediate state.
fn find_roko_dir_from(start: &Path) -> Option<PathBuf> {
    for ancestor in start.ancestors() {
        let candidate = ancestor.join(ROKO_DIR_NAME);
        if candidate.is_dir() {
            return Some(candidate);
        }
    }
    None
}
```

```rust
/// Path to the persistent PID file:
/// `<discovered .roko/>/runtime/agent-pids.json`.
///
/// Discovery walks upward from the current working directory looking for
/// an existing `.roko/` ancestor (see [`find_roko_dir_from`]). When no
/// ancestor matches — typically a fresh checkout where `.roko/` has not
/// been created yet — the function falls back to `<cwd>/.roko/`, which
/// preserves the pre-BI_45 behavior so first-run cleanup still works.
fn agent_pids_path() -> Option<PathBuf> {
    let cwd = std::env::current_dir().ok()?;
    let roko_dir = find_roko_dir_from(&cwd).unwrap_or_else(|| cwd.join(ROKO_DIR_NAME));
    Some(roko_dir.join(PID_REGISTRY_SUBPATH))
}
```
