# BI_48: Remove or merge legacy chat.rs into chat_inline.rs

## Task
**No-op write** for the source code of `crates/roko-cli/src/chat.rs` itself. After verification, the legacy `chat.rs` file is **still actively referenced** from default-build paths (4 distinct call sites across `chat_inline.rs`, `dispatch_direct.rs`, `agent_serve.rs`, and `commands/agent.rs`). Per the task spec for BI_48 ("If actively used, add a TODO inventory of remaining call sites and DON'T remove (mark this batch as no-op with documented reason in commit message)"), this batch:

1. Adds a TODO-style call-site inventory at the top of `chat.rs` (and only there) so the next reviewer sees the merge plan without re-running ripgrep.
2. Leaves `chat.rs`, `chat_inline.rs`, and `lib.rs` source code otherwise unchanged.
3. Does NOT remove `pub mod chat;` from `lib.rs`.
4. Documents in the commit message that this is intentionally a no-op pending downstream caller migrations.

The follow-up batches that migrate each caller (and finally remove `chat.rs`) are explicitly **out of scope**.

## Runner Context
Group **BI_COD**, batch 48 of 56. Closes MASTER-INDEX § **S10.5**. Deps: none.

## Problem
`crates/roko-cli/src/chat.rs` (920 lines) and `crates/roko-cli/src/chat_inline.rs` (5507 lines) both exist. The audit hypothesis was that `chat.rs` was legacy and could be removed once `chat_inline.rs` covered the use case. Verification at HEAD shows the four `pub` items in `chat.rs` are still imported from default-build paths:

| Symbol | Defined at | Live caller(s) |
|--------|------------|----------------|
| `pub fn lookup_sidecar_url(agent_id, workdir) -> Option<String>` | `chat.rs:76` | `chat_inline.rs:1157` (inside `resolve_chat_backend`) |
| `pub async fn run_chat_repl(agent_id, serve_url) -> Result<()>` | `chat.rs:96` | `chat_inline.rs:1168` (fallback when `should_use_inline()` is false) |
| `pub async fn run_direct_provider_chat(...)` | `chat.rs:247` | `commands/agent.rs:31` |
| `pub fn extract_clean_text(raw: &str) -> String` | `chat.rs:566` | `chat_inline.rs:28, 4376`; `dispatch_direct.rs:30, 205`; `agent_serve.rs:514` |

Removing `chat.rs` (or its `pub mod chat;` line at `lib.rs:49`) would break the default build immediately. Per `00-RULES.md`'s default-build assumption ("If your prompt sends you there by accident, stop"), the right move is to leave `chat.rs` in place and document the planned merge.

`dispatch_direct.rs` is gated by `feature = "legacy-direct-dispatch"` (`lib.rs:64-65`), so its `extract_clean_text` use does not affect the default build — but it still affects compilation under that feature, and the runner's scope is "default build", so we should not break it either.
