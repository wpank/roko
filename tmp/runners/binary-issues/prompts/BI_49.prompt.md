# BI_49: Unify roko init and roko config init

## Task
Make `roko init` always (a) bootstrap the workspace (`.roko/` layout, `roko.toml` template, demo seed), then (b) if and only if no global config has ever been written (`~/.roko/config.toml` is absent), invoke the existing `run_init_wizard` for first-ever onboarding. `roko config init` is repurposed as a thin wrapper that always runs only the wizard. The existing `cmd_init` and `run_init_wizard` functions stay where they live; only their composition and the dispatch in `main.rs` change.

## Runner Context
Group **BI_COD**, batch 49 of 56. Closes MASTER-INDEX § **S10.6**. Deps: none.

## Problem
There are two onboarding entry points today:

1. `crates/roko-cli/src/commands/util.rs:104` — `pub(crate) async fn cmd_init(path, cloud, profile, demo) -> Result<()>` performs the **workspace** bootstrap: `.roko/` layout, `roko.toml` from template, optional demo seed, post-init hints.
2. `crates/roko-cli/src/config_cmd.rs:51` — `pub fn run_init_wizard(target, inputs) -> Result<PathBuf>` runs the interactive **global** wizard that detects installed LLM CLIs and writes `~/.roko/config.toml`.

`roko init` is dispatched at `main.rs:2070-2078` and only calls `cmd_init`. `roko config init` is dispatched at `commands/config_cmd.rs:19-57` (the `dispatch_config` function's `ConfigCmd::Init` arm) and only calls `run_init_wizard`. A first-time user who runs `roko init` against a fresh repo gets a `.roko/` layout but no global config — the next command (`roko run …`) then errors out about a missing provider.

Per the task spec for BI_49: unify `roko init` so it (1) always runs the workspace bootstrap, then (2) **only on the first-ever invocation on this machine** (no `~/.roko/config.toml` yet) prompts for the global wizard. `roko config init` becomes an alias that triggers only the wizard (its current behaviour, kept for muscle memory).

The existing helper `crate::config::global_config_path() -> PathBuf` (defined at `crates/roko-cli/src/config.rs:2713`) already returns the canonical global path; we just need to check `.exists()` on it.

## Required Changes

### Step 1: Make `cmd_init` chain the global wizard on first run

In `crates/roko-cli/src/commands/util.rs`, leave the existing body of `cmd_init` (lines 104-216) untouched up to and including the `// Check for interrupted session …` block. Just before the final `Ok(())` at line 218, insert:

```rust
    // BI_49: First-ever-run onboarding.
    //
    // `roko init` was historically workspace-only; the user then had to
    // discover `roko config init` (the global wizard). To eliminate that
    // hidden second step, run the wizard automatically iff the user has
    // never written a global config on this machine.
    //
    // Subsequent `roko init` calls in other workspaces skip the wizard
    // (the global config already exists), so this is idempotent and never
    // re-prompts established users.
    let global_path = crate::config::global_config_path();
    if !global_path.exists() {
        println!();
        println!(
            "no global config found at {} — running first-time setup wizard...",
            global_path.display()
        );
        let inputs = roko_cli::WizardInputs::default();
        let _ = roko_cli::run_init_wizard(None, &inputs)
            .with_context(|| "first-time global config wizard")?;
        println!();
        println!("global config written. you can re-run the wizard any time with:");
        println!("    roko config init");
    } else {
        println!();
        println!(
            "global config already present at {} (skipping wizard).",
            global_path.display()
        );
    }
```

`run_init_wizard` and `WizardInputs` are already re-exported from `crate::lib` via `pub use config_cmd::{EditTarget, WizardInputs, run_init_wizard};` (`crates/roko-cli/src/lib.rs:129`). `crate::config::global_config_path` is `pub` (`crates/roko-cli/src/config.rs:2713`). `Context as _` is already imported at the top of `commands/util.rs` (the file already uses `with_context`); no new imports are required for the snippet above.
