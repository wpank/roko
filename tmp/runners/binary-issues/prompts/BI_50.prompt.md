# BI_50: Switch dispatcher audit-signals lock to parking_lot::Mutex

## Task
Replace the `std::sync::Mutex<Vec<Engram>>` used by the in-test `CollectAuditSink` helper inside `crates/roko-agent/src/dispatcher/mod.rs` with `parking_lot::Mutex<Vec<Engram>>`, and drop the `expect("audit signals lock")` panics on every lock site. The `parking_lot` lock returns the guard directly (no poisoning), so the `.expect(...)` becomes a `.lock()` with no `?` / `unwrap` / `expect`.

## Runner Context
Group **BI_MTX**, batch 50 of 56. Closes MASTER-INDEX § **S11.1**. Deps: none.

## Problem
`crates/roko-agent/src/dispatcher/mod.rs:779,786-788` calls `.lock().expect("audit signals lock")` against a `std::sync::Mutex`. Per `00-RULES.md` rule **M1** (`parking_lot::Mutex` over `std::sync::Mutex`) and **M3** (`expect("…")` is a panic), this should be `parking_lot::Mutex` whose `lock()` returns the guard directly.

The lock lives inside the `#[cfg(test)] mod tests` block (the `CollectAuditSink` test helper, declared at `dispatcher/mod.rs:772-790`). Even in test code the rule still applies, because the `expect("audit signals lock")` panic message has been the audit's flagged hit and there is no reason for the test sink to keep poisoning semantics — flipping the type is mechanical and removes one source of `.expect()`.

`parking_lot` is already a workspace dependency (verified: `Cargo.toml:159` declares `parking_lot = "0.12"`) and the agent crate already pulls it in (verified: `crates/roko-agent/Cargo.toml:31` has `parking_lot = { workspace = true }`). No `Cargo.toml` changes are required.
