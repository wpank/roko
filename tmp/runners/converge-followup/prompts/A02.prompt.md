# A02: Delete Runtime-Local AffectPolicy

## Task

Remove the duplicate `AffectPolicy`, `AffectContext`, and `DispatchModulation` definitions from
`crates/roko-runtime/src/effect_driver.rs` and use the canonical versions from
`crates/roko-core/src/foundation.rs` directly.

Also delete the CLI policy bridge in `crates/roko-cli/src/run.rs` that exists only to translate
between the two trait versions.

## Changes Required

### In `crates/roko-runtime/src/effect_driver.rs`:

1. Delete the local `AffectContext`, `DispatchModulation`, and `AffectPolicy` trait definitions
   (approximately lines 31-87).

2. Add `use roko_core::foundation::{AffectPolicy, DispatchModulation};` (and `AffectContext` if
   it exists in foundation, otherwise define a minimal one or use the foundation equivalent).

3. Update `EffectDriver` to use the foundation trait. The driver stores a `Box<dyn AffectPolicy>`
   — make sure the trait bounds match the foundation version.

4. Where `EffectDriver` calls `policy.modulate(...)`, adapt the call to match the foundation
   trait's method signature. If the signatures differ, align the runtime call to use the
   foundation types.

### In `crates/roko-cli/src/run.rs`:

1. Delete the `RuntimePolicyAdapter` struct and its `impl` block (approximately lines 319-405)
   that bridges between foundation and runtime policy traits.

2. Where the adapter was constructed and passed to `EffectDriver`, pass the daimon policy
   directly since it already implements the foundation `AffectPolicy`.

## Acceptance Criteria

- `grep -rn 'pub trait AffectPolicy' crates/` returns ONLY `crates/roko-core/src/foundation.rs`.
- No `RuntimePolicyAdapter` or equivalent bridge struct exists.
- `cargo check -p roko-runtime -p roko-cli` passes.

## Do NOT

- Change the foundation trait signature.
- Add new trait bounds beyond what the foundation trait requires.
