# B02: Apply Affect Modulation

## Task

Make the `DispatchModulation` computed by `EffectDriver` actually affect the model request
and execution behavior, instead of being computed and discarded.

## Problem

At approximately lines 133-138, `EffectDriver` calls `policy.modulate(&context)` to get a
`DispatchModulation` value. But the modulation is never applied — the `ModelCallRequest` at
~lines 176-189 is built with empty/default values regardless.

## Changes Required

### In `crates/roko-runtime/src/effect_driver.rs`:

1. After computing `modulation`, apply it to the `ModelCallRequest`:
   - `modulation.model_override` → set request `model` field
   - `modulation.temperature_override` → set request temperature/extra args
   - `modulation.budget_limit` → set request `budget_remaining`
   - `modulation.max_turns` → set request max turns or similar field
   - `modulation.routing_hints` → set request `routing_hints`

2. If `DispatchModulation` doesn't have these fields (check `foundation.rs`), use whatever
   fields it does have. The point is: the modulation output must visibly change the request.

3. If the modulation suggests `skip` or `defer`, honor that by returning early without
   spawning the agent.

4. Log the applied modulation at debug level so it's observable.

## Acceptance Criteria

- The `modulation` variable is used after computation (not just computed and dropped).
- A test or code path shows that a non-default modulation changes the `ModelCallRequest`.
- `cargo check -p roko-runtime` passes.

## Do NOT

- Change the `AffectPolicy` trait (it's in core foundation now, from A02).
- Change the `DispatchModulation` type definition.
