# B03: Typed Gate Config — Fix Shell/Custom Name Mismatch

## Task

Fix the gate name mismatch between CLI and `GateService`, and remove the no-op custom gate.

## Problem

1. CLI passes `"shell"` for shell gates at `crates/roko-cli/src/commands/util.rs:245-250`
2. `GateService` recognizes `custom` and `custom:shell` at `crates/roko-gate/src/gate_service.rs:49-60`
3. The `custom` gate always returns `true` at `crates/roko-gate/src/gate_service.rs:74-77`

Result: shell gates either fail silently (wrong name) or pass silently (no-op).

## Changes Required

### In `crates/roko-gate/src/gate_service.rs`:

1. Replace the no-op custom gate at ~lines 74-77:
   ```rust
   // BEFORE (bad):
   "custom" => Ok(GateResult { passed: true, ..Default::default() })
   ```
   With an explicit error or typed custom gate config:
   ```rust
   // AFTER (good):
   "custom" => anyhow::bail!("custom gate requires explicit command configuration")
   ```

2. Add a `"shell"` gate type that maps to the same handling as `"custom:shell"`, so both
   `"shell"` and `"custom:shell"` work.

3. If `custom:shell` has real behavior, make sure `"shell"` delegates to it. If it doesn't,
   add real shell command execution (read a command from gate config, run it, check exit code).

### In `crates/roko-cli/src/commands/util.rs`:

1. At ~lines 245-250, verify the gate name being passed matches what `GateService` expects.
   If the code passes `"shell"`, and `GateService` now handles `"shell"`, this should work.

2. If gate configs include a command string, make sure it's passed through to `GateService`.

## Acceptance Criteria

- `GateService` does NOT return `Ok(passed: true)` for unconfigured custom gates.
- The gate name used by CLI matches the name handled by `GateService`.
- Shell gates either execute a real command or return an explicit error.
- `cargo check -p roko-gate -p roko-cli` passes.

## Do NOT

- Change the gate pipeline ordering or rung system.
- Add new gate types beyond fixing the existing ones.
