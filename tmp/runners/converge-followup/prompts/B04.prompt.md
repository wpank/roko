# B04: Resume and Checkpoint Correctness

## Task

Fix resume behavior so it emits the recovered phase transition and uses a non-empty checkpoint path.

## Problem

1. Resume at ~`workflow_engine.rs:312-313` starts from `resumed_output` without emitting the
   first recovered phase transition event.
2. Resume emits `StateCheckpointed` with an empty path at ~`workflow_engine.rs:417-421`.
3. `EffectDriver::save_checkpoint` correctly writes an atomic file and emits the path at
   ~`effect_driver.rs:381-410`, but the resume path doesn't use the same effect.

## Changes Required

### In `crates/roko-runtime/src/workflow_engine.rs`:

1. When resuming from a checkpoint, emit a `PhaseTransition` event for the recovered state
   before continuing execution. This lets observers know the engine resumed and what phase
   it's in.

2. Replace the empty checkpoint path in `StateCheckpointed` events with the actual file path
   from the checkpoint operation. If the checkpoint path is available from `EffectDriver`,
   use it directly.

3. Route checkpoint save during resume through the same `EffectDriver::save_checkpoint` path
   used during normal execution. Don't emit a separate `StateCheckpointed` with different
   (empty) metadata.

4. Use `RuntimeEventEnvelope` (from A01) with `source: "workflow_engine"` for phase events.

## Acceptance Criteria

- Resuming from a checkpoint emits at least one `PhaseTransition` event with the recovered phase.
- No `StateCheckpointed` event has an empty checkpoint path.
- Checkpoint save during resume uses the same code path as during normal execution.
- `cargo check -p roko-runtime` passes.

## Do NOT

- Change the checkpoint file format.
- Add new checkpoint trigger points.
