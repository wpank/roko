# B05: Workflow Completion Feedback

## Task

Emit workflow completion/failure feedback events and flush them through the feedback sink.

## Problem

`WorkflowEngine` has TODOs at ~lines 206-207 and 224-225 where workflow final feedback should
be emitted. Currently, workflow completion does not record feedback, so the learning loop
cannot learn from normal v2 runs.

## Changes Required

### In `crates/roko-runtime/src/workflow_engine.rs`:

1. At the points marked TODO for workflow feedback (~lines 206-207, 224-225), emit
   `FeedbackEvent` with:
   - `event_type`: "workflow_completed" or "workflow_failed"
   - `run_id`: from the workflow context
   - `model`: the model used for the primary agent call
   - `success`: whether the workflow succeeded
   - `token_usage`: total tokens consumed across all agent calls
   - `cost`: total estimated cost
   - `duration`: elapsed wall-clock time

2. After emitting the feedback event, call `feedback_sink.record(event)` if a sink is
   available. The engine likely holds an `Option<Box<dyn FeedbackSink>>` or similar.

3. Call `feedback_sink.flush()` after the final feedback event to ensure it's persisted
   before the engine returns.

4. If the engine doesn't currently hold a feedback sink, add one to `WorkflowEngine::new()`
   as an optional parameter. Don't make it mandatory yet (that's Wave C).

## Acceptance Criteria

- Successful workflow completion emits a feedback event with real metadata.
- Failed workflow completion also emits a feedback event (with `success: false`).
- Feedback is flushed before the engine returns.
- `cargo check -p roko-runtime` passes.

## Do NOT

- Make feedback mandatory in the constructor (that's C01's job).
- Add learning or analytics logic — just emit the event.
