# B06: Typed Review and Commit Outcomes

## Task

Replace hardcoded review approval and error-as-hash commit results with typed outcomes.

## Problem

1. At ~`workflow_engine.rs:471-477`, any `AgentCompleted` review result is mapped to
   `ReviewApproved`. Real review semantics are lost.

2. At ~`effect_driver.rs:311-378`, commit errors are encoded as `CommitDone { hash: "error: ..." }`.
   This makes invalid state look valid.

## Changes Required

### In `crates/roko-runtime/src/workflow_engine.rs`:

1. Parse the review agent's output to determine the actual review decision:
   - If the output contains explicit approval language or a structured approval signal,
     emit `ReviewApproved`.
   - If the output contains rejection or revision requests, emit `ReviewRejected { reason }`.
   - If the output is ambiguous, emit `ReviewUnclear` and let the pipeline decide
     (default to requesting revision, not approving).

2. Add `ReviewRejected` and `ReviewUnclear` as new variants if they don't exist, or use
   whatever typed review outcome variants already exist in `RuntimeEvent`.

### In `crates/roko-runtime/src/effect_driver.rs`:

1. Replace the error-in-success pattern for commits:
   ```rust
   // BEFORE (bad):
   CommitDone { hash: format!("error: {err}") }

   // AFTER (good):
   CommitFailed { error: format!("{err}") }
   ```

2. Add a `CommitFailed` variant to the commit result type or use `Result<CommitDone, CommitError>`.

3. Propagate commit failures upward so the workflow engine can handle them (retry, skip, fail).

## Acceptance Criteria

- Review outcomes are NOT hardcoded to approval.
- Commit errors are NOT encoded inside success variants.
- `grep -n 'CommitDone.*error' crates/roko-runtime/` returns nothing.
- `cargo check -p roko-runtime` passes.

## Do NOT

- Implement an actual LLM judge for reviews (that's separate).
- Change the commit logic beyond error handling.
