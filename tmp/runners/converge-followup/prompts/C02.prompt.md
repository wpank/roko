# C02: ModelCallService Completeness

## Task

Attach feedback, gateway writer, router, cascade, knowledge, and event consumers to
`ModelCallService` through the service factory.

## Problem

`ModelCallService` at `model_call_service.rs` has optional feedback. If no sink is attached
(~lines 413-426), it returns success and records nothing. Knowledge advice is computed but
mostly logged into debug metadata (~lines 1314-1317). Fallback models are empty (~line 1237).

## Changes Required

### In `crates/roko-agent/src/model_call_service.rs`:

1. Where the service is constructed (via the factory from C01), ensure these are attached:
   - `FeedbackSink` — for recording model call outcomes
   - `GatewayEventWriter` — for durable gateway events (if available from `gateway_events.rs`)
   - Cascade router config — so `fallback_models` is populated from config
   - Knowledge store — so knowledge advice actually influences prompt or routing

2. When feedback is attached, the call path at ~lines 413-426 should always record:
   - Model used, provider, tokens, cost, duration, success/failure
   - Prompt section ids and knowledge ids (from the `ModelCallRequest` metadata added in A03)

3. If `knowledge_advice` is computed at ~lines 465-531, make it do something meaningful:
   - Include relevant knowledge in the system prompt or routing hints
   - Or at minimum, set `request.knowledge_ids` so feedback can correlate outcomes

4. Populate `fallback_models` from the workspace config cascade router settings instead of
   `Vec::new()` at ~line 1237.

## Acceptance Criteria

- Production `ModelCallService` instances have non-empty feedback sinks.
- Model calls record feedback events with model, tokens, cost, and provenance metadata.
- `fallback_models` is populated from config (not empty).
- `cargo check -p roko-agent` passes.

## Do NOT

- Change the provider dispatch logic.
- Add new provider backends.
