# C06: Cache Key Preserves Message Order, Budget Correctness

## Task

Fix the prompt cache key to preserve message order, and document budget scope.

## Problem

1. Cache key at `model_call_service.rs:672-677` **sorts** messages before hashing. This makes
   different conversations produce the same cache key if they contain the same messages in
   different order. Message order is semantically meaningful in chat.

2. Budget tracking at `model_call_service.rs:742-801` is process-local and cumulative. It's
   not a durable, reserved, cross-run budget. This should at minimum be documented.

## Changes Required

### In `crates/roko-agent/src/model_call_service.rs`:

1. Fix the cache key at ~lines 672-677:
   ```rust
   // BEFORE (bad): sorts messages, loses ordering
   let mut messages = request.messages.clone();
   messages.sort_by(|a, b| a.content.cmp(&b.content));

   // AFTER (good): hash messages in order
   for msg in &request.messages {
       hasher.update(msg.role.as_bytes());
       hasher.update(msg.content.as_bytes());
   }
   ```

2. The cache key should include: model name + messages in order + any relevant parameters
   (temperature, max_tokens, tools). Do NOT sort or reorder.

3. For budget tracking at ~lines 742-801:
   - Add a comment documenting that budget is process-local and best-effort
   - If concurrent calls can race on the budget counter, use `AtomicU64` or `tokio::sync::Mutex`
   - Do NOT attempt to make budget durable in this batch (too invasive)

## Acceptance Criteria

- A test proves that swapping message order changes the cache key.
- Cache key includes model name and message order.
- Budget code has a documentation comment about its scope.
- `cargo check -p roko-agent` passes.

## Do NOT

- Change the cache eviction strategy.
- Implement persistent budget tracking.
