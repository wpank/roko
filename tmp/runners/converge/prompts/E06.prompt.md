/// Runtime lifecycle events emitted directly by the workflow engine.
#[derive(Debug, Clone)]
pub enum WorkflowEvent {
    WorkflowStarted { run_id: String, template: String, prompt: String },
    PhaseTransition { run_id: String, from: String, to: String },
    WorkflowCompleted { run_id: String, outcome: WorkflowOutcome },
}

/// Consume workflow lifecycle events for side effects such as logging or UI updates.
pub trait WorkflowEventConsumer: Send + Sync {
    fn consume(&self, event: &WorkflowEvent);
}

pub struct WorkflowEngine {
    services: EffectServices,
    consumers: Vec<Arc<dyn WorkflowEventConsumer>>,
}

impl WorkflowEngine {
    pub fn new(services: EffectServices) -> Self { ... }
    pub fn add_consumer(&mut self, consumer: Arc<dyn WorkflowEventConsumer>) { ... }
    pub async fn run(&self, config: WorkflowRunConfig) -> Result<WorkflowResult> { ... }

    fn emit(&self, event: WorkflowEvent) {
        for consumer in &self.consumers {
            consumer.consume(&event);
        }
        emit_runtime_event(event);
    }
}

The current `run()` loop emits three `WorkflowEvent`s: `WorkflowStarted` (before the loop),
`PhaseTransition` (each iteration when phase changes), and `WorkflowCompleted` / halt (in
the `Done` / `Halt` arms).

### Task

After F06 lands, the engine uses `RuntimeEvent` and `EventConsumer` from roko-core. This
batch extends the event emission to cover agent spawns and gate runs so that consumers
receive a complete picture of execution.

#### 1. Emit `RuntimeEvent::WorkflowStarted` at the start of `run()`

This already happens — verify the call exists and uses `RuntimeEvent::WorkflowStarted`
(not the old `WorkflowEvent::WorkflowStarted`).

#### 2. Emit `RuntimeEvent::AgentSpawned` before each agent dispatch

In the `run()` loop, before each `driver.spawn_agent(...)` call, emit:

```rust
self.emit(RuntimeEvent::AgentSpawned {
    run_id: run_id.clone(),
    agent_id: String::new(),  // agent_id not known until driver assigns it
    role: role.to_string(),
    model: String::new(),     // model not known until ModelCaller routes
});
```

The driver's `spawn_agent` already emits its own `AgentSpawned` (with agent_id and model
filled in after the call). This engine-level emit is a pre-dispatch notification so
consumers can show a "spawning..." indicator before the potentially long model call.

Add this pre-dispatch emit before each of the four `spawn_agent` call sites:
- `SpawnStrategist` arm: role = `"strategist"`
- `SpawnImplementer` arm: role = `"implementer"`
- `SpawnAutoFixer` arm: role = `"autofix"`
- `SpawnReviewer` arm: role = `"reviewer"`

#### 3. Emit `RuntimeEvent::GateStarted` before gates run

Before `driver.run_gates(...)`:

```rust
self.emit(RuntimeEvent::GateStarted {
    run_id: run_id.clone(),
    gate_name: "pipeline".to_string(),
    rung: 0,
});
```

The per-gate `GatePassed`/`GateFailed` events are emitted by the `EffectDriver` internally.
This engine-level event is a coarse-grained "gates starting" notification.

#### 4. Emit `RuntimeEvent::PhaseTransition` after each `pipeline.step()`

This already happens — verify the `if old_phase != new_phase` check is still correct and
emits `RuntimeEvent::PhaseTransition` (not the old `WorkflowEvent::PhaseTransition`).

#### 5. `add_consumer` takes `Arc<dyn EventConsumer>`

After F06, the `consumers` field is `Vec<Arc<dyn EventConsumer>>` where `EventConsumer`
is `roko_core::foundation::EventConsumer`. Verify `add_consumer` accepts the right type.

#### 6. No structural changes to the run loop

Do not reorder the `match output` arms. Do not change the `strategy_input` / `reviewer_input`
adapter functions. Do not change `generate_run_id`, `commit_message`, `template_name`, or
`truncate`.
