# Batch G09 — Wire force_backend to CascadeRouter learning

## Write Scope

- `crates/roko-agent/src/model_call_service.rs`
- `crates/roko-learn/src/cascade_router.rs`

## Task

When a caller uses `force_backend` (an explicit model override in the request), the gateway
should record this as a forced observation in the `CascadeRouter` so the routing bandit
learns from manual model selections. This closes the UX34 gap documented in the codebase.

### Background

The `CascadeRouter` already has a `record_override_outcome()` method:

```rust
/// Record a force_backend override outcome for learning (UX34).
pub fn record_override_outcome(
    &self,
    model_slug: &str,
    ctx: &RoutingContext,
    success: bool,
) -> bool {
    // ... dampened reward at OVERRIDE_LEARNING_RATE ...
}
```

This method exists but is never called from `ModelCallService`. The model call service needs
to detect when a caller explicitly specified a model (as opposed to letting the router
choose) and, after the call completes, feed that observation back to the router.

### Changes to ModelCallService

1. Add an optional `CascadeRouter` reference to `ModelCallService`:

```rust
pub struct ModelCallService {
    // ... existing fields ...
    /// Optional cascade router for recording forced model observations.
    cascade_router: Option<Arc<CascadeRouter>>,
}
```

2. Add a builder method:

```rust
impl ModelCallService {
    /// Attach a CascadeRouter for force_backend learning (UX34).
    #[must_use]
    pub fn with_cascade_router(mut self, router: Arc<CascadeRouter>) -> Self {
        self.cascade_router = Some(router);
        self
    }
}
```

3. In `ModelCallService::call()`, after the provider call completes successfully, check
   whether the request specified an explicit model (non-empty `req.model`). If so, and if
   a `CascadeRouter` is available, call `record_override_outcome()`:

```rust
// After successful provider call:
if !req.model.is_empty() {
    if let Some(ref router) = self.cascade_router {
        let routing_ctx = RoutingContext {
            task_category: TaskCategory::Implementation,
            complexity: TaskComplexityBand::Standard,
            iteration: 1,
            role: AgentRole::Implementer,
            // ... defaults for fields not available from the request ...
            ..RoutingContext::default()
        };
        router.record_override_outcome(&model_used, &routing_ctx, true);
    }
}
```

For failed calls, pass `success: false`:

```rust
// After provider call failure:
if !req.model.is_empty() {
    if let Some(ref router) = self.cascade_router {
        let routing_ctx = RoutingContext::default();
        router.record_override_outcome(&model_used, &routing_ctx, false);
    }
}
```

### Existing Code Context

`CascadeRouter::record_override_outcome` (lines 904-925 of cascade_router.rs):

```rust
pub fn record_override_outcome(
    &self,
    model_slug: &str,
    ctx: &RoutingContext,
    success: bool,
) -> bool {
    let Some(model_idx) = self.model_index_for_slug(model_slug) else {
        return false;
    };
    let raw_reward = if success { 1.0 } else { 0.0 };
    let dampened_reward = raw_reward * OVERRIDE_LEARNING_RATE;
    self.observe_internal(
        &ctx.to_features(),
        model_idx,
        dampened_reward,
        success,
        None,
        None,
    );
    true
}
```

### Tests

Add these tests in `model_call_service.rs`:

1. `force_backend_records_when_router_present` — create a `ModelCallService` with a
   `CascadeRouter`, make a call with an explicit model, verify the router's confidence
   stats were updated. (This requires the `CascadeRouter` to be constructable in a test.
   Use `CascadeRouter::load_or_new` with a temp path and a small model list.)

2. `force_backend_noop_when_no_router` — create a `ModelCallService` without a router,
   make a call with an explicit model, verify no panic.

If `CascadeRouter` construction in tests is too complex due to required state, test the
recording logic via a dedicated unit test that calls `record_override_outcome()` directly
on a test-constructed router, and add a `// TODO(gateway): integration test for force_backend
learning` comment.

## DO NOT

- Change the `record_override_outcome` method signature or its dampening logic. The
  `OVERRIDE_LEARNING_RATE` is already tuned.
- Add `roko-learn` as a dependency of `roko-core`. The dependency goes on `roko-agent` only.
- Record observations when `req.model` is empty (the router chose the model, not the user).
- Make the router recording synchronous-blocking in a way that could slow down the call path.
  The `record_override_outcome` call is already non-blocking (it updates in-memory state).
- Touch `crates/roko-serve/` or `crates/roko-cli/`. The router wiring from those surfaces
  into `ModelCallService` happens when they construct the service (already covered by G04/G05).

## Done Criteria

1. `cargo check -p roko-agent` passes.
2. `cargo check -p roko-learn` passes.
3. `ModelCallService` has a `with_cascade_router()` builder method.
4. When `req.model` is non-empty and a router is attached, `record_override_outcome()` is
   called after the provider returns.
5. Success calls pass `success: true`; failures pass `success: false`.
6. When no router is attached, the override recording is silently skipped.
7. The UX34 gap is closed: manual model overrides now feed back into routing learning.
