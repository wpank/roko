# Batch K05 — Knowledge confidence update loop

## Write Scope
- **MODIFY**: `crates/roko-neuro/src/lib.rs`

## Task

Add methods to `KnowledgeStore` (and/or the `NeuroStore` trait) for updating knowledge confidence based on usage outcomes, creating a closed feedback loop between model routing decisions and durable knowledge.

### 1. Add `update_confidence()` to KnowledgeStore

```rust
/// Adjust the confidence score of a knowledge entry by `delta`.
///
/// The resulting confidence is clamped to `[0.0, 1.0]`. If the entry
/// is not found, this is a no-op (returns Ok(false)).
pub fn update_confidence(&mut self, knowledge_id: &str, delta: f64) -> Result<bool>
```

Implementation:
1. Load all entries from the JSONL backing file.
2. Find the entry with matching `id`.
3. If found, adjust `entry.confidence = (entry.confidence + delta).clamp(0.0, 1.0)`.
4. Rewrite the backing file with the updated entries.
5. Return `Ok(true)` if found and updated, `Ok(false)` if not found.

### 2. Add `record_usage()` to KnowledgeStore

```rust
/// Record a usage outcome for a knowledge entry.
///
/// - `succeeded = true`: bump confidence by +0.02 (small positive reinforcement)
/// - `succeeded = false`: reduce confidence by -0.05 (stronger negative signal)
///
/// Entries that drop below confidence 0.1 after repeated failures will be
/// candidates for the next GC pass.
pub fn record_usage(&mut self, knowledge_id: &str, succeeded: bool) -> Result<()>
```

Implementation:
1. Compute delta: `if succeeded { 0.02 } else { -0.05 }`.
2. Call `self.update_confidence(knowledge_id, delta)?`.
3. Log via `tracing::debug!`:
   ```rust
   tracing::debug!(
       knowledge_id,
       succeeded,
       delta,
       "recorded knowledge usage outcome"
   );
   ```

### 3. Add `batch_record_usage()` for efficiency

```rust
/// Record usage outcomes for multiple knowledge entries at once.
///
/// More efficient than calling `record_usage` in a loop because it
/// performs a single load-modify-write cycle.
pub fn batch_record_usage(
    &mut self,
    outcomes: &[(String, bool)],  // (knowledge_id, succeeded)
) -> Result<usize>
```

Implementation:
1. Load all entries once.
2. For each `(id, succeeded)` in outcomes, find the matching entry and adjust confidence.
3. Rewrite once.
4. Return the count of entries that were actually found and updated.

### 4. Add tier promotion/demotion logic

Add a private method:

```rust
/// Check if an entry's confidence warrants tier promotion or demotion.
fn maybe_adjust_tier(entry: &mut KnowledgeEntry)
```

Rules:
- If `entry.confidence >= 0.9` and current tier is below `Validated`, promote to `Validated`.
- If `entry.confidence <= 0.2` and current tier is above `Provisional`, demote to `Provisional`.
- If `entry.confidence <= 0.05`, mark the entry's `half_life_days` to a very short value (e.g., 1.0 day) so decay eliminates it quickly.

Call `maybe_adjust_tier` from within `update_confidence()` after adjusting the confidence value.

Check `KnowledgeEntry` for the actual tier field name -- it may be `validation_tier`, `tier`, or similar. Use whatever field exists. If no tier field exists, skip this step and document it as a gap.
