# Batch L04 — Add layer-check to CI workflow

## Write Scope
- **MODIFY**: `.github/workflows/ci.yml`

## Task

Add a CI job that runs the `roko layer-check` subcommand after the build, ensuring layer dependency violations are caught in pull requests.

### 1. Add the layer-check job

Add a new job to `.github/workflows/ci.yml` after the existing `test` job:

```yaml
  layer-check:
    name: Layer Check
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
      - name: Build roko-cli
        run: cargo build -p roko-cli
      - name: Run layer check
        run: cargo run -p roko-cli -- layer-check
```

### 2. Placement

The new job should be independent of the `test` and `fmt` jobs (it can run in parallel). Place it after the `fmt` job definition.

### Existing Code Context

Current `.github/workflows/ci.yml`:
```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:

env:
  CARGO_TERM_COLOR: always
  RUSTFLAGS: "-D warnings"

jobs:
  test:
    name: Test + Clippy
    runs-on: ubuntu-latest
    timeout-minutes: 60
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          components: clippy
      - uses: Swatinem/rust-cache@v2
      - run: cargo clippy --workspace --no-deps -- -D warnings
      - run: cargo test --workspace

  fmt:
    name: Formatting
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@nightly
        with:
          components: rustfmt
      - run: cargo fmt --all --check
```

### Notes

- The layer-check job uses `cargo run -p roko-cli -- layer-check` rather than a pre-built binary because this is a workspace tool, not an external dependency.
- The `rust-cache` action is included for build speed.
- The timeout is 15 minutes, which is generous for a metadata-only check (the bulk of time is compilation).
- The job does NOT install `cargo-deny` -- that is a separate concern (L03 creates the config, but CI integration for cargo-deny can be added later or as a separate step).

## DO NOT

- Remove or rename any existing CI configuration
- Add cargo-deny as a CI step (that is a separate task)
- Change the `on:` trigger configuration
- Add secrets, environment variables, or permissions beyond what already exists
- Create a new workflow file -- add to the existing `ci.yml`
