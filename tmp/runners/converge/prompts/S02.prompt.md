The existing provider adapters registered in `adapter_for_kind`:
```rust
pub fn adapter_for_kind(kind: ProviderKind) -> &'static dyn ProviderAdapter {
    match kind {
        ProviderKind::OpenAiCompat  => &OPENAI_COMPAT_ADAPTER,
        ProviderKind::ClaudeCli     => &CLAUDE_CLI_ADAPTER,
        ProviderKind::AnthropicApi  => &ANTHROPIC_API_ADAPTER,
        ProviderKind::CursorAcp     => &CURSOR_ACP_ADAPTER,
        ProviderKind::PerplexityApi => &PERPLEXITY_ADAPTER,
        ProviderKind::GeminiApi     => &GEMINI_ADAPTER,
    }
}
```

`ProviderKind` variants in roko-core (from `roko_core::agent`):
- `ClaudeCli` — spawns `claude` CLI subprocess
- `AnthropicApi` — direct Anthropic REST API
- `OpenAiCompat` — OpenAI-compatible REST endpoint
- `CursorAcp`, `PerplexityApi`, `GeminiApi`

Model slug routing convention: models prefixed with `claude-` route to either
`ClaudeCli` (when `agent.command = "claude"`) or `AnthropicApi` (when `ANTHROPIC_API_KEY`
is set and no command override). OpenAI and compatible models (e.g., `gpt-4o`, `glm-5-1`)
route to `OpenAiCompat`.

`ModelCallService.config` is already a `RokoConfig`. `create_agent_for_model` consults
`config.models` and `config.providers` to determine which adapter to use. When those maps
are empty (default config), it falls back to the `agent.command` field or the `ExecAgent`
fallback.

### Task

1. Add a `with_anthropic_api_key(key: String) -> Self` builder method that sets the
   `ANTHROPIC_API_KEY` env var entry in a stored `Vec<(String, String)>` on the service, and
   passes it via `AgentOptions.env` in the `call` method.

2. Add a `with_openai_base_url(url: String) -> Self` builder method that stores the base URL
   and injects it as a provider config entry keyed to `"openai-compat"` when building the
   agent options. Use the existing pattern from `provider/mod.rs` tests:
   ```rust
   config.providers.insert("openai-compat", ProviderConfig {
       kind: ProviderKind::OpenAiCompat,
       base_url: Some(url),
       api_key_env: Some("OPENAI_API_KEY".into()),
       ..ProviderConfig::default()  // check what default looks like
   });
   ```

3. In the `call` method, when `model` starts with `"claude-"` and neither `config.models`
   nor `config.providers` has an explicit entry for it, AND `ANTHROPIC_API_KEY` is set in the
   stored env entries, inject a minimal `AnthropicApi` provider config so
   `create_agent_for_model` routes to `AnthropicApiAdapter` instead of falling back to
   `ExecAgent`. The injected config should use `timeout_ms: Some(120_000)`.

4. Add a unit test `routes_claude_model_to_anthropic_api_when_key_set` that constructs a
   `ModelCallService` with `with_anthropic_api_key("sk-test".into())`, creates a request
   with `model: "claude-haiku-4"`, and asserts that `create_agent_for_model` is called with
   a config that has an `AnthropicApi` provider entry. (Use the mock env approach from the
   existing provider tests, or simply call `resolve_model` + inspect `config`.)
