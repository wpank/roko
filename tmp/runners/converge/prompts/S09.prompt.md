### Existing Code Context

Current `PromptAssemblyService` struct (from `prompt_assembly_service.rs`):
```rust

}
```

Current `assemble` method:
```rust
#[async_trait]
impl PromptAssembler for PromptAssemblyService {
    async fn assemble(&self, spec: PromptSpec) -> Result<String> {
        let role = resolve_role(spec.role.as_deref());
        let identity = role_identity_for(role);

        let mut builder = SystemPromptBuilder::new(identity);

        if let Some(conventions) = conventions_for_spec(&spec, self.default_conventions.as_deref()) {
            builder = builder.with_conventions(conventions);
        }

        if let Some(task) = spec.task {
            builder = builder.with_task(task);
        }

        for feedback in spec.gate_feedback {
            builder = builder.with_gate_feedback_text(feedback);
        }

        if !spec.anti_patterns.is_empty() {
            builder = builder.with_anti_patterns(spec.anti_patterns);
        }

        Ok(builder.build())
    }
}
```

`SystemPromptBuilder` relevant methods:
```rust
pub fn with_token_budget(mut self, budget: usize) -> Self
pub fn build(self) -> String
```

The builder assembles 9 layers in order:
1. Role identity
2. Conventions
3. Domain / context
4. Task / gate feedback
5. Tools
6. Skills / playbooks
7. Anti-patterns
8. Affect state
9. (cache markers — structural, not content)

Each layer contributes a section of text. The `with_token_budget` applies a hard cap on
total output length. There is no per-section budget control in the builder itself — that
logic must live in the service.

### Task

1. Add a new optional field to `PromptAssemblyService`:
   ```rust
   section_effectiveness: Option<HashMap<String, f64>>,
   ```

2. Add a builder method:
   ```rust
   /// Set per-section effectiveness scores.
   ///
   /// Keys are section names matching the 9 builder layers:
   /// `"identity"`, `"conventions"`, `"domain"`, `"context"`, `"task"`,
   /// `"gate_feedback"`, `"tools"`, `"playbooks"`, `"anti_patterns"`.
   ///
   /// Values are effectiveness scores in `[0.0, 1.0]`. Sections with
   /// score < 0.1 are skipped entirely. Scores are used to scale the
   /// section's share of the token budget.
   pub fn with_section_effectiveness(mut self, scores: HashMap<String, f64>) -> Self
   ```

3. In `assemble`, before building the prompt, apply the effectiveness filter. Check each
   section against the scores map. If a section has a score below 0.1, skip injecting it
   into the builder entirely. This means wrapping each builder call in a guard:
   ```rust
   fn should_include(&self, section: &str) -> bool {
       self.section_effectiveness
           .as_ref()
           .and_then(|scores| scores.get(section))
           .map_or(true, |&score| score >= 0.1)
   }
   ```

4. Apply the guard around each section injection. For example:
   ```rust
   if self.should_include("conventions") {
       if let Some(conventions) = conventions_for_spec(&spec, self.default_conventions.as_deref()) {
           builder = builder.with_conventions(conventions);
       }
   }

   if self.should_include("task") {
       if let Some(task) = spec.task {
           builder = builder.with_task(task);
       }
   }
   // ... etc for gate_feedback, anti_patterns
   ```
   The `"identity"` section is NEVER skipped (it is always injected via `SystemPromptBuilder::new`).

5. If a token budget is configured, scale it by section effectiveness. Compute a weighted
   budget: sum the effectiveness scores of included sections, then set the token budget to
   `base_budget * (sum_of_included / total_sections)`. This gives lower-effectiveness
   prompts a smaller total budget:
   ```rust
   if let Some(base_budget) = self.token_budget {
       let (included_weight, total) = self.effective_budget_ratio();
       let scaled = ((base_budget as f64) * (included_weight / total)).ceil() as usize;
       builder = builder.with_token_budget(scaled.max(256)); // floor at 256 tokens
   }
   ```

6. Write the helper:
   ```rust
   /// Returns (sum of included section scores, total section count).
   fn effective_budget_ratio(&self) -> (f64, f64) {
       let sections = ["identity", "conventions", "domain", "context", "task",
                        "gate_feedback", "tools", "playbooks", "anti_patterns"];
       let total = sections.len() as f64;
       let included: f64 = sections.iter()
           .filter(|s| self.should_include(s))
           .map(|s| {
               self.section_effectiveness
                   .as_ref()
                   .and_then(|scores| scores.get(*s).copied())
                   .unwrap_or(1.0)
           })
           .sum();
       (included, total)
   }
   ```

7. Add a unit test `effectiveness_skips_low_scoring_sections`:
   - Build a service with `with_section_effectiveness` where `"conventions"` has score `0.05`
     and `"anti_patterns"` has score `0.0`
   - Set `default_conventions` to `"VISIBLE_CONVENTIONS"` and anti_patterns to
     `["VISIBLE_ANTI_PATTERN"]`
   - Assemble a prompt and assert the output does NOT contain `"VISIBLE_CONVENTIONS"` or
     `"VISIBLE_ANTI_PATTERN"`
   - Assert the output still contains the role identity (not skippable)

8. Add a unit test `effectiveness_includes_high_scoring_sections`:
   - Build a service with `with_section_effectiveness` where `"task"` has score `0.9`
   - Assemble with a task `"BUILD_THE_FEATURE"` and assert the output contains it

9. Add a unit test `no_effectiveness_includes_everything`:
   - Build a service without effectiveness scores
   - Assemble with conventions, task, and anti_patterns
   - Assert all sections appear in the output
