## Existing Code Context

```rust
// crates/roko-runtime/src/workflow_engine.rs (abridged)

// Standard workflow phases (from pipeline_state.rs):
// Pending -> Implementing -> Gating -> Reviewing -> Committing -> Complete
//
// WorkflowConfig::standard() produces:
//   WorkflowConfig { has_strategy: false, has_review: true,
//                    max_iterations: 2, max_autofix_attempts: 2 }
//
// The engine dispatches:
//   PipelineOutput::SpawnImplementer  -> driver.spawn_agent("implementer", ...) -> ModelCaller::call()
//   PipelineOutput::RunGates          -> driver.run_gates()                     -> GateRunner::run_gates()
//   PipelineOutput::SpawnReviewer     -> driver.spawn_agent("reviewer", ...)    -> ModelCaller::call()
//   PipelineOutput::Commit            -> driver.commit()  (runs `git` — SKIP with a real git repo)
//
// NOTE: driver.commit() calls `git add -A` and `git commit` in the workdir.
// When workdir is "." inside a test the commit will fail with
// "nothing to commit" which returns PipelineInput::CommitDone { hash: "noop" }
// — that is fine; the state machine still transitions to Complete.

fn reviewer_input(input: PipelineInput) -> PipelineInput {
    match input {
        PipelineInput::AgentCompleted { output, .. } => {
            PipelineInput::ReviewApproved { summary: output }
        }
        input => input,
    }
}
```

Key facts:
- `GateReport { verdicts: vec![] }` means all gates passed (no failures).
- Returning a single `GateVerdict { gate_name: "compile".into(), passed: true,
  output: "".into(), duration_ms: 0 }` is also acceptable and makes the test
  more explicit.
- After gates pass, because `has_review: true`, the engine spawns a reviewer.
  `MockModelCaller::call()` returns the same canned response for all roles.
  The `reviewer_input()` adapter converts `AgentCompleted` to `ReviewApproved`.
- After review approves, `Commit` is requested. The `driver.commit()` method
  runs real `git` commands but handles "nothing to commit" gracefully by
  returning `CommitDone { hash: "noop" }`.

## Task

Implement two tests inside the existing `mod tests` block:

### `workflow_engine_standard_passes_through_review`

1. Reuse or define the four mock service structs.
2. Build `EffectServices` and `WorkflowRunConfig` with

   - `result.outcome` matches `WorkflowOutcome::Success { .. }`.

### `workflow_engine_standard_gate_pass_triggers_review_phase`

Same setup. Attach an event consumer that records `PhaseTransition` events.
After the run, assert the collected transitions include both
`from: "implementing"` -> `to: "gating"` and `from: "gating"` -> `to: "reviewing"`.

Hint: filter the collected events for `WorkflowEvent::PhaseTransition { from, to, .. }`
and check that both expected transitions appear (order does not need to be
contiguous, just present).
