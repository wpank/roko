## Problem
`FeedbackService::record_knowledge_usage()` records which knowledge entry IDs influenced a task's routing and prompt composition, but it is NEVER called from the orchestration loop. The method exists and is tested in isolation (see `feedback_service.rs:300-335`), but the orchestrator never invokes it after gate checks complete. This means the learning subsystem cannot correlate knowledge entries with task outcomes, so knowledge scoring and knowledge-informed routing remain inert.

## Architecture Contract
- The orchestrator already has a `knowledge_store: KnowledgeStore` field on `PlanRunner`.
- Knowledge entry IDs are available when building the task prompt — they come from `build_knowledge_routing_advice()` and `render_neuro_chunk()` which query the KnowledgeStore.
- The call site must be AFTER gate checks so we know `gate_passed: bool`.
- `FeedbackService` is not currently a field on `PlanRunner`. You will need to create one during `PlanRunner::build()` initialization, or use an existing `LearningRuntime` method if one wraps this functionality.
- The `record_knowledge_usage()` call is synchronous and append-only, so it can be called inline.

### Step 3 — Track knowledge IDs used per task dispatch

In the task dispatch path, knowledge IDs come from `build_knowledge_routing_advice()` and `render_neuro_chunk()`. These return knowledge entry references. You need to collect the IDs into a `Vec<String>` and store them in a local variable before the gate check.

Search for where `render_neuro_chunk` or `build_knowledge_routing_advice` is called. After the call, extract the knowledge entry IDs:

```rust
    let knowledge_ids: Vec<String> = knowledge_advice
        .entries
        .iter()
        .map(|e| e.id.clone())
        .collect();
```

### Step 4 — Call record_knowledge_usage after gate checks

After the gate pipeline returns `gate_passed`, call:

```rust
    if let Err(e) = self.feedback_service.record_knowledge_usage(
        &format!("{plan_id}/{task_id}"),
        knowledge_ids.clone(),
        gate_passed,
        &model_slug,
    ) {
        tracing::warn!(
            plan_id = %plan_id,
            task_id = %task_id,
            error = %e,
            "[orchestrate] failed to record knowledge usage feedback"
        );
    }
```

Place this call in the post-gate section where episodes and efficiency events are already being recorded, so it is consistent with the existing pattern.
