## Problem
Two backend reliability features are built but never wired into the orchestration loop:

1. **Knowledge-informed retry:** When a task fails and enters the retry path, the orchestrator does NOT query the neuro knowledge store for entries relevant to the failure. This means retries use the exact same prompt context as the first attempt, reducing the chance of success.

2. **Auto-dream trigger at plan completion:** The `maybe_auto_dream()` method exists on PlanRunner (orchestrate.rs:7371) and checks episode count thresholds, but it is only called at plan completion. The threshold check works, but the dream cycle itself can fail silently because the DreamCycle construction path may not find the required stores. We need to verify the auto-dream trigger fires reliably and that knowledge entries from the current plan run are available for consolidation.

## Architecture Contract
- Knowledge queries go through `self.knowledge_store` which is a `KnowledgeStore` on `PlanRunner`.
- `KnowledgeStore` has `query(&self, topic: &str) -> Vec<KnowledgeEntry>` and `query_by_kind(&self, kind: KnowledgeKind) -> Vec<KnowledgeEntry>`.
- The retry path is where gate failure triggers a re-dispatch. Search for `gate_failure` or `replan` or `retry` in orchestrate.rs.
- `DreamCycle::new()` requires `Arc<EpisodeLogger>`, `Arc<KnowledgeStore>`, `Arc<PlaybookStore>`, `Arc<dyn AgentDispatcher>`.
- `maybe_auto_dream()` (orchestrate.rs:7371) already reads episodes and checks thresholds.
- Anti-knowledge patterns (gate failures that should be avoided) are already recorded via `query_anti_knowledge_patterns()`.

Search orchestrate.rs for the task retry/replan path. Look for where `build_gate_failure_plan_revision` is called or where a failed task is re-dispatched. Before the retry dispatch, query the knowledge store for relevant entries.

**File:** `crates/roko-cli/src/orchestrate.rs`

In the retry path, after determining the task has failed and before re-dispatching, add:

```rust
    // Query knowledge store for entries relevant to this failure.
    let failure_knowledge: Vec<String> = {
        let topic = format!("{} {} failure", task_def.title, task_def.domain.as_deref().unwrap_or("general"));
        let entries = self.knowledge_store.query(&topic);
        let anti_patterns = query_anti_knowledge_patterns(&self.knowledge_store, &task_id);
        let mut context_parts = Vec::new();

        if !entries.is_empty() {
            context_parts.push(format!(
                "## Prior Knowledge ({} relevant entries)\n{}",
                entries.len(),
                entries.iter()
                    .take(5)
                    .map(|e| format!("- [{}] {}", e.kind, e.content))
                    .collect::<Vec<_>>()
                    .join("\n")
            ));
        }

        if !anti_patterns.is_empty() {
            context_parts.push(format!(
                "## Anti-Patterns to Avoid\n{}",
                anti_patterns.iter()
                    .take(3)
                    .map(|e| format!("- AVOID: {}", e.content))
                    .collect::<Vec<_>>()
                    .join("\n")
            ));
        }

        context_parts
    };

    // Inject failure knowledge into the retry prompt context.
    if !failure_knowledge.is_empty() {
        let knowledge_context = failure_knowledge.join("\n\n");
        tracing::info!(
            plan_id = %plan_id,
            task_id = %task_id,
            entries = failure_knowledge.len(),
            "[orchestrate] injecting knowledge-informed retry context"
        );
        // Append to the system prompt or task context that gets passed to the retry dispatch.
        // The exact injection point depends on how the retry prompt is assembled.
    }
```
