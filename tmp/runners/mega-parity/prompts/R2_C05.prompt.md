## Task

Learning/dashboard: don't count skipped as pass

## Runner Context

You are working in runner `mega-parity`, batch R2_C05.
This batch is part of Runner 2: execution-contract — Make CLI execution contracts truthful enough that demo scenarios and agent sessions can rely on them.

## Problem

After C04 adds `skipped` to `GateVerdict`, two consumers still treat skipped gates as passes:

1. **orchestrate.rs adaptive threshold observation** (line 16176): feeds `passed` from every
   verdict — including skipped ones — into the EMA. A skipped gate with `passed=false` would
   incorrectly degrade the adaptive threshold for that rung.

2. **orchestrate.rs gate summary/reporting**: any summary that counts gates needs to separate
   `passed` / `failed` / `skipped` counts.

3. **runtime_feedback.rs** `GateOutcomeRecord`: the `passed` field receives raw gate results.
   If a skipped gate record is written, the pass rate calculation is inflated or deflated.

## Prerequisite State (from C04)

After C04, `roko_core::foundation::GateVerdict` has:
```rust
pub struct GateVerdict {
    pub gate_name: String,
    pub passed: bool,       // false for both real failures AND skipped
    pub output: String,
    pub duration_ms: u64,
    #[serde(default)]
    pub skipped: bool,      // true only when gate was not executed
    #[serde(default)]
    pub skip_reason: Option<String>,
}
```

### Change 1: orchestrate.rs — filter skipped gates from adaptive threshold observation

File: `/Users/will/dev/nunchi/roko/roko/crates/roko-cli/src/orchestrate.rs`
Lines: 16174–16177

Before:
```rust
for recorded in &recorded_verdicts {
    self.adaptive_thresholds
        .observe(recorded.rung.as_index(), recorded.verdict.passed);
}
```

**IMPORTANT: `recorded.verdict` is of type `roko_core::Verdict`, NOT `GateVerdict`.**
`roko_core::Verdict` does NOT have a `skipped` field. C04 adds `skipped` to `GateVerdict`
(in `roko_core::foundation`), not to `Verdict`. Therefore, filtering at
`recorded.verdict.skipped` will NOT compile.

**The filter must happen at the `GateVerdict` level in `gate_service.rs::run_gates()`**,
not at the `Verdict` level in `orchestrate.rs`. Look for where `GateVerdict` results are
collected in `run_gates()` at line 210-253 and filter skipped verdicts there before they
reach the adaptive threshold observer.

After (in `gate_service.rs::run_gates()`, where `GateVerdict` structs are collected):
```rust
// Filter skipped gates before passing to adaptive threshold observer.
// GateVerdict.skipped is the correct field (added by C04).
let non_skipped_verdicts: Vec<_> = verdicts.iter().filter(|v| !v.skipped).collect();
```

Then in `orchestrate.rs`, change the observer loop to only iterate over the non-skipped
verdicts that `run_gates()` returns (or add a `skipped` field to `RecordedGateVerdict` that
mirrors `GateVerdict.skipped`). The simplest approach is to add a `skipped: bool` field to
`RecordedGateVerdict` that is set from the `GateVerdict.skipped` value, then filter on it:

After (in orchestrate.rs):
```rust
for recorded in &recorded_verdicts {
    // Do not update adaptive thresholds from skipped gates — skipped
    // verdicts carry no signal about real rung difficulty.
    if !recorded.skipped {
        self.adaptive_thresholds
            .observe(recorded.rung.as_index(), recorded.verdict.passed);
    }
}
```

Note: `recorded.skipped` (on `RecordedGateVerdict`) is populated from `GateVerdict.skipped`,
NOT from `Verdict.skipped` (which does not exist).

### Change 2: orchestrate.rs — three-way gate summary

Find the gate summary log line in `orchestrate.rs`. Search for:
```bash
grep -n "all_passed\|gates_passed\|gate.*passed\|\"passed\".*gate\|gate.*summary" \
    crates/roko-cli/src/orchestrate.rs | head -20
```

Identify where the gate results are logged or formatted into a summary string. Add a three-way
breakdown. The pattern should be:

```rust
// Count three-way breakdown for logging.
let passed_count = verdicts.iter().filter(|v| !v.skipped && v.passed).count();
let failed_count = verdicts.iter().filter(|v| !v.skipped && !v.passed).count();
let skipped_count = verdicts.iter().filter(|v| v.skipped).count();
tracing::info!(
    passed = passed_count,
    failed = failed_count,
    skipped = skipped_count,
    all_passed = all_passed,
    "gate pipeline complete"
);
```

Note: `all_passed` (line 16078) already correctly uses `verdicts.iter().all(|verdict| verdict.passed)`.
After C04, skipped verdicts have `passed=false`, so `all_passed` will be `false` when any gate
is skipped. You may want to change the semantics of `all_passed` to mean "no real failures" by
excluding skipped gates:

```rust
// Before (line 16078):
let all_passed = aggregate_verdict.passed && verdicts.iter().all(|verdict| verdict.passed);

// After:
let all_passed = aggregate_verdict.passed
    && verdicts.iter().filter(|v| !v.skipped).all(|verdict| verdict.passed);
```

However, this is a semantic change — it means "the plan advances even if some gates are skipped."
Only make this change if the acceptance criteria require it. Otherwise leave `all_passed` as-is
and only add the logging.
