## Problem

Unmatched `/api/*` routes return the SPA HTML fallback instead of a JSON 404. API clients (demo app, agents, curl) that hit a typo'd or non-existent API endpoint get HTML back and parse-fail instead of getting a clean JSON error they can handle programmatically.

**Root cause**: In `crates/roko-serve/src/routes/mod.rs` lines 135–146, the outer router has
`.fallback(crate::embedded::serve_embedded)` which catches ALL unmatched routes including
`/api/*` paths, returning `index.html` (HTML). The `/api` sub-router has no fallback of its own.

## Architecture Contract

- API routes return JSON, always
- SPA catch-all only applies to non-API paths

```rust
/// Fallback handler for unmatched `/api/*` routes.
///
/// Returns a JSON 404 so API clients (demo app, curl, agents) get a machine-readable
/// error instead of the SPA `index.html` response.
async fn api_not_found(req: axum::extract::Request) -> impl axum::response::IntoResponse {
    let path = req.uri().path().to_string();
    (
        axum::http::StatusCode::NOT_FOUND,
        [("content-type", "application/json")],
        serde_json::json!({
            "error": "not_found",
            "message": format!("No route matches {path}")
        })
        .to_string(),
    )
}
```

## Why No Change to Outer Fallback

The outer `router.fallback(crate::embedded::serve_embedded)` at line 146 does NOT need to
change. Axum's `nest("/api", api)` routes requests with the `/api` prefix to the `api`
sub-router exclusively. Once `/api` is matched, the outer fallback is never reached for those
paths. The `api_not_found` fallback on the inner `api` router handles all unmatched `/api/*`
paths before Axum would fall through to the outer router.
