### build_command output (claude_cli_agent.rs lines 302–359)

```
claude
  [extra_args]
  --print
  --verbose
  --output-format stream-json
  --model <model>
  --effort <effort>
  --settings <settings_json>
  [--dangerously-skip-permissions]
  [--max-turns <n>]
  [--fallback-model <slug>]
  [--append-system-prompt <text>]
  [--tools <csv>]
  [--mcp-config <path>]
  [--strict-mcp-config]
  [--resume <session_id>]
```

The agent reads the session_id from the `"result"` stream-json event — but the ClaudeCliAgent's `run()` method does not return the session_id. The caller must read it from the output Engram or from the raw stdout (which ClaudeCliAgent does not expose). **This means `send_turn` must either parse the session_id from the agent output or ClaudeCliAgent needs a way to expose it.**

**Design decision for the contract**: `send_turn` must call `ClaudeCliAgent::run()` and then parse the `session_id` from the raw stdout by reusing the same stream-json parsing that `dispatch_direct::dispatch_claude_cli` already does (lines 174–242 of dispatch_direct.rs). Alternatively: expose session_id via an extended return path. The simplest approach is to re-parse it from the Engram body text (which contains the raw text output) — but stream-json events are not in the Engram body. **The contract must specify that `send_turn` spawns the agent via `ClaudeCliAgent::build_command()` directly, not via `ClaudeCliAgent::run()`, so it can capture all stream-json events including `"result"` for session_id extraction.**
