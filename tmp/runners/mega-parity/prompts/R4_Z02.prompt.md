## Problem
Plan generation from PRDs and plan validation are separate code paths that lack repo-awareness. The generator produces `tasks.toml` files that may reference nonexistent crates, use invalid model aliases, or duplicate existing workspace members. The validator checks structural correctness but not semantic grounding. Before adding grounding, we need a complete map of what is and isn't validated.

## Architecture Contract
This is a read-only audit batch. No code changes. The output is a written analysis document that identifies:
1. The plan generation flow from `prd plan <slug>` through to `tasks.toml` output
2. The validation checks in `plan validate` (what IS checked)
3. The items that are NOT validated (file existence, crate existence, model validity, role presence)
4. How `plan regenerate` works and what context it receives
5. Where generated plans are written (output path)
6. What learning records are created during plan generation
