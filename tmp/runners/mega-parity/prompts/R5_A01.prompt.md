## Task
Parse Claude CLI stream-json result usage

## Runner Context
You are working in runner `mega-parity`, batch R5_A01.
This batch is part of Runner 5: telemetry-learning — Make cost, usage, episodes, learning, and cascade router feedback truthful enough that dashboards show real data and self-improvement actually works.

## Problem
The Claude CLI emits a stream-json result event containing usage data (input_tokens, output_tokens, cache tokens, cost_usd, model), but the current parser either ignores these fields or stores absent values as zero, making cost tracking unreliable.

## Architecture Contract
- Absent fields = None, NOT zero
- All usage fields are Option types
- UsageSource tracks whether data came from provider or is unknown
- Model from result event supersedes any guess

## Changes Required
1. In `claude_cli_agent.rs`, locate the existing stream parser: `fn parse_stream_events(stdout: &str) -> Option<Vec<Value>>` (around line 278). This parses each JSON line from stdout into a `Vec<Value>`. The downstream consumer is `fn output_text()` which calls `BackendResponse::StreamJson(events).extract_text()`. The `result` event in stream-json format includes `usage` and `total_cost_usd` fields. Extract these into `Option<T>` fields.
2. Extract fields from the result event: `input_tokens`, `output_tokens`, `cache_creation_tokens`, `cache_read_tokens`, `cost_usd` (from `total_cost_usd`), `model`
3. All fields are `Option<T>` -- absent in JSON means None, present means Some(value)
4. Extend `AgentResult.usage` to carry these parsed values
5. If no result event is received at all, all fields remain None with UsageSource::Unknown
6. **Note**: The codebase has an existing `Usage` type in `roko_core::chat_types` (with `input_tokens: u32`, `output_tokens: u32`, `cache_read_tokens: u32`, `cache_create_tokens: u32`, `cost_usd: f32`, `wall_ms: u64`). This batch should populate that type's fields from stream data where possible. R5_A02 will create the new `UsageObservation` wrapper with proper `Option<T>` semantics.
