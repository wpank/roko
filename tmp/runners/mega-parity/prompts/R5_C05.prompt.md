## Step-by-Step Changes

### Step 1 ‚Äî Add `DataQuality` struct to `projection_contract.rs`

**Before:** No `DataQuality` type exists.

**After:** Add near the top of `projection_contract.rs`:

```rust
/// Data quality summary attached to responses that read from on-disk telemetry.
///
/// Consumers should check `has_real_data` before rendering charts or metrics.
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct DataQuality {
    /// Whether the source file existed and contained at least one entry.
    pub has_real_data: bool,
    /// Total number of entries read from the source.
    pub entry_count: usize,
    /// Number of entries where `cost_usd` was zero or absent
    /// (these inflate total cost calculations to 0 rather than true cost).
    pub null_cost_count: usize,
}

impl DataQuality {
    /// Compute quality from a slice of efficiency events.
    pub fn from_efficiency_events(events: &[AgentEfficiencyEvent]) -> Self {
        let null_cost_count = events.iter().filter(|e| e.cost_usd == 0.0).count();
        Self {
            has_real_data: !events.is_empty(),
            entry_count: events.len(),
            null_cost_count,
        }
    }
}
```

### Step 4 ‚Äî Ensure null costs are represented as null in JSON, not 0.0

In event serialization, when `cost_usd == 0.0` and the field was not actually
observed (no real token cost data), use `Option<f64>` with `serde(skip_serializing_if = "Option::is_none")`.

For the `AgentEfficiencyEvent` type (in `roko-learn`), if cost is not yet tracked,
the field should serialize as `null` rather than `0.0`. If modifying `roko-learn` is
out of scope, at minimum the API response should annotate the `null_cost_count` so
consumers know how many zero values are data gaps.
