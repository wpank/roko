## Problem
The CascadeRouter learns which models work best for which task categories by observing outcomes. It uses LinUCB (contextual bandit) to balance exploration vs exploitation. ACP dispatches produce outcomes (success/failure, latency, cost) but NEVER report them to the router.

Editor usage is likely the highest-volume dispatch path in production. If the router can't learn from it, it can't optimize model selection. This makes the entire cascade routing system half-blind.

## Architecture Contract
- **TL-5:** Learning reads what execution writes
- **ACP-3:** No silent learning gaps — every dispatch feeds the router
- **TL-3:** Model is known before logging — never observe "unknown-model"
- **P-5:** Model must flow to EffectDriver — resolved model used for both dispatch AND observation

### Step 3: Add `observe_cascade_router()` helper

In `bridge_events.rs`, add a free function:

```rust
/// Record a dispatch outcome in the CascadeRouter.
/// Non-blocking: uses spawn_blocking for sync I/O.
/// Never fails the caller.
fn record_cascade_observation(
    router_path: std::path::PathBuf,
    model_slug: String,
    mode: String,
    prompt_text: String,
    success: bool,
    wall_ms: u64,
    output_tokens: Option<u64>,
    model_slugs: Vec<String>,
) {
    tokio::task::spawn_blocking(move || {
        let router = CascadeRouter::load_or_new(&router_path, model_slugs);
        let routing_ctx = acp_routing_context(&mode, &prompt_text);

        let Some(model_idx) = router.model_index_for_slug(&model_slug) else {
            tracing::debug!(
                model = %model_slug,
                "skipping cascade observation: model not in router arms"
            );
            return;
        };

        let context_vec = routing_ctx.to_features();
        let reward = compute_acp_reward(success, wall_ms, output_tokens);
        router.observe(context_vec, model_idx, reward);

        // Persist — sync I/O inside spawn_blocking
        if let Err(e) = router.save(&router_path) {
            tracing::warn!(
                path = %router_path.display(),
                error = %e,
                "failed to persist cascade router after ACP observation"
            );
        }
    });
}

/// Compute a composite reward for the bandit from ACP outcomes.
fn compute_acp_reward(success: bool, wall_ms: u64, output_tokens: Option<u64>) -> f64 {
    if !success {
        return 0.0;
    }

    // Base reward for success
    let latency_bonus = if wall_ms < 5_000 { 0.15 } else if wall_ms < 15_000 { 0.05 } else { 0.0 };
    let token_bonus = match output_tokens {
        Some(t) if t < 2_000 => 0.05,
        Some(t) if t < 5_000 => 0.02,
        _ => 0.0,
    };
    (0.8 + latency_bonus + token_bonus).min(1.0)
}
```

## Do NOT
- Use async for `router.save()` — it's sync; wrap in `spawn_blocking`
- Use `model_slug: &str` in `record_cascade_observation` directly — take `String` to move into `spawn_blocking`
- Assert or unwrap `model_index_for_slug` — if slug not found, skip with debug log
- Create CascadeRouter with empty model slugs — `new()` panics; use at least `vec!["default".to_string()]` or load from persisted file
- Record observation with fabricated `context_vec` — always use `RoutingContext::to_features()`
- Use `observe_multi_objective` for initial wiring — use `observe()` (single reward) which is simpler and sufficient; `observe_multi_objective` requires normalized_cost/latency calculations that need R5_F03's cost data
- Block dispatch on router I/O — `spawn_blocking` returns immediately to the async caller
- Record twice for the same dispatch — one observation per dispatch
- Fabricate nonexistent task category variants (e.g. `Planning`, `SimpleEdit`, `Review`) — use `Implementation` as fallback for unknown categories. Real variants: Scaffolding, Implementation, Integration, Verification, Research, Refactor, Infra, Docs
