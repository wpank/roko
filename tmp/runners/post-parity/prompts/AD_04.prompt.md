# AD_04: Reflect dream routing-advice updates as ConfigOptionUpdate

## Task
When AD_02 emits the dream card and the report contains routing recommendations (`report.routing_recommendations > 0` AND `roko_dreams::routing_advice::*` returns advice), translate the advice into one or more `ConfigOptionUpdate` notifications so the editor's status bar / model selector reflects the new defaults.

## Runner Context
Runner AD_A (ACP: Dream Journal at Session Start), batch 4 of 4. Depends on AD_02 (does NOT depend on AD_03 — these can run in parallel).

## Problem
`tmp/solutions/acp/08-NOVEL-WORKFLOWS.md` Workflow 3 ends with:

```
💡 Routing updated: Opus preferred for error handling tasks
```

Today the dreams crate persists routing advice (`crates/roko-dreams/src/routing_advice.rs`) and the cascade router can pick it up at dispatch time, but ACP never SHOWS the user that defaults changed. The dream card includes a count line ("Routing updated: 3 recommendations") but no granular per-task advice surfaces.

### Step 1: Helper to translate advice → updates

Append to `crates/roko-acp/src/dream_card.rs`:

```rust
use crate::types::{SessionUpdate, ContentBlock};

/// Translate dream routing advice into a list of ConfigOptionUpdate
/// notifications. Empty when no advice is present.
pub fn dream_routing_updates(workdir: &std::path::Path) -> Vec<SessionUpdate> {
    let Ok(advice) = load_routing_advice(workdir) else { return Vec::new() };
    advice.into_iter().map(|a| {
        SessionUpdate::ConfigOptionUpdate {
            option_id: format!("routing.{}", a.task_kind),
            value: serde_json::json!(a.preferred_model),
            label: Some(format!(
                "{} preferred for {} tasks (confidence {:.2})",
                a.preferred_model, a.task_kind, a.confidence
            )),
        }
    }).collect()
}

/// Wrapper around the dreams crate's routing advice loader.
/// Adapt to the actual public symbol — most likely
/// `roko_dreams::routing_advice::load_advice` or similar.
fn load_routing_advice(
    workdir: &std::path::Path,
) -> std::io::Result<Vec<roko_dreams::routing_advice::RoutingAdvice>> {
    // VERIFY THE EXACT MODULE PATH BEFORE COMMIT.
    Ok(roko_dreams::routing_advice::load_advice(workdir)?)
}
```

### Step 3: Generic update emitter

If a `send_session_update` helper doesn't exist, factor one out of `emit_dream_card`:

```rust
async fn send_session_update<R, W>(
    transport: &mut StdioTransport<R, W>,
    session_id: &str,
    update: SessionUpdate,
) -> Result<()>
where
    R: AsyncRead + Unpin,
    W: AsyncWrite + Unpin,
{
    let notification = JsonRpcNotification {
        jsonrpc: JSON_RPC_VERSION.to_owned(),
        method: "session/update".to_owned(),
        params: Some(serde_json::json!({
            "sessionId": session_id,
            "update": update,
        })),
    };
    transport.send_notification(notification).await
        .context("send session/update notification")
}
```

Refactor `emit_dream_card` to call `send_session_update`.

### Step 4: Bound the number of updates

If a single dream cycle produces 50 routing recommendations, we don't want 50 updates. Cap at 10 and add a summary line if exceeded:

```rust
const MAX_ROUTING_UPDATES_PER_DREAM: usize = 10;

let updates = crate::dream_card::dream_routing_updates(&sessions_workdir);
let total = updates.len();
for update in updates.into_iter().take(MAX_ROUTING_UPDATES_PER_DREAM) {
    send_session_update(transport, &session_id, update).await?;
}
if total > MAX_ROUTING_UPDATES_PER_DREAM {
    let chunk = SessionUpdate::AgentMessageChunk {
        content: ContentBlock::Text {
            text: format!(
                "_…and {} more routing recommendation(s) — see `roko dreams journal`._\n",
                total - MAX_ROUTING_UPDATES_PER_DREAM,
            ),
        },
        _meta: None,
    };
    send_session_update(transport, &session_id, chunk).await?;
}
```

## Acceptance Criteria
- `dream_routing_updates(workdir) -> Vec<SessionUpdate>` returns empty Vec on missing file
- Each update has `option_id = "routing.<task_kind>"`, value = preferred model slug
- Cap of 10 updates with summary chunk for overflow
- Emitted AFTER the dream card and BEFORE the `last-dream-shown` marker is written (so a crash mid-emit results in a re-show, not a silent loss)
- Updates do not crash the session if the dreams crate's API changes — wrap in `Result` and fall back silently
