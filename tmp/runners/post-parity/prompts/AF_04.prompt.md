# AF_04: Auto-escalate model on persistent frustration via CascadeRouter

## Task
When the daimon's `behavioral_state` is `Struggling` for 2+ consecutive iterations, automatically escalate the model tier on the next dispatch via the existing `CascadeRouter`. This implements the "Mood Ring" auto-escalation arrow shown in `tmp/solutions/acp/08-NOVEL-WORKFLOWS.md` Workflow 2.

## Runner Context
Runner AF_B (ACP: Affect-Driven Modulation), batch 1 of 2. Depends on AF_03.

## Problem
After AF_03, the daimon receives task/gate outcomes and the behavioral state is queryable. AF_03 also emits the Mood Ring card. But routing decisions on the next dispatch don't yet read the state â€” there's no link between "agent is frustrated" and "next dispatch picks a stronger model".

`DaimonPolicy::modulate_dispatch` already maps `BehavioralState::Struggling` to `tier_bias = +0.5` and `turn_limit_factor = 1.3` (`crates/roko-daimon/src/policy.rs:73-99`), but ACP's dispatch path constructs `Dispatcher` requests without consulting `DispatchModulation`.

## Exact Changes

### Step 1: Track consecutive struggling count

The daimon already tracks state but ACP needs a lightweight per-session count to decide "persistent". Add to `AcpSession` (`crates/roko-acp/src/session.rs`):

```rust
pub struct AcpSession {
    // existing fields...

    /// Count of consecutive iterations the daimon has been Struggling.
    /// Reset to 0 on any non-Struggling state.
    /// Used by AF_04 to trigger auto-escalation.
    #[serde(default)]
    pub consecutive_struggling: u32,

    /// Last applied tier override applied via auto-escalation, if any.
    /// Cleared when state returns to non-Struggling.
    #[serde(default)]
    pub auto_escalated_tier: Option<String>,
}
```

Add accessor:

```rust
impl AcpSession {
    /// Returns true if the daimon is in struggling for at least `n` iterations.
    pub fn is_persistently_struggling(&self, n: u32) -> bool {
        self.consecutive_struggling >= n
    }
}
```

### Step 2: Update count after each phase

In `crates/roko-acp/src/runner.rs`, after the AF_03 call to `record_task_to_affect`, query the daimon and update the session:

```rust
async fn refresh_struggling_count(
    affect: &Option<SharedAffect>,
    session: &mut AcpSession,
) {
    let Some(handle) = affect.as_ref() else { return };
    let state = {
        let guard = handle.lock().await;
        guard.behavioral_state()
    };
    use roko_core::BehavioralState as B;
    if matches!(state, B::Struggling) {
        session.consecutive_struggling += 1;
    } else {
        session.consecutive_struggling = 0;
        session.auto_escalated_tier = None;  // demote escalation when daimon recovers
    }
}
```

`AcpSession` is held via `&mut` inside the bridge_events path; you may need to:
- pass a `&mut AcpSession` into `run_workflow_pipeline`, OR
- emit a `CognitiveEvent::StrugglingCountChanged(u32)` and let `bridge_events.rs` mutate the session.

The cleanest route is the second: `run_workflow_pipeline` already emits `CognitiveEvent` events to `bridge_events.rs`, and the consumer there owns the session.

### Step 3: Add a routing-override hook

Inside `crates/roko-acp/src/bridge_events.rs`, in the dispatch resolution path (where `resolved` is computed), insert:

```rust
// AF_04: auto-escalate when daimon has been Struggling for 2+ iterations.
if session.is_persistently_struggling(2) {
    if let Some(escalated) = escalate_via_cascade(&resolved.slug, workdir) {
        if escalated.slug != resolved.slug {
            tracing::info!(
                from = %resolved.slug,
                to = %escalated.slug,
                struggling_for = session.consecutive_struggling,
                "AF_04 auto-escalating model due to persistent struggle"
            );
            session.auto_escalated_tier = Some(escalated.slug.clone());
            resolved = escalated;
        }
    }
}
```

Define `escalate_via_cascade` as a helper:

```rust
fn escalate_via_cascade(
    current_slug: &str,
    workdir: &Path,
) -> Option<ResolvedModel> {
    use roko_learn::cascade_router::CascadeRouter;
    let path = workdir.join(".roko").join("learn").join("cascade-router.json");
    let model_slugs = cascade_router_model_slugs(/* ... */);  // existing fn at bridge_events.rs:459
    let router = CascadeRouter::load_or_new(&path, model_slugs);

    // Pick the next-tier model: cascade router orders arms cheapâ†’expensive,
    // so escalation = pick the arm AFTER current_slug if available.
    let arms = router.arms();
    let pos = arms.iter().position(|m| m.as_str() == current_slug)?;
    let next = arms.get(pos + 1)?.clone();
    resolve_model(workdir, &next).ok()  // existing resolver
}
```

Verify the `arms()` accessor exists; if not, expose one in `roko-learn`. If `CascadeRouter` doesn't have ordered arms, fall back to a static tier table (haiku â†’ sonnet â†’ opus). Document the chosen path in a `// TODO(cascade-arms-order):` comment if you take the static fallback.
