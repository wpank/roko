# CD_04: Remove unsafe env::set_var calls

## Goal

Replace the `unsafe { std::env::set_var(...) }` call in
`crates/roko-cli/src/commands/util.rs` with a safe alternative.

## Files to Edit

- `crates/roko-cli/src/commands/util.rs` (primary)

## Steps

### 1. Locate the unsafe call

In `crates/roko-cli/src/commands/util.rs`, line 236:

```rust
unsafe { std::env::set_var("ROKO_PROVIDER", p) };
```

### 2. Understand the purpose

The `ROKO_PROVIDER` env var is set so downstream code can read it. Determine
what reads `ROKO_PROVIDER`:

```bash
grep -rn 'ROKO_PROVIDER' crates/ --include='*.rs' | grep -v target/
```

### 3. Replace with config-based passing

Instead of setting an environment variable, pass the provider through the
config/context chain. Options:

**Option A: Thread through function parameters**

If the provider is only read by code in the same call chain, add a parameter:

```rust
// Before:
fn configure_provider(provider: &str) {
    unsafe { std::env::set_var("ROKO_PROVIDER", provider) };
}

// After:
fn configure_provider(provider: &str) -> String {
    provider.to_string()
}
```

**Option B: Use a thread-local or static**

If the provider needs to be globally accessible:

```rust
use std::sync::OnceLock;

static PROVIDER_OVERRIDE: OnceLock<String> = OnceLock::new();

fn set_provider(provider: &str) {
    let _ = PROVIDER_OVERRIDE.set(provider.to_string());
}

pub fn get_provider_override() -> Option<&'static str> {
    PROVIDER_OVERRIDE.get().map(|s| s.as_str())
}
```

**Option C: If env var is truly needed**

If downstream code (especially non-Rust processes) reads the env var, keep
`set_var` but limit the unsafe scope and add a safety comment:

```rust
// SAFETY: This is called once at CLI startup before any threads are spawned.
// `set_var` is unsafe in multi-threaded programs because concurrent reads of
// the environment are not synchronized. We ensure this call happens on the
// main thread before tokio::main starts.
unsafe { std::env::set_var("ROKO_PROVIDER", p) };
```

### 4. Search for other unsafe set_var calls

```bash
grep -rn 'set_var' crates/roko-cli/ --include='*.rs' | grep -v target/
```

Fix all occurrences with the same approach.

### 5. Remove the need for env vars

If `ROKO_PROVIDER` is only read by Rust code in the same process, replace
the reads with the config chain:

```rust
// Before:
let provider = std::env::var("ROKO_PROVIDER").ok();
// After:
let provider = get_provider_override();
```

## Verification

```bash
cargo check -p roko-cli
cargo clippy -p roko-cli --no-deps 2>&1 | grep "unsafe"
# Should output: 0 unsafe-related warnings
```

## Constraints

- If the env var is needed for subprocess communication (e.g., spawned
  processes that read it), keep `set_var` with a clear SAFETY comment.
- Do NOT use `unsafe` without a SAFETY comment explaining the preconditions.
- Prefer `OnceLock` over `Mutex` for one-time initialization.
- Thread the value through config/parameters when possible.
