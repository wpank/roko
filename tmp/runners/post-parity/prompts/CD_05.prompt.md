# CD_05: Unify gate config format in schema

## Goal

Ensure the gate configuration in `roko.toml` (`[gates]` section) and the
foundation `GateConfig` struct use a consistent format, eliminating any
divergence between the CLI config schema and the runtime gate config.

## Files to Edit

- `crates/roko-core/src/config/schema.rs` (primary)

## Also Read

- `crates/roko-core/src/foundation.rs` — `GateConfig`, `ShellGateCommand`

## Steps

### 1. Identify the schema gate config

In `crates/roko-core/src/config/schema.rs`, find the `[gates]` section config.
It may be a `GatesConfig` struct or fields on `RokoConfig`:

```bash
grep -n 'gates\|Gates' crates/roko-core/src/config/schema.rs | head -20
```

### 2. Compare with foundation GateConfig

The foundation `GateConfig` (in `foundation.rs`) has:
- `workdir: PathBuf`
- `enabled_gates: Vec<String>`
- `shell_gates: Vec<ShellGateCommand>`
- `max_rung: Option<u8>`
- `complexity: Option<f64>` (from G1_01)
- `prior_failures: u32` (from G1_01)

The schema config should map to these fields.

### 3. Add `max_rung` to the schema config

If the schema `[gates]` section does not have `max_rung`, add it:

```rust
    /// Maximum gate rung to execute (0-6). Higher rungs are skipped.
    #[serde(default)]
    pub max_rung: Option<u8>,
```

### 4. Add shell gate configuration

If shell gates are not configurable via TOML, add:

```rust
    /// Custom shell gates to run after built-in gates.
    #[serde(default)]
    pub shell_gates: Vec<ShellGateConfig>,
```

With a matching config struct:

```rust
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct ShellGateConfig {
    /// Program to invoke.
    pub program: String,
    /// Arguments to pass.
    #[serde(default)]
    pub args: Vec<String>,
    /// Timeout in milliseconds (default: 30000).
    #[serde(default = "default_shell_gate_timeout")]
    pub timeout_ms: u64,
}

const fn default_shell_gate_timeout() -> u64 {
    30_000
}
```

### 5. Add conversion function

Add a function that converts the schema config to the foundation config:

```rust
impl GatesConfig {
    /// Convert to a foundation `GateConfig` for a given workdir.
    pub fn to_gate_config(&self, workdir: PathBuf) -> roko_core::foundation::GateConfig {
        roko_core::foundation::GateConfig {
            workdir,
            enabled_gates: self.enabled_gates(),
            shell_gates: self.shell_gates.iter().map(|sg| {
                roko_core::foundation::ShellGateCommand {
                    program: sg.program.clone(),
                    args: sg.args.clone(),
                    timeout_ms: sg.timeout_ms,
                }
            }).collect(),
            max_rung: self.max_rung,
            complexity: None,
            prior_failures: 0,
        }
    }
}
```

### 6. Update example generation

In `write_example_gates` (or similar), include the new fields:

```rust
    let _ = writeln!(out, "[gates]");
    let _ = writeln!(out, "# max_rung = 4  # 0=compile, 1=clippy, 2=test, 3=diff, 4=fmt, 5=shell, 6=judge");
    let _ = writeln!(out, "# [[gates.shell_gates]]");
    let _ = writeln!(out, "# program = \"./scripts/lint.sh\"");
    let _ = writeln!(out, "# args = [\"--strict\"]");
    let _ = writeln!(out, "# timeout_ms = 60000");
```

## Verification

```bash
cargo check -p roko-core
cargo test -p roko-core --lib
```

Both must succeed. Existing configs without the new fields parse correctly
due to `#[serde(default)]`.

## Constraints

- All new fields must be `#[serde(default)]` for backwards compatibility.
- Do NOT change the foundation `GateConfig` struct — only the schema config.
- The conversion function (`to_gate_config`) bridges the two formats.
- Do NOT remove existing gate config fields — only add new ones.
