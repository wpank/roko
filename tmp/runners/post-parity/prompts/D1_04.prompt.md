# D1_04: Wire CascadeRouter into roko run

## Goal

Load a `CascadeRouter` at the start of `roko run` and pass it into the
workflow model selection and service factory construction.

## Files to Edit

- `crates/roko-cli/src/run.rs` (primary)

## Also Read

- `crates/roko-cli/src/model_selection.rs` — `load_cascade_router`

## Steps

### 1. Import `load_cascade_router`

Add to the imports at the top of `run.rs`:

```rust
use crate::model_selection::load_cascade_router;
```

### 2. Find the workflow model resolution path

Locate `resolve_workflow_model_selection` or the function that builds
`EffectServices` for the workflow engine. This is typically
`build_workflow_effect_services` or similar.

### 3. Load the CascadeRouter before model resolution

Before the model is resolved, load the cascade router:

```rust
let config = load_roko_config(workdir)?;
let cascade_router = load_cascade_router(workdir, &config);
```

### 4. Thread into ServiceFactory

The `ServiceFactory::build()` in `roko-orchestrator/src/service_factory.rs`
already loads its own CascadeRouter internally. However, for the CLI path
to use the same router instance (so the router loaded for model selection is
the same one used for observation recording), pass it via `ServiceConfig`.

If `ServiceConfig` does not have a cascade router field, the simplest approach
is to let `ServiceFactory` continue using its own internal router. The CLI
model selection and the service factory router are both loaded from the same
JSON file, so they will have the same state.

### 5. Alternatively: pass to `resolve_effective_model_key`

If `resolve_effective_model_key` now accepts a cascade router (from D1_02),
pass the loaded router:

```rust
let model_key = resolve_effective_model_key(
    workdir,
    cli_model,
    None, // role
    "roko run",
    Some(&cascade_router),
)?;
```

### 6. Update `run_workflow_engine_report_with_hub` if needed

If this function calls `resolve_effective_model_key` internally, ensure the
cascade router is threaded through.

## Verification

```bash
cargo check -p roko-cli
```

Must succeed. No runtime behavior change since the router was already loaded
inside ServiceFactory — this just ensures the CLI model selection also
consults the router.

## Constraints

- The CascadeRouter is loaded from disk once and shared — do NOT load it twice.
- If `resolve_effective_model_key` does not yet accept a router (D1_02 not
  merged), pass `None` and leave a TODO comment.
- Do NOT modify `ServiceFactory` internals — it has its own router loading.
