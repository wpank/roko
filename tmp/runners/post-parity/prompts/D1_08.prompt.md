# D1_08: Add Entry Point Tag to Episode Records

## Goal

Add an `entry_point` field to `RuntimeEvent` variants so episode records can
distinguish between `roko run`, `roko chat`, `roko plan run`, and ACP sessions.

## Files to Edit

- `crates/roko-core/src/runtime_event.rs` (primary)

## Also Read

- `crates/roko-agent/src/model_call_service.rs` — caller field

## Steps

### 1. Add `entry_point` to `WorkflowStarted`

In `crates/roko-core/src/runtime_event.rs`, update the `WorkflowStarted`
variant:

```rust
    WorkflowStarted {
        run_id: String,
        template: String,
        prompt: String,
        /// CLI entry point that started this workflow (e.g., "run", "chat", "plan_run", "acp").
        #[serde(default)]
        entry_point: Option<String>,
    },
```

### 2. Add `entry_point` to `AgentSpawned`

```rust
    AgentSpawned {
        run_id: String,
        agent_id: String,
        role: String,
        model: String,
        /// CLI entry point tag for episode attribution.
        #[serde(default)]
        entry_point: Option<String>,
    },
```

### 3. Add `entry_point` to `FeedbackRecorded`

```rust
    FeedbackRecorded {
        run_id: String,
        kind: String,
        summary: String,
        /// CLI entry point tag for episode attribution.
        #[serde(default)]
        entry_point: Option<String>,
    },
```

### 4. Update the `run_id()` match arms

The `run_id()` method matches all variants — the new `entry_point` field does
not affect the pattern since it uses `..` rest patterns. Verify each arm
still compiles.

### 5. Update the `kind()` method

No changes needed — `kind()` returns static strings based on variant names.

### 6. Define entry point constants

Add a module for entry point constants:

```rust
/// Entry point tag constants for episode attribution.
pub mod entry_point {
    pub const RUN: &str = "run";
    pub const CHAT: &str = "chat";
    pub const PLAN_RUN: &str = "plan_run";
    pub const ACP: &str = "acp";
    pub const SERVE: &str = "serve";
    pub const RESEARCH: &str = "research";
}
```

### 7. Update existing event construction sites

Search for `RuntimeEvent::WorkflowStarted`, `RuntimeEvent::AgentSpawned`, and
`RuntimeEvent::FeedbackRecorded` in the codebase. Add `entry_point: None` to
each construction site to maintain backwards compatibility.

Key files to update:
- `crates/roko-runtime/src/workflow_engine.rs` — `WorkflowStarted` emission
- `crates/roko-runtime/src/effect_driver.rs` — `AgentSpawned`, `FeedbackRecorded`
- Any test files that construct these events

## Verification

```bash
cargo check --workspace
```

Must succeed. The `#[serde(default)]` on `entry_point` ensures existing
serialized events deserialize correctly.

## Constraints

- All new fields must be `Option<String>` with `#[serde(default)]` to maintain
  backwards compatibility with existing JSONL logs.
- Do NOT change the `kind()` return values — they are stable API.
- Set `entry_point: None` at all existing construction sites; the actual
  entry point values are set by the CLI command handlers, not by the runtime.
