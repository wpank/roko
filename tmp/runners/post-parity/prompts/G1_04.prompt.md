# G1_04: Generate feedback inside GateService

## Goal

After running the gate pipeline, auto-populate `GateReport.feedback` with
actionable guidance derived from the failure verdicts, and set the
`failure_classification`.

## Files to Edit

- `crates/roko-gate/src/gate_service.rs` (primary)

## Also Read

- `crates/roko-core/src/foundation.rs` — `GateReport`, `FailureClassification`

## Steps

### 1. Use `GateReport::from_verdicts` in the return path

In the `GateRunner::run_gates` implementation, replace the manual
`GateReport { verdicts }` construction with:

```rust
Ok(GateReport::from_verdicts(verdicts))
```

This automatically populates `feedback` and `failure_classification`.

### 2. Enrich feedback with actionable guidance

After constructing the report via `from_verdicts`, add role-specific guidance:

```rust
let mut report = GateReport::from_verdicts(verdicts);

// Add actionable guidance based on failure classification
if let Some(ref classification) = report.failure_classification {
    match classification {
        FailureClassification::CompileError => {
            report.feedback.push(
                "Focus on fixing type errors and missing imports. \
                 Check that all referenced symbols exist and have the correct types."
                    .to_string(),
            );
        }
        FailureClassification::LintFailure => {
            report.feedback.push(
                "Run `cargo clippy --fix` or `cargo fmt` to auto-fix lint issues. \
                 Check for unused imports, variables, and naming conventions."
                    .to_string(),
            );
        }
        FailureClassification::TestFailure => {
            report.feedback.push(
                "Review the failing test assertions. Check that the implementation \
                 matches the expected behavior described in test names."
                    .to_string(),
            );
        }
        FailureClassification::ReviewRejection => {
            report.feedback.push(
                "Address the reviewer's specific concerns. Focus on the \
                 points raised rather than making unrelated changes."
                    .to_string(),
            );
        }
        FailureClassification::MultipleFailures => {
            report.feedback.push(
                "Multiple gates failed. Start by fixing compile errors, \
                 then address lint issues, then fix tests."
                    .to_string(),
            );
        }
        _ => {}
    }
}
```

### 3. Include error snippets in feedback

For compile and test failures, extract the most relevant error lines:

```rust
for verdict in &report.verdicts {
    if !verdict.passed && !verdict.skipped && !verdict.output.is_empty() {
        // Extract first 10 lines of error output
        let snippet: String = verdict
            .output
            .lines()
            .take(10)
            .collect::<Vec<_>>()
            .join("\n");
        report.feedback.push(format!(
            "Error from '{}' gate:\n```\n{}\n```",
            verdict.gate_name, snippet
        ));
    }
}
```

## Verification

```bash
cargo check -p roko-gate
cargo test -p roko-gate --lib
```

Both must succeed.

## Constraints

- Feedback strings must be human-readable and suitable for prompt injection.
- Error snippets are capped at 10 lines to prevent prompt bloat.
- The feedback order should be: classification guidance first, then error
  snippets.
