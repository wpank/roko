# OB_01: Add tracing spans to hot-path functions

## Task
Add `#[instrument]` attributes to key dispatch boundary functions in `roko-agent/src/dispatcher/mod.rs` so that tracing captures structured spans for tool dispatch, safety checks, and handler execution.

## Runner Context
Runner OB_A (Observability: Tracing), batch 1 of 4. No dependencies.

## Problem
The tool dispatch pipeline (validate, authorize, resolve handler, execute, truncate) has no tracing spans. When diagnosing latency or failures, operators must rely on log lines. Structured spans enable tracing visualization, latency attribution, and automated alerting.

## Current Code

**ToolDispatcher::dispatch_one** and related methods in `crates/roko-agent/src/dispatcher/mod.rs`:
- `dispatch_one` (or similar) -- dispatches a single tool call
- `dispatch_batch` -- dispatches a batch of calls (parallel + serial)
- Safety check methods called from dispatch

These functions use `tracing::{debug, warn}` for logging but do not create spans.

## Exact Changes

### Step 1: Add `#[instrument]` to dispatch_one

```rust
#[tracing::instrument(
    name = "tool.dispatch",
    skip(self, call, ctx),
    fields(
        tool.name = %call.name,
        tool.id = %call.id,
        otel.kind = "CLIENT",
    )
)]
pub async fn dispatch_one(
    &self,
    call: &ToolCall,
    ctx: &ToolContext,
) -> ToolResult {
    // existing body
}
```

### Step 2: Add `#[instrument]` to dispatch_batch

```rust
#[tracing::instrument(
    name = "tool.dispatch_batch",
    skip(self, calls, ctx),
    fields(
        batch.size = calls.len(),
    )
)]
pub async fn dispatch_batch(
    &self,
    calls: &[ToolCall],
    ctx: &ToolContext,
) -> Vec<ToolResult> {
    // existing body
}
```

### Step 3: Add spans around safety checks

Wrap the safety check section in a span:

```rust
let _safety_span = tracing::info_span!(
    "tool.safety_check",
    tool.name = %call.name,
).entered();
// existing safety check code
```

### Step 4: Add spans around handler execution

```rust
let _exec_span = tracing::info_span!(
    "tool.handler_execute",
    tool.name = %call.name,
).entered();
// existing handler.execute() call
```

### Step 5: Record result status

After dispatch completes:
```rust
tracing::Span::current().record("tool.result", match &result {
    ToolResult::Ok { .. } => "ok",
    ToolResult::Err { .. } => "error",
});
```

## Acceptance Criteria
- `dispatch_one` and `dispatch_batch` have `#[instrument]` spans
- Safety check and handler execution wrapped in named spans
- Span fields include tool name, batch size, result status
- `otel.kind = "CLIENT"` on dispatch spans for OpenTelemetry compat
- No behavioral changes to dispatch logic

## Do NOT
- Add new dependencies (tracing is already a dependency)
- Change dispatch behavior or return types
- Add spans to non-hot-path functions
