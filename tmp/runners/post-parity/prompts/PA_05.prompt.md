# PA_05: Audit and remove remaining reqwest::Client::new() calls

## Task
Find and eliminate all remaining `reqwest::Client::new()` calls outside the shared client factory.

## Runner Context
Runner PA, batch 5 of 5. Depends on PA_02, PA_03, PA_04.

## Problem
After wiring the shared client into the main consumers (PA_02–PA_04), many straggler call sites still create throwaway clients. This negates connection pooling and causes TLS renegotiation latency.

## Exact Changes

### Step 1: Fix roko-serve routes (biggest win — 4 clients in one handler)

In `routes/secrets.rs`, replace lines 182,206,228,247:
```rust
// BEFORE (each test function)
let client = reqwest::Client::new();

// AFTER (use state.http_client)
let client = &state.http_client;
```

### Step 2: Fix roko-serve feedback.rs

```rust
// BEFORE: feedback.rs:67
let client = Client::builder().user_agent("roko-serve-feedback/0.1").build()?;

// AFTER: accept state.http_client, add user-agent per-request via .header()
```

### Step 3: Fix roko-agent streaming backends

For `openai_compat_backend.rs:389` and `cursor_agent.rs:588`, add a `client: reqwest::Client` parameter to the streaming function and pass it from the agent's shared poster.

### Step 4: Fix roko-cli one-off callers

For `agent_serve.rs`, `chain_handler.rs`, `commands/job.rs` — thread the shared client from the CLI startup context.

### Step 5: Leave blocking clients alone

`config_cmd.rs:259` (blocking), `tui/app.rs:3183` (blocking), `roko-mcp-slack/main.rs:649` (blocking) — these use `reqwest::blocking::Client` which can't share with async. Leave these as-is.
