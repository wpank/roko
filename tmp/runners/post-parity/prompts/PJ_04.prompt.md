# PJ_04: Add fsync to JSONL append and wire recovery at startup

## Task
Add `sync_data()` to `append_jsonl()` for durability, and wire JSONL recovery at runner startup for all critical JSONL files.

## Runner Context
Runner PJ, batch 4 of 5. No dependencies.

## Problem
PV-4 anti-pattern: "Flush is not fsync." `append_jsonl()` at persist.rs:150-164 calls `file.flush()` which pushes data from Rust's buffer to the OS — but does NOT call `sync_data()` (fsync), so the OS can still lose data on power loss. Additionally, while `recover_jsonl()` exists at persist.rs:241-337, verify it's called at startup for all critical files.

## Current Code

**append_jsonl()** — `persist.rs:150-164`:
```rust
pub fn append_jsonl(path: &Path, value: &impl Serialize) -> Result<()> {
    let mut line = serde_json::to_string(value).context("serializing JSONL value")?;
    line.push('\n');
    let mut file = OpenOptions::new()
        .create(true)
        .append(true)
        .open(path)?;
    file.write_all(line.as_bytes())?;
    file.flush()?;   // ← flush only, no fsync
    Ok(())
}
```

**atomic_write()** — `persist.rs:141-147`:

## Exact Changes

### Step 1: Add sync_data() to append_jsonl()

In `persist.rs:150-164`, after `file.flush()`:

```rust
pub fn append_jsonl(path: &Path, value: &impl Serialize) -> Result<()> {
    let mut line = serde_json::to_string(value).context("serializing JSONL value")?;
    line.push('\n');
    let mut file = OpenOptions::new()
        .create(true)
        .append(true)
        .open(path)?;
    file.write_all(line.as_bytes())?;
    file.flush()?;
    file.sync_data()?;  // ← ADD: fsync for durability
    Ok(())
}
```

### Step 2: Add sync_data() to atomic_write()

In `persist.rs:141-147`, fsync the tmp file before rename:

```rust
pub fn atomic_write(path: &Path, content: &[u8]) -> Result<()> {
    let tmp = path.with_extension("tmp");
    {
        let mut file = fs::File::create(&tmp)?;
        file.write_all(content)?;
        file.sync_data()?;  // ← ADD: fsync before rename
    }
    fs::rename(&tmp, path)?;
    Ok(())
}
```

### Step 3: Verify recover_jsonl is called at startup for all JSONL files

In the runner startup path (event_loop.rs), verify that `recover_jsonl()` is called for:
- `events.jsonl` (runner events)
- `episodes.jsonl` (episode log)
- `efficiency.jsonl` (efficiency events)

If not called for all three at startup (not just on resume), add:

```rust
// At runner startup, before main loop:
for path in [&paths.events_jsonl, &paths.episodes_jsonl, &paths.efficiency_jsonl] {
    if path.exists() {
        if let Ok(recovery) = persist::recover_jsonl::<serde_json::Value, _>(path, |line| {
            serde_json::from_str(line)
        }) {
            if recovery.truncated_lines > 0 {
                warn!(
                    path = %path.display(),
                    truncated = recovery.truncated_lines,
                    "recovered truncated JSONL lines at startup"
                );
            }
        }
    }
}
```

## Acceptance Criteria
- `append_jsonl()` calls `sync_data()` after flush
- `atomic_write()` fsyncs the tmp file before rename
- All critical JSONL files recovered at startup (not just on resume)
- Recovery truncates only the last line if it's incomplete
- Existing valid data never lost

## Do NOT
- Replace JSONL with a database
- Add file locking (single-writer model is correct)
- Change the recover_jsonl() validation logic
- Remove flush() (keep both flush + sync_data)
