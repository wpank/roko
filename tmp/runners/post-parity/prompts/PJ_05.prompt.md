# PJ_05: Improve resume drift detection to re-queue instead of hard fail

## Task
Change resume drift detection from hard error to re-queue: drifted completed tasks should be marked as "needs re-run" instead of blocking resume entirely.

## Runner Context
Runner PJ, batch 5 of 5. No dependencies.

## Problem
PV-5 anti-pattern: "Resume should be resilient." Currently, `prepare_resume()` at resume.rs:151-192 compares task fingerprints via FNV-1a hash. When a task's content has changed since the snapshot, it produces a `TaskMismatch` which is a **hard error** — the entire resume fails. This is too aggressive: drifted tasks should be re-queued while non-drifted completed tasks are preserved.

## Current Code

**TaskDefFingerprint struct** — `persist.rs:131-138`:
```rust

    pub plan_id: String,
    pub task_id: String,

```

**Fingerprint computation** — `persist.rs:346-388`:

`id, title, role, tier, domain, depends_on, depends_on_plan, verify, acceptance, max_loc, max_retries, timeout_secs`

## Exact Changes

### Step 1: Change TaskMismatch from error to warning with re-queue

In `resume.rs`, replace the hard-error behavior with a soft re-queue:

```rust
// BEFORE: mismatches → ResumeError
// AFTER: mismatches → warning + re-queue

pub struct ResumeReport {
    // ... existing fields ...
    pub drifted_tasks: Vec<DriftedTask>,  // ← ADD
}

pub struct DriftedTask {
    pub plan_id: String,
    pub task_id: String,
    pub old_fingerprint: String,
    pub new_fingerprint: String,
}
```

In the validation loop at resume.rs:151-192:

```rust
let mut drifted = Vec::new();
for (plan_id, tasks) in plans {
    for task in tasks {
        let actual = TaskDefFingerprint::from_task(task, plan_id);
        let key = (plan_id.as_str(), task.id.as_str());
        let Some(expected) = snapshot_index.get(&key) else { continue; };
        if expected.fingerprint != actual.fingerprint {
            drifted.push(DriftedTask {
                plan_id: plan_id.clone(),
                task_id: task.id.clone(),
                old_fingerprint: expected.fingerprint.clone(),
                new_fingerprint: actual.fingerprint.clone(),
            });
        }
    }
}
report.drifted_tasks = drifted;
```

### Step 2: Remove drifted tasks from completed set

In the event loop resume path, after getting the ResumeReport:

```rust
for dt in &resume_report.drifted_tasks {
    if let Some(completed) = state.completed_tasks.get_mut(&dt.plan_id) {
        completed.retain(|id| id != &dt.task_id);
        warn!(
            plan = %dt.plan_id,
            task = %dt.task_id,
            "task definition drifted since snapshot — re-queuing"
        );
    }
}
```

### Step 3: Add --force-resume flag

In `types.rs` (RunConfig), add:
```rust
pub force_resume: bool,  // default: false
```

When `force_resume` is true, skip the drift check entirely (log at info level):
```rust
if config.force_resume {
    info!("--force-resume: skipping task drift validation");
    report.drifted_tasks = Vec::new();
}
```

### Step 4: Log summary of drift detection

```rust
if !resume_report.drifted_tasks.is_empty() {
    warn!(
        drifted_count = resume_report.drifted_tasks.len(),
        "detected {} drifted tasks — re-queuing for re-execution",
        resume_report.drifted_tasks.len()
    );
    for dt in &resume_report.drifted_tasks {
        info!(plan = %dt.plan_id, task = %dt.task_id, "re-queued (definition changed)");
    }
}
```

## Acceptance Criteria
- Drifted tasks detected via fingerprint comparison (same hash logic)
- Drifted completed tasks removed from completed set → re-queued for execution
- Non-drifted tasks preserved as completed
- `--force-resume` bypasses drift check entirely
- No drift → resume as before (no behavioral change)
- Warning log shows which tasks drifted

## Do NOT
- Block resume entirely on drift (re-queue instead)
- Change the fingerprint format or hash algorithm
- Change canonical_task_payload field list
