# PK_04: Wire dream trigger worker to drain after plan completion

## Task
After a plan completes, drain pending dream triggers and run the dream consolidation cycle (awaited, not fire-and-forget).

## Runner Context
Runner PK, batch 4 of 6. No dependencies.

## Problem
KR-4 anti-pattern: "Triggers accumulate but never fire." Dream triggers are spawned during plan execution (event_loop.rs:670-703 uses `tokio::spawn`). But `tokio::spawn` without awaiting the JoinHandle means the dream task may be cancelled when the runner exits. The dream cycle produces routing advice and knowledge distillation that feeds PK_05.

## Current Code

**Dream trigger spawn** — `event_loop.rs:670-703`:
```rust
tokio::spawn({
    let dream_runner = DreamRunner::new(&config.workdir.join(".roko"), dream_config);
    async move {
        dream_runner.consolidate_now().await;
    }
});
```
This spawns an un-awaited tokio task. If the runner exits before the dream task completes, the dream is cancelled.

**IMPORTANT**: `DreamRunner::consolidate_now()` signature — check whether it's:
- `async fn consolidate_now(&mut self) -> Result<DreamReport>` (async — can be awaited directly)
- `fn consolidate_now(&mut self) -> Result<DreamReport>` (sync — calls `block_on` internally)

If it's synchronous with internal `block_on()`, you MUST use `tokio::task::spawn_blocking()` to avoid panicking when calling `block_on` inside a tokio runtime. Do NOT directly `.await` a synchronous method.

**DreamCycleReport** — `crates/roko-dreams/src/cycle.rs:72-85`:
```rust
pub struct DreamCycleReport {
    started_at,
    completed_at,
    total_episodes,
    processed_episodes,
    // ...
}
```

## Exact Changes

### Step 1: Replace fire-and-forget spawn with awaited dream

In `event_loop.rs`, after the main loop exits (plan completion):

```rust
// BEFORE (fire-and-forget, line 670-703):
tokio::spawn(async move { dream_runner.consolidate_now().await; });

// AFTER (awaited with timeout):
if config.learning.dream_on_completion.unwrap_or(true) {
    debug!("running dream consolidation after plan completion");
    let dream_runner = DreamRunner::new(&config.workdir.join(".roko"), dream_config);

    // Check whether consolidate_now is async or sync:
    // If ASYNC: await directly with timeout
    let dream_result = tokio::time::timeout(
        Duration::from_secs(120),
        dream_runner.consolidate_now(),  // if async
    ).await;

    // If SYNC (calls block_on internally): use spawn_blocking to avoid panic
    // let dream_result = tokio::time::timeout(
    //     Duration::from_secs(120),
    //     tokio::task::spawn_blocking(move || dream_runner.consolidate_now()),
    // ).await;

    match dream_result {
        Ok(Ok(report)) => {
            info!(
                episodes = report.processed_episodes,
                "dream consolidation completed"
            );
        }
        Ok(Err(err)) => {
            warn!(%err, "dream consolidation failed — plan results unaffected");
        }
        Err(_) => {
            warn!("dream consolidation timed out after 120s — skipping");
        }
    }
}
```

### Step 2: Verify consolidate_now is async-safe

Before implementing Step 1, check the method signature:

```bash
grep -n "fn consolidate_now" crates/roko-dreams/src/runner.rs
```

- If it's `pub async fn consolidate_now(...)` → use the async path above
- If it's `pub fn consolidate_now(...)` (sync, uses `block_on`) → use `spawn_blocking` path
- If it's `pub async fn consolidate_async(...)` with sync wrapper → call `consolidate_async()` directly

### Step 3: Make dream opt-in via config

Add `dream_on_completion` to the learning config if it doesn't already exist:

```rust
// In config types (check roko-core/src/config/schema.rs learning section):
#[serde(default)]
pub dream_on_completion: Option<bool>,  // default true
```

### Step 4: Ensure dream doesn't affect plan exit code

The dream runs AFTER the plan's success/failure is determined but BEFORE the runner returns:

```rust
// Plan result is already determined
let plan_result = if all_passed { PlanResult::Success } else { PlanResult::Failed(failures) };

// Dream is best-effort post-processing — errors logged, not propagated
run_dream_consolidation_if_enabled(&config).await;

plan_result  // return original result regardless of dream outcome
```

## Acceptance Criteria
- Dream consolidation awaited after plan completion (not fire-and-forget)
- Dream has 120-second timeout (doesn't hang the runner)
- If `consolidate_now()` is sync, uses `spawn_blocking` (not direct `.await`)
- Dream failure doesn't change plan result or exit code
- Opt-out via `learning.dream_on_completion = false`
- Dream report logged (episodes processed count)

## Do NOT
- Directly `.await` a synchronous method that uses `block_on` (causes panic)
- Run dreams during plan execution (only after completion)
- Add a cron/scheduler for dreams
- Change the DreamCycle or DreamRunner internals
