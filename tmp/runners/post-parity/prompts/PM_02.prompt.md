# PM_02: Add merge conflict detection and automatic retry with rebase

## Task
Detect merge conflicts in PlanMerger and retry with rebase before declaring failure.

## Runner Context
Runner PM, batch 2 of 3. Depends on PM_01.

## Problem
MQ-2 anti-pattern: "Merge conflicts are silent data loss." When two parallel tasks modify overlapping files and their branches are merged sequentially, the second merge may conflict. Without detection, git merge failures are silent errors. The merger needs conflict detection and rebase-based resolution.

## Current Code

After PM_01, `execute_git_merge()` exists and returns `MergeResult::Conflict`. This batch adds the retry-with-rebase logic.

## Exact Changes

### Step 1: Detect conflict from git merge exit code

In `execute_git_merge()` (from PM_01), the conflict case is:
```rust
if stderr.contains("CONFLICT") || stderr.contains("Automatic merge failed") {
    // Abort the failed merge
    Command::new("git")
        .args(["merge", "--abort"])
        .current_dir(workdir)
        .output()?;

    Ok(MergeResult::Conflict { branch, stderr })
}
```

### Step 2: Add rebase retry on conflict

```rust
fn handle_merge_conflict(
    workdir: &Path,
    task_branch: &str,
    plan_branch: &str,
) -> Result<MergeResult> {
    info!(
        task = task_branch,
        plan = plan_branch,
        "merge conflict detected — attempting rebase"
    );

    // Checkout the task branch
    let checkout = Command::new("git")
        .args(["checkout", task_branch])
        .current_dir(workdir)
        .output()?;
    if !checkout.status.success() {
        return Ok(MergeResult::RebaseFailed {
            branch: task_branch.to_string(),
            reason: "checkout failed".into(),
        });
    }

    // Rebase onto the plan branch
    let rebase = Command::new("git")
        .args(["rebase", plan_branch])
        .current_dir(workdir)
        .output()?;

    if rebase.status.success() {
        // Rebase succeeded — now merge should be fast-forward
        let _ = Command::new("git")
            .args(["checkout", plan_branch])
            .current_dir(workdir)
            .output()?;
        let merge = Command::new("git")
            .args(["merge", "--no-ff", task_branch])
            .current_dir(workdir)
            .output()?;
        if merge.status.success() {
            return Ok(MergeResult::RebasedSuccess);
        }
    }

    // Rebase failed — abort and report
    let _ = Command::new("git")
        .args(["rebase", "--abort"])
        .current_dir(workdir)
        .output();
    let _ = Command::new("git")
        .args(["checkout", plan_branch])
        .current_dir(workdir)
        .output();

    Ok(MergeResult::RebaseFailed {
        branch: task_branch.to_string(),
        reason: String::from_utf8_lossy(&rebase.stderr).to_string(),
    })
}
```

### Step 3: Integrate into merge flow

```rust
fn perform_merge(workdir, task_branch, plan_branch) -> Result<MergeResult> {
    match execute_git_merge(workdir, task_branch, plan_branch)? {
        MergeResult::Success => Ok(MergeResult::Success),
        MergeResult::Conflict { .. } => {
            handle_merge_conflict(workdir, task_branch, plan_branch)
        }
        other => Ok(other),
    }
}
```

### Step 4: Re-queue task on failed rebase

In the event loop, when `MergeResult::RebaseFailed` is returned:

```rust
match merge_result {
    MergeResult::RebaseFailed { branch, reason } => {
        warn!(
            branch = %branch,
            reason = %reason,
            "merge conflict unresolvable — re-queuing task for re-execution"
        );
        // Mark task as needing re-execution on updated base
        task_dag.mark_for_rerun(&plan_id, &task_id);
        state.merge_conflict_count += 1;
    }
    // ...
}
```

## Acceptance Criteria
- Merge conflicts detected (not silently corrupted)
- Rebase attempted as first resolution strategy
- Failed rebase → task re-queued for re-execution with updated base
- Clean merges unaffected
- Conflict count tracked in RunState for logging

## Do NOT
- Implement automatic conflict resolution (agents handle that on re-execution)
- Block the entire plan on one conflict
- Add interactive rebase or complex resolution strategies
