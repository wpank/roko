# PN_07: Fix /api/bench/runs POST path to match demo app expectations

## Task
Ensure the bench run creation endpoint matches what the demo app calls.

## Runner Context
Runner PN, batch 7 of 7. No dependencies.

## Problem
DS-7 anti-pattern: "Path mismatch between frontend and backend." Demo calls `POST /api/bench/runs` (plural) but server registers `POST /api/bench/run` (singular). Path mismatch causes bench creation to fail silently.

## Current Code

**Bench route registration** — `routes/bench.rs:31-46`:
```rust
.route("/bench/run", post(start_bench_run))   // singular
.route("/bench/runs", get(list_bench_runs))    // plural (GET only)
```

**Demo expectation** — `useBench.ts:204`:
Calls `POST /api/bench/runs` (plural).

## Exact Changes

### Step 1: Add plural POST alias

The simplest fix is to register the same handler under both paths:

```rust
.route("/bench/run", post(start_bench_run))
.route("/bench/runs", get(list_bench_runs).post(start_bench_run))  // ← ADD .post()
```

This registers:
- `GET /api/bench/runs` → list (existing)
- `POST /api/bench/runs` → create (NEW, matches demo)
- `POST /api/bench/run` → create (existing, for backward compat)

### Step 2: Verify response format

Check that `start_bench_run` returns `{id: "run-id-here"}` matching `useBench.ts:204` expectations:

```rust
// In start_bench_run handler:
Json(serde_json::json!({
    "id": run_id,  // must be "id", not "run_id" or "runId"
}))
```

### Step 3: Verify the demo creates bench runs successfully

After the fix, `POST /api/bench/runs` should:
1. Create a new bench run
2. Return `{id: "..."}`
3. The run appears in `GET /api/bench/runs` list
