# PO_03: Publish TaskStatusEvent per task

## Task
Emit task status transitions (pending → running → gating → done/failed) as StateHub events.

## Runner Context
Runner PO, batch 3 of 6. No dependencies.

## Problem
OB-3 anti-pattern: "Task status exists in memory but invisible." TUI and demo app have no way to show per-task progress during a plan run. Task status changes only exist in the event loop's internal state.

## Current Code

**DashboardEvent** — already has `TaskStarted` and `TaskCompleted` variants. Check if they include enough detail (task_id, plan_id, status enum).

**Task lifecycle in event loop**:
1. Task selected from DAG → dispatch
2. Agent spawned → agent working
3. Agent completes → gate execution
4. Gates pass/fail → task done/retry

## Exact Changes

### Step 1: Verify existing TaskStarted/TaskCompleted events

If `DashboardEvent::TaskStarted` and `TaskCompleted` already exist, verify they're published at each transition point:

```rust
// At task dispatch:
tui.publish(DashboardEvent::TaskStarted {
    task_id: task.id.clone(),
    plan_id: plan_id.clone(),
    title: task.title.clone(),
    phase: "running".into(),
});

// At gate start:
tui.publish(DashboardEvent::TaskStarted {
    task_id: task.id.clone(),
    plan_id: plan_id.clone(),
    title: task.title.clone(),
    phase: "gating".into(),
});

// At task completion:
tui.publish(DashboardEvent::TaskCompleted {
    task_id: task.id.clone(),
    plan_id: plan_id.clone(),
    success: gate_passed,
    phase: "completed".into(),
});
```

### Step 2: Add status field if missing

If the existing variants don't carry enough status info, add a `status` field:

```rust
TaskStatusChanged {
    task_id: String,
    plan_id: String,
    status: String,  // "pending", "running", "gating", "completed", "failed", "retrying"
    title: Option<String>,
    timestamp_ms: u64,
},
```

### Step 3: Publish at every transition

Ensure events are emitted at all 6 status transitions:
1. **pending** → when task is first selected from DAG
2. **running** → when agent is spawned
3. **gating** → when agent completes and gates start
4. **completed** → when gates pass
5. **failed** → when gates fail and max retries exceeded
6. **retrying** → when gates fail but retries remain

## Write Scope
- `crates/roko-core/src/dashboard_snapshot.rs` (verify/enhance task event variants)
- `crates/roko-cli/src/runner/event_loop.rs` (publish at each transition)


## Verify
```bash
cargo build --workspace 2>&1 | head -30
cargo test --workspace 2>&1 | tail -20
```
## Acceptance Criteria
- Task status events emitted at every transition
- Demo/TUI can build a task checklist from events alone
- Status values: pending, running, gating, completed, failed, retrying
- Events include task_id, plan_id, and human-readable title

## Do NOT
- Change the task DAG's internal status tracking
- Add per-task event storage (events are ephemeral, streamed via SSE)
