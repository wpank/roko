# PO_05: Publish ResumeStarted and ResumeCompleted events

## Task
Emit events when a plan resume begins and completes, so UIs can show resume status.

## Runner Context
Runner PO, batch 5 of 6. No dependencies.

## Problem
OB-5 anti-pattern: "Resume is invisible." Resume happens silently. Users see tasks re-running but don't know the system is resuming from a checkpoint. No UI indication of which tasks were skipped vs re-executed.

## Current Code

**prepare_resume()** — `resume.rs:130-214`:
Loads snapshot, validates fingerprints, runs JSONL recovery. No events emitted.

## Exact Changes

### Step 1: Add resume event variants

In `dashboard_snapshot.rs`:

```rust
ResumeStarted {
    checkpoint_path: String,
    tasks_completed_before: usize,
    tasks_remaining: usize,
    timestamp_ms: u64,
},
ResumeCompleted {
    tasks_skipped: usize,
    drifted_tasks: usize,
    timestamp_ms: u64,
},
ResumeFailed {
    reason: String,
    timestamp_ms: u64,
},
```

### Step 2: Publish in resume path

In `event_loop.rs`, around the resume handling code:

```rust
// Before prepare_resume():
tui.publish(DashboardEvent::ResumeStarted {
    checkpoint_path: snapshot_path.display().to_string(),
    tasks_completed_before: snapshot.tasks_completed,
    tasks_remaining: snapshot.tasks_total - snapshot.tasks_completed,
    timestamp_ms: now_ms(),
});

// After successful resume:
tui.publish(DashboardEvent::ResumeCompleted {
    tasks_skipped: resume_report.tasks_skipped,
    drifted_tasks: resume_report.drifted_tasks.len(),
    timestamp_ms: now_ms(),
});

// On resume failure:
tui.publish(DashboardEvent::ResumeFailed {
    reason: format!("{}", err),
    timestamp_ms: now_ms(),
});
```
