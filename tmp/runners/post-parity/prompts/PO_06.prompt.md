# PO_06: Publish ModelFallbackEvent when CascadeRouter falls back

## Task
Emit an event when the CascadeRouter uses a fallback model after the primary fails.

## Runner Context
Runner PO, batch 6 of 6. No dependencies.

## Problem
OB-6 anti-pattern: "Fallback is invisible." When model X fails and the router falls back to model Y, there's no trace. Users can't understand why a different model was used or diagnose provider issues.

## Current Code

**Model fallback** — occurs in the dispatch path when a provider returns an error and the router selects an alternative. Currently this is either:
- In `ModelCallService`'s `ProviderCallCell::execute` fallback chain
- In the runner's retry logic after task failure

## Exact Changes

### Step 1: Add ModelFallback variant

In `dashboard_snapshot.rs`:

```rust
ModelFallback {
    requested_model: String,
    fallback_model: String,
    reason: String,       // "rate_limited", "auth_error", "timeout", etc.
    task_id: Option<String>,
    timestamp_ms: u64,
},
```

### Step 2: Publish when fallback occurs

In the dispatch path (event_loop.rs), when a model call fails and the retry uses a different model:

```rust
// When fallback is selected:
tui.publish(DashboardEvent::ModelFallback {
    requested_model: original_model.clone(),
    fallback_model: actual_model.clone(),
    reason: classify_error(&err),  // from PS_03's ProviderErrorKind
    task_id: Some(state.current_task_id.clone()),
    timestamp_ms: now_ms(),
});
```

### Step 3: Classify the reason

Use a simple string classification:

```rust
fn classify_fallback_reason(err: &anyhow::Error) -> String {
    let msg = err.to_string().to_lowercase();
    if msg.contains("rate limit") || msg.contains("429") {
        "rate_limited".into()
    } else if msg.contains("auth") || msg.contains("401") || msg.contains("403") {
        "auth_error".into()
    } else if msg.contains("timeout") {
        "timeout".into()
    } else if msg.contains("503") || msg.contains("unavailable") {
        "service_unavailable".into()
    } else {
        "unknown".into()
    }
}
```

### Step 4: Only emit when fallback actually occurs

No event when the primary model succeeds (no noise):

```rust
if actual_model != requested_model {
    tui.publish(DashboardEvent::ModelFallback { ... });
}
```

## Acceptance Criteria
- Fallback events emitted with original and fallback model names
- Reason for fallback included (rate_limited, auth, timeout, etc.)
- No event when primary model succeeds (no noise)
- TUI/SSE subscribers can show "Fell back: sonnet → haiku (rate_limited)"

## Do NOT
- Change the fallback selection logic
- Add model exclusion based on events
