# PS_04: Wire provider error classification into retry strategy

## Task
Use `ProviderErrorKind` from PS_03 to make differentiated retry decisions in the v2 runner.

## Runner Context
Runner PS, batch 4 of 4. Depends on PS_03.

## Problem
PD-4 anti-pattern: "Same backoff for all failures." `backoff_for_attempt` (task_dag.rs:115-135) uses exponential backoff regardless of failure type. Rate limits should respect `retry-after`; auth failures should not retry; network timeouts should use longer backoff.

## Current Code

**backoff_for_attempt** — `task_dag.rs:115-135`:
```rust
pub fn backoff_for_attempt(&self, attempt: u32) -> Duration {
    let base = Duration::from_secs(2);
    let multiplier = 2u64.pow(attempt.min(5));
    base * multiplier as u32
    // max: 2 * 32 = 64 seconds
}
```

## Exact Changes

### Step 1: Change backoff_for_attempt to accept error kind

```rust
pub fn backoff_for_attempt(
    attempt: u32,
    error_kind: Option<&ProviderErrorKind>,
) -> Option<Duration> {
    match error_kind {
        // Rate limit: use retry-after if available, else short exponential
        Some(ProviderErrorKind::RateLimit { retry_after: Some(d) }) => Some(*d),
        Some(ProviderErrorKind::RateLimit { retry_after: None }) => {
            let base = Duration::from_secs(1);
            Some(base * 2u32.pow(attempt.min(4)))  // 1, 2, 4, 8, 16s
        }

        // Network timeout: longer base backoff
        Some(ProviderErrorKind::NetworkTimeout) => {
            let base = Duration::from_secs(5);
            Some(base * 2u32.pow(attempt.min(3)))  // 5, 10, 20, 40s
        }

        // Service unavailable: standard exponential
        Some(ProviderErrorKind::ServiceUnavailable) => {
            let base = Duration::from_secs(2);
            Some(base * 2u32.pow(attempt.min(5)))  // 2, 4, 8, 16, 32, 64s
        }

        // Non-retryable errors: no backoff, don't retry
        Some(ProviderErrorKind::AuthFailure) |
        Some(ProviderErrorKind::InvalidRequest) |
        Some(ProviderErrorKind::ModelNotFound) |
        Some(ProviderErrorKind::QuotaExceeded) |
        Some(ProviderErrorKind::ContextLengthExceeded) => None,

        // Unknown: default exponential
        _ => {
            let base = Duration::from_secs(2);
            Some(base * 2u32.pow(attempt.min(5)))
        }
    }
}
```

### Step 2: Classify error before retry decision in event loop

In the runner event loop, when a task fails:

```rust
let error_kind = classify_provider_error(&err);

match backoff_for_attempt(attempt, Some(&error_kind)) {
    Some(backoff) => {
        debug!(
            task = %task_id,
            error = ?error_kind,
            backoff_ms = backoff.as_millis(),
            "retrying after backoff"
        );
        tokio::time::sleep(backoff).await;
        // re-queue task
    }
    None => {
        warn!(
            task = %task_id,
            error = ?error_kind,
            "non-retryable error — failing task immediately"
        );
        // mark task as permanently failed
    }
}
```

## Acceptance Criteria
- Rate-limited requests respect `retry-after` header when available
- Auth failures fail immediately without retry
- Network timeouts use longer backoff (5s base) than rate limits (1s base)
- Provider health records error classification
- Unknown errors use default exponential backoff (backward compatible)

## Do NOT
- Change the max retry count logic (that's separate from backoff strategy)
- Add infinite retry for any error type
