# PV_02: Fill PlanCompleted feedback with real task counts and cost totals

## Task
Populate the `FeedbackEvent::PlanCompleted` event with actual task counts and cumulative cost instead of zeros.

## Runner Context
Runner PV, batch 2 of 5. Depends on PV_01.

## Problem
`event_loop.rs:1347-1358` emits zero task counts:

```rust
// event_loop.rs:1353-1357 — CURRENT (broken)
Some(FeedbackEvent::PlanCompleted {
    plan_id: plan_id.clone(),
    succeeded,
    tasks_completed: 0,   // always zero
    tasks_failed: 0,      // always zero
    total_cost_usd: 0.0,  // always zero
})
```

## Exact Changes

### Step 1: Thread `RunState` map into the `PlanCompleted` arm

The `PlanCompleted` event is produced in the `RunnerEvent::PlanCompleted` arm of `runner_event_to_feedback`. After PV_01 passes `run_states`, use it to compute totals:

```rust
RunnerEvent::PlanCompleted { plan_id, succeeded } => {
    let plan_tasks: Vec<&RunState> = run_states.values()
        .filter(|s| s.plan_id == *plan_id)
        .collect();
    let tasks_completed = plan_tasks.iter().filter(|s| s.succeeded).count() as u32;
    let tasks_failed = plan_tasks.iter().filter(|s| !s.succeeded).count() as u32;
    let total_cost_usd: f64 = plan_tasks.iter().map(|s| s.cost_usd).sum();

    Some(FeedbackEvent::PlanCompleted {
        plan_id: plan_id.clone(),
        succeeded: *succeeded,
        tasks_completed,
        tasks_failed,
        total_cost_usd,
    })
}
```

### Step 2: If `RunState` doesn't have a `plan_id` field

Check the `RunState` struct. If it doesn't track which plan a task belongs to, add `plan_id: String` to `RunState` and populate it when the state is created from `TaskDef`.
