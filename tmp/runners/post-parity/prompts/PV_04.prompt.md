# PV_04: Fix started_at_ms to emit Unix timestamp instead of elapsed duration

## Task
Fix the `started_at_ms` field in `save_snapshot` to emit a Unix epoch timestamp, not elapsed milliseconds since process start.

## Runner Context
Runner PV, batch 4 of 5. No dependencies.

## Problem
`event_loop.rs:1432`:

```rust
// CURRENT (wrong)
started_at_ms: state.started_at.elapsed().as_millis().saturating_sub(0) as u64,
```

`state.started_at` is a `std::time::Instant` (monotonic clock). `.elapsed()` returns how long the run has been alive, not a wall-clock timestamp. The field is named `started_at_ms` and every other `timestamp_ms` field in the codebase is a Unix epoch millis. The `.saturating_sub(0)` is a no-op.

## Exact Changes

### Step 1: Compute Unix timestamp from Instant

```rust
// AFTER (correct)
started_at_ms: {
    let elapsed = state.started_at.elapsed();
    let now_unix = std::time::SystemTime::now()
        .duration_since(std::time::UNIX_EPOCH)
        .unwrap_or_default()
        .as_millis() as u64;
    now_unix.saturating_sub(elapsed.as_millis() as u64)
},
```

This computes: "current Unix time minus how long the run has been alive = when the run started in Unix time."

### Step 2: Verify `tui_bridge.rs:177` consistency

Check that `timestamp_now_ms()` in `tui_bridge.rs:174-179` uses the same pattern. It currently does `SystemTime::now().duration_since(UNIX_EPOCH).unwrap_or_default()` which is correct (returns current Unix time), but the `unwrap_or_default()` returns 0 on failure — add a comment noting this edge case.
