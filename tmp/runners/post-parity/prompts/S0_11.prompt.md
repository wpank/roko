# S0_11: Forward streaming events to chat TUI

## Task
Verify that live streaming events from the Session dispatch path are rendered in the chat TUI, and fix any remaining discard paths.

## Runner Context
Post-parity Phase 0 (Stability), batch 11 of 25. No dependencies. Group S0C (Learning Wiring).

## Problem
The chat TUI in `crates/roko-cli/src/chat_inline.rs` has a `streaming_event_rx` channel
(line 768) that receives `AgentRuntimeEvent` values from the session dispatch path. The
event loop (around line 1207) already processes these events:

```rust
if let Some(ref mut event_rx) = session.streaming_event_rx {
    loop {
        match event_rx.try_recv() {
            Ok(event) => match &event {
                roko_agent::AgentRuntimeEvent::MessageDelta { text } => {
                    // Renders text to TUI viewport
                }
                // ... other variants may be silently dropped
            },
```

The issue is that non-`MessageDelta` event variants may be silently dropped. Events like
`ToolUse`, `ToolResult`, `ThinkingDelta`, `Error`, and `Usage` should either be rendered
or explicitly logged.

## Exact Changes

### Step 1: Audit AgentRuntimeEvent variants

In `crates/roko-agent/src/lib.rs` or wherever `AgentRuntimeEvent` is defined, read all
variants. Common ones:
- `MessageDelta { text }` -- already handled
- `ThinkingDelta { text }` -- should update thinking indicator
- `ToolUse { name, input }` -- should show tool activity
- `ToolResult { name, output }` -- should show tool result
- `Error { message }` -- should display error
- `Usage { input_tokens, output_tokens }` -- should update cost meter

### Step 2: Add handlers for unhandled variants

In the event processing loop at `chat_inline.rs:1210`, expand the match arms:

```rust
Ok(event) => match &event {
    roko_agent::AgentRuntimeEvent::MessageDelta { text } => {
        // existing: render text to viewport
    }
    roko_agent::AgentRuntimeEvent::ThinkingDelta { text } => {
        if session.phase == Phase::Thinking {
            // Update the thinking label with latest thought
            session.streaming.update_thinking(text);
        }
    }
    roko_agent::AgentRuntimeEvent::ToolUse { name, .. } => {
        session.streaming.update_status(&format!("using {name}..."));
    }
    roko_agent::AgentRuntimeEvent::ToolResult { .. } => {
        // Tool result received; no special rendering needed
    }
    roko_agent::AgentRuntimeEvent::Usage { input_tokens, output_tokens } => {
        session.cost.add_tokens(*input_tokens, *output_tokens);
    }
    _ => {
        tracing::trace!("unhandled streaming event: {event:?}");
    }
}
```

Adjust variant names and field names to match the actual `AgentRuntimeEvent` enum.

### Step 3: Apply same fix to the second event loop

There is a duplicate event loop at around line 1586 (the direct REPL variant). Apply
the same expanded match arms there.
