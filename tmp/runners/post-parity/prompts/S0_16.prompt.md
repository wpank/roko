# S0_16: Remove unsafe set_var() provider override (duplicate elimination)

## Task
Remove any remaining `unsafe { std::env::set_var() }` calls for provider override in CLI entry points beyond what S0_08 covered.

## Runner Context
Post-parity Phase 0 (Stability), batch 16 of 25. No dependencies. Group S0D (Env Var Elimination).

## Problem
S0_08 removes the `set_var("ROKO_PROVIDER", ...)` in `util.rs`. But there may be other
`set_var` calls scattered across CLI entry points that set environment variables for
provider selection, model override, or API key injection.

These are unsafe in multi-threaded contexts and should all be replaced with config threading.

## Exact Changes

### Step 1: Audit all unsafe set_var calls

Search the entire CLI crate for `set_var` calls:

```bash
grep -rn 'set_var\|std::env::set_var' crates/roko-cli/src/ --include='*.rs'
```

For each hit:
1. If it sets a config-related env var (ROKO_*, ANTHROPIC_*, provider vars), replace with
   direct config mutation after `resolve_config_for_workdir()`.
2. If it sets a PATH or non-config var, leave it but add a comment explaining why.
3. If it is in test code, leave it.

### Step 2: Replace each set_var with config mutation

For each unsafe set_var found, apply the pattern from S0_08:

**Before:**
```rust
#[allow(unsafe_code)]
unsafe { std::env::set_var("ROKO_SOME_VAR", value) };
```

**After:**
```rust
// Apply override directly to config (replaces unsafe set_var).
config.some_field = value.clone();
```

### Step 3: Document any set_var calls that MUST remain

If any `set_var` calls are genuinely needed (e.g., setting `PATH` for child processes),
wrap them with a clear safety comment:

```rust
// SAFETY: This set_var is intentional -- it must be visible to child processes
// spawned by the agent. It is called before the tokio runtime starts threads.
#[allow(unsafe_code)]
unsafe { std::env::set_var("PATH", new_path) };
```

### Step 4: Search other crates too

Also check `roko-acp`, `roko-agent`, and `roko-serve` for similar patterns:

```bash
grep -rn 'set_var' crates/ --include='*.rs' | grep -v target/ | grep -v test
```

```bash
grep -rn '#\[allow(unsafe_code)\]' crates/roko-cli/src/ --include='*.rs'
# Each remaining allow should have a SAFETY comment
```

## Acceptance Criteria
- No `unsafe { std::env::set_var }` for config-related vars in production CLI code
- Any remaining `set_var` calls have clear SAFETY documentation
- Config overrides applied via direct struct mutation after loading
- Tests may still use `set_var` (they run single-threaded)
- `#[allow(unsafe_code)]` only present with documented justification

## Do NOT
- Remove `set_var` calls in test code
- Change the `apply_env()` function in roko-core
- Add new config fields (use existing ones)
- Break child process environment setup for agents
- Touch any files outside the identified scope
