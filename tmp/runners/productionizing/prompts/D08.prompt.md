# D08: Feature-gated OpenTelemetry export

## Goal

Add an `otel` feature to `roko-cli` that, when enabled and `OTEL_EXPORTER_OTLP_ENDPOINT` is set, attaches an OpenTelemetry layer to the existing `tracing` subscriber. Default build does NOT pull `opentelemetry-*` crates.

## Files to Edit

- `crates/roko-cli/Cargo.toml` (add optional deps + `otel` feature)
- `crates/roko-cli/src/main.rs` (or `crates/roko-cli/src/telemetry.rs` if it already exists) — wire the layer
- `crates/roko-cli/src/telemetry.rs` (NEW if missing)

## Also Read (do NOT modify)

- The existing tracing subscriber init — search:

```bash
grep -rn 'tracing_subscriber::registry\|tracing_subscriber::fmt\|EnvFilter' crates/roko-cli/src/main.rs
```

## Step 1: Cargo.toml

Add optional deps and the feature. Keep existing entries; insert near the bottom:

```toml
[dependencies]
# ... existing ...
opentelemetry         = { version = "0.27", optional = true }
opentelemetry_sdk     = { version = "0.27", features = ["rt-tokio"], optional = true }
opentelemetry-otlp    = { version = "0.27", features = ["grpc-tonic", "trace"], optional = true }
tracing-opentelemetry = { version = "0.28", optional = true }

[features]
otel = [
    "dep:opentelemetry",
    "dep:opentelemetry_sdk",
    "dep:opentelemetry-otlp",
    "dep:tracing-opentelemetry",
]
```

Pin to versions that the workspace already uses elsewhere if any. If unsure, the `0.27 / 0.28` set above is mutually compatible.

## Step 2: telemetry.rs

If `crates/roko-cli/src/telemetry.rs` exists:

```bash
ls crates/roko-cli/src/telemetry.rs
```

…edit it. Otherwise create it.

```rust
//! Tracing subscriber setup, with optional OpenTelemetry export.
//!
//! The OTEL layer is feature-gated (`otel`) and additionally requires the
//! `OTEL_EXPORTER_OTLP_ENDPOINT` env var. When either is missing, this
//! module behaves identically to a plain `tracing_subscriber::fmt()` setup.

use tracing_subscriber::layer::SubscriberExt;
use tracing_subscriber::util::SubscriberInitExt;
use tracing_subscriber::EnvFilter;

pub fn init_tracing() -> anyhow::Result<()> {
    let filter = EnvFilter::try_from_default_env()
        .unwrap_or_else(|_| EnvFilter::new("info,roko=debug"));

    let fmt_layer = tracing_subscriber::fmt::layer().compact();

    #[cfg(feature = "otel")]
    {
        if let Some(layer) = build_otel_layer()? {
            tracing_subscriber::registry()
                .with(filter)
                .with(fmt_layer)
                .with(layer)
                .init();
            return Ok(());
        }
    }

    tracing_subscriber::registry()
        .with(filter)
        .with(fmt_layer)
        .init();
    Ok(())
}

#[cfg(feature = "otel")]
fn build_otel_layer() -> anyhow::Result<Option<tracing_opentelemetry::OpenTelemetryLayer<
    tracing_subscriber::Registry,
    opentelemetry_sdk::trace::Tracer,
>>> {
    let endpoint = match std::env::var("OTEL_EXPORTER_OTLP_ENDPOINT") {
        Ok(e) => e,
        Err(_) => return Ok(None),
    };
    let service_name =
        std::env::var("OTEL_SERVICE_NAME").unwrap_or_else(|_| "roko".to_string());

    use opentelemetry::KeyValue;
    use opentelemetry_otlp::WithExportConfig;
    use opentelemetry_sdk::Resource;

    let exporter = opentelemetry_otlp::new_exporter().tonic().with_endpoint(endpoint);
    let provider = opentelemetry_otlp::new_pipeline()
        .tracing()
        .with_exporter(exporter)
        .with_trace_config(
            opentelemetry_sdk::trace::Config::default()
                .with_resource(Resource::new(vec![KeyValue::new("service.name", service_name)])),
        )
        .install_batch(opentelemetry_sdk::runtime::Tokio)?;
    Ok(Some(tracing_opentelemetry::layer().with_tracer(provider)))
}
```

If `tracing-opentelemetry` exposes a different builder API in the version you pin, match it. The shape above is the 0.27/0.28 idiom. If newer versions land in the workspace, adjust.

## Step 3: main.rs

Find the existing tracing init (it might already be `tracing_subscriber::fmt::init();`). Replace with a call to the new `init_tracing()`:

```rust
mod telemetry;

fn main() -> anyhow::Result<()> {
    telemetry::init_tracing()?;
    // ... existing main body ...
}
```

If `main.rs` already has a more elaborate init (e.g. it sets up file logging), preserve that and *add* the OTEL layer beside it instead of replacing.

## Step 4: opt-in only

Default build (`cargo build -p roko-cli`) MUST NOT compile any of the `opentelemetry-*` crates. Verify by reading the cargo features — `dep:` prefix is critical for that.

`OTEL_EXPORTER_OTLP_ENDPOINT` unset = no OTEL layer attached even with the feature on.

## Step 5: instrumentation scope

Do NOT add `#[tracing::instrument]` to every function in this batch. The existing tracing instrumentation is enough; OTEL just exports it. Hot paths to focus on (instrument later, separate batch):

- `orchestrate.rs::dispatch_agent_with`
- `model_call_service.rs::call_model`
- the gate pipeline

Adding instrumentation in this batch would balloon the diff. Skip it.

## Verification

```bash
grep -nE '\[features\]|^otel ' crates/roko-cli/Cargo.toml
grep -nE 'opentelemetry|tracing-opentelemetry' crates/roko-cli/Cargo.toml
ls crates/roko-cli/src/telemetry.rs
grep -n 'init_tracing' crates/roko-cli/src/main.rs
```

## Acceptance Criteria

- [ ] `[features] otel = [...]` in `roko-cli/Cargo.toml`
- [ ] `opentelemetry`, `opentelemetry_sdk`, `opentelemetry-otlp`, `tracing-opentelemetry` are `optional = true`
- [ ] `crates/roko-cli/src/telemetry.rs` exists with `init_tracing`
- [ ] OTEL layer added only when both feature is on AND `OTEL_EXPORTER_OTLP_ENDPOINT` is set
- [ ] Default build does not pull any `opentelemetry-*` crate
- [ ] `main.rs` calls `telemetry::init_tracing()`

## Do NOT

- Do not enable the feature by default.
- Do not pull metrics or logs OTEL — traces only.
- Do not add per-function `#[instrument]` in this batch.
- Do not expose endpoint configuration via `roko.toml` — env var only (operators expect this).
- Do not run `cargo *`.
