# P10: Recover from poisoned Mutex in warm_pool, orchestrate, gate_runner

## Goal

Replace 8 `expect("poisoned")` / `expect("...lock...")` calls on `std::sync::Mutex::lock()` with poison-recovering variants that log at `warn!` level and continue. After any panic that happened to be holding one of these locks, the rest of the system can continue to function instead of cascade-crashing on the next access.

## Files to Edit

- `crates/roko-cli/src/dispatch/warm_pool.rs` (4 sites: lines 109, 123, 140, 155)
- `crates/roko-cli/src/orchestrate.rs` (3 sites: lines 1437, 1441, 17498)
- `crates/roko-cli/src/gate_runner.rs` (1 site: line 187)

## The replacement pattern

BEFORE:
```rust
let mut guard = self.inner.lock().expect("poisoned");
```

AFTER:
```rust
let mut guard = self.inner.lock().unwrap_or_else(|poisoned| {
    tracing::warn!(target: "roko::warm_pool", "warm pool mutex was poisoned; recovering");
    poisoned.into_inner()
});
```

For non-mutable access (`let guard = ...`), drop the `mut` keyword — keep the rest the same.

The `target: "..."` argument is optional but useful for filtering logs. Use:

| File | suggested target |
|---|---|
| `dispatch/warm_pool.rs` | `roko::warm_pool` |
| `orchestrate.rs` (enrichment stats) | `roko::orchestrate::enrichment` |
| `orchestrate.rs` (gate sink) | `roko::orchestrate::gate_sink` |
| `gate_runner.rs` | `roko::gate_runner` |

The log message should reference what was poisoned, briefly.

## Site-by-site

### `crates/roko-cli/src/dispatch/warm_pool.rs`

Confirm the targets:

```bash
grep -n 'expect("poisoned")' crates/roko-cli/src/dispatch/warm_pool.rs
```

Expected lines (pattern `self.inner.lock().expect("poisoned")`):
- Line 109 — `WarmPool::insert` — uses `let mut guard = ...`
- Line 123 — `WarmPool::take` — uses `let mut guard = ...`
- Line 140 — `WarmPool::evict_expired` — uses `let mut guard = ...`
- Line 155 — `WarmPool::stats` — uses `let guard = ...` (immutable)

For all four, swap `.expect("poisoned")` for the recovering pattern with `target: "roko::warm_pool"` and message `"warm pool mutex was poisoned; recovering"`.

If you'd rather avoid four near-identical `tracing::warn!` blocks, extract a private helper inside `warm_pool.rs`:

```rust
fn lock_or_recover<'a, T>(
    m: &'a std::sync::Mutex<T>,
) -> std::sync::MutexGuard<'a, T> {
    m.lock().unwrap_or_else(|poisoned| {
        tracing::warn!(target: "roko::warm_pool", "warm pool mutex was poisoned; recovering");
        poisoned.into_inner()
    })
}
```

Then call sites become:

```rust
let mut guard = lock_or_recover(&self.inner);
```

Either approach is fine. Pick the one that keeps the diff smallest.

## Do NOT

- Do not migrate to `tokio::sync::Mutex` (different semantics; mixed sync/async is a deadlock source).
- Do not change function signatures.
- Do not silently `unwrap()` — recovery must log.
- Do not run `cargo *`.
