# P11: Auto-trigger `GcEngine.should_auto_gc()` in long-running serve

## Goal

Wire the existing `roko_fs::gc::GcEngine` into a periodic background task in `roko-serve` so unbounded JSONL logs (`episodes.jsonl`, `efficiency.jsonl`, `signals.jsonl`) compact themselves on long-running deployments.

The GC engine and policy already exist (`crates/roko-fs/src/gc.rs:186` exposes `should_auto_gc`). We are NOT writing a GC algorithm — we're wiring an existing one to a tokio interval.

## Files to Edit

- `crates/roko-serve/src/lib.rs` (start a background task during server boot)
- `crates/roko-serve/src/state.rs` (only if `AppState` needs a `GcEngine` field; usually not)

## Also Read (do NOT modify)

- `crates/roko-fs/src/gc.rs` — for the `GcEngine`, `GcPolicy`, `should_auto_gc()` signatures

## Step 3: lib.rs — add the auto-gc task

Open `crates/roko-serve/src/lib.rs`. Find `run_server_with_state` (around line 668) — the function that binds the listener and runs the Axum server. Just before the server is awaited, spawn the background task.

Add at top of file (alphabetical with other imports):

```rust
use std::time::Duration;
use tokio::time::interval;
```

Then a free function near `log_provider_credential_status` (around line 762):

```rust
/// Spawn a tokio task that periodically runs the storage GC engine when
/// `.roko/` exceeds the configured size threshold.
///
/// Cancels when the returned `tokio::task::JoinHandle` is dropped (the server
/// owns it; on shutdown the task is dropped and stops on the next tick).
fn spawn_auto_gc_task(state: Arc<AppState>) -> tokio::task::JoinHandle<()> {
    tokio::spawn(async move {
        let mut tick = interval(Duration::from_secs(60));
        // First tick fires immediately — skip it to avoid GC on cold start.
        tick.tick().await;

        loop {
            tick.tick().await;

            let layout = match roko_fs::Layout::for_workdir(&state.workdir) {
                Ok(l) => l,
                Err(err) => {
                    tracing::warn!(error = %err, "auto-gc: failed to resolve layout");
                    continue;
                }
            };
            let policy = roko_fs::gc::GcPolicy::from_config(&state.config.gc);
            let engine = roko_fs::gc::GcEngine::new(layout, policy);

            match engine.should_auto_gc().await {
                Ok(false) => continue,
                Ok(true) => {
                    tracing::info!("auto-gc: threshold exceeded; running compaction");
                    match engine.run().await {
                        Ok(report) => tracing::info!(
                            removed = report.removed_count,
                            failed = report.failed_count,
                            "auto-gc: complete"
                        ),
                        Err(err) => tracing::warn!(error = %err, "auto-gc: run failed"),
                    }
                }
                Err(err) => tracing::warn!(error = %err, "auto-gc: should_auto_gc check failed"),
            }
        }
    })
}
```

The exact call signatures (`Layout::for_workdir`, `GcPolicy::from_config`, `engine.run`) should match what's actually in `roko-fs`. Read the file to find the real names. Common variants:

| Concept | Likely API |
|---|---|
| Build layout | `Layout::new(workdir)` or `Layout::for_workdir(workdir)` or `Layout::open(workdir)` |
| Build policy | `GcPolicy::from_config(&cfg)` or `GcPolicy::default()` |
| Run | `engine.run()` or `engine.gc()` or `engine.compact_all()` |

If `state.config.gc` doesn't exist (the config struct may live elsewhere), look at how `crates/roko-cli/src/orchestrate.rs` constructs a `GcEngine` and copy the call.

## Step 4: spawn the task in run_server_with_state

Open `crates/roko-serve/src/lib.rs`, find `run_server_with_state` (around line 668). After the listener is bound and before `axum::serve(...)` is awaited, spawn:

```rust
let _gc_handle = spawn_auto_gc_task(Arc::clone(&state));
```

Don't await it; don't `.abort()` it on shutdown — letting the task drop with the runtime is fine for now.

## Step 5: do NOT trigger GC on every write

`maybe_auto_gc` at the end of every plan task would tank latency. Stick to a 60-second interval driven by tokio. If the user wants more aggressive GC, they can lower `gc.size_threshold_mb` in `roko.toml`.

## Acceptance Criteria

- [ ] `spawn_auto_gc_task` defined in `crates/roko-serve/src/lib.rs`
- [ ] Spawned exactly once during `run_server_with_state` boot
- [ ] Task ticks every 60s and only runs GC when `should_auto_gc()` returns true
- [ ] Errors logged at `warn!`; success logged at `info!`
- [ ] Shutdown is clean (the task drops with the runtime — no explicit `abort` needed for this batch)
- [ ] No new dependency

## Do NOT

- Do not implement GC logic — `roko_fs::gc::GcEngine` already does it.
- Do not call GC on every write — interval-driven only.
- Do not lower the threshold below `gc.size_threshold_mb` in config.
- Do not add a `cli` flag for this — it's automatic, governed by config.
- Do not run `cargo *`.
