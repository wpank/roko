## Mechanical recipe

For a typical conductor alert:

BEFORE:
```rust
eprintln!(
    "[ALERT] agent {} unhealthy: {} (failures={})",
    agent_id, reason, failure_count
);
```

AFTER:
```rust
tracing::warn!(
    agent_id = %agent_id,
    reason = %reason,
    failure_count,
    "agent unhealthy"
);
```

Notes:
- `%var` for `Display`-formatted values (strings, paths, errors).
- `?var` for `Debug`-formatted values (non-trivial structs).
- Bare `var` for primitive types (`u64`, `bool`, etc.).
- Drop the `[TAG]` prefixes — `tracing` carries the level and target separately.

For a "✗ task failed" line like:

BEFORE:
```rust
eprintln!("  [{done}/{total}] \u{2717} {task_id} \u{2014} {reason_brief}");
```

AFTER:
```rust
tracing::error!(
    done, total,
    task = %task_id,
    reason = %reason_brief,
    "task failed"
);
```

For the bare blank-line `eprintln!()` at line 7803, just **delete it** (a blank line between log records is not how `tracing` works).

For multi-line summary headers (lines 7804, 11269, 11273), preserve the information but use one `tracing::info!`/`error!` per record — don't try to recreate ASCII tables in structured logs.

## Worked examples for the trickier lines

### Lines 6098 / 6113 (conductor alerts)

Look at the surrounding 8 lines to see what's being printed. Typical patterns:

```rust
eprintln!(
    "[CONDUCTOR ALERT] {} agents stalled in {} for {}s — {}",
    stalled_count, role, secs_stalled, action
);
```

Convert to:

```rust
tracing::warn!(
    target: "roko::conductor",
    stalled = stalled_count,
    role = %role,
    secs_stalled,
    action = %action,
    "conductor alert"
);
```

### Lines 7803 / 7804 (blank line + summary)

```rust
eprintln!();
eprintln!("=== Plan summary: {} tasks, {:.2}s total ===", n, secs);
```

Becomes:

```rust
tracing::info!(
    target: "roko::summary",
    tasks = n,
    duration_secs = secs,
    "plan summary"
);
```

### Lines 11269 / 11273 (failure summary)

If the messages explicitly include "failed" / "error", emit at `error!`. If they list cumulative stats only, `warn!` is fine.

### Lines 13376 / 13378 (per-task failure progress in a loop)

These print as the plan executes. Use `error!` so production deploys notice them in log aggregators.
