# P18: Add Railway auto-deploy job to docker-publish.yml

## Goal

Extend the existing GitHub Actions workflow that publishes to GHCR so that, after a successful image push, the workflow ALSO deploys the new image to Railway. Only on push-to-main, not on PRs.

## Files to Edit

- `.github/workflows/docker-publish.yml`

## Also Read (do NOT modify)

- The other files under `.github/workflows/` for naming conventions

## Step 1: locate the workflow

```bash
ls .github/workflows/
cat .github/workflows/docker-publish.yml
```

If the file doesn't exist, create it with both the build and deploy jobs:

```yaml
name: Build & deploy

on:
  push:
    branches: [main, wp-arch2]
  pull_request:
    branches: [main, wp-arch2]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: nunchi-trade/roko

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to GHCR
        if: github.event_name == 'push'
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: .
          file: Dockerfile
          push: ${{ github.event_name == 'push' }}
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy-railway:
    needs: build
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Railway CLI
        run: npm i -g @railway/cli

      - name: Deploy from GHCR image
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
        run: railway up --image ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
```

## Step 2: if the file already exists, append only the deploy job

The file likely already has a `build` (or similarly named) job that pushes to GHCR. Read it. The `deploy-railway` job below must:

1. Be a sibling job (top-level under `jobs:`)
2. Have `needs: build` (matching whatever the build job is actually called)
3. Be guarded by `if: github.ref == 'refs/heads/main' && github.event_name == 'push'`
4. Pull the GHCR image tag the build job produced

If the build job tags differently (e.g., `:sha-${{ github.sha }}` only), use the same scheme:

```yaml
- name: Deploy from GHCR image
  env:
    RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}
  run: railway up --image ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
```

Or whichever tag the build pushed (`:main`, `:${{ github.sha }}`, etc.). Match whatever the build job actually produces.

## Step 3: branch trigger

Triggers on push to `main` only — NOT on `wp-arch2` or PRs. The build job can run on more branches; only the deploy job is gated.

## Step 4: Railway token

This requires a `RAILWAY_TOKEN` secret in the GitHub repo settings. Document this in a comment at the top of the workflow (for humans to read):

```yaml
# Requires `RAILWAY_TOKEN` in repo secrets.
# Get one at: https://railway.app/account/tokens
```

We do NOT add the secret here — it has to be added through the GitHub UI.

## Step 5: keep deploy non-blocking for the rest of CI

The deploy job is independent from any test or lint job. If a hypothetical `test` job exists in the same file (or another), `deploy-railway` only `needs: build`. Don't add `needs: test` — make deploy depend on build only, so production deploys aren't blocked by flaky integration tests.

If there's a strong policy reason to gate on tests, do it via branch protection rules, not the workflow.

## Verification

```bash
# Job exists:
grep -n 'deploy-railway:' .github/workflows/docker-publish.yml

# Guarded to main + push:
grep -n "github.ref == 'refs/heads/main'" .github/workflows/docker-publish.yml

# Uses the secret:
grep -n 'RAILWAY_TOKEN' .github/workflows/docker-publish.yml

# Pulls from GHCR (does not rebuild):
grep -nE 'railway up --image' .github/workflows/docker-publish.yml
```

## Acceptance Criteria

- [ ] `deploy-railway` job exists in `docker-publish.yml`
- [ ] Triggered only on push to `main`
- [ ] `needs: build` (or whatever the build job is named)
- [ ] Pulls the image from GHCR — does not rebuild
- [ ] Reads `RAILWAY_TOKEN` from secrets
- [ ] Comment in file documents the secret
- [ ] Does NOT block on test jobs

## Do NOT

- Do not embed `RAILWAY_TOKEN` in plaintext.
- Do not deploy on PRs or other branches.
- Do not gate the deploy on test or lint job success.
- Do not change the build job's tagging scheme.
- Do not run `cargo *`.
