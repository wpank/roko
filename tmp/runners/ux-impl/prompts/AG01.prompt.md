# AG01: Capture legacy mirage-rs response fixtures

## Goal

Create golden fixture files capturing exactly the JSON shape of the
mirage-rs `/api/knowledge/*` and `/api/pheromones/*` routes. AG09
will diff aggregator responses against these.

## Context

The compat shim's whole job is preserving the response shape so the
nunchi-dashboard (Wave DB) can swap base URLs without changing client
code. We can't preserve a shape we haven't captured.

This batch creates *static* JSON files committed to the repo. It does
not invoke a running mirage-rs (Codex can't spawn services). Instead
it constructs hand-curated fixtures from
`apps/mirage-rs/src/http_api/knowledge.rs` and
`apps/mirage-rs/src/http_api/pheromone.rs` source.

## Changes Required

1. Read `apps/mirage-rs/src/http_api/knowledge.rs` end to end. Identify
   the response struct(s) for each route:
   - `/api/knowledge/entries` → likely `Vec<KnowledgeEntryResponse>` or `{ items, total, ... }`.
   - `/api/knowledge/edges`.
   - `/api/knowledge/search`.
   - `/api/knowledge/kinds`.

2. Read `apps/mirage-rs/src/http_api/pheromone.rs` similarly:
   - `/api/pheromones`.
   - `/api/pheromones/{kind}`.
   - `/api/pheromones/topology`.
   - `/api/pheromones/active`.
   - `/api/pheromones/decay/{kind}`.

3. For each route, hand-author a fixture JSON file that:
   - Demonstrates every field the response struct emits (use Rust
     serde derives as the spec).
   - Uses representative dummy data (e.g. addresses
     `0x1111111111111111111111111111111111111111`, hashes
     `0xaaaa...`, timestamps `1717000000`).
   - Includes both populated and edge-case rows where the response is
     a list (one with all fields, one with optional fields null).

4. Write each fixture under `tmp/runners/ux-impl/fixtures/`:

   ```
   tmp/runners/ux-impl/fixtures/
     README.md
     knowledge-entries.json
     knowledge-kinds.json
     knowledge-edges.json
     knowledge-search.json
     pheromones.json
     pheromones-topology.json
     pheromones-active.json
     pheromones-decay-threat.json
   ```

5. `README.md` in that directory documents:
   - Source of each fixture (the `pub fn` that produced the response
     in `http_api/`).
   - That AG09 uses these as golden references.
   - That changing a fixture is allowed only if the *legacy* mirage
     shape changes — never to match a new aggregator quirk.

## Acceptance Criteria

- [ ] All fixture files listed above exist under `tmp/runners/ux-impl/fixtures/`.
- [ ] Each is valid JSON (`jq . < <file> > /dev/null` succeeds).
- [ ] Each documents every field present in the matching mirage
  response struct.
- [ ] `tmp/runners/ux-impl/fixtures/README.md` exists with the source
  + immutability notes.

## Do NOT

- Do not run `cargo run -p mirage-rs` to capture fixtures live; this
  batch executes inside Codex without a fork.
- Do not invent fields that the legacy struct didn't have. If
  unsure, leave a TODO comment in the README pointing at the
  ambiguity; AG09 will surface the bug.
- Do not place fixtures under any path other than the one above.
