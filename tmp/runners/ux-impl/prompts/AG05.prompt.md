# AG05: Replace knowledge handlers in aggregator with KnowledgeSource

## Goal

Make the four `/api/knowledge/*` aggregator handlers in
`crates/roko-serve/src/routes/aggregator.rs` dispatch on
`state.knowledge_source` instead of unconditionally reading from
`KnowledgeStore::for_layout(...)`.

## Context

Today, `list_knowledge_entries` (around line 441) calls
`KnowledgeStore::for_layout(&state.layout)` directly. The aggregator
also has `list_knowledge_kinds` (around line 595) that hard-codes
counts to 0. Both must consult `state.knowledge_source` (added in
AG04).

The captured fixtures in `tmp/runners/ux-impl/fixtures/knowledge-*.json`
(from AG01) define the response shape; do not regress it.

## Changes Required

1. In `crates/roko-serve/src/routes/aggregator.rs`, replace the body
   of each handler that previously called `KnowledgeStore::for_layout`:

   ```rust
   async fn list_knowledge_entries(
       State(state): State<Arc<AppState>>,
   ) -> Result<Json<Value>, ApiError> {
       let cache_key = "aggregator:knowledge:entries";
       if let Some(cached) = state.cached_json(cache_key).await {
           return Ok(Json(cached));
       }

       let entries: Vec<Value> = match &state.knowledge_source {
           KnowledgeSource::Local(store) => entries_from_local(store)?,
           KnowledgeSource::Chain(reader, _fetcher) => entries_from_chain(reader).await?,
           KnowledgeSource::Hybrid { chain, cache, .. } => {
               // Chain is authoritative; fall back to local cache if
               // chain is briefly unreachable (logged at debug).
               match entries_from_chain(chain).await {
                   Ok(v) => v,
                   Err(err) => {
                       tracing::debug!(?err, "chain unreachable; falling back to local cache");
                       entries_from_local(cache)?
                   }
               }
           }
       };

       let body = paginate_response(entries, ...);   // adapt to existing helper
       state.cache_json(cache_key, &body, KNOWLEDGE_TTL).await;
       Ok(Json(body))
   }
   ```

   Implement the helpers `entries_from_local` and
   `entries_from_chain` as private `async fn` in the same module. The
   *output shape* of each helper must match the captured fixture
   `tmp/runners/ux-impl/fixtures/knowledge-entries.json`.

2. Repeat for `list_knowledge_edges`, `search_knowledge`,
   `list_knowledge_kinds`.

3. For `list_knowledge_kinds`: replace the hard-coded `count: 0`
   values with real counts from the source. For `Chain` and `Hybrid`,
   query `reader.count()` once and group by `kind` — InsightBoard
   doesn't store kind metadata directly, so the count goes against a
   single `"insight"` row and the rest stay zero. Document this
   limitation in a `// NOTE:` comment for future expansion.

4. Use the existing TTL constants (`KNOWLEDGE_TTL = 30s` is already
   defined near the top of `aggregator.rs`). If they don't exist, add
   them inline.

5. Remove any direct `roko_neuro::knowledge_store::KnowledgeStore::for_layout`
   call from `aggregator.rs`. The store is only accessed through
   `KnowledgeSource`.

## Acceptance Criteria

- [ ] No occurrence of `KnowledgeStore::for_layout` in `aggregator.rs`.
- [ ] Each `/api/knowledge/*` handler dispatches on `KnowledgeSource`.
- [ ] When `state.knowledge_source` is `Local`, the response shape
  matches the fixture exactly (use `serde_json::to_value` and
  `serde_json::from_str` to compare).
- [ ] When `state.knowledge_source` is `Chain` or `Hybrid`, the
  response includes any insights present on chain.
- [ ] `cargo build -p roko-serve` succeeds.
- [ ] `cargo test -p roko-serve` passes (AG09 adds full compat tests; existing tests must not break).

## Do NOT

- Do not block the request thread on chain calls without a timeout.
  Use `tokio::time::timeout(Duration::from_secs(5), reader.list())`
  or rely on the reader's internal timeout.
- Do not change the public route signatures. The handlers' return type,
  parameters, and HTTP method must match what the legacy mirage shim
  did.
- Do not paginate differently than the legacy fixture. If the fixture
  uses `{ items, total, offset, limit, has_more }`, preserve that.
- Do not introduce a new error type. Reuse `ApiError` already in
  `crates/roko-serve/src/error.rs`.
