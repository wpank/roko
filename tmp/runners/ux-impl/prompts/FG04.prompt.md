# FG04: CLAUDE.md "Default-built" column + docs/v2/WORKSPACE.md

## Goal

Surface the default-members vs all-members split so contributors know
which crates ship and which don't.

## Changes Required

1. `CLAUDE.md` "Key crates" table:
   - Add a new column: `Default-built`.
   - For each crate row, mark `yes` (in `default-members`) or `no` (Phase 2+).

2. Create `docs/v2/WORKSPACE.md`:

```markdown
# Workspace shape

Roko's Cargo workspace has two slices:

- **default-members** — the shipped crates. Built by `cargo build --workspace`.
- **members** — every workspace member, including Phase 2+. Built by
  `cargo build --workspace --all`.

## Why

CI on every PR runs `cargo build --workspace`. Phase 2 crates
(`roko-dreams`, `roko-daimon`, etc.) are work-in-progress; including
them in default CI would block PRs on partial work.

A weekly `phase2-build` workflow runs the broader `--all` set so
Phase 2 crates don't accumulate breakage between promotions.

## Promoting a crate from Phase 2 to default

1. Verify the crate has shipped tests and at least one external
   call site.
2. Add the crate's path to `default-members` in the root `Cargo.toml`.
3. Update its row in `CLAUDE.md` "Key crates" Default-built column to `yes`.
4. Remove its entry from `tmp/ux/ux-followup/08-phase-2-vision.md`.

## Building only the shipped slice

```bash
cargo build --workspace                  # default-members
cargo build --workspace --all            # everything
cargo test --workspace                   # default-members tests
cargo test --workspace --all             # everything's tests
```

## See also

- `.github/workflows/phase2-build.yml` — weekly cron on `--all`.
- `tmp/ux/ux-followup/08-phase-2-vision.md` — Phase 2 entry conditions.
```

3. Update `CLAUDE.md` Build commands section (if present) to mention
