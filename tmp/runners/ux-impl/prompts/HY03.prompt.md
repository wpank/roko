# HY03: Fill `# Errors` / `# Panics` in roko-core, roko-runtime

## Goal

Remove crate-level `#[allow(clippy::missing_errors_doc)]` and
`#[allow(clippy::missing_panics_doc)]` from `roko-core` and
`roko-runtime`. Every public function gains the missing section.

## Changes Required

1. Grep:

   ```bash
   rg -t rust '#\\[allow\\(clippy::missing_(errors|panics)_doc\\)\\]' \
     crates/roko-core crates/roko-runtime
   ```

2. For each hit:
   - If at file head (`#![allow(...)]`): remove it. Then run
     `cargo clippy -p roko-{core,runtime} --no-deps -- -D warnings` and
     fix every flagged function.
   - If on a single function (`#[allow(...)] pub fn ...`): remove it
     and add the missing doc.

3. For functions returning `Result<T, E>`, add a `# Errors` section:

   ```rust
   /// Build a config from the given paths.
   ///
   /// # Errors
   ///
   /// Returns an error if any path is unreadable, or if the merged
   /// config fails validation.
   pub fn from_paths(...) -> Result<Self, ConfigError> { ... }
   ```

4. For functions that may panic (uses `unwrap`, `expect`, indexing,
   `assert!`), add a `# Panics` section:

   ```rust
   /// Render a prompt for the given role.
   ///
   /// # Panics
   ///
   /// Panics if `role` is not present in `cfg.agent.roles`.
   pub fn render_for_role(cfg: &RokoConfig, role: &str) -> String { ... }
   ```

5. If a public function never panics (no `unwrap`, no `expect`, no
   indexing on slices, no `assert!`), do not add `# Panics`. The
   `missing_panics_doc` lint only fires when the body suggests a panic
   path.

## Acceptance Criteria

- [ ] `rg -t rust '#\\[allow\\(clippy::missing_(errors|panics)_doc\\)\\]' crates/roko-core crates/roko-runtime` returns 0 hits.
- [ ] `cargo clippy -p roko-core --no-deps -- -D warnings` is green.
- [ ] `cargo clippy -p roko-runtime --no-deps -- -D warnings` is green.
- [ ] `cargo build -p roko-core -p roko-runtime` is green.
- [ ] No new `unwrap()` introduced (HY3 must not weaken safety to
  silence panic docs).

## Do NOT

- Do not silence the lint per-function with a fresh `#[allow(...)]`.
  Add the doc.
- Do not add a generic `# Errors: returns an error if anything goes
  wrong` line. Be specific.
- Do not change function signatures.
- Do not introduce new `unwrap()` calls; if a function's panic
  surface is unclear, refactor to remove the panic before documenting.
