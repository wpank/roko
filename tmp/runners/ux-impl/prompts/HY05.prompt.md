# HY05: Flaky timeout-based tests fix (mock time / CI scaling)

## Goal

Replace timing-based assertions in `crates/roko-agent/src/exec.rs` (and
siblings) with mock time or CI-scaled timeouts.

## Context

Followup item 58. `with_timeout_ms(100)` style tests fail
intermittently on slow CI.

## Changes Required

1. Grep:

   ```bash
   rg -t rust 'with_timeout_ms\\(\\s*[0-9]+' crates/ apps/
   ```

2. For each hit, decide:

   **Strategy A (preferred): mock time.** Use `tokio::time::pause` /
   `advance`:

   ```rust
   #[tokio::test(start_paused = true)]
   async fn fast_timeout() {
       let job = my_thing_with_timeout(Duration::from_millis(100));
       tokio::time::advance(Duration::from_millis(101)).await;
       assert!(job.timed_out());
   }
   ```

   **Strategy B (fallback): CI-scaled timeouts.** Use a helper:

   ```rust
   fn timeout_for_test() -> Duration {
       if std::env::var("CI").is_ok() {
           Duration::from_secs(5)   // 50× normal under CI
       } else {
           Duration::from_millis(100)
       }
   }
   ```

3. For tests that intentionally race (rare), document with a comment
   *and* add to a suspect list at
   `crates/<crate>/tests/.flaky-allow.txt`. CI doesn't fail on these
   today, but reviewers see them.

4. After the fix, run the suspect tests 10× locally:

   ```bash
   for i in $(seq 1 10); do cargo test -p roko-agent --release || break; done
   ```

## Acceptance Criteria

- [ ] `rg -t rust 'with_timeout_ms\\(\\s*100\\b' crates/ apps/` returns
  0 hits in tests (production code may legitimately have 100 ms timeouts).
- [ ] `cargo test -p roko-agent --release` passes 10× in a row locally.
- [ ] Where Strategy A applies, the test uses `start_paused = true`
  and `tokio::time::advance`.

## Do NOT

- Do not fix flakiness with a retry decorator. That hides the bug.
- Do not bump every timeout to 5 seconds blindly. Strategy A is
  always better when the test logic permits.
- Do not skip tests with `#[ignore]` to make CI green.
