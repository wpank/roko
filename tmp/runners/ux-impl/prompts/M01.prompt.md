# M01: Audit no live consumers depend on mirage REST routes

## Goal

Produce `docs/v2/MIRAGE-CONSUMER-AUDIT.md` enumerating every active
consumer of `apps/mirage-rs`'s `http_api/*` routes. The deletion in M04
must not be merged until this audit shows zero in-tree consumers.

## Context

Plan 01 (`tmp/ux/implementation-plans/01-mirage-extraction-final.md`)
calls for deleting `apps/mirage-rs/src/{http_api,chain,roko_bridge}/`
once consumers have moved to `roko-serve`'s aggregator. This batch is
the verification step — it does not delete anything.

The active aggregator surface is at
`crates/roko-serve/src/routes/aggregator.rs`. The local demo at
`demo/demo-app/` already routes everything through roko-serve at port
6677 (see `demo/demo-app/src/lib/serve-url.ts`). The sibling
nunchi-dashboard still hits mirage on `:8545` for REST — that work is
Wave DB and is out of scope for this batch.

## Changes Required

1. Create `docs/v2/MIRAGE-CONSUMER-AUDIT.md` with the structure below.

2. For each in-repo consumer of mirage REST routes, classify it as
   one of:
   - `IN_TREE_REST` — a Rust crate that calls `/api/*` on mirage-rs.
     Must be migrated before M04.
   - `IN_TREE_RPC` — a Rust crate that calls JSON-RPC (eth_*, mirage_*)
     on mirage-rs. **Stays.** Mirage retains JSON-RPC after Wave M.
   - `DEMO_APP_REST` — `demo/demo-app/` REST hits. Must be migrated to
     roko-serve before M04.
   - `DEMO_APP_RPC` — `demo/demo-app/` JSON-RPC hits. Stays.
   - `SIBLING_DASHBOARD` — `nunchi-dashboard/` (sibling repo). Tracked
     under Wave DB (manual). Note in audit but does not block M04.
   - `OBSOLETE` — referenced in dead code or comments. Note as such.

3. Use these grep commands and record their output verbatim under a
   "Raw scan" appendix:

   ```bash
   rg -t rust -t toml --no-heading 'mirage_rs::http_api|mirage-rs.*\\b/api/' \
     -- crates/ apps/

   rg --no-heading 'http://[^"]*:8545/api|VITE_CHAIN_URL\\s*\\)\\s*\\?\\s*\\?\\s*"[^"]*api' \
     demo/

   rg --no-heading 'http://[^"]*:8545|VITE_CHAIN_URL' \
     demo/demo-app/src/
   ```

4. Produce the summary table:

   ```markdown
   | Consumer | Path | Class | Action |
   |----------|------|-------|--------|
   | <crate or file> | <path:line> | <CLASS> | <Migrate / Stay / Delete> |
   ```

5. Conclude with one of:
   - `READY FOR M04` — all `IN_TREE_REST` and `DEMO_APP_REST` rows are
     `Migrate` and migrated, OR there are zero such rows.
   - `BLOCKED` — there is at least one `IN_TREE_REST` consumer that has
     not migrated. List the blockers explicitly.

## Acceptance Criteria

- [ ] `docs/v2/MIRAGE-CONSUMER-AUDIT.md` exists.
- [ ] Contains the three raw grep outputs.
- [ ] Contains the classified summary table.
- [ ] Contains an explicit `READY FOR M04` or `BLOCKED` conclusion.
- [ ] No code files modified. (This is a documentation-only batch.)

## Do NOT

- Do not modify any source file. This audit only writes one new
  Markdown file.
- Do not classify `nunchi-dashboard` as a blocker for M04 — that
  migration runs in Wave DB and is independent.
- Do not silently elide a hit. Every line from the grep must appear in
  the audit.
