# M06: Rewire scenario.rs if it touches ChainContext

## Goal

Make `apps/mirage-rs/src/scenario.rs` compile and pass tests after M04
deleted `chain/`. If the scenario runner historically pre-seeded
InsightEntry / pheromone state for replay, that logic must move to
JSON-RPC-only setup or to a fixture file.

## Context

The original scenario runner was advertised as "Scenario manifests,
assertion checking, scenario runner". After Wave M, scenarios should
be EVM-only: load fork, optionally `setBalance` / `setStorageAt`,
replay transactions, assert on the resulting EVM state.

If `scenario.rs` already compiles with no `chain::` imports, this
batch is a no-op — note that in the commit message and exit cleanly.

## Changes Required

1. `cargo check -p mirage-rs` after M04. If `scenario.rs` is clean,
   skip to step 4.

2. If `scenario.rs` references `crate::chain::...` (or
   `crate::http_api::...`), classify each reference:

   | Reference | Action |
   |-----------|--------|
   | `chain::ChainContext::new(...)` | Delete; the scenario doesn't need it |
   | `chain::Pheromone` / `chain::InsightEntry` setup | Move to a fixture file under `apps/mirage-rs/tests/fixtures/<name>.json` |
   | `chain_*` JSON-RPC method invocation | Delete; replace with `mirage_*` or `eth_*` equivalent if applicable |

3. For any moved fixture data, write a `.json` file with the shape:

   ```json
   {
     "scenario": "demo-replay",
     "preconditions": [
       {"type": "set_balance", "address": "0x...", "wei": "1000000000000000000"},
       {"type": "set_storage_at", "address": "0x...", "slot": "0x...", "value": "0x..."}
     ],
     "transactions": [...],
     "assertions": [
       {"type": "balance_eq", "address": "0x...", "wei": "..."},
       {"type": "storage_eq", "address": "0x...", "slot": "0x...", "value": "0x..."}
     ]
   }
   ```

   Update the runner to consume the fixture via standard JSON-RPC
   calls (`anvil_setBalance` / `anvil_setStorageAt` if upstream
   supports them, or the `mirage_set_balance` / `mirage_set_storage_at`
   extensions).

4. If a scenario test asserted "after replay, InsightBoard contains
   X", *do not* port it inside mirage. Move that test to
   `crates/roko-serve/tests/aggregator_e2e.rs` (or leave it for
   AG09 which writes the compat tests). Note in the commit message
   that the assertion was moved out of mirage.

5. `cargo test -p mirage-rs --lib scenario` passes. If a test is
   structurally about chain state and can't be ported, delete it with a
   commit-message note pointing to where the equivalent test now
   lives.

## Acceptance Criteria

- [ ] `cargo build -p mirage-rs` succeeds.
- [ ] `cargo test -p mirage-rs --lib scenario` passes (or the test
  module has been intentionally trimmed with a documented reason).
- [ ] No `use crate::chain::` or `use crate::http_api::` lines remain
  in `scenario.rs`.
- [ ] Any fixture files added are under `apps/mirage-rs/tests/fixtures/`
  and committed.

## Do NOT

- Do not re-create `ChainContext` or `KnowledgeStore` types inside
  `scenario.rs`. Those concepts no longer live in mirage.
- Do not import `roko-neuro` or `roko-chain` from mirage to "fill the
  gap". Mirage stays free of application-state crates.
- Do not skip running the tests with `#[ignore]`. If a test must be
  removed, remove it and explain why.
