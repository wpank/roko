# Plan Run 0/10 Tasks -- Root Cause Analysis

**Date:** 2026-05-04

## Root Cause 1: Model Hint Resolution Failure

Tasks generated by `prd plan` contain `model_hint` values like `"claude-sonnet-4-6"`,
`"claude-opus-4-6"`, or `"claude-haiku-4-5"`. These are baked into `tasks.toml` at
plan generation time.

When the user runs `plan run` with `--model cerebras-70b`:

1. `resolve_effective_model()` in `crates/roko-cli/src/model_selection.rs` applies a
   6-step precedence chain: `CliOverride > TaskModel > RoleConfig > CascadeRouter >
   ProjectDefault > BuiltInDefault`.
2. `--model cerebras-70b` becomes `cli_model` (step 1), which wins over `task_model_hint`
   (step 3). So the CLI override correctly takes precedence.
3. However, when `--model` is NOT passed and no CLI override exists, the `task_model_hint`
   from `tasks.toml` wins at step 3 (line 238 of `model_selection.rs`).
4. `select_provider()` then looks up the provider for the resolved model. For
   `model_hint = "claude-sonnet-4-6"`, it resolves to provider `"anthropic"` (kind
   `claude_cli`). If the `anthropic` provider is not configured or `ANTHROPIC_API_KEY`
   is not set, the dispatch fails.
5. The error propagates through `dispatch_agent_with()` (orchestrate.rs:15050-15054)
   where `hard_model_override` validation uses `?` to return early.
6. The event loop marks the task as `Fatal` via `ExecutorEvent::Fatal(...)` and moves on.
7. All 10 tasks fail the same way -> 0/10 execute.

**The core issue:** model hints in `tasks.toml` are generated assuming the Claude CLI
provider is available. When running with a different provider, there is no fallback --
the hint wins over the project default and the resolution error cascades to task failure.

The fallback path at orchestrate.rs:15018-15032 catches `resolve_effective_model` errors
and falls back to `agent.default_model`, but this only helps when the task hint itself
causes the error. When the hint resolves successfully to a model whose provider is
unavailable, the failure happens later in the dispatch pipeline.

**Config context:** `roko.toml` has `default_model = "opus"` which maps to provider
`"anthropic"` (kind `claude_cli`). The `cerebras-70b` model maps to provider `"cerebras"`
(kind `cerebras_api`). No `[models."claude-sonnet-4-6"]` entry exists by that exact key,
but the slug match (resolve_model step 2, agent.rs:298-302) finds `[models.claude-sonnet]`
with `slug = "claude-sonnet-4-6"` and `provider = "anthropic"`.

## Root Cause 2: Stale Executor State (Secondary)

The `--fresh` flag (plan.rs:243-268) only removes `.roko/state/executor.json`. It does
NOT remove:

- `.roko/state/orchestrator.json` -- aggregate orchestrator snapshot
- `.roko/state/run-state.json` -- runner-owned cost/token/completed-task state
- `.roko/state/events.json` -- event log for replay

All three are defined in `PersistPaths` (runner/persist.rs:25-42) and consumed during
resume. The event loop's `load_resume()` (event_loop.rs:1695-1727) reads from
`executor.json`, but stale `orchestrator.json` data could mark plans as terminal
or cause plan-ID mismatch errors.

The stale snapshot detection (event_loop.rs:1735-1739) checks for plan-ID overlap
and starts fresh if there's no match. This mitigates the worst case but doesn't cover
the scenario where plan IDs overlap but task states are stale (e.g., all tasks marked
completed from a previous run).

## Root Cause 3: TOML Quality from Cheaper Models

When `prd plan` uses a cheaper/faster model (e.g., cerebras-70b via `llama3.1-8b`) to
generate `tasks.toml`, the output quality degrades:

- **Truncated field names:** `denied_tool` instead of `denied_tools`, `deni_tools`,
  `stus` instead of `status`
- **Merged values:** fields concatenated across line boundaries, e.g.,
  `model_hint = "claude-sonnet-4-ax_loc = 45"` (model hint merged with max_loc)
- **Missing required fields:** `title`, `description`, or `role` omitted entirely
- **Invalid TOML syntax:** unclosed strings, mismatched brackets

The plan validation step (`validate_before_run`, plan.rs:231-233) catches parse-level
TOML errors but does NOT validate semantic correctness of individual fields. A task with
`model_hint = "claude-sonnet-4-ax_loc = 45"` parses as valid TOML (it's a legal string
value) but fails at resolution time because no model with that slug exists.

## Demo App Issues

### Port 8545 WebSocket Failures

The demo app connects to `ws://localhost:8545` (serve-url.ts:61) for the Mirage chain
endpoint. This is the local Anvil/Mirage development chain defined in `roko.toml`
(`[chain] rpc_url = "http://127.0.0.1:8545"`). When mirage-rs is not running, these
connections fail with WebSocket errors. This is expected and non-blocking -- the UI
falls back to HTTP projections via roko-serve.

### execCmd Timeouts

`execCmd` (useTerminal.ts:255-286) wraps shell commands with an invisible OSC 7777
sideband marker. The command is wrapped as:

```
${cmd}; __rk_ec=$?; printf '\033]7777;D;%d;${marker}\033\\' "$__rk_ec"; (exit $__rk_ec)
```

xterm.js intercepts the OSC 7777 escape via `term.parser.registerOscHandler(7777, ...)`
(useTerminal.ts:143). Timeouts occur when:

1. The shell doesn't emit the OSC marker (e.g., command crashes before printf runs)
2. xterm.js parser doesn't intercept the sequence (terminal state corruption)
3. The command itself takes longer than the timeout (default 30s, some overrides at 4-8s)

The timeout fires a `setTimeout` (useTerminal.ts:279-284) that resolves the promise with
`{ ok: false, exitCode: -1 }`.

### ensureWorkspaceCwd Failures

`ensureWorkspaceCwd` (terminal-session.ts:115-126) runs `cd <dir>` via `execCmd` with a
5-second timeout. Failures happen when:

1. The directory doesn't exist (server-side workspace creation hasn't completed)
2. The `cd` command itself times out due to OSC marker issues (see above)
3. The PTY session is in a broken state from a previous command failure

### workflow-api ERR_EMPTY_RESPONSE

`fetchWorkflowSnapshot` (workflow-api.ts:109-138) sends GET requests to
`${SERVE_URL}/api/workflows/${id}?root=${root}` with a 3-second `AbortSignal.timeout`.

`ERR_EMPTY_RESPONSE` occurs when:

1. `roko serve` is not running on :6677
2. The serve process is running but returns an empty body (route handler returns before
   writing response bytes)
3. The request is aborted by the 3-second timeout before the server responds
4. The workspace root doesn't exist on the server yet (returns 400, retried up to 3 times
   with 500ms backoff)

After 3 consecutive failures, `serveReachable` is set to false (workflow-api.ts:130)
and subsequent calls short-circuit with `null` for 30 seconds.
