## Summary

**Most of the v1 corpus is real code, and a surprising amount of what v1's own STATUS.md calls "Specified" has since been built.** Of the 22 sections, roughly 9 are wired end-to-end (architecture kernel, orchestration, agents, composition, verification, learning core, neuro, conductor, daimon, code intelligence), 7 are partial (dreams, safety, interfaces, coordination, heartbeat, lifecycle, tools, deployment), 3 are built-but-not-wired (chain, identity-economy, technical analysis), and 1 is a bibliography (references). Nothing in v1 is entirely vapor except the chain-dependent economy layer and the frontier TA oracles — and even those have substantial scaffolds (`roko-chain` has 14 subsystem modules; `roko-primitives` has `tda.rs`/`manifold.rs`/`sheaf.rs`/`tropical.rs`).

**The dominant failure mode is not missing code — it is naming drift and stale documentation.** The kernel data type is still `Engram` (`crates/roko-core/src/engram.rs:63`); `Signal` is a one-line forward-compat re-export (`crates/roko-core/src/signal.rs:6`). The v1 docs' six verb traits (Substrate/Scorer/Gate/Router/Composer/Policy) were renamed in code to Store/Score/Verify/Route/Compose/React (`crates/roko-core/src/traits.rs:37,167,214,242,285,339`), with v2's Bus/Observe/Connect/Trigger added (`traits.rs:385,400,408,420`) and the v2 `Cell` trait live (`crates/roko-core/src/cell.rs`). Storage is split-brained: `.roko/` carries both `engrams.jsonl` (the *main* log, `crates/roko-fs/src/file_substrate.rs:48`) and `signals.jsonl` (explicitly labeled "legacy signal log", `crates/roko-fs/src/layout.rs:217`) — the rename history is Signal (v0) → Engram (v1) → Signal (v2), and the code sits mid-flight.

**Execution is triple-engined.** The v1-documented orchestrator (`roko-orchestrator` ParallelExecutor/MergeQueue/DAG) is built and reachable, Runner v2 is the main *live* execution path (via `--engine runner-v2`, and it is where recent self-dev commits actually run), and the current `roko plan run` Clap default is the Graph Engine (`main.rs:1361` `default_value = "graph"`). That default is misleading today because the graph path delegates plan tasks through `TaskExecutorCell` dry-run scaffolding (`crates/roko-graph/src/cells/task_executor.rs:81-86`) unless users explicitly select Runner v2. **Crucially, as of the 07-08 re-verification, the Runner v2 event loop is no longer a thin executor: daimon, dreams, playbooks, efficiency, neuro/episode ingestion, and knowledge-routing writebacks now fire inside `runner/event_loop.rs` — only the conductor's `check_all` remains orchestrate.rs-only.** The 23,676-line `orchestrate.rs` monolith still hosts the conductor OODA loop, oracles, and c-factor summary, so those v1 subsystems risk being stranded on runner-v2 runs unless ported. The Graph Engine's cognitive-loop cells are still mostly thin wrappers or passthroughs (`PassthroughCell`, `TaskExecutorCell` dry-run).

**Verdict: ~72% of v1 is implemented, ~60% is wired to a runtime path (up from ~55% now that runner-v2 carries the cross-cuts), and both `docs/v1/STATUS.md` and root `CLAUDE.md` materially understate the code** (heartbeat, dreams, lifecycle, MCP, safety-contract hardening, and runner-v2 cross-cut integration are all further along than documented) while overstating others (parallel Graph execution, plugin SPI, universal tool dispatch, and the Graph-Engine-as-default framing). The single biggest remaining runtime gap for v1 parity is **conductor coverage on the live runner-v2 path** and **live TaskExecutorCell dispatch on the graph default path**.

## Per-section notes
