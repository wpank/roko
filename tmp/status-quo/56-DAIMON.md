## Summary

roko-daimon is a 7.4K-LOC affect engine whose state *can* measurably change model selection, turn limits, cascade-tier routing, tool allowlists, prompt content, context-retrieval ranking, task-queue ordering, and replan decisions — **but only on the runner-v2 and `roko serve` paths, which are no longer the default** (see Engine Drift above). Live evidence that it *did* run: `.roko/daimon/affect.json` has 919 appraisal ticks, 10 merged somatic markers, and a persisted `struggling` behavioral state (confidence 0.187) from real runs (last write 2026-05-08 — i.e. **before** the Graph Engine became default; no daimon writes since).

The crate splits into three strata: (1) a **wired core (runner-v2/serve only)** — PAD + ALMA three-layer state, appraisal events, behavioral-state machine with hysteresis, 8D somatic landscape (kiddo k-d tree), dispatch modulation, tier thresholds; (2) **built-but-unwired machinery** — OCC/Scherer structured appraisal, four-factor `RetrievalWeights`, knowledge-store contrarian injection, collective contagion, fatigue responses, error-pattern scaling; (3) **internal-only speculative modules** — goals, life review, mortality/Nietzsche phases, somatic-TA/IIT-Phi (zero external callers). The v2 "affect as functor" model (PAD as Signal metadata, appraisal as React Cell) remains target-state: today's design is exactly the standalone-struct + method-call pattern that `18-affect-as-functor.md §1` critiques.

Persistence is fragmented across **three paths**: canonical `.roko/daimon/affect.json` (runner-v2/serve), a second independent affect state at `.roko/state/daimon.json` (WorkflowEngine/`roko run` path), and a dead `pad-state.json` path in roko-neuro. The Graph Engine writes **none** of them.

## Open questions

0. **Is daimon-on-Graph-Engine wanted, or is affect deliberately a runner-v2/serve feature?** The default plan-run path silently drops all modulation. If daimon is meant to be load-bearing, it must be ported to a graph Cell/middleware; if it's an experimental opt-in, CLAUDE.md and `roko explain daimon` should say so and the "Wired" status should be downgraded to "Wired (runner-v2/serve only)". This decision gates most of the P0/P1 work below.
1. **Which affect state is authoritative when both loops run?** `roko run` (WorkflowEngine, `.roko/state/daimon.json`) and `roko plan run` (`.roko/daimon/affect.json`) evolve independent moods for the same workspace. Merge policy needed (last-write? ALMA-layer blend?).
2. **Is daimon model-override churn desirable?** `modulate()` can demote a router-selected model (Coasting) after CascadeRouter already applied a behavioral tier shift from the *same* state — the two mechanisms (tier shift at routing + promote/demote at dispatch) can double-apply or fight. Should one own the decision?
3. **Live state is stale-Struggling** (conf 0.187 since 2026-05-08, pleasure 0.94 / dominance 0.95 — an odd "confident-but-helpless" corner from EMA saturation). With 4h half-life, decay only applies on next appraisal; should a heartbeat decay tick exist so a workspace doesn't wake up Struggling weeks later? (No cron/heartbeat touches daimon today.)
4. **Hysteresis tracker vs `AffectState.behavioral_state` divergence**: `query()` swaps in `behavioral_tracker.current_state` (lib.rs:2323-2328) but the serialized `state.behavioral_state` may disagree after reload reclassification (pe09_regression.rs:58-81 asserts reclassify-on-load) — which is canonical for external consumers?
5. **Does the 15% landscape contrarian blend satisfy v1 doc 07's intent**, or was the knowledge-retrieval injection (unwired) the load-bearing half (anti-rumination)?
6. Should `AffectBehaviorModulation`'s five profiles (incl. `sharing_threshold`, `trigger_dream_cycles`) replace the ad-hoc `modulate()` match — they encode the documented behavior table but are dead code today.
