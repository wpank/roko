Implementation:
- Design and invariants: both `ServerBuilder::start_background` and public `run_server_with_state` call one startup normalization/validation helper before restore, watcher/subscriber/bridge/saver/job/archive task creation, router/listener work, or any other fallible startup operation. All PUT, explicit reload, and unchanged watcher reload calls cross one process-wide ownership gate and monotonic generation counter. Async HTTP callers execute the synchronous transaction on Tokio's blocking pool; the watcher retains its public synchronous API; no blocking mutex is held across an `.await`. A PUT transaction selects raw project source under that gate, merges the request without raw-only normalization, derives the prospective effective candidate through the ordinary runtime-layer pipeline, then constructs one dispatch index from the complete effective model namespace. Effective references are normalized and strictly validated against that index first; source references are canonicalized against the same exact-key-first index without copying any overlay field and without rejecting an unresolved raw value masked by a valid runtime field override. Both projections are serialized before any write, only source bytes are atomically persisted and synchronously replicated, and only effective state is infallibly published before ownership release. Once the blocking transaction starts, request-future cancellation cannot cancel it; the gate prevents an older accepted generation from propagating after a newer generation. For live dispatch, trimmed exact keys win, unique aliases become canonical keys, duplicate/unresolved effective references are fatal, and an empty registry retains compatibility. Tier and role keys remain sorted before validation.
- Source/effective separation: `resolve_runtime_layers` is the single loader pipeline for ordinary, provenance, and in-memory candidate resolution: global merge, named env, hierarchical env, interpolation, file-secret resolution, and post-merge provider validation retain the same order. Reload publishes that effective object without a source write. PUT never snapshots effective `AppState` for persistence; it reads `workdir/roko.toml`, or when absent seeds the new local file from unified discovery's `ValidatedConfig.raw`, then resolves an owned clone in memory before any source canonicalization. The complete effective registry governs identity in both projections, preventing a project-only slug alias from overriding a global exact key. Exact keys and unique aliases still canonicalize in retained source, while source references masked by runtime field overrides remain verbatim if unresolved. No global registry, environment value, interpolation result, or file-secret literal is promoted into project or replica source.
